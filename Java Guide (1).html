Java Basics — Introduction
8 Qs

Java is a high-level, class-based, object-oriented programming language developed by James Gosling at Sun Microsystems (now owned by Oracle), first released in 1995. It follows the principle of "Write Once, Run Anywhere" (WORA) — Java programs are compiled to an intermediate format called bytecode, which can run on any device or operating system that has a Java Virtual Machine (JVM) installed, without any modification.

Key characteristics of Java:

  • Object-Oriented: Everything in Java (except primitive types) is an object. The language is built around the principles of classes, objects, encapsulation, inheritance, and polymorphism.
  • Platform Independent: Java source code is compiled to bytecode — an intermediate representation that is not tied to any specific OS or hardware. The JVM on each platform translates bytecode to native machine code at runtime.
  • Statically Typed: Variable types must be declared explicitly and are checked at compile time — catching type errors before the program runs.
  • Strongly Typed & Memory Safe: Java has no pointers (unlike C/C++), automatic garbage collection, and strict type checking — making it significantly safer and less prone to memory leaks and crashes.
  • Multi-threaded: Java has built-in support for concurrent programming via threads — allowing programs to perform multiple tasks simultaneously.
  • Widely used in: Android app development, enterprise backend systems (Spring Boot), web servers, big data (Hadoop, Spark), financial systems, and embedded systems.
java — hello world
public class HelloWorld {
    public static void main(String[] args) {
        System.out.println("Hello, World!"); // prints to console
    }
}

These three acronyms describe the layered architecture that makes Java work. They are nested — JDK contains JRE which contains JVM.

  • JVM (Java Virtual Machine): The innermost layer — an abstract computing machine that executes Java bytecode. The JVM is what makes Java platform-independent: each OS has its own JVM implementation, and the JVM translates bytecode into native machine code for that specific platform at runtime. It also manages memory (garbage collection), handles exceptions, and provides the runtime environment. You cannot use it alone without the JRE.
  • JRE (Java Runtime Environment): The middle layer — contains the JVM plus all the standard Java class libraries (like java.lang, java.util, java.io) needed to run Java programs. If you only want to run (not develop) a Java application, you install the JRE.
  • JDK (Java Development Kit): The outermost layer — contains the JRE plus development tools: the Java compiler (javac), debugger, profiler, javadoc, and other utilities. If you want to write and compile Java code, you install the JDK. Most developers install the JDK.

In Java's lifecycle, code passes through two distinct phases before producing output. Understanding these phases helps you categorise errors and understand when different checks happen.

  • Compile-time: The phase when the Java compiler (javac) processes your .java source files and translates them into .class bytecode files. During this phase, the compiler performs: syntax checking (correct Java grammar?), type checking (are you passing a String where an int is expected?), semantic analysis (are variables declared before use?), and code generation (bytecode output). Errors caught at compile-time are called compile-time errors or compilation errors. You cannot run the program at all until all compile-time errors are fixed.
  • Run-time: The phase when the JVM actually executes the compiled bytecode. Run-time is when the program produces actual results — but it's also when issues that the compiler couldn't predict may surface: dividing by zero, accessing a null reference, running out of memory, array index out of bounds. These are called run-time errors or exceptions. Unlike compile-time errors (which you fix before running), run-time errors occur during execution and can crash the program if not handled with try-catch.
java — compile-time vs run-time errors
// COMPILE-TIME ERROR — caught by javac before execution
int x = "hello";  // ❌ Type mismatch: cannot convert String to int
undeclaredMethod();  // ❌ Cannot resolve method

// RUN-TIME ERROR — only occurs during execution
int[] arr = new int[3];
System.out.println(arr[5]); // ❌ ArrayIndexOutOfBoundsException (runtime)

String s = null;
s.length();  // ❌ NullPointerException (runtime)

int result = 10 / 0; // ❌ ArithmeticException: / by zero (runtime)

Java has endured for nearly 30 years as one of the world's most popular languages. Here are its key features and why they matter:

  • Platform Independence (WORA): Compile once, run anywhere. Java bytecode runs on any JVM on any OS — Windows, Mac, Linux — without recompiling. Critical for enterprise software deployed across diverse environments.
  • Object-Oriented: Java is built around OOP — encapsulation, inheritance, polymorphism, and abstraction. This promotes modular, reusable, and maintainable code design.
  • Automatic Garbage Collection: Java's JVM automatically manages memory — allocating it when objects are created and reclaiming it when they're no longer used. Developers don't manually allocate or free memory (unlike C/C++), eliminating a large class of bugs.
  • Strong Type System & Security: Java is strongly and statically typed. It has no raw pointers, a security manager, and a bytecode verifier — making it significantly more secure than languages like C.
  • Multi-threading: Built-in support for concurrent programming with the Thread class, Runnable interface, and the java.util.concurrent package. Essential for high-performance server applications.
  • Rich Standard Library (Java API): Java ships with an enormous standard library — collections, I/O, networking, concurrency, database connectivity (JDBC), and more — reducing the need for third-party dependencies for common tasks.
  • Massive Ecosystem & Community: Decades of libraries, frameworks (Spring, Hibernate, Maven, Gradle), and one of the world's largest developer communities. Nearly every enterprise problem has a Java solution.
  • Backward Compatibility: Java maintains strong backward compatibility — code written decades ago still runs on modern JVMs.

Platform independence is Java's defining characteristic — summarised as "Write Once, Run Anywhere" (WORA). Here's why Java achieves this while languages like C do not.

Why C is platform-dependent: When you compile a C program, the compiler produces native machine code — binary instructions specific to the CPU architecture and OS it was compiled on (e.g., x86 Linux). That binary won't run on ARM macOS or Windows x64 without recompilation. Different platforms need different binaries.

How Java achieves platform independence: Java uses a two-step compilation model with an intermediate format called bytecode:

  1. Step 1 — Java compiler (javac): Compiles your .java source code into .class files containing bytecode — not native machine code, but a compact, standardised set of instructions for an imaginary "virtual machine."
  2. Step 2 — JVM on each platform: Each OS/hardware combination has its own JVM implementation. The JVM reads the same bytecode and translates it into the native machine code for that specific platform at runtime using a JIT (Just-In-Time) compiler. Same bytecode → different native code on each platform → same result everywhere.

Why bytecode (not direct compilation to machine code)?

  • Bytecode is a universal intermediate format — not tied to any hardware or OS.
  • A single compiled .jar file can be distributed and executed on Windows, macOS, Linux, Android without modification.
  • The JVM also performs runtime optimisations (JIT compilation), often making repeated code run faster than simple interpretation would.

The main method is the entry point of every Java application. When you run a Java program, the JVM looks for this exact method signature and starts execution from there. Every standalone Java application must have exactly one main method.

The complete signature: public static void main(String[] args)

Role of each keyword:

  • public: An access modifier. Making main public means it is accessible from anywhere — specifically, from the JVM itself. The JVM is external to your program and needs to call main to start it. If main were private or package-private, the JVM couldn't call it and your program couldn't start.
  • static: Makes main a class-level method — it belongs to the class itself, not to any instance (object). This is critical because when the JVM starts, no objects exist yet. The JVM calls main before creating any instances, so main must be callable without creating an object first. Without static, you'd need an object of the class to call main — but you need main to create objects — a chicken-and-egg problem.
  • void: The return type of main. It returns nothing back to the JVM. The program communicates its exit status to the OS via System.exit(0) (success) or System.exit(1) (failure) — not via a return value from main.
  • String[] args: An array of Strings — the command-line arguments passed to the program when it's launched. For example: java Main hello world gives args = ["hello", "world"]. The JVM passes this array to main automatically.
java — main method and command line args
public class Main {

    // Entry point — JVM calls this to start the program
    public static void main(String[] args) {

        // args contains command-line arguments
        System.out.println("Number of args: " + args.length);

        for (String arg : args) {
            System.out.println("Arg: " + arg);
        }

        System.out.println("Program started!");
    }
}

Understanding the spectrum of code "levels" helps you appreciate why Java's bytecode approach is elegant.

High-Level Code: Programming languages designed for humans — readable, expressive, and abstract from the underlying hardware. Concepts like variables, loops, classes, and methods exist. Java, Python, JavaScript, and C# are high-level languages. High-level code must be translated to something the machine can execute — it's not directly runnable by hardware.

Low-Level Code: Code that is close to the hardware and specific to a particular CPU architecture. Two forms:

  • Assembly Language: Human-readable but hardware-specific instructions (MOV AX, 1, ADD AX, BX). One step above machine code — each instruction maps directly to a machine operation.
  • Machine Code: The lowest level — pure binary (0s and 1s) that the CPU executes directly. All programs eventually become machine code before the CPU can run them.

Java Bytecode — the middle ground: Java bytecode is an intermediate representation that sits between high-level Java source code and low-level native machine code. It is:

  • Produced by the Java compiler (javac) from .java source files.
  • Stored in .class files in a compact, binary format (not human-readable).
  • Not tied to any specific CPU or OS — it's instructions for the JVM (an imaginary machine).
  • Executed by the JVM, which translates it to native machine code using JIT compilation.
  • More compact and efficient than high-level source code, but not as fast as directly-compiled native code.

// HIGH-LEVEL (Java) — human readable, abstract

Variables & Data Types
4 Qs

A variable is a named memory location that stores a value of a specific type. In Java, every variable must be declared with a data type before use — the compiler reserves an appropriate amount of memory and enforces that only compatible values are stored there.

A data type defines what kind of data a variable can hold — numbers, text, true/false values, objects, etc. — and how much memory is allocated for it.

Two major categories of data types in Java:

  • Primitive Data Types (8 types): The most basic building blocks — they hold a single value directly in memory. They are not objects and do not have methods. Java's 8 primitives are: byte, short, int, long, float, double, char, boolean.
  • Reference (Non-Primitive) Data Types: Variables that store a reference (memory address) to an object — not the object itself. The actual data lives in the heap. Examples: String, arrays, classes, interfaces, and all objects created with new.
java — declaring and using variables
// Syntax: dataType variableName = value;
int     age    = 25;
double  price  = 99.99;
boolean active = true;
char    grade  = 'A';
String  name   = "Riya";  // reference type

// Declaration without initialisation
int count;  // declared but not yet assigned
count = 10; // assigned later

// Multiple variables of same type
int x = 1, y = 2, z = 3;

Java has exactly 8 primitive data types — the fundamental types built into the language. They hold actual values directly in memory (stack), not references to objects. They are lowercase by convention.

  • byte — 8-bit signed integer. Range: -128 to 127. Used for memory-efficient integer storage (e.g., file I/O, network data). Default: 0.
  • short — 16-bit signed integer. Range: -32,768 to 32,767. Rarely used explicitly. Default: 0.
  • int — 32-bit signed integer. Range: ~-2.1 billion to ~2.1 billion. The most commonly used integer type for general calculations. Default: 0.
  • long — 64-bit signed integer. Range: ~-9.2 quintillion to ~9.2 quintillion. Use when int isn't large enough (e.g., timestamps, very large numbers). Literals need an L suffix: 100L. Default: 0L.
  • float — 32-bit IEEE 754 floating-point. Single precision. Use when memory is a concern and full precision isn't needed. Literals need f suffix: 3.14f. Default: 0.0f.
  • double — 64-bit IEEE 754 floating-point. Double precision. The default choice for decimal numbers. Default: 0.0.
  • char — 16-bit Unicode character. Range: '\u0000' (0) to '\uffff' (65,535). Stores a single character. Literal: 'A'. Default: '\u0000'.
  • boolean — Stores only true or false. Used for flags and conditions. Default: false.

Reference (Non-Primitive) data types are variables that don't store the actual data directly — they store a reference (memory address/pointer) to where the data lives in the heap memory. Everything that is not one of the 8 primitives is a reference type in Java.

Main categories of reference types:

  • String: The most commonly used reference type. Represents a sequence of characters. Strings in Java are immutable — once created, they cannot be changed (any operation that appears to modify a String actually creates a new one). Stored in the String pool for memory efficiency.
  • Arrays: A fixed-size, ordered collection of elements of the same type. Declared with type[] — can hold primitives or reference types. Arrays are objects in Java.
  • Classes: Any user-defined or library class (e.g., Scanner, ArrayList, HashMap). Variables of a class type hold a reference to an object instance created with new.
  • Interfaces: An interface variable can hold a reference to any object that implements that interface.

When a reference variable holds null, it means it's not pointing to any object. Attempting to use a null reference causes a NullPointerException at runtime.

java — reference types
// String — reference type (immutable)
String name = "Riya";
String city = new String("Mumbai");
name = name + " Sharma"; // creates NEW string, 'name' now points to new object

// Arrays — fixed-size collection
int[]    scores  = { 90, 85, 78 };
String[] fruits  = { "Apple", "Mango", "Banana" };
int[]    arr     = new int[5]; // initialised with 0s

System.out.println(scores[0]);   // 90
System.out.println(fruits.length); // 3

// Class instance (object)
Scanner sc = new Scanner(System.in);

// null reference
String s = null;
System.out.println(s.length()); // ❌ NullPointerException at runtime

Understanding this distinction is essential — it explains behaviour around memory, assignment, comparison, and method parameters.

  • Storage: Primitives store their actual value directly in the variable's memory location (stack). Reference types store a memory address — the actual object data lives in the heap.
  • Memory location: Primitive variables live on the stack — fast, automatically managed. Reference type objects live on the heap — managed by the garbage collector.
  • Default values: Primitives have default values (int = 0, boolean = false, double = 0.0). Reference types default to null (no object).
  • Methods: Primitive types have no methods. Reference types (being objects) have methods — String.length(), array.clone(), etc.
  • Assignment (copy vs reference): Assigning a primitive copies the value — two independent variables. Assigning a reference copies the address — both variables point to the same object. Changing the object through one variable affects what the other variable sees.
  • Comparison: == on primitives compares values. == on reference types compares memory addresses (are they the same object?). Use .equals() to compare reference type content.
  • Nullability: Primitives cannot be null — they always have a value. Reference types can be null.
  • Size: Primitives have a fixed, known size (int = 4 bytes, double = 8 bytes). Reference type object sizes vary depending on their contents.
java — primitive vs reference: key behaviour differences
// ── Primitive: copy by VALUE ──
int a = 10;
int b = a;    // b gets a COPY of 10
b = 20;
System.out.println(a); // 10 — a is unaffected

// ── Reference: copy by REFERENCE (address) ──
int[] arr1 = { 1, 2, 3 };
int[] arr2 = arr1;  // arr2 points to the SAME array
arr2[0] = 99;
System.out.println(arr1[0]); // 99 — arr1 is affected!

// ── Comparison difference ──
String s1 = new String("hello");
String s2 = new String("hello");
System.out.println(s1 == s2);        // false (different objects)
System.out.println(s1.equals(s2));  // true  (same content)

int x = 5, y = 5;
System.out.println(x == y);          // true  (same value)
Operators — Introduction
8 Qs

An operator is a special symbol (or keyword) that performs an operation on one or more values (called operands) and produces a result. Java has a rich set of operators, categorised by their purpose and the number of operands they act on.

Types of operators in Java:

  • Arithmetic Operators: + - * / % — mathematical calculations.
  • Assignment Operators: = += -= *= /= %= — assign values to variables.
  • Comparison (Relational) Operators: == != > < >= <= — compare two values, return boolean.
  • Logical Operators: && || ! — combine boolean expressions.
  • Unary Operators: + - ++ -- ! — operate on a single operand.
  • Ternary Operator: condition ? valueIfTrue : valueIfFalse — compact if-else.
  • Bitwise Operators: & | ^ ~ << >> >>> — operate on individual bits.
  • instanceof Operator: Tests whether an object is an instance of a specific class.

Arithmetic operators perform mathematical calculations on numeric operands (int, double, long, etc.). Java has 5 arithmetic operators:

  • + (Addition): Adds two operands. Also used for String concatenation — if either operand is a String, + converts the other to String and concatenates.
  • - (Subtraction): Subtracts the right operand from the left.
  • * (Multiplication): Multiplies two operands.
  • / (Division): Divides left by right. Important: If both operands are integers, the result is an integer (fractional part is truncated). Use at least one double for decimal division.
  • % (Modulus/Remainder): Returns the remainder after integer division. Commonly used to check if a number is even/odd (n % 2 == 0), or to cycle through a range.
java — arithmetic operators
int a = 10, b = 3;

System.out.println(a + b);  // 13
System.out.println(a - b);  // 7
System.out.println(a * b);  // 30
System.out.println(a / b);  // 3  (integer division — truncates)
System.out.println(a % b);  // 1  (10 = 3*3 + 1, remainder is 1)

// Decimal division — cast to double
System.out.println((double) a / b); // 3.3333...

// String concatenation with +
String name = "Riya";
System.out.println("Hello, " + name);  // "Hello, Riya"
System.out.println("Score: " + 90);   // "Score: 90"

// Useful modulus patterns
System.out.println(10 % 2); // 0 — even number
System.out.println(7  % 2); // 1 — odd number
System.out.println(13 % 5); // 3 — remainder cycling 0-4

Assignment operators assign a value to a variable. The basic assignment operator is =. Java also provides compound assignment operators that combine an arithmetic operation with assignment — shorthand for common patterns like x = x + 5.

  • = — Simple assignment: assigns right-side value to left-side variable.
  • += — Add and assign: x += 5 means x = x + 5.
  • -= — Subtract and assign: x -= 3 means x = x - 3.
  • *= — Multiply and assign: x *= 2 means x = x * 2.
  • /= — Divide and assign: x /= 4 means x = x / 4.
  • %= — Modulus and assign: x %= 3 means x = x % 3.
java — assignment operators
int x = 10; // simple assignment

x += 5;  System.out.println(x); // 15
x -= 3;  System.out.println(x); // 12
x *= 2;  System.out.println(x); // 24
x /= 4;  System.out.println(x); // 6
x %= 4;  System.out.println(x); // 2

// Compound operators also work on strings
String msg = "Hello";
msg += " World";
System.out.println(msg); // "Hello World"

Comparison (Relational) operators compare two values and return a boolean result — either true or false. They are the foundation of decision-making in Java — used in if statements, while loops, and any condition that controls program flow.

  • == — Equal to: returns true if both operands are equal.
  • != — Not equal to: returns true if operands are different.
  • > — Greater than.
  • < — Less than.
  • >= — Greater than or equal to.
  • <= — Less than or equal to.

When to use: Any time you need to make a decision based on comparing values — validating user input, checking loop termination, comparing prices, ages, scores, etc. Important: For comparing objects (like Strings), always use .equals() instead of == — == compares references (addresses), not content.

java — comparison operators
int a = 10, b = 20;

System.out.println(a == b);  // false
System.out.println(a != b);  // true
System.out.println(a >  b);  // false
System.out.println(a <  b);  // true
System.out.println(a >= 10); // true
System.out.println(a <= 5);  // false

// Used in conditions
int age = 18;
if (age >= 18) {
    System.out.println("Adult");
}

// String comparison — NEVER use == for content
String s1 = new String("Java");
String s2 = new String("Java");
System.out.println(s1 == s2);       // false — different objects
System.out.println(s1.equals(s2)); // true  — same content ✅

Logical operators combine or invert boolean expressions to form more complex conditions. They operate on boolean values and return a boolean result.

  • && (Logical AND): Returns true only if both operands are true. If the first operand is false, the second is not evaluated (short-circuit evaluation) — saving unnecessary computation.
  • || (Logical OR): Returns true if at least one operand is true. If the first operand is true, the second is not evaluated (short-circuit evaluation).
  • ! (Logical NOT): Reverses a boolean value. !true → false, !false → true. Used to invert conditions.

When to use: Any time a condition depends on multiple criteria — a user must be both logged in AND have admin role; a number must be either positive OR even; a field must NOT be null.

java — logical operators
boolean isLoggedIn = true;
boolean isAdmin    = false;
int     age        = 20;

// AND — both must be true
if (isLoggedIn && isAdmin) {
    System.out.println("Access granted");
} else {
    System.out.println("Access denied"); // ← this runs
}

// OR — at least one must be true
if (age < 13 || age > 65) {
    System.out.println("Discounted ticket");
}

// NOT — invert the condition
if (!isAdmin) {
    System.out.println("Not an admin"); // ← this runs
}

// Short-circuit — second operand not evaluated if result already known
String s = null;
if (s != null && s.length() > 0) { // ✅ safe — s.length() skipped if s is null
    System.out.println("Not empty");
}

Unary operators operate on a single operand (unlike binary operators which need two). Java has 5 unary operators:

  • + (Unary plus): Indicates a positive value. Rarely used explicitly — numbers are positive by default.
  • - (Unary minus/negation): Negates a numeric value. -x changes positive to negative and vice versa.
  • ++ (Increment): Increases a variable's value by 1. Two forms: pre-increment (++x — increments first, then uses the value) and post-increment (x++ — uses the value first, then increments).
  • -- (Decrement): Decreases a variable's value by 1. Same pre/post distinction as ++.
  • ! (Logical NOT): Inverts a boolean value.
java — unary operators
int x = 5;

System.out.println(-x);   // -5  (negation)
System.out.println(+x);   // 5   (unary plus)

// Pre-increment: increment FIRST, then use value
int a = 5;
System.out.println(++a);  // 6 (a becomes 6 first, then printed)
System.out.println(a);    // 6

// Post-increment: use value FIRST, then increment
int b = 5;
System.out.println(b++);  // 5 (5 printed first, then b becomes 6)
System.out.println(b);    // 6

// Decrement
int c = 10;
System.out.println(--c);  // 9
System.out.println(c--);  // 9 (printed, then c becomes 8)
System.out.println(c);    // 8

// Logical NOT
boolean flag = true;
System.out.println(!flag); // false

The ternary operator (?:) is Java's only operator that takes three operands. It is a concise, inline alternative to a simple if-else statement — useful when you need to choose between two values based on a condition and want to do so in a single expression.

Syntax: condition ? valueIfTrue : valueIfFalse

  • The condition is evaluated first — it must produce a boolean.
  • If the condition is true, the expression returns valueIfTrue.
  • If the condition is false, the expression returns valueIfFalse.

When to use: For simple, short decisions where the result is a single value — setting a variable, choosing a display string, selecting a default. Avoid nesting ternary operators — they become unreadable quickly. For complex logic, prefer a regular if-else block.

java — ternary operator
// Basic ternary
int age = 20;
String status = (age >= 18) ? "Adult" : "Minor";

The instanceof operator is used to test whether an object is an instance of a specific class, subclass, or interface. It returns true if the object is of that type (or a subtype), and false otherwise. If the object reference is null, instanceof always returns false.

When to use:

  • Before casting: Before casting an object to a specific type, use instanceof to verify the cast is safe and won't throw a ClassCastException at runtime.
  • Polymorphic behaviour: When a method receives a parent type reference but needs to behave differently based on the actual runtime type of the object.
  • Pattern matching (Java 16+): Modern Java allows instanceof with a pattern variable — the cast and variable binding happen in one expression.
java — instanceof operator
// Class hierarchy
class Animal {}
class Dog extends Animal {}

Animal a = new Dog();

System.out.println(a instanceof Dog);    // true  — actual type is Dog
System.out.println(a instanceof Animal); // true  — Dog is-a Animal

// Safe casting — check before cast

    Dog dog = (Dog) a; // safe cast ✅
    System.out.println("It's a dog!");
}

// null check — instanceof with null is always false

System.out.println(nullAnimal instanceof Animal); // false (no NPE)

// Java 16+ Pattern Matching — check AND cast in one step

    System.out.println(dog.getClass().getSimpleName()); // Dog
}
Control Statements — Basics
5 Qs

By default, Java executes statements sequentially — top to bottom, one after another. Control statements (also called control flow statements) allow you to alter this default sequential execution — making programs dynamic and capable of making decisions, repeating actions, and branching based on conditions.

Without control statements, a program would always do the same thing in the same order — completely rigid. Control statements are what make programs intelligent and responsive.

Three categories of control statements:

  • Conditional / Selection Statements: Execute a block of code only if a condition is true — choosing between paths. if, if-else, if-else if-else, switch.
  • Looping / Iteration Statements: Repeat a block of code multiple times — as long as a condition holds true. for, while, do-while, for-each.
  • Jump / Transfer Statements: Immediately transfer control to another part of the program. break (exit a loop or switch), continue (skip current iteration), return (exit a method).

Conditional statements allow a program to make decisions — executing different blocks of code depending on whether a condition is true or false. They are the foundation of program logic.

Forms of the if statement:

  • if: Executes a block only if the condition is true. If false, the block is skipped entirely and execution continues after.
  • if-else: Provides two paths — one for true, one for false. Exactly one of the two blocks executes, never both.
  • if-else if-else: Chains multiple conditions. Conditions are evaluated top to bottom. The first one that is true executes its block. The rest are skipped. The final else is a catch-all that runs if no condition was true.
  • Nested if: An if statement inside another if block — for multi-dimensional conditions.
java — if / if-else / if-else if-else
// Simple if
int temp = 35;
if (temp > 30) {
    System.out.println("It's hot!");
}

// if-else — two paths
int age = 16;
if (age >= 18) {
    System.out.println("Can vote");
} else {
    System.out.println("Too young to vote"); // ← runs
}

// if-else if-else — multiple conditions
int score = 72;
if      (score >= 90) { System.out.println("Grade A"); }
else if (score >= 80) { System.out.println("Grade B"); }
else if (score >= 70) { System.out.println("Grade C"); } // ← runs
else if (score >= 60) { System.out.println("Grade D"); }
else                  { System.out.println("Fail");    }

// Nested if
int num = 15;
if (num > 0) {
    if (num % 2 == 0) {
        System.out.println("Positive and Even");
    } else {
        System.out.println("Positive and Odd"); // ← runs
    }
}

Looping statements (iteration statements) allow a block of code to be executed repeatedly as long as a specified condition remains true. Instead of writing the same code multiple times, a loop lets you write it once and control how many times it executes.

Java provides four types of loops: while, for, do-while, and the enhanced for-each loop.

The while loop: A pre-test loop — the condition is evaluated before each iteration. If the condition is false from the start, the loop body never executes. The loop continues as long as the condition remains true. You must ensure the condition eventually becomes false, or you'll have an infinite loop.

Structure: while (condition) { // body }

When to use while: When you don't know in advance how many iterations are needed — the loop should run until a certain condition is met. Example: reading user input until they enter "quit", polling until a resource is available.

java — while loop
// Count 1 to 5
int i = 1;
while (i <= 5) {
    System.out.println(i);  // 1, 2, 3, 4, 5
    i++;                      // update — must move toward false condition
}

// Sum of 1 to 100
int sum = 0, n = 1;
while (n <= 100) {
    sum += n;
    n++;
}
System.out.println("Sum: " + sum); // 5050

// Condition false from start — body never runs
int x = 10;
while (x < 5) {
    System.out.println("Never prints");
}

// ⚠️ Infinite loop — be careful!
// while (true) { System.out.println("infinite"); }
// Use break to exit: while (true) { if (condition) break; }

Both while and for are pre-test loops — they check the condition before each iteration. The fundamental difference is in structure and intended use case, not capability — anything you can do with for, you can do with while and vice versa.

  • for loop: Has all three loop control elements (initialisation, condition, update) packed into a single compact line. Best used when you know in advance exactly how many iterations are needed. Very readable for counting loops.
    for (initialisation; condition; update) { // body }
  • while loop: Only has the condition in the loop header. Initialisation is done before the loop and update happens inside the body. Best used when the number of iterations is unknown — the loop runs until a condition changes.
    while (condition) { // body + update }
  • Variable scope: In a for loop, the initialisation variable (int i = 0) is scoped to the loop — not accessible after. In a while loop, the counter variable is declared before the loop and accessible after.
  • Readability: for is more concise and readable for count-based loops. while is clearer for condition-based loops where count is irrelevant.
java — for vs while: same result, different style
// for loop — best for known iteration count
for (int i = 1; i <= 5; i++) {
    System.out.println(i); // 1, 2, 3, 4, 5
}
// System.out.println(i); ← i not accessible here

// while loop — same output, different structure
int i = 1; // initialisation outside
while (i <= 5) {
    System.out.println(i);
    i++; // update inside body
}
System.out.println(i); // ← 6, accessible here

// while better for unknown count — read until -1
Scanner sc = new Scanner(System.in);
int num = sc.nextInt();
while (num != -1) {
    System.out.println("Got: " + num);
    num = sc.nextInt();
}

// for better for arrays/collections
int[] arr = { 10, 20, 30 };
for (int j = 0; j < arr.length; j++) {
    System.out.println(arr[j]);
}

Both break and continue are jump statements that alter the normal flow inside a loop (or switch), but they do very different things:

  • break — exits the loop entirely: When executed, it immediately terminates the innermost enclosing loop (or switch statement) and transfers control to the statement immediately following the loop. All remaining iterations are abandoned — the loop is done.
  • continue — skips the current iteration: When executed, it immediately jumps to the next iteration of the loop — skipping the rest of the current loop body. The loop itself continues; only the current cycle is cut short. In a for loop, the update expression still runs after continue.
java — break vs continue
// break — stops the loop when i == 4
for (int i = 1; i <= 10; i++) {
    if (i == 4) break;
    System.out.print(i + " "); // 1 2 3 (stops before 4)
}
System.out.println("After loop"); // executes — break exits to here

// continue — skips even numbers, prints only odd
for (int i = 1; i <= 10; i++) {
    if (i % 2 == 0) continue; // skip even — jump to i++
    System.out.print(i + " "); // 1 3 5 7 9
}

// Practical: search in array — break when found
int[] nums = { 5, 12, 3, 8, 21 };
for (int n : nums) {
    if (n == 8) {
        System.out.println("Found 8!");
        break; // no point checking remaining elements
    }
}
Control Statements — Advanced
5 Qs

Both loops repeat a block of code, but they differ in a critical way: when the condition is checked.

  • while loop (pre-test / entry-controlled): The condition is checked before the loop body executes. If the condition is false at the start, the loop body executes zero times — it never runs.
  • do-while loop (post-test / exit-controlled): The loop body executes first, and then the condition is checked at the end. This guarantees the loop body executes at least once — regardless of whether the condition is true or false.

The key distinction in one line: while may execute 0 or more times. do-while always executes at least once.

When to choose do-while: Whenever you need the code to run at least once before checking if it should repeat — classic example: showing a menu to a user. You always need to display the menu at least once before asking if they want to continue.

java — while vs do-while
// while — body may NEVER execute
int x = 10;
while (x < 5) {
    System.out.println("while runs"); // NEVER prints
}

// do-while — body executes AT LEAST ONCE
int y = 10;
do {
    System.out.println("do-while runs"); // prints ONCE despite false condition
} while (y < 5); // condition checked AFTER body — false, so stops

// Classic use case: menu system
Scanner sc = new Scanner(System.in);
int choice;
do {
    System.out.println("1. View  2. Edit  3. Exit");
    System.out.print("Enter choice: ");
    choice = sc.nextInt();
    // process choice...
} while (choice != 3); // keeps looping until user enters 3
// Menu always shows at least once — perfect for do-while

Choosing the right loop makes code more readable, maintainable, and expressive. Here's the decision framework:

  • Use for when: You know the exact number of iterations in advance — or you're working with a range/index. The loop variable, condition, and update are all clearly defined upfront. Examples: iterate over an array by index, count from 1 to N, repeat exactly 5 times, process every element with index access.
  • Use while when: The number of iterations is unknown ahead of time — the loop runs until some condition changes. The stopping condition depends on runtime events or external input. Examples: reading until end of file, processing until a flag is set, polling until a resource is ready, game loop that runs until the player quits.
  • Use do-while when: The loop body must execute at least once, regardless of the condition — and the condition to continue is only known after the first run. Examples: displaying a menu (always show first, then ask to repeat), asking for user input with validation (get input, check if valid, repeat if not), connection retry (try connecting first, then check if successful).
java — choosing the right loop
// for — known count: print multiplication table of 5
for (int i = 1; i <= 10; i++) {
    System.out.println("5 x " + i + " = " + (5 * i));
}

// while — unknown count: keep asking until valid positive number
Scanner sc = new Scanner(System.in);
int num = -1;
while (num < 0) {
    System.out.print("Enter a positive number: ");
    num = sc.nextInt();
}
System.out.println("Got: " + num);

// do-while — must run at least once: validated input
int password;
do {
    System.out.print("Enter PIN (4 digits): ");
    password = sc.nextInt();
} while (password < 1000 || password > 9999); // repeat until valid 4-digit
System.out.println("Valid PIN entered");

The for-each loop (also called the enhanced for loop) was introduced in Java 5 to provide a cleaner, simpler syntax for iterating over arrays and collections. It removes the need to manage an index variable.

  • Regular for loop: Uses an index counter — you have full control over the starting value, step size, and direction. You can access elements by index, iterate in reverse, skip elements, or modify elements by index. More verbose but more flexible.
  • For-each loop: Automatically iterates over every element from first to last — no index management needed. The element variable directly receives each value. Cleaner syntax, harder to get wrong (no off-by-one errors). But: no access to the current index, cannot iterate in reverse, and cannot modify the collection while iterating.

When to use for-each: When you need to visit every element in order, don't need the index, and don't need to modify the array/collection structure. Basically: reading or processing each element — not searching by position, not reversing, not inserting/deleting.

java — for vs for-each
int[] numbers = { 10, 20, 30, 40, 50 };

// Regular for — has index, can go backwards, can modify by index
for (int i = 0; i < numbers.length; i++) {
    System.out.println("Index " + i + ": " + numbers[i]);
    numbers[i] *= 2; // ✅ can modify elements
}

// for-each — cleaner, element directly available, no index needed
String[] fruits = { "Apple", "Mango", "Banana" };
for (String fruit : fruits) {
    System.out.println(fruit); // Apple, Mango, Banana
}

// for-each with collections
ArrayList<Integer> list = new ArrayList<>();
list.add(1); list.add(2); list.add(3);
for (int num : list) {
    System.out.println(num);
}

// When to use regular for instead of for-each:
// - Need the index: for (int i = ...) { arr[i] }
// - Reverse iteration: for (int i = arr.length-1; i >= 0; i--)
// - Skip elements: for (int i = 0; i < arr.length; i += 2)
// - Modify array elements by index

The switch statement is a multi-way branch statement — it evaluates an expression and executes the code block associated with the matching case value. It is an alternative to a long if-else if-else if chain when you're comparing the same variable against many specific constant values.

Key concepts:

  • switch(expression): The expression is evaluated once. It can be of type int, byte, short, char, String (Java 7+), or enum. Not float, double, or long.
  • case value: Each case specifies a constant value to compare against. If it matches, execution begins there.
  • break: Critical — exits the switch block. Without break, execution "falls through" to the next case, running its code too (sometimes intentional, often a bug).
  • default: Optional catch-all case — executes if no case matches. Like the final else in an if-else chain.
  • Switch Expression (Java 14+): The modern switch expression syntax using -> arrows — no fall-through, no explicit break needed, and can return a value.
java — switch statement and modern switch expression
// Traditional switch statement
int day = 3;
switch (day) {
    case 1: System.out.println("Monday");    break;
    case 2: System.out.println("Tuesday");   break;
    case 3: System.out.println("Wednesday"); break; // ← runs
    case 4: System.out.println("Thursday");  break;
    case 5: System.out.println("Friday");    break;
    default: System.out.println("Weekend");
}

// Fall-through (intentional) — multiple cases same action
int month = 4;
switch (month) {
    case 2:
        System.out.println("28 or 29 days"); break;
    case 4: case 6: case 9: case 11:  // fall-through for all 4 cases
        System.out.println("30 days"); break;
    default:
        System.out.println("31 days");
}

// Modern Switch Expression (Java 14+) — cleaner, no fall-through
String dayName = switch (day) {
    case 1 -> "Monday";
    case 2 -> "Tuesday";
    case 3 -> "Wednesday";
    case 4 -> "Thursday";
    case 5 -> "Friday";
    default -> "Weekend";
};

Choosing the right conditional construct makes code more readable, maintainable, and efficient. Here's the practical decision guide:

  • Use if (single) when: You have one condition and only want to execute code when it's true — no action needed when it's false. Example: log a warning only if an error occurs. Simple, one-directional check.
  • Use if-else when: There are exactly two mutually exclusive outcomes — something should happen for both the true and false case. Example: show "Admin Panel" or "User Dashboard" based on user role.
  • Use if-else if-else when: You have multiple conditions to check — ranges, grades, tiers — and the conditions are evaluated sequentially. Best when conditions involve ranges or complex boolean expressions that don't map to simple equality. Example: assigning letter grades (A, B, C, D, F) based on score ranges.
  • Use switch when: You're comparing the same single variable against many specific, discrete constant values (not ranges). Switch is cleaner, faster to read, and sometimes slightly more performant than a long if-else chain. Example: routing based on a menu choice, handling different HTTP methods, mapping day numbers to names.
  • Use ternary operator when: The logic is a simple binary choice and the result fits in one line — setting a variable to one of two values. Example: String label = isAdmin ? "Admin" : "User";. Avoid for complex or nested conditions — use if-else for readability.
java — choosing the right conditional
// if — one condition, no else needed
if (temperature > 40) {
    System.out.println("⚠️ Heatwave alert!");
}

// if-else — exactly two outcomes
if (balance >= amount) { transfer(amount); }
else                   { System.out.println("Insufficient funds"); }

// if-else if-else — ranges / complex conditions
if      (score >= 90) grade = 'A';
else if (score >= 80) grade = 'B';
else if (score >= 70) grade = 'C';
else                   grade = 'F';

// switch — same variable, multiple specific values
switch (httpMethod) {
    case "GET":    handleGet();    break;
    case "POST":   handlePost();   break;
    case "DELETE": handleDelete(); break;
    default:       handleError();
}

// ternary — simple one-liner binary choice
String msg = (items.isEmpty()) ? "Cart is empty" : "Items in cart: " + items.size();
String & StringBuilder
5 Qs

In Java, String objects are immutable — once created, their content cannot be changed. Every concatenation creates a brand-new String object. StringBuilder is mutable — you can modify it in place without creating new objects, making it far more efficient for building strings repeatedly.

  • Mutability: String is immutable; StringBuilder is mutable.
  • Performance: String concatenation in a loop is O(n²); StringBuilder.append() is O(n).
  • Thread Safety: String is thread-safe (immutable); StringBuilder is NOT. Use StringBuffer if multiple threads share it.
java — string vs stringbuilder
// String — each + creates a NEW object
String s = "Hello";
s = s + " World"; // new object created

// ❌ Inefficient loop
String result = "";
for (int i = 0; i < 1000; i++) result += i; // 1000 new objects!

// ✅ StringBuilder — modifies same object
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) sb.append(i);
String finalStr = sb.toString();

// StringBuilder methods
StringBuilder b = new StringBuilder("Hello");
b.append(" World");  // "Hello World"
b.insert(5, ",");    // "Hello, World"
b.delete(5, 6);    // "Hello World"
b.reverse();        // "dlroW olleH"

Use String when: text doesn't change after creation, simple one-off concatenation, map keys/set elements, multi-threaded sharing (immutable = safe).

Use StringBuilder when: building strings in a loop, frequent modifications (insert/delete/replace), performance-critical code, generating CSV rows, SQL queries, or large reports dynamically.

java — real-world usage
// ✅ String — simple fixed text
String greeting = "Hello, " + name + "!"; // compiler optimises

// ✅ StringBuilder — building in a loop
String[] words = { "Java", "is", "awesome" };
StringBuilder sentence = new StringBuilder();
for (String w : words) {
    if (sentence.length() > 0) sentence.append(" ");
    sentence.append(w);
}
System.out.println(sentence); // "Java is awesome"

The String Pool is a special memory region in the JVM heap where Java stores String literals to reuse them. When you write a literal like "Java", the JVM checks if it already exists in the pool — if so, it returns the existing reference instead of creating a new object. new String("Java") always creates a new heap object outside the pool.

java — string pool
String s1 = "Java";  String s2 = "Java";
System.out.println(s1 == s2);       // true  — same pool reference
System.out.println(s1.equals(s2)); // true

String s3 = new String("Java");  String s4 = new String("Java");
System.out.println(s3 == s4);       // false — different heap objects
System.out.println(s3.equals(s4)); // true

String s5 = s3.intern(); // forces pool lookup — returns pool ref
System.out.println(s1 == s5); // true
  • == — compares references (memory addresses). Returns true only if both variables point to the exact same object.
  • .equals() — compares content character by character. Returns true if both strings have the same characters in the same order.

Rule: Always use .equals() for String content comparison. To avoid NullPointerException, put the known non-null string first: "expected".equals(variable).

java — == vs equals()

String input = null;
System.out.println("hello".equals(input)); // false — no NPE ✅

Arrays — Introduction

An Array is a fixed-size, ordered collection of elements of the same data type stored in contiguous memory. Each element is accessed by its zero-based index.

Why we need arrays: Instead of 100 separate variables (score1, score2, ...), declare one array int[] scores = new int[100]. Process all values with a single loop, find max/min/average, sort data, store pixels, manage inventory. Arrays are the foundation of all other data structures (ArrayList, Stack, Queue). Accessing any element by index is O(1) — instant.

java — why arrays
int[] temperatures = { 32, 35, 30, 28, 33, 36, 29 };

System.out.println("Avg: " + (double) sum / temperatures.length);

Two ways to declare: with new (elements get default values: 0, false, null), or with an initialiser list (values provided directly). Access elements with arrayName[index]. Indices: 0 to length-1. Accessing an invalid index throws ArrayIndexOutOfBoundsException.

int[]    primes = { 2, 3, 5, 7, 11 };

OOPS — Classes, Objects & Packages

Object-Oriented Programming (OOPS) organises code around objects — bundles of data and behaviour — rather than functions and procedures. Java is primarily OOP.

4 main pillars:

  • Encapsulation: Bundling data + methods in a class; restrict access with private. Protect internal state.
  • Inheritance: Child class inherits from parent using extends. Code reuse + hierarchy.
  • Polymorphism: Same method, different behaviour depending on the object. Overriding + overloading.
  • Abstraction: Hide complex implementation; expose only essentials via abstract classes/interfaces.

A Class is a blueprint/template defining fields (data) and methods (behaviour). It is a logical entity — no memory allocated. An Object is a concrete instance created from a class using new — it lives in heap memory with its own copy of field values.

Why use them: Model real-world entities naturally (Account, Product, Customer), reuse code (one class → many objects), encapsulate state, make code intuitive and maintainable.

OOPS — Inheritance
7 Qs

Inheritance is an OOP mechanism where a child class (subclass) acquires the properties and methods of a parent class (superclass) using the extends keyword. The child inherits all public and protected members, can reuse them, add new ones, or override them.

When to use in real applications:

  • IS-A relationship exists: Use inheritance when the child truly IS a type of the parent. Dog extends Animal — a dog IS an animal. SavingsAccount extends BankAccount — a savings account IS a bank account. If the relationship is HAS-A (a Car HAS-A Engine), use composition instead.
  • Code reuse: Multiple subclasses need the same behaviour. Define it once in the parent — all children get it for free. Example: all account types need deposit() and getBalance().
  • Extending existing functionality: When you need a specialised version of an existing class — PremiumUser extends User adds premium-specific features without touching the base User class.
java — inheritance example
// Parent class
class BankAccount {
    protected String owner;
    protected double balance;
    public BankAccount(String owner, double balance) {
        this.owner = owner; this.balance = balance;
    }
    public void deposit(double amt) { balance += amt; }
    public double getBalance()      { return balance; }
}
// Child class — IS-A BankAccount
class SavingsAccount extends BankAccount {
    private double interestRate;
    public SavingsAccount(String owner, double bal, double rate) {
        super(owner, bal); // call parent constructor
        this.interestRate = rate;
    }
    public void addInterest() { balance += balance * interestRate; } // new behaviour
}
SavingsAccount sa = new SavingsAccount("Riya", 10000, 0.05);
sa.deposit(5000);   // ✅ inherited from BankAccount
sa.addInterest();   // ✅ SavingsAccount-specific

Inheritance is implemented using the extends keyword. Key rules: a class can extend only one parent class; the child's constructor must call super() if the parent has a parameterised constructor; super.method() calls the parent version of an overridden method.

java — implementing inheritance
class Shape {
    protected String color;
    public Shape(String color) { this.color = color; }
    public String getInfo() { return "Color: " + color; }
    public double area() { return 0; }
}

class Circle extends Shape {

    public Circle(String color, double radius) {

        this.radius = radius;

    public String getInfo() {

    }

  • Single Inheritance: One child, one parent. Dog extends Animal. Most common and simplest — use when a class specialises exactly one parent concept.
  • Multilevel Inheritance: A → B → C chain. Animal → Mammal → Dog. Use for progressively specialised hierarchies. Each level adds more specific behaviour.
  • Hierarchical Inheritance: One parent, multiple children. Animal → Dog, Cat, Bird. Use when multiple specialised types share common base behaviour.
  • Multiple Inheritance (via Interfaces): One class implements multiple interfaces. Java doesn't support multiple class inheritance but supports multiple interface implementation. Use when a class needs to fulfil multiple contracts.
  • Hybrid Inheritance: Combination of above types — achieved only via interfaces in Java.

Multiple inheritance is when a class inherits from more than one parent class simultaneously — getting the members of all parents. Java does NOT support multiple inheritance of classes — a class can only extend one class. However, Java does support multiple inheritance through interfaces — a class can implement any number of interfaces.

java — multiple inheritance: not allowed vs allowed
// ❌ Multiple class inheritance — NOT allowed in Java
class A { void hello() { System.out.println("A"); } }
class B { void hello() { System.out.println("B"); } }
// class C extends A, B { } // ← COMPILE ERROR

// ✅ Multiple interface implementation — ALLOWED
interface Printable  { void print(); }
interface Saveable   { void save(); }
interface Exportable { void export(); }
class Report implements Printable, Saveable, Exportable {
    public void print()  { System.out.println("Printing..."); }
    public void save()   { System.out.println("Saving..."); }
    public void export() { System.out.println("Exporting..."); }
}

Java deliberately excluded multiple class inheritance to avoid the Diamond Problem — a classic ambiguity that arises when a class inherits from two parents that both inherit from a common grandparent.

The Diamond Problem: Imagine class D extends both B and C. Both B and C extend A and both override A's show() method. Now when D calls show(), which version should be used — B's or C's? The compiler has no way to know, creating a fundamental ambiguity. The inheritance hierarchy forms a diamond shape, hence the name.

java — diamond problem illustrated
//         A (show())
//        / //       B   C    Both override show()
//        \ /
//         D     ← Which show() does D.show() call? Ambiguous!
//
// class A { void show() { System.out.println("A"); } }
// class B extends A { void show() { System.out.println("B"); } }
// class C extends A { void show() { System.out.println("C"); } }
// class D extends B, C { } // ← Java says NO — prevents this ambiguity

// ✅ Interfaces ALLOW this because:
// - Interface methods were abstract (no implementation to conflict)
// - Java 8 default methods: class MUST override if two interfaces conflict
interface A { default void show() { System.out.println("A"); } }
interface B extends A { default void show() { System.out.println("B"); } }
interface C extends A { default void show() { System.out.println("C"); } }
class D implements B, C {
    public void show() { B.super.show(); } // must override — resolve explicitly
}

Java provides two clean alternatives to multiple class inheritance:

  • Multiple Interface Implementation: A class can implement as many interfaces as needed. This gives the class multiple contracts/types to fulfil without the diamond ambiguity — because interface methods are abstract (the class writes its own implementation) so there's nothing to conflict.
  • Composition (HAS-A relationship): Instead of inheriting from multiple classes, a class holds references to objects of those classes as fields and delegates calls to them. This is often considered better design than inheritance for complex scenarios — it's more flexible and avoids tight coupling.
java — composition as alternative to multiple inheritance
// Instead of: class SmartPhone extends Phone, Camera (not allowed)
class Phone  { void call() { System.out.println("Calling..."); } }
class Camera { void capture() { System.out.println("Capturing..."); } }

// ✅ Composition — SmartPhone HAS-A Phone and HAS-A Camera
class SmartPhone {
    private Phone  phone  = new Phone();
    private Camera camera = new Camera();
    void call()    { phone.call(); }     // delegate to Phone
    void capture() { camera.capture(); } // delegate to Camera
}

SmartPhone sp = new SmartPhone();
sp.call();     // "Calling..."
sp.capture(); // "Capturing..."

Use the final keyword on the class declaration. A final class cannot be extended — any attempt to extend it causes a compile error. Similarly, final on a method prevents it from being overridden in subclasses (while still allowing the class to be inherited).

Why make a class final: Security (prevent subclasses from altering behaviour — e.g., Java's String, Integer, Math are all final), performance (JVM can optimise final method calls), and design intent (signals the class is complete and not meant to be extended).

OOPS — Polymorphism
12 Qs

Polymorphism (Greek: "many forms") is the OOP ability for a single interface or method name to behave differently depending on the object or context. The same method call can produce different results based on which object's class implements it. It is what allows you to write flexible, extensible code — adding new types without modifying existing code that uses the parent type.

java — polymorphism in action
class Animal  { void speak() { System.out.println("..."); } }
class Dog    extends Animal { void speak() { System.out.println("Woof!"); } }
class Cat    extends Animal { void speak() { System.out.println("Meow!"); } }
class Parrot extends Animal { void speak() { System.out.println("Hello!"); } }

// All stored as Animal type — polymorphism at work
Animal[] animals = { new Dog(), new Cat(), new Parrot() };
for (Animal a : animals) a.speak(); // Woof! Meow! Hello!
// Same method call, 3 different behaviours
  • Compile-time Polymorphism (Static / Early Binding): Resolved by the compiler at compile time. Achieved through method overloading — same method name, different parameter lists. The compiler picks the correct version based on the arguments at compile time.
  • Runtime Polymorphism (Dynamic / Late Binding): Resolved by the JVM at runtime. Achieved through method overriding — a child class redefines a parent's method. Which version runs depends on the actual object type at runtime, not the declared reference type.
java — compile-time vs runtime polymorphism
// Compile-time — overloading, resolved at compile time
class Calculator {
    int    add(int a, int b)       { return a + b; }
    double add(double a, double b) { return a + b; }
    int    add(int a, int b, int c){ return a+b+c; }
}

// Runtime — overriding, resolved at runtime based on actual type
Animal a = new Dog();  // reference=Animal, object=Dog
a.speak(); // "Woof!" — Dog's version called at runtime

Method Overloading is defining multiple methods in the same class with the same name but different parameter lists. The compiler selects the right version based on the arguments provided at the call site. Overloaded methods must differ in: number of parameters, type of parameters, or order of parameters. Return type alone is NOT enough.

When to use: When the same operation applies to different types or different numbers of inputs — print(int), print(double), print(String). Makes the API cleaner — callers don't need to remember different method names for the same logical operation.

java — method overloading
class MathUtils {
    // Overloaded by parameter count
    static int    add(int a, int b)            { return a+b; }
    static int    add(int a, int b, int c)      { return a+b+c; }
    // Overloaded by parameter type
    static double add(double a, double b)       { return a+b; }
    // Overloaded by type order
    static String add(String s, int n)          { return s + n; }
    static String add(int n, String s)          { return n + s; }
}
System.out.println(MathUtils.add(2, 3));        // 5
System.out.println(MathUtils.add(1, 2, 3));     // 6
System.out.println(MathUtils.add(1.5, 2.5));    // 4.0

Overloading is called compile-time polymorphism or early binding because the compiler resolves which overloaded method to call at compile time — before the program ever runs. The compiler examines the method name, the number of arguments, and their types at the call site, matches them against the available overloaded signatures, and hard-codes the correct method reference into the bytecode. No decision needs to be made at runtime — everything is determined during compilation.

java — compile-time resolution
MathUtils.add(2, 3);       // compiler sees: 2 ints → calls add(int,int)
MathUtils.add(1.5, 2.5);  // compiler sees: 2 doubles → calls add(double,double)
// Decision made at COMPILE TIME — that's why it's "early binding"
// The correct method is known before the program starts running

A method can be overloaded in 3 ways — by changing the parameter signature (NOT the return type):

  • 1. Different number of parameters: add(int a, int b) vs add(int a, int b, int c)
  • 2. Different types of parameters: add(int a, int b) vs add(double a, double b)
  • 3. Different order of parameter types: info(String name, int age) vs info(int age, String name)

NOT valid for overloading: Different return type alone, different access modifiers alone, or different exception declarations alone — these do not constitute overloading and the compiler will raise a duplicate method error.

java — 3 ways to overload
class Demo {
    void show(int a)                   {} // 1 int
    void show(int a, int b)            {} // 2 ints — different count
    void show(double a)               {} // double — different type
    void show(String s, int n)        {} // String,int — different order
    void show(int n, String s)        {} // int,String — different order

    // ❌ NOT valid — return type alone doesn't overload
    // int  show(int a) { return a; } // compile error: duplicate method
}

No — two methods with the same name and same parameter list but different return types are NOT overloaded methods. They are duplicate methods and cause a compile error: "method is already defined in class".

The reason: when calling a method, the return type is not part of the disambiguation — the compiler looks at the method name and parameter types to select a method. If two methods have the same name and same parameters, the compiler has no way to distinguish which one you're calling. The return type you plan to use on the caller side is not visible to the resolver.

java — return type alone is not enough
class Test {
    int    getValue() { return 1; }
    double getValue() { return 1.0; } // ❌ COMPILE ERROR — duplicate
}
// Why? At the call site:  test.getValue();
// Compiler cannot determine which to call — return type is ambiguous

// ✅ Valid overloading requires parameter difference:
int    getValue(int x)    { return x; }
double getValue(double x) { return x; } // ✅ different param type

Method Overriding is when a child class provides its own implementation of a method that is already defined in its parent class. The method in the child class must have the same name, same parameter list, and same or covariant return type as the parent's method. It enables runtime polymorphism — when the method is called on a parent-type reference, the child's version runs.

Rules for overriding:

  • The method signature (name + params) must match exactly.
  • Return type must be the same or a subtype (covariant return).
  • Access modifier cannot be more restrictive than the parent's (can be same or more permissive).
  • Cannot override static, final, or private methods.
  • Use @Override annotation — it lets the compiler verify you're actually overriding and not accidentally overloading.
java — method overriding
class Vehicle {
    public void fuelUp() { System.out.println("Fill with petrol"); }
    public void move()   { System.out.println("Vehicle moving"); }
}
class ElectricCar extends Vehicle {
    @Override
    public void fuelUp() { System.out.println("Charge the battery"); } // overridden
    // move() inherited as-is from Vehicle
}
Vehicle v = new ElectricCar(); // reference=Vehicle, object=ElectricCar
v.fuelUp(); // "Charge the battery" — runtime picks ElectricCar's version
v.move();   // "Vehicle moving" — inherited

Overriding is what enables runtime polymorphism — the ability to write code that works generically with a parent type but automatically uses the correct subclass behaviour. Different method names would break this entirely.

  • Uniform interface: Code that works with Animal objects (like a loop calling a.speak()) doesn't need to know about Dog, Cat, or Parrot — it just calls speak() and each animal responds correctly. If Dog used bark() and Cat used meow(), you couldn't write generic code.
  • Extensibility without modification: Adding a new animal type (say, Lion) only requires overriding speak() in Lion. All existing code that uses Animal references automatically works with Lion — no changes needed (Open/Closed Principle).
  • Contract enforcement: Parent classes (and interfaces) define a contract — all subclasses must honour it. Overriding ensures each subclass provides its own appropriate implementation of that contract.
java — implementing overriding correctly
class Payment {
    public String process(double amount) {
        return "Processing ₹" + amount;
    }
}
class UPIPayment extends Payment {
    @Override  // ← annotation: compiler verifies this actually overrides
    public String process(double amount) { // same name + params
        return "UPI: ₹" + amount + " via riya@upi";
    }
}
class CardPayment extends Payment {
    @Override
    public String process(double amount) {
        return "Card: ₹" + amount + " via **** 4321";
    }
}
// Polymorphism — same code, different behaviour
Payment[] payments = { new UPIPayment(), new CardPayment() };
for (Payment p : payments)
    System.out.println(p.process(1000));
// UPI: ₹1000.0 via riya@upi
// Card: ₹1000.0 via **** 4321

Overriding is called runtime polymorphism or late binding because which method body to execute is determined at runtime — not at compile time. At compile time, the compiler only knows the declared type of the reference. It verifies that the method exists in that type and generates a virtual method call. The actual decision — "which class's version of this method?" — is made by the JVM at runtime, based on the actual object type stored in that reference.

java — late binding demonstrated
Animal a;  // declared type = Animal

// At compile time: compiler only knows 'a' is Animal, verifies speak() exists
// It generates a VIRTUAL call — actual target decided later

if (someCondition) a = new Dog();
else               a = new Cat();

// At RUNTIME: JVM checks actual object type, calls correct version
a.speak(); // JVM decides: Dog.speak() or Cat.speak() depending on actual object
// Compiler doesn't know which will run — that's why it's "late binding"
  • 1. Definition: Overloading — same method name, different parameters in the same class. Overriding — same method name and parameters in parent and child class.
  • 2. Binding: Overloading = compile-time / early binding (resolved by compiler). Overriding = runtime / late binding (resolved by JVM).
  • 3. Inheritance required: Overloading does NOT require inheritance — happens within one class. Overriding REQUIRES an inheritance relationship (parent-child).
  • 4. Parameters: Overloading MUST differ in parameters. Overriding MUST have the SAME parameters.
  • 5. Return type: Overloading can have any return type (return type alone doesn't distinguish). Overriding must have the same or covariant (subtype) return type.
java — overloading vs overriding at a glance
class Parent {
    void show(int x)    {}   // original
    void show(String s)  {}   // ← OVERLOADING (same class, diff param)
}
class Child extends Parent {
    @Override
    void show(int x)    {}   // ← OVERRIDING (same sig, child class)
    void show(double d)  {}   // ← OVERLOADING (different param type)
}

Annotations are metadata tags that provide information about code to the compiler, JVM, or frameworks — without directly affecting the program's logic. They begin with @ and can be applied to classes, methods, fields, parameters, and packages.

Common built-in annotations:

  • @Override — Tells the compiler this method is intended to override a parent method. The compiler verifies it — catches typos and signature mismatches that would otherwise silently become overloading instead of overriding.
  • @Deprecated — Marks a method/class as outdated. The compiler generates a warning when it's used.
  • @SuppressWarnings — Suppresses specific compiler warnings.
  • @FunctionalInterface — Marks an interface as having exactly one abstract method (for lambda compatibility).
  • Framework annotations: @SpringBootApplication, @RestController, @Autowired (Spring), @Entity, @Table (JPA/Hibernate).

OOPS — Encapsulation & Abstraction
5 Qs

Encapsulation is the OOP principle of bundling data (fields) and the methods that operate on that data together inside a class, while restricting direct access to the internal data using access modifiers. External code can only interact with an object through its defined public interface (methods) — not by directly reading or modifying its fields.

Think of it like a capsule (tablet) — the medicine is bundled inside a protective coating. You take the capsule and benefit from the medicine without directly handling the raw chemicals inside.

Why encapsulation matters: It protects the object's internal state from accidental or malicious corruption. It allows the internal implementation to change without breaking external code (as long as the public interface stays the same). It makes bugs easier to find because state can only change through known, controlled pathways.

Encapsulation in Java is achieved by combining two things: making fields private (hiding the data) and providing public getter/setter methods (controlled access).

java — full encapsulation example
class Employee {
    // Step 1: Make all fields private — hidden from outside
    private String name;
    private int    age;
    private double salary;

    // Step 2: Provide public getters — controlled READ access
    public String getName()   { return name; }
    public int    getAge()    { return age; }
    public double getSalary() { return salary; }

    // Step 3: Provide public setters with VALIDATION — controlled WRITE access
    public void setName(String name) {
        if (name == null || name.trim().isEmpty())
            throw new IllegalArgumentException("Name cannot be blank");
        this.name = name;
    }
    public void setAge(int age) {
        if (age < 18 || age > 65) throw new IllegalArgumentException("Invalid age");
        this.age = age;
    }
    public void setSalary(double salary) {
        if (salary < 0) throw new IllegalArgumentException("Negative salary?");
        this.salary = salary;
    }
}
Employee e = new Employee();
e.setName("Riya");    // ✅ goes through validation
e.setSalary(-100);   // ❌ throws IllegalArgumentException
// e.salary = -100;   // ❌ compile error — private field
  • Data protection: Private fields can only be changed through setters — invalid values (negative ages, null names) are rejected before they corrupt object state.
  • Flexibility and maintainability: Internal implementation can change without affecting code that uses the class. Rename a field, change its type, derive it from other fields — only the getter/setter changes; callers are unaffected.
  • Controlled access: You decide the level of access — read-only (getter only), write-only (setter only), or read-write. Public fields give no such control.
  • Testability: Encapsulated objects have a clear, defined public interface that's easy to test and mock in unit tests.
  • Debugging ease: Set a breakpoint in a setter to catch every modification of a field — impossible with direct field access.
  • Framework integration: Spring, Hibernate, and JavaBeans rely on getter/setter conventions to work via reflection.

Abstraction is the OOP principle of hiding complex implementation details and exposing only the essential, relevant features to the user. You show what an object does, not how it does it. Users interact with a simple interface without needing to understand the internal workings.

Real-world analogy: When you drive a car, you use the steering wheel, pedals, and gear — you don't need to understand combustion engines, transmission systems, or fuel injection. The car abstracts that complexity away.

Two ways to implement abstraction in Java:

  • Abstract Classes: Declared with abstract keyword. Cannot be instantiated. Can have abstract methods (no body — subclass must implement) and concrete methods (with body). Use when related classes share some common implementation.
  • Interfaces: All methods are implicitly abstract (Java 7) or can have default implementations (Java 8+). A class implements an interface and provides all method bodies. Use when unrelated classes need to share a contract.
java — abstraction with abstract class and interface
// Abstract class — partial abstraction
abstract class Shape {
    protected String color;
    public Shape(String color) { this.color = color; }

    abstract double area();       // ← abstract: no body, MUST override
    abstract double perimeter(); // ← abstract: no body

    void display() {               // ← concrete: has body, can be inherited
        System.out.println("Color: " + color + ", Area: " + area());
    }
}
class Circle extends Shape {
    double r;
    public Circle(String c, double r) { super(c); this.r = r; }
    double area()      { return Math.PI * r * r; }
    double perimeter() { return 2 * Math.PI * r; }
}
// Shape s = new Shape(); // ❌ Cannot instantiate abstract class
Shape s = new Circle("Red", 5); // ✅ polymorphism
s.display(); // "Color: Red, Area: 78.53..."

// Interface — full abstraction (contract only)
interface Payable {
    boolean processPayment(double amount); // abstract — no body
    default void printReceipt() { System.out.println("Receipt printed"); }
}
class UPI implements Payable {
    public boolean processPayment(double amount) {
        System.out.println("UPI: ₹" + amount); return true;
    }
}

These two concepts are often confused because they both involve "hiding" — but they hide different things and serve different purposes:

  • Abstraction — hides complexity (WHAT): Focuses on hiding the implementation details from the outside world. Shows only what an object does — not how it does it. It's about the design of the interface. Achieved using abstract classes and interfaces.
  • Encapsulation — hides data (HOW to protect): Focuses on hiding the internal data of an object from direct access. Bundles data and methods and controls access using access modifiers. It's about data protection and access control. Achieved using private fields + public getters/setters.

Simple analogy: A TV remote is abstraction — you press "Volume Up" without knowing the signal frequencies used. The TV's internal circuit is encapsulated — the electronic components are sealed inside the TV casing, protected from direct access.

An abstract class is a class declared with the abstract keyword that serves as an incomplete blueprint — it can have both abstract methods (no body; subclass must implement) and concrete methods (with body; inherited as-is). An abstract class cannot be instantiated directly — you must extend it and implement all abstract methods before creating an object.

  • Can have abstract and concrete methods, constructors, fields, and static members.
  • A subclass must implement ALL abstract methods — unless it is also abstract.
  • Promotes code reuse by putting shared logic in concrete methods while forcing subclasses to define their own versions of abstract methods.
java — abstract class
abstract class Vehicle {
    protected String brand;
    public Vehicle(String brand) { this.brand = brand; }

    abstract void fuelType();       // no body — must override
    abstract double maxSpeed();

class ElectricCar extends Vehicle {

Vehicle v = new ElectricCar("Tesla"); // ✓ polymorphism

v.fuelType(); // "Electric"

Example: Employee abstract class with shared name, id, getDetails() — subclasses FullTimeEmployee, PartTimeEmployee each implement calculateSalary() differently.
  • Enforce a contract with partial implementation: You want subclasses to implement specific methods (abstract) but also want to provide utility methods they all share (concrete).
  • Template Method pattern: Define the skeleton of an algorithm in the abstract class. Fixed steps are concrete methods; variable steps are abstract. Subclasses fill in the variable steps.
  • When you need a constructor or instance state: Interfaces can't have instance fields or constructors. Abstract classes can — use them when subclasses need shared initialised state.

An interface is a completely abstract type that defines a contract — a set of method signatures that implementing classes must provide. It specifies what a class can do, without specifying how. A class implements an interface and provides the body for all its abstract methods. Crucially, a class can implement multiple interfaces.

  • All fields in an interface are implicitly public static final (constants).
  • All methods without a body are implicitly public abstract.
  • Java 8+ allows default methods (with body) and static methods.
  • Java 9+ allows private methods (helper methods for default methods).
java — interface definition and implementation
interface Drawable {
    int VERSION = 1;          // implicitly public static final
    void draw();               // implicitly public abstract
    void resize(double factor);

    default void printInfo() { // default method — optional to override
        System.out.println("Drawable v" + VERSION);
    }
    static void getVersion() { System.out.println("v" + VERSION); }
}
interface Colorable { void setColor(String color); }

    public void draw()               { System.out.println("Drawing circle"); }

    public void setColor(String c)  { this.color = c; }
}

  • Multiple contracts for one class: A Report class can implement Printable, Saveable, and Exportable — something not possible with multiple class inheritance.
  • API design / loose coupling: Program to an interface, not an implementation. List<String> list = new ArrayList<>() — you can swap ArrayList for LinkedList without changing anything that uses the List interface.
  • Dependency injection and testability: Pass interfaces as parameters — in tests, inject mock implementations. In production, inject real ones. Spring uses interfaces everywhere for this reason.
  • Callback / event listener patterns: Runnable, Comparator, ActionListener — all interfaces used to pass behaviour as a parameter.
  • 1. Methods: Abstract class can have both abstract and concrete methods. Interface: all methods are abstract by default; can have default and static methods (Java 8+).
  • 2. Multiple inheritance: A class can extend only ONE abstract class. A class can implement MULTIPLE interfaces.
  • 3. Fields / State: Abstract class can have instance fields (non-static, non-final). Interface fields are implicitly public static final (constants only).
  • 4. Constructor: Abstract class CAN have constructors (called via super() in subclass). Interface CANNOT have constructors.
  • 5. Access modifiers: Abstract class methods can be public, protected, or package-private. Interface methods are implicitly public.
  • 6. Use relationship: Abstract class = IS-A (inheritance, partial implementation shared). Interface = CAN-DO / contract (unrelated classes share behaviour).

Default methods (introduced in Java 8) are methods inside an interface that have a concrete implementation using the default keyword. They allow interfaces to have method bodies without breaking existing classes that implement the interface. An implementing class can use the default implementation as-is or override it.

When to use:

  • Utility methods on the interface: Helper methods that use other interface methods — like Comparator.reversed() or Predicate.and().

A constructor is a special method that is automatically called when a new object is created using the new keyword. Its job is to initialise the object's state — setting field values so the object is ready to use the moment it's created.

Key characteristics:

  • Has the same name as the class — mandatory.
  • No return type — not even void. If you add a return type, it becomes a regular method, not a constructor.
  • Called automatically by the JVM when new ClassName() is used.
  • Can be overloaded — a class can have multiple constructors with different parameter lists.
  • If you define no constructor, the Java compiler automatically provides a default no-arg constructor.
java — constructor basics
class Book {
    String title;
    String author;
    double price;

    // Constructor — same name as class, no return type
    public Book(String title, String author, double price) {
        this.title  = title;
        this.author = author;
        this.price  = price;
    }
}
// Constructor is called automatically here:
Book b = new Book("Clean Code", "Robert Martin", 599.0);
System.out.println(b.title + " by " + b.author); // Clean Code by Robert Martin

Java has three types of constructors: default, no-arg, and parameterised.

  • Default Constructor: Auto-generated by the Java compiler when you define NO constructor at all in your class. It takes no arguments and assigns default values (0, null, false) to fields. As soon as you write any constructor yourself, the compiler stops generating this — you must explicitly write a no-arg constructor if you still want one.
  • No-arg Constructor: A constructor you explicitly write with no parameters. Unlike the compiler-generated default, you control what happens inside — you can initialise fields to non-default values or run setup logic.
  • Parameterised Constructor: A constructor with parameters — the most commonly written constructor. Lets callers provide specific values when creating an object.
java — types of constructors
class Example {
    int    x; String s;
    // If NO constructor written — compiler generates: Example() { }
    // x = 0, s = null (default values)
}

class Config {
    String host;
    int    port;

    // No-arg — explicit, sets sensible defaults
    public Config() { this.host = "localhost"; this.port = 8080; }

    // Parameterised
    public Config(String host, int port) { this.host = host; this.port = port; }

Config c1 = new Config();               // localhost:8080
Config c2 = new Config("prod.server", 443); // custom

When to use: When an object requires certain data to be meaningful from the start. A Student without a name and roll number isn't a valid student. A BankAccount without an owner and initial balance is incomplete. The parameterised constructor enforces these required fields at object creation.

Constructor overloading is defining multiple constructors in the same class with different parameter lists. Java distinguishes them by their signature (number and type of parameters). It gives callers flexibility — they can create objects in different ways based on what information they have available.

When to use: When creating an object should work with different amounts of information — create a User with just a name, or with name + email, or with name + email + phone. Constructor overloading avoids forcing callers to always provide all data.

Constructor chaining is calling one constructor from another constructor, forming a chain. It avoids duplicating initialisation code across multiple overloaded constructors — you write the full initialisation logic once and all other constructors delegate to it.

Two forms:

  • this(args) — Call another constructor in the SAME class. Must be the FIRST statement in the constructor body.
  • super(args) — Call the parent class constructor from a child class. Also must be the FIRST statement.
java — constructor chaining with this() and super()
class Person {
    String name; int age; String city;

    // Master constructor — full initialisation
    public Person(String name, int age, String city) {
        this.name = name; this.age = age; this.city = city;
    }
    // Chains to master — provides default city
    public Person(String name, int age) {
        this(name, age, "Unknown"); // this() must be FIRST line
    }
    // Chains to above
    public Person(String name) { this(name, 0); }
}

class Employee extends Person {
    String company;
    public Employee(String name, int age, String company) {
        super(name, age);  // calls Person(name,age) — must be FIRST
        this.company = company;
    }
}

A copy constructor is a constructor that takes an object of the same class as a parameter and creates a new object that is a copy of the given object. Java does not provide one automatically (unlike C++) — you write it yourself. It creates a deep copy — a completely independent new object with the same field values.

When to use: When you need a completely independent duplicate of an object — modifying the copy must not affect the original. Use when: creating an initial state snapshot, passing a safe copy to external code, implementing "undo" functionality, or cloning complex objects with multiple fields.

java — copy constructor

    int x, y;

    public Point(int x, int y) { this.x = x; this.y = y; } // normal

        this.x = other.x;  // copies values from other
        this.y = other.y;
    }
}

Point copy     = new Point(original); // deep copy — independent object

System.out.println(copy.x);     // 100 — only copy changed

// Contrast with reference copy (NOT a copy constructor):
Point ref = original; // SAME object — changing ref changes original!

The super keyword in Java is a reference to the parent class. It has three distinct uses:

  • 1. Call parent class constructor — super(args): Used inside a child class constructor to invoke the parent's constructor. Must be the FIRST statement.   • 2. Access parent class method — super.methodName(): When a child overrides a parent method, super.method() explicitly calls the parent's version. Useful when you want to extend parent behaviour rather than completely replace it.

Exception Handling — Basics
6 Qs

An exception is an unexpected event that occurs during program execution, disrupting the normal flow — division by zero, null pointer access, file not found, network timeout. Without handling, an exception crashes the program and shows a stack trace.

Exception Handling is a mechanism to gracefully detect, respond to, and recover from such runtime errors — keeping the program running or failing gracefully with a meaningful message instead of crashing abruptly.

Java implements exception handling with 5 keywords:

  • try: Encloses the risky code that might throw an exception.
  • catch: Catches a specific exception type and handles it. Can have multiple catch blocks.
  • finally: Optional block that ALWAYS executes — whether an exception occurred or not. Used for cleanup.
  • throw: Manually throw an exception from your code.
  • throws: Declares that a method may throw a certain exception — letting callers know to handle it.
java — try-catch-finally structure
public class Main {
    public static void main(String[] args) {
        try {
            // Risky code — might throw
            int[] arr = { 1, 2, 3 };
            System.out.println(arr[5]);   // ArrayIndexOutOfBoundsException
            int  result = 10 / 0;           // ArithmeticException
        } catch (ArrayIndexOutOfBoundsException e) {
            System.out.println("Array index error: " + e.getMessage());
        } catch (ArithmeticException e) {
            System.out.println("Math error: " + e.getMessage());
        } finally {
            System.out.println("Always runs — cleanup here");
        }
        System.out.println("Program continues normally"); // ✓ runs
    }
}

The finally block is an optional block after catch that always executes — regardless of whether an exception was thrown or caught, whether the try block completed normally, or even if a return statement was executed in try/catch. The only scenarios where finally doesn't run: System.exit() is called, or the JVM crashes.

Purpose: Resource cleanup — closing database connections, file streams, network sockets, releasing locks. These must happen whether the operation succeeded or failed — that's exactly what finally guarantees.

java — finally always executes
public static void divide(int a, int b) {
    try {
        System.out.println("Result: " + (a / b));
        return; // even with return, finally runs!
    } catch (ArithmeticException e) {
        System.out.println("Cannot divide by zero");
    } finally {
        System.out.println("finally: always executes"); // ALWAYS runs
    }
}
divide(10, 2);  // Result: 5 / finally: always executes
divide(10, 0);  // Cannot divide by zero / finally: always executes

// Classic resource cleanup pattern with finally
FileReader reader = null;
try {
    reader = new FileReader("data.txt");
    // read file...
} catch (IOException e) {
    System.out.println("File error: " + e.getMessage());
} finally {
    if (reader != null) reader.close(); // always close — even if error
}

finally should be used whenever you acquire resources that must be released regardless of success or failure:

  • Database connections: Always close the connection — unclosed connections exhaust the connection pool and cause server failures under load.
  • File streams: Always close FileReader, BufferedWriter, etc. — unclosed files cause resource leaks and lock the file.
  • Network sockets: Always close socket connections.
  • Lock release: If you acquire a lock in a concurrent program, always release it in finally — a lock never released causes deadlocks.

Modern alternative — try-with-resources (Java 7+): For classes implementing AutoCloseable, try-with-resources automatically calls close() at the end — cleaner than manual finally. Still use explicit finally for non-closeable resources or for guaranteed logic (logging, metrics).

When to use multiple catch blocks:

  • Different exceptions require different handling logic — a FileNotFoundException (file missing) needs a different response than an IOException (file corrupted/locked).
  • You want to provide specific, meaningful error messages to users rather than a generic "something went wrong."
  • Different recovery strategies — retry for network errors, skip for format errors, abort for security exceptions.

Important ordering rule: Catch more specific (child) exceptions BEFORE less specific (parent) ones. If you put Exception first, it catches everything and the specific catches below it become unreachable — the compiler will give an error.

  • When it IS acceptable: As the LAST catch block in a chain — after all specific exceptions are handled — to act as a safety net for truly unexpected errors. Always log the full exception here (don't swallow it silently). Also acceptable at the top-most application layer (main method, REST controller) to prevent unhandled crashes — log and show a generic error message.

The throw keyword is used to explicitly throw an exception from your code — at any point you determine something has gone wrong that the current method cannot handle. You write throw new ExceptionType("message") and the exception propagates up the call stack until it's caught or crashes the program.

When to use throw:

  • Validate inputs / enforce business rules: When a method receives invalid data that would lead to incorrect or dangerous behaviour — throw an appropriate exception to immediately signal the problem to the caller.
  • Signal impossible states: When your code reaches a state that should never happen (e.g., a switch default that should be unreachable).
  • Re-throw exceptions: Catch an exception, wrap it in a more meaningful custom exception, and re-throw — adding context while preserving the original cause.
  • Custom domain exceptions: Throw your own exception classes (InsufficientFundsException, UserNotFoundException) that model your business domain clearly.
java — throw keyword usage
// Validate business rule — throw when invalid
public void withdraw(double amount) {
    if (amount <= 0) {
        throw new IllegalArgumentException("Amount must be positive: " + amount);
    }
    if (amount > balance) {
        throw new IllegalStateException("Insufficient funds. Balance: " + balance);
    }
    balance -= amount;
}

// Custom domain exception

    public UserNotFoundException(int userId) {
        super("User not found with id: " + userId);
    }
}

    User user = db.findById(id);
    if (user == null) throw new UserNotFoundException(id);
    return user;
}

// Re-throw with context
try {
    conn = getConnection();
} catch (SQLException e) {
    throw new RuntimeException("Failed to connect to database", e); // wraps original
}

The throws keyword is used in a method signature to declare that the method might throw one or more checked exceptions — without handling them inside the method. It is a warning to callers: "this method can produce these exceptions; you must handle them."

Key points:

  • throws appears in the method signature — after the parameter list.
  • It does NOT throw an exception itself — it only advertises that the method could throw.
  • Required for checked exceptions (compile-time exceptions like IOException, SQLException). Not required for unchecked exceptions (RuntimeException and its subclasses).
  • When a method declares throws, its callers must either catch the exception in a try-catch or also declare throws to propagate it further up.
java — throws keyword

    BufferedReader reader = new BufferedReader(new FileReader(path));
    return reader.readLine(); // may throw IOException — not handled here
}

// Multiple exceptions declared
public void saveToDb(User user) throws SQLException, IOException {
    // ... may throw either exception
}

// Caller MUST handle or re-declare throws
public void processFile() {
    try {
        String content = readFile("data.txt"); // must handle IOException
        System.out.println(content);
    } catch (IOException e) {
        System.out.println("File error: " + e.getMessage());
    }
}

// Or propagate further
public void processFile2() throws IOException { // pass the responsibility up
    String content = readFile("data.txt");
}
  • Purpose: throw actually throws an exception — triggers it. throws declares that a method might throw — just an announcement.
  • Location: throw is used inside a method body. throws is used in the method signature (declaration).
  • Number of exceptions: throw can throw only ONE exception at a time. throws can declare MULTIPLE exceptions: throws IOException, SQLException.
  • What follows: throw is followed by an exception instance: throw new RuntimeException(). throws is followed by exception class names: throws IOException.
  • Checked vs Unchecked: throw can throw both checked and unchecked exceptions. throws is mainly used for checked exceptions (for unchecked, it's optional).

Java's exception hierarchy starts with Throwable at the root, which has two main branches:

  • Error: Serious problems that are typically unrecoverable — JVM-level issues. Examples: OutOfMemoryError, StackOverflowError, VirtualMachineError. You should generally NOT catch Errors — if the JVM runs out of memory, your catch block probably won't even run. Let the JVM handle them.
  • Exception: Problems that programs can potentially recover from. Divided into two categories:
    • Checked Exceptions (direct subclasses of Exception, excluding RuntimeException): IOException, SQLException, FileNotFoundException, ClassNotFoundException. The compiler FORCES you to handle or declare them.
    • Unchecked Exceptions (subclasses of RuntimeException): NullPointerException, ArrayIndexOutOfBoundsException, ArithmeticException, ClassCastException, NumberFormatException, IllegalArgumentException. Compiler does NOT force you to handle them.

Checked Exceptions are exceptions that the Java compiler forces you to handle at compile time — you MUST either wrap the calling code in a try-catch OR declare the method with throws. They represent predictable, recoverable problems external to the application — file not found, network unavailable, database connection failed. The compiler insists you think about recovery.

Unchecked Exceptions are subclasses of RuntimeException. The compiler does NOT require you to handle them. They typically represent programming errors — bugs in logic like accessing a null reference, out-of-bounds array access, type cast errors. They should be fixed in code, not caught and "handled."

Key differences:

  • Compiler enforcement: Checked = must handle or declare. Unchecked = optional to handle.
  • Nature: Checked = external/environmental failures. Unchecked = programming bugs.
  • Hierarchy: Checked = direct subclasses of Exception (excluding RuntimeException). Unchecked = subclasses of RuntimeException.
  • Usage: Checked for predictable recoverable failures (IOException). Unchecked for logic errors you should fix (NullPointerException).

Collections — Basics
10 Qs

A Collection in Java is a container object that groups multiple elements into a single unit. Unlike arrays — which are fixed-size, store only one type, and have no built-in methods — collections are dynamic, resizable, and come with a rich set of built-in operations for adding, removing, searching, sorting, and iterating elements.

Why collections are essential in real applications:

  • Dynamic size: Collections grow and shrink automatically as you add/remove elements. Arrays need a fixed size declared upfront.
  • Rich operations: Built-in methods for add, remove, contains, sort, iterate, search — no need to write algorithms from scratch.
  • Multiple data structures: Java provides ArrayList (dynamic array), LinkedList (chain of nodes), HashSet (unique elements), HashMap (key-value pairs), TreeSet (sorted), PriorityQueue (ordered), and more — each optimised for different use cases.
  • Generics support: Type-safe containers with compile-time checks — List<String>, Map<Integer, User>.
  • Real use: Shopping cart items, list of users from a database, set of unique tags, product-price mappings, task queues.

The Java Collections Framework is organised around a hierarchy of interfaces and their implementing classes:

  • List (ordered, allows duplicates): ArrayList — dynamic array, fast random access. LinkedList — doubly-linked list, fast insert/delete. Vector — legacy thread-safe list.
  • Set (unique elements, no duplicates): HashSet — no order, O(1) operations. LinkedHashSet — insertion order maintained. TreeSet — sorted natural order.
  • Queue (FIFO ordering): LinkedList — basic queue. PriorityQueue — elements ordered by priority. ArrayDeque — double-ended queue.
  • Map (key-value pairs, unique keys): HashMap — no order, O(1) access. LinkedHashMap — insertion order. TreeMap — sorted by key. Hashtable — legacy, thread-safe.

  • Collection<T>: Extends Iterable. The root interface of the collection hierarchy proper. A Map stores data as key-value pairs. Each key is unique — you cannot have two entries with the same key (adding a duplicate key replaces the existing value). Values may be duplicated. Map does NOT extend the Collection interface — it is a separate hierarchy.

  • LinkedHashMap: Maintains insertion order. Slightly slower than HashMap. Good for LRU caches.
  • TreeMap: Sorted by keys in natural order (or custom Comparator). O(log n) operations. Great for ordered key iteration.
  • Hashtable: Legacy, thread-safe version of HashMap. Slower due to synchronisation. Use ConcurrentHashMap instead in modern code.

  • Access: HashSet: check presence with contains(element). HashMap: retrieve value with get(key).
  • Use case: HashSet — store unique items, check membership. HashMap — store lookups, key-value relationships.

Collections — Advanced
6 Qs

A LinkedList is a data structure where elements (nodes) are stored in sequence, but — unlike an array — they are NOT in contiguous memory. Each node holds a value AND a reference (pointer) to the next node (and previous in doubly linked). Java's LinkedList class is a doubly linked list.

  • Singly Linked List: Each node has a value and a reference to the NEXT node only. You can only traverse forward. Insertion/deletion at head is O(1). No backward traversal.
  • Doubly Linked List: Each node has a value, a reference to the NEXT node, AND a reference to the PREVIOUS node. Java's LinkedList is this type. Enables traversal in both directions. Insertion/deletion at head and tail is O(1). Used to implement both List and Deque (double-ended queue).
java — linkedlist usage
LinkedList<String> ll = new LinkedList<>();

// Add at head and tail — O(1)
ll.addFirst("First");   // ["First"]
ll.addLast("Last");     // ["First", "Last"]
ll.add("Middle");       // ["First", "Last", "Middle"]
ll.addFirst("NewFirst"); // ["NewFirst", "First", "Last", "Middle"]

// Access head and tail
System.out.println(ll.getFirst()); // "NewFirst"
System.out.println(ll.getLast());  // "Middle"

// Remove from head/tail — O(1)
ll.removeFirst(); // removes "NewFirst"
ll.removeLast();  // removes "Middle"

// Use as Queue (FIFO): offer = enqueue, poll = dequeue
ll.offer("task1"); ll.offer("task2");
System.out.println(ll.poll()); // "First" — FIFO

ArrayList vs LinkedList key differences:

  • Internal structure: ArrayList = dynamic array (contiguous memory). LinkedList = doubly-linked nodes (non-contiguous).
  • Random access (get by index): ArrayList is O(1) — direct memory offset. LinkedList is O(n) — must traverse from head.
  • Insert/Delete at head or tail: LinkedList is O(1). ArrayList is O(n) — must shift all elements.
  • Insert/Delete in the middle: Both O(n) — LinkedList must find the position (O(n) traversal), ArrayList must shift elements.
  • Memory: ArrayList uses less memory — just the elements. LinkedList uses more — each node stores element + two pointers (next + prev).
  • Use case: ArrayList for random access and read-heavy operations. LinkedList for frequent add/remove at head/tail, implementing queues/stacks/deques.
java — when linkedlist wins over arraylist
// ✓ LinkedList — fast at head/tail operations (O(1))

tasks.addLast("normal task");  // O(1)
tasks.removeFirst();          // O(1)

// ✓ ArrayList — fast random access (O(1))
ArrayList<String> products = new ArrayList<>();
products.add("Laptop"); products.add("Phone");
System.out.println(products.get(0)); // O(1) — direct access

// ❌ LinkedList — slow random access (O(n))
tasks.get(50); // must traverse 50 nodes — use ArrayList for this

Multithreading — Overview
3 Qs

Understanding the difference between a process and a thread is fundamental to understanding multithreading.

  • Process: A process is an independent, self-contained program in execution — it has its own dedicated memory space, its own heap, stack, and system resources (file handles, network sockets). Processes are isolated from each other; one process cannot directly access another's memory. A Java application you launch is a process. Your browser, VS Code, and Spotify running simultaneously are separate processes. Starting a process is expensive — the OS must allocate a full memory space.
  • Thread: A thread is the smallest unit of execution within a process — a lightweight "worker" that executes instructions. Threads exist inside a process and share the same memory space (heap) and resources of their parent process. A process can have many threads running concurrently. Threads are cheap to create relative to processes because they share resources instead of duplicating them.

Key differences:

  • Memory: Each process has its own isolated memory. Threads within a process share the same heap memory.
  • Communication: Processes communicate via IPC (Inter-Process Communication — pipes, sockets). Threads communicate directly through shared memory — faster but requires synchronisation.
  • Creation cost: Creating a process is heavy. Creating a thread is lightweight.
  • Failure isolation: A crashing process doesn't directly affect other processes. A crashing thread can crash the entire process it belongs to.
concept — process vs thread
// One JVM process — multiple threads inside it
//
// ┌─────────────────────────────────┐
// │         JVM Process              │
// │  ┌──────────────────────────┐   │
// │  │    Shared Heap Memory    │   │  ← all threads share this
// │  └──────────────────────────┘   │
// │                                 │
// │  Thread-1    Thread-2  Thread-3 │  ← each has own stack
// │  (main)      (worker)  (daemon) │
// └─────────────────────────────────┘
//
// Separate processes — isolated memory
// ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
// │  Chrome     │  │  VS Code    │  │  Java App   │
// │  (process)  │  │  (process)  │  │  (process)  │
// └─────────────┘  └─────────────┘  └─────────────┘

Multithreading is a programming technique where a single program runs multiple threads concurrently — each performing its own task simultaneously (or in rapid interleaving on a single CPU core). Java has built-in support for multithreading via the Thread class, Runnable interface, and the java.util.concurrent package.

Advantages of multithreading:

  • Better CPU utilisation: While one thread is waiting (I/O, network, database), another thread can run — the CPU is never idle. A single-threaded program waiting for a file to load is wasted time; a multi-threaded program uses that wait time to do other work.
  • Improved responsiveness: In GUI applications, the UI thread stays responsive while background threads do heavy work. Without multithreading, a "Save" operation would freeze the UI until complete.
  • Parallel processing on multi-core CPUs: Modern CPUs have multiple cores. Multithreading allows true parallel execution — each core runs a different thread simultaneously, dramatically speeding up CPU-intensive tasks (image processing, ML training, bulk data crunching).
  • Resource sharing: Threads share the process's memory — sharing data between threads is faster than between processes (no serialisation or IPC overhead).
  • Real-world necessity: Web servers handle thousands of simultaneous requests — each request gets its own thread. Without multithreading, requests would queue up and users would wait.
java — multithreading conceptual example
// Single-threaded — tasks run one after another (slow)
downloadFile();   // wait 5 seconds...
processImage();   // wait 3 seconds...
sendEmail();      // wait 2 seconds...
// Total: ~10 seconds (sequential)

// Multi-threaded — tasks run concurrently (fast)
new Thread(() -> downloadFile()).start();
new Thread(() -> processImage()).start();
new Thread(() -> sendEmail()).start();
// Total: ~5 seconds (limited by slowest task)

// Thread lifecycle states:
// NEW → RUNNABLE → RUNNING → (BLOCKED/WAITING) → TERMINATED
  • Main Thread: When a Java application starts, the JVM creates one thread automatically — the main thread. This is the thread that executes the main() method. It is the parent of all other threads you create. The JVM terminates when the main thread finishes AND all non-daemon threads have finished. You can get a reference to it using Thread.currentThread() inside main().
  • Daemon Thread: A daemon thread is a background service thread — it exists to serve other threads. The key characteristic: the JVM automatically terminates daemon threads when all non-daemon (user) threads have finished, even if the daemon thread is still running. Daemon threads do not prevent the JVM from shutting down. Set a thread as daemon BEFORE starting it with thread.setDaemon(true). Examples: JVM's garbage collector runs as a daemon thread, background logging threads, heartbeat monitors.
java — main thread and daemon thread
public class Main {
    public static void main(String[] args) {

        // Main thread info
        Thread mainThread = Thread.currentThread();
        System.out.println("Thread: " + mainThread.getName()); // "main"
        System.out.println("Daemon: " + mainThread.isDaemon()); // false

        // Create a daemon thread
        Thread daemon = new Thread(() -> {
            while (true) {
                System.out.println("Daemon running...");
                try { Thread.sleep(500); } catch (InterruptedException e) { }
            }
        });
        daemon.setDaemon(true); // must set BEFORE start()
        daemon.start();
        System.out.println("Main thread ending...");
        // When main thread finishes, JVM terminates the daemon thread too
        // Even though daemon has an infinite loop — it doesn't block shutdown
    }
}
Multithreading — Implementation
5 Qs

Java provides three primary ways to create and run threads:

  • 1. Extending the Thread class: Create a subclass of Thread, override its run() method, and call start(). Simple but inflexible — since Java doesn't support multiple inheritance, your class cannot extend anything else once it extends Thread.
  • 2. Implementing the Runnable interface: Implement Runnable, define the run() method, wrap it in a Thread object and call start(). More flexible — your class can still extend another class. This is the preferred approach for most use cases.
  • 3. Implementing the Callable interface with ExecutorService: Unlike Runnable, Callable can return a result and throw checked exceptions. Used with ExecutorService and Future for advanced thread pool management. Best for production-grade applications.

Additionally, Java 8+ lambdas make Runnable implementations very concise since Runnable is a functional interface.

java — 3 ways to implement threads
// 1. Thread class
class MyThread extends Thread {
    public void run() { System.out.println("Thread class"); }
}
new MyThread().start();

// 2. Runnable interface
class MyRunnable implements Runnable {
    public void run() { System.out.println("Runnable"); }
}
new Thread(new MyRunnable()).start();

// 2b. Runnable with lambda (Java 8+)
new Thread(() -> System.out.println("Lambda Runnable")).start();

// 3. Callable + ExecutorService + Future
ExecutorService exec = Executors.newFixedThreadPool(2);
Future<Integer> future = exec.submit(() -> {
    Thread.sleep(1000);
    return 42; // Callable can return a value — Runnable cannot
});
System.out.println("Result: " + future.get()); // 42 (blocks until done)
exec.shutdown();

To implement multithreading using the Thread class: create a subclass that extends Thread, override the run() method with the task logic, and call start() (not run()) on the object to launch the thread.

Critical distinction: Call start(), not run(). Calling run() directly does NOT create a new thread — it executes the method on the current thread like a normal method call. Calling start() tells the JVM to create a new OS thread and execute run() on it concurrently.

java — thread class implementation
class DownloadTask extends Thread {
    private String fileName;

    public DownloadTask(String fileName) {
        this.fileName = fileName;
    }

    @Override
    public void run() {  // define the work this thread does
        System.out.println(Thread.currentThread().getName()
            + " downloading: " + fileName);
        try {
            Thread.sleep(2000); // simulate 2-second download
        } catch (InterruptedException e) {
            System.out.println("Download interrupted!");
        }
        System.out.println(fileName + " download complete");
    }
}

public class Main {
    public static void main(String[] args) {
        DownloadTask t1 = new DownloadTask("video.mp4");
        DownloadTask t2 = new DownloadTask("music.mp3");
        DownloadTask t3 = new DownloadTask("image.jpg");

        t1.start(); // ✓ creates new thread, runs concurrently
        t2.start(); // ✓ another new thread
        t3.start(); // ✓ another new thread
        // All 3 download simultaneously — not one-by-one

        // ❌ WRONG: t1.run() — NOT a new thread, runs on main thread!
    }
}

To implement multithreading using Runnable: implement the Runnable interface, define the run() method with the task logic, then pass the Runnable to a Thread constructor and call start(). Runnable is a functional interface (one abstract method) — Java 8+ lets you use lambda expressions instead of a full class.

java — runnable interface implementation
// Classic Runnable implementation
class PrintTask implements Runnable {
    private String message;
    private int    count;

    public PrintTask(String message, int count) {
        this.message = message; this.count = count;
    }

    @Override
    public void run() {
        for (int i = 1; i <= count; i++) {
            System.out.println(Thread.currentThread().getName()
                + ": " + message + " [" + i + "]");
        }
    }
}

public class Main {
    public static void main(String[] args) {
        // Runnable object passed to Thread constructor
        Thread t1 = new Thread(new PrintTask("Hello", 3), "Thread-A");
        Thread t2 = new Thread(new PrintTask("World", 3), "Thread-B");
        t1.start(); t2.start();

        // ✓ Lambda — most concise (Java 8+)
        Thread t3 = new Thread(() -> {
            for (int i = 0; i < 3; i++)
                System.out.println("Lambda task: " + i);
        });
        t3.start();

        // RunTask can still extend another class — Runnable doesn't block inheritance
        // class PrintTask extends SomeBaseClass implements Runnable { } ✓
    }
}
  • Inheritance: Extending Thread means your class cannot extend any other class (Java's single inheritance). Implementing Runnable leaves your class free to extend another class — preferred for flexibility.
  • Code design: Thread couples the task (what to do) with the thread mechanism (how to run it) in one class. Runnable separates the task from the thread — better separation of concerns.
  • Reusability: A Runnable object can be passed to multiple Thread instances or to an ExecutorService. A Thread object can only be started once.
  • Object sharing: A single Runnable can be shared across multiple threads (useful for shared-state tasks). Thread subclasses are one-per-thread.
  • Modern usage: Runnable works naturally with ExecutorService, thread pools, CompletableFuture, and lambda expressions. Thread class is lower-level and less compatible with modern concurrency utilities.
  • Return value: Neither Thread nor Runnable can return a result from run(). Use Callable<T> with Future<T> for tasks that return values.
java — thread class vs runnable
// ❌ Thread class — inheriting Thread blocks further extends
class Worker extends Thread {
    public void run() { System.out.println("Working"); }
    // Cannot extend Animal, Vehicle, etc. now
}

// ✓ Runnable — class still free to extend something else
class SmartWorker extends Employee implements Runnable {
    public void run() { System.out.println("Smart working"); }
}

// Runnable works with thread pools — Thread class doesn't
ExecutorService pool = Executors.newFixedThreadPool(3);
pool.submit(new SmartWorker()); // ✓ Runnable goes to thread pool
pool.submit(() -> System.out.println("Lambda task")); // ✓ lambda
pool.shutdown();

The java.lang.Thread class provides many methods for controlling and inspecting threads:

  • start() — Creates a new thread and begins executing run() on it. Cannot be called twice on the same Thread object.
  • run() — Contains the task logic. Do NOT call this directly — call start() instead.
  • sleep(long millis) — Static method. Pauses the current thread for the specified milliseconds. Throws InterruptedException. Does NOT release any locks the thread holds.
  • join() — Makes the calling thread wait for this thread to finish before continuing. Useful for synchronising threads — wait for worker threads to complete before using their results.
  • interrupt() — Signals a thread to stop what it's doing. Sets the thread's interrupted flag. The thread must check isInterrupted() or catch InterruptedException to respond.
  • isAlive() — Returns true if the thread has been started and has not yet terminated.
  • getName() / setName() — Get/set the thread's name for identification in logs.
  • getPriority() / setPriority(int) — Thread priority from 1 (MIN) to 10 (MAX), default 5 (NORM). Hints to the scheduler — not guaranteed.
  • currentThread() — Static method. Returns a reference to the currently executing thread.
  • isDaemon() / setDaemon(boolean) — Check/set daemon status. Must be set before start().
java — important thread methods in action
public class ThreadMethods {
    public static void main(String[] args) throws InterruptedException {

        Thread worker = new Thread(() -> {
            System.out.println("Worker: " + Thread.currentThread().getName());
            try {
                Thread.sleep(2000); // sleep 2 seconds
            } catch (InterruptedException e) {
                System.out.println("Worker interrupted!");
            }
            System.out.println("Worker done");
        }, "MyWorker"); // give it a name

        worker.setPriority(Thread.MAX_PRIORITY); // priority 10
        worker.setDaemon(false);  // user thread (default)

        System.out.println("Alive before start: " + worker.isAlive()); // false
        worker.start();
        System.out.println("Alive after start:  " + worker.isAlive()); // true

        worker.join(); // main waits for worker to finish
        System.out.println("Alive after join:   " + worker.isAlive()); // false
        System.out.println("Main continues after worker finished");
    }
}
Generics — Basics
5 Qs

Generics allow you to write classes, methods, and interfaces that work with any data type while maintaining compile-time type safety. Instead of hardcoding a specific type (like String or Integer), you use a type parameter (like T) as a placeholder — the actual type is supplied when the class or method is used.

The core problems generics solve:

  • Type safety at compile time: Without generics, collections stored Object — you could accidentally add a String to a list of Integers. The bug would only surface at runtime as a ClassCastException. Generics catch this at compile time.
  • Eliminate manual casting: Without generics, every element retrieved from a collection had to be cast: (String) list.get(0). Generics eliminate this boilerplate — the type is already known.
  • Code reuse without duplication: Write one generic Pair<T, U> class that works for any combination of types — instead of writing StringPair, IntPair, IntStringPair separately.
java — why generics: without vs with
// WITHOUT generics (pre-Java 5) — unsafe, verbose
List rawList = new ArrayList();
rawList.add("hello");
rawList.add(42);           // different type added — no compile error!
String s = (String) rawList.get(1); // ❌ ClassCastException at runtime

// WITH generics — type-safe, no casting needed
List<String> safeList = new ArrayList<>();
safeList.add("hello");
// safeList.add(42); // ❌ COMPILE ERROR — only String allowed
String result = safeList.get(0); // ✓ no cast needed — type known

A generic method is a method that declares its own type parameter(s) independently of the class. The type parameter is declared in angle brackets <T> BEFORE the return type. Generic methods can exist in both generic and non-generic classes — they are self-contained type definitions.

The type is inferred from the arguments at the call site — you usually don't need to specify it explicitly.

java — generic methods
public class Utils {

    // Generic method — <T> declared before return type
    public static <T> void printArray(T[] array) {
        for (T item : array) System.out.print(item + " ");
        System.out.println();
    }

    // Generic method — returns T
    public static <T> T getFirst(T[] array) {
        if (array == null || array.length == 0)
            throw new IllegalArgumentException("Empty array");
        return array[0];
    }

    // Generic method — multiple type params
    public static <K, V> void printPair(K key, V value) {
        System.out.println(key + " -> " + value);
    }
}

// Usage — type inferred from arguments
Integer[] nums    = {1, 2, 3};
String[]  words   = {"Java", "Generics"};
Utils.<Integer>printArray(nums);  // explicit — rarely needed
Utils.printArray(words);          // inferred: T = String
System.out.println(Utils.getFirst(nums));  // 1 (Integer)
Utils.printPair(101, "Riya");  // K=Integer, V=String

A generic class declares one or more type parameters in angle brackets after the class name: class ClassName<T>. The type parameter T can then be used as a type throughout the class — for fields, method parameters, and return types. When you instantiate the class, you provide the actual type: new Box<String>().

java — generic class implementation
// Generic class with one type parameter T
class Box<T> {
    private T value;  // T used as field type

    public Box(T value)   { this.value = value; }
    public T getValue()   { return value; }
    public void setValue(T value) { this.value = value; }

    @Override
    public String toString() { return "Box[" + value + "]"; }
}

// Generic class with TWO type parameters
class Pair<K, V> {
    private K key;
    private V value;

    public Pair(K key, V value) { this.key = key; this.value = value; }
    public K getKey()           { return key; }
    public V getValue()         { return value; }
    public String toString()   { return key + "=" + value; }
}

// Usage — T is replaced with actual type
Box<String>  strBox  = new Box<>("Hello");
Box<Integer> intBox  = new Box<>(42);
Box<double[]> arrBox = new Box<>(new double[]{1.1, 2.2});

System.out.println(strBox.getValue());  // "Hello"
System.out.println(intBox.getValue() * 2); // 84

Pair<Integer, String> student = new Pair<>(101, "Riya");
System.out.println(student); // 101=Riya

Pair<String, List<Integer>> complex = new Pair<>("scores", Arrays.asList(90,85));
  • Type Parameter: The placeholder declared in the class or method definition — the T in class Box<T>. It's a variable that represents a type, not a value. It can be named anything, but conventions exist: T (Type), E (Element — in collections), K (Key), V (Value), N (Number), R (Return type). Multiple: <K, V>, <T, R>.
  • Type Argument: The actual concrete type you supply when instantiating a generic class or calling a generic method — the String in new Box<String>(). Type arguments replace all occurrences of the type parameter at compile time.
  • What is T: T is simply a naming convention for a generic type parameter. It is NOT a special keyword — you could write class Box<Apple> and use Apple as the type placeholder (though that would be confusing). T stands for "Type" and is the standard convention for a single general-purpose type parameter.
java — type parameters vs type arguments
// TYPE PARAMETER — declared in the class definition
//                       ↓ T is the type parameter
class Container<T> {  // T is a placeholder here
    T data;
    public Container(T data) { this.data = data; }
}

// TYPE ARGUMENT — supplied at instantiation

Bounded type parameters restrict what types can be used as type arguments. Instead of accepting any type T, you specify an upper or lower bound — ensuring the type has certain methods available or fits within a class hierarchy.

  • Upper Bound (<T extends SomeClass>): T must be SomeClass or a subclass of it. This lets you call SomeClass's methods on T inside the generic code. Example: <T extends Number> means T can be Integer, Double, Long, etc. — you can call t.doubleValue() safely.
  • Multiple bounds (<T extends ClassA & InterfaceB>): T must extend ClassA AND implement InterfaceB. The class must come first; interfaces after. Example: <T extends Number & Comparable<T>>.
  • Wildcards with bounds:
    • <? extends Type> — upper bounded wildcard — accepts Type and its subtypes (read-only use).
    • <? super Type> — lower bounded wildcard — accepts Type and its supertypes (write-friendly).
    • <?> — unbounded wildcard — accepts any type.
java — bounded type parameters

public static <T extends Number> double sum(List<T> list) {
    double total = 0;
    for (T n : list) total += n.doubleValue(); // Number method — safe
    return total;
}
System.out.println(sum(Arrays.asList(1, 2, 3)));        // 6.0 (Integer)
System.out.println(sum(Arrays.asList(1.5, 2.5, 3.0)));  // 7.0 (Double)
// sum(Arrays.asList("a","b")); // ❌ String not a Number

// Multiple bounds
public static <T extends Number & Comparable<T>> T clamp(T val, T min, T max) {
    if (val.compareTo(min) < 0) return min;
    if (val.compareTo(max) > 0) return max;
    return val;
}

Type safety means that the compiler guarantees each variable only ever holds values of its declared type — preventing type mismatch errors from silently corrupting data or crashing the program at runtime. A type-safe program either refuses to compile incorrect type usage, or makes it impossible by construction.

How generics provide type safety:

  • Compile-time type checking: When you declare List<String>, the compiler enforces that only Strings are added. An attempt to add an Integer causes an immediate compile error — the bug is caught before the code ever runs.
  • No unsafe casts in calling code: Generic types carry type information, so retrieved values are already the correct type — no need to cast, and no risk of a wrong cast causing ClassCastException at runtime.
  • Type inference: With the diamond operator <> (Java 7+), the compiler infers the type from context, reducing verbosity without sacrificing safety.

// Bug caught at compile time — before deployment

Type casting (also called explicit type conversion) is the act of telling the compiler to treat a value as a different type — using the syntax (TargetType) expression. It is necessary when you have a reference typed as a parent class/Object but know the actual object is of a more specific type. If the cast is wrong, a ClassCastException is thrown at runtime.

How generics eliminate casting: Before generics, Java collections stored Object — the only common type. Every retrieval required a cast: (String) list.get(0). If you cast to the wrong type, the error only appeared at runtime. Generics parameterise the collection with the actual type — the compiler knows the type at compile time, so no cast is needed when retrieving elements. The cast is effectively done once internally, safely, by the compiler.

A lambda expression is a concise, anonymous function introduced in Java 8 that can be treated as a value — passed as an argument, stored in a variable, or returned from a method. It represents a block of code (behaviour) without needing a full class or method declaration. Lambdas are the foundation of Java's functional programming support.

The lambda operator is the arrow -> which separates the parameter list from the body:

(parameters) -> { body }

Syntax variants:

  • No params, single statement: () -> System.out.println("Hello")
  • One param (parens optional): x -> x * 2 or (x) -> x * 2
  • Multiple params: (a, b) -> a + b
  • Block body (multiple statements): (a, b) -> { int sum = a + b; return sum; }
java — lambda syntax variations
// No parameters

r.run(); // "Lambda running!"

// One parameter — parentheses optional

// Two parameters
BinaryOperator<Integer> adder = (a, b) -> a + b;
System.out.println(adder.apply(3, 4)); // 7

// Block body — multiple statements, explicit return
Comparator<String> byLength = (s1, s2) -> {
    int diff = s1.length() - s2.length();
    return diff != 0 ? diff : s1.compareTo(s2);
};

// Before lambdas — verbose anonymous class
Runnable old = new Runnable() {
    public void run() { System.out.println("Old way"); }
};
// After lambdas — concise
Runnable modern = () -> System.out.println("Modern way");

  • Functional Interface: An interface with exactly ONE abstract method (also called a SAM — Single Abstract Method interface). It can have any number of default or static methods, but only one abstract. Because there's only one method to implement, a lambda expression can represent it completely. Annotate with @FunctionalInterface — the compiler then verifies the constraint (error if you add a second abstract method).

Types of Classes — Inner Class & Final Class

Java supports several types of classes, each serving different design purposes:

  • Concrete Class: A regular class with full implementation of all methods. Can be instantiated directly. The most common type.
  • Abstract Class: Declared with abstract. Has abstract (unimplemented) and/or concrete methods. Cannot be instantiated — must be subclassed.
  • Final Class: Declared with final. Cannot be extended/inherited. Used for immutable classes and security. Examples: String, Integer, Math.
  • Inner Class (Non-static nested class): A class defined inside another class. Has access to the outer class's members including private ones.
  • Static Nested Class: A nested class declared static. Does not have access to the outer class's instance members. Acts like a top-level class but is logically grouped inside the outer.
  • Anonymous Class: A class without a name, defined and instantiated in one expression. Often used to implement interfaces or extend classes inline — replaced by lambdas for functional interfaces in Java 8+.
  • Enum: A special class type representing a fixed set of constants. Implicitly final. Has its own methods, fields, and constructors.
  • Record (Java 14+): A compact, immutable data class that auto-generates constructor, getters, equals, hashCode, and toString.

An inner class (non-static nested class) is a class defined inside another class body. It logically belongs to the outer class and can directly access all members of the outer class — including private fields and methods — without any special syntax. This tight coupling is useful for helper classes that are implementation details of the outer class.

Types of inner/nested classes:

  • Member inner class: Declared directly inside the outer class body (not in a method). Requires an instance of the outer class to instantiate.
  • Local inner class: Defined inside a method. Only accessible within that method.
  • Anonymous class: A local class without a name — defined and instantiated in one expression.
  • Static nested class: Inner class with static modifier — does NOT need outer instance.

A final class is a class declared with the final keyword that cannot be subclassed (extended). No class can use extends on a final class. Attempting to extend it causes a compile error.

Why make a class final:

  • Immutability: String, Integer, Double are final — if they could be subclassed, a subclass could override methods like equals() or hashCode() in unpredictable ways, breaking contracts. Finality guarantees the behaviour is exactly as documented.
  • Security: Prevents malicious subclasses from overriding critical methods. Encryption classes, authentication tokens, and similar security-sensitive classes are often final.
  • Performance: The JVM can optimise calls to final class methods more aggressively (method inlining) since it knows no override exists.
  • Design intent: Signals that the class is complete and not designed for extension.

Examples of final classes in Java: java.lang.String, java.lang.Integer, java.lang.Math, java.lang.System.

A final method is a method declared with the final keyword that cannot be overridden in any subclass. The class itself can be inherited and extended, but this specific method's implementation is locked — subclasses cannot provide their own version.

Why make a method final:

  • Lock critical algorithms: The Template Method pattern — you define the overall algorithm skeleton as final, allowing subclasses to fill in the variable steps but never change the overall flow.
  • Prevent unintended overriding: If a method's behaviour is central to the class's contract and changing it would break invariants, make it final.
  • Performance: The JVM can inline final method calls — slightly faster execution.

  • final method: Cannot be overridden in subclasses. The implementation is locked.
  • final class: Cannot be extended/subclassed. The class hierarchy stops here.

Types of Classes — Static Class & Enum

A static method belongs to the class itself, not to any individual instance (object). It can be called directly on the class name — no object needs to be created first: ClassName.methodName().

Key characteristics:

  • Can only directly access static fields and methods. Cannot access instance fields or methods (no this reference exists).
  • Called on the class, not an object: Math.sqrt(25), Arrays.sort(arr), Collections.sort(list).
  • Loaded into memory when the class is loaded — before any objects are created.
  • Can be overloaded but NOT overridden (hiding is possible but not polymorphic overriding).

Common uses: Utility/helper methods (Math, Arrays, Collections), factory methods (Integer.valueOf()), the main() entry point, constants access, and singleton pattern getInstance methods.

A static nested class is a class defined inside another class with the static modifier. Unlike a non-static inner class, it does NOT hold a reference to an instance of the outer class — it behaves like a regular top-level class but is logically namespaced inside the outer class for organisational purposes.

Key differences from non-static inner class:

  • Does NOT need an outer class instance to be instantiated — created directly: new Outer.Nested().
  • Cannot directly access the outer class's instance fields or methods (only static members).
  • No implicit reference to the outer instance — lighter weight, no memory leak risk.

When to use: When a helper class logically belongs to the outer class but doesn't need access to its instance state. Builder pattern implementations, configuration objects, and test helper classes are commonly static nested classes.

class University {
    private static String uniName = "Tech University"; // static field
    private        String location = "Mumbai";         // instance field

    static class Department {  // static nested class
        String name;
        public Department(String name) { this.name = name; }
        void display() {
            System.out.println(uniName + " - Dept: " + name); // ✓ static field
            // System.out.println(location); // ❌ instance field not accessible
        }
    }
}

// No outer instance needed — direct instantiation

dept.display(); // "Tech University - Dept: Computer Science"

The static keyword means a member belongs to the class rather than any specific instance. There is only ONE copy of a static member shared across all instances of the class.

static can be applied to:

  • static field (class variable): Shared by all instances. Changes made by one instance are visible to all. Stored in the class area of memory. Use for: counters, shared config, constants (static final).
  • static method: Called on the class. No this. Can only access static members. Use for: utilities, factory methods, main().
  • static block (static initialiser): Runs once when the class is first loaded. Used to initialise complex static fields.
  • static nested class: Nested class with no implicit outer reference. Logically grouped but independent.
  • static import: Import static members to use without class qualification: import static java.lang.Math.PI.

An Enum (Enumeration) is a special class type that represents a fixed, finite set of named constants. Enums are type-safe replacements for the old public static final int constant pattern. Each enum constant is an instance of the enum class itself.

Why enums are better than constants:

  • Type safety: A method taking a Direction enum cannot accidentally receive an invalid integer — the compiler enforces it.
  • Readability: Direction.NORTH is far clearer than 1 or 0.
  • Rich features: Enums can have fields, constructors, and methods — making constants self-descriptive.
  • switch compatibility: Enums work perfectly with switch statements and expressions.
  • Built-in values() and valueOf(): Iterate all constants, convert String to enum constant.

Real-world uses: Day of week, month, HTTP methods (GET/POST/PUT/DELETE), order status (PENDING/SHIPPED/DELIVERED/CANCELLED), user roles (ADMIN/EDITOR/VIEWER), seasons, directions, database transaction types.

  3. Identify the pattern/approach: Is it a single-pass (iterate once)? Two-pointer? Sorting first? Nested loops? Recognising the pattern is the key skill to develop.
  4. Write the logic in plain English (pseudocode): "Loop through the array, keep track of the current minimum, update if a smaller value is found." Only then translate to code.

public static int arraySum(int[] arr) {
    if (arr == null || arr.length == 0) return 0; // edge case

    int sum = 0;            // accumulator — start at 0
    for (int num : arr) {   // visit every element
        sum += num;          // add to running total
    }
    return sum;
}

Approach: Initialise a min variable to the first element (NOT to 0 or Integer.MAX_VALUE — using the first element is cleanest and correct). Loop from index 1 onward. Whenever you find an element smaller than current min, update min. Return min after the loop. O(n) time, O(1) space.

java — find smallest element
public static int findMin(int[] arr) {
    if (arr == null || arr.length == 0)
        throw new IllegalArgumentException("Array cannot be empty");

    int min = arr[0]; // start with first element as assumed minimum

    for (int i = 1; i < arr.length; i++) { // start from index 1
        if (arr[i] < min) {
            min = arr[i]; // found a new minimum — update
        }
    }
    return min;
}

Approach (single pass, O(n)): Track two variables — first (largest seen) and second (second largest seen). Initialise both to Integer.MIN_VALUE. For each element: if it's greater than first, push first down to second and update first. Else if it's greater than second (and not equal to first — handle duplicates), update second. Key edge case: must handle duplicate maximums.

java — second largest element
public static int secondLargest(int[] arr) {
    if (arr == null || arr.length < 2)
        throw new IllegalArgumentException("Need at least 2 elements");

    int first  = Integer.MIN_VALUE; // largest
    int second = Integer.MIN_VALUE; // second largest

    for (int num : arr) {
        if (num > first) {
            second = first; // old first becomes second
            first  = num;   // new first
        } else if (num > second && num != first) {
            second = num;   // new second (not equal to first)
        }
    }

    if (second == Integer.MIN_VALUE)
        throw new RuntimeException("No distinct second largest (all elements equal?)");
    return second;
}

Approach: Two arrays are equal if: they have the same length AND every corresponding element is equal. Check length first (quick fail). Then loop and compare element by element — return false the moment any mismatch is found. Return true if the entire loop completes without mismatch. Note: arr1 == arr2 only checks reference equality (same object), not content — you must compare element by element or use Arrays.equals().

// Manual implementation
public static boolean arraysAreEqual(int[] a, int[] b) {
    if (a == null && b == null) return true;
    if (a == null || b == null) return false;
    if (a.length != b.length)    return false; // different size = not equal

    for (int i = 0; i < a.length; i++) {
        if (a[i] != b[i]) return false; // mismatch found — stop immediately
    }
    return true; // all elements matched
}

Approach: An array is sorted in ascending order if every element is less than or equal to the next one. Loop from index 0 to length-2 (comparing pairs: arr[i] and arr[i+1]). If you ever find arr[i] > arr[i+1], it's NOT sorted — return false immediately. If the loop completes without finding such a pair, return true. Arrays of 0 or 1 elements are trivially sorted.

java — check if array is sorted ascending
public static boolean isSortedAscending(int[] arr) {
    if (arr == null || arr.length <= 1) return true; // 0 or 1 element = sorted

    for (int i = 0; i < arr.length - 1; i++) { // compare adjacent pairs
        if (arr[i] > arr[i + 1]) {
            return false; // found a violation — not sorted
        }
    }
    return true; // no violation found
}

Approach (simple — sort after merge): Create a new array of size a.length + b.length. Copy all elements of a, then all elements of b into the new array. Sort the combined array. Return it. O((m+n) log(m+n)).

Approach (optimal — merge sorted arrays): If both input arrays are already sorted, use the two-pointer merge technique for O(m+n) time — maintain a pointer for each array, compare pointed elements, pick the smaller one into the result, advance that pointer.

java — merge two arrays into sorted array
// Simple approach — works for any arrays (sorted or not)
public static int[] mergeSorted(int[] a, int[] b) {
    int[] merged = Arrays.copyOf(a, a.length + b.length); // copy a
    System.arraycopy(b, 0, merged, a.length, b.length);     // append b
    Arrays.sort(merged);                                    // sort combined
    return merged;
}

// Optimal — two-pointer merge (only when both arrays are already sorted)
public static int[] mergeTwoSorted(int[] a, int[] b) {
    int[] result = new int[a.length + b.length];
    int i = 0, j = 0, k = 0; // i=pointer into a, j=pointer into b, k=result index

    while (i < a.length && j < b.length) {
        if (a[i] <= b[j]) result[k++] = a[i++]; // a's element is smaller
        else               result[k++] = b[j++]; // b's element is smaller
    }
    // Copy remaining elements (one array may have leftover)
    while (i < a.length) result[k++] = a[i++];
    while (j < b.length) result[k++] = b[j++];

    return result;
}

Important note: Java arrays have fixed size — you cannot truly "remove" an element. Instead, you create a new array of size (original.length - occurrences), copy all elements except the target value, and return the new array.

Approach: First count how many times the target appears (to size the new array). Then copy all non-matching elements into the new array. Alternatively — and often simpler in interviews — collect non-matching elements and return a new array. For removing ALL occurrences, the approach below works cleanly.

java — remove specific element from array
// Remove ALL occurrences of target value
public static int[] removeElement(int[] arr, int target) {
    if (arr == null || arr.length == 0) return arr;

    // Step 1: Count remaining elements (not equal to target)
    int count = 0;
    for (int num : arr) if (num != target) count++;

    // Step 2: Create result array and fill with non-target elements
    int[] result = new int[count];
    int   idx    = 0;
    for (int num : arr) {
        if (num != target) result[idx++] = num;
    }
    return result;
}

// Remove first occurrence only
public static int[] removeFirst(int[] arr, int target) {
    int foundAt = -1;
    for (int i = 0; i < arr.length; i++) {
        if (arr[i] == target) { foundAt = i; break; }
    }
    if (foundAt == -1) return arr; // not found — return original

    int[] result = new int[arr.length - 1];
    for (int i = 0, j = 0; i < arr.length; i++) {
        if (i != foundAt) result[j++] = arr[i]; // skip the found index
    }
    return result;
}

Approach 1 (StringBuilder): Use StringBuilder's built-in reverse() method — simplest and most efficient.

java — reverse a string
// Approach 1: StringBuilder.reverse() — simplest
public static String reverseString(String str) {
    if (str == null || str.length() <= 1) return str;
    return new StringBuilder(str).reverse().toString();
}

// Approach 2: manual loop — iterate backward
public static String reverseManual(String str) {
    if (str == null || str.length() <= 1) return str;
    StringBuilder sb = new StringBuilder();
    for (int i = str.length() - 1; i >= 0; i--) {
        sb.append(str.charAt(i)); // read from end to start
    }
    return sb.toString();
}

// Approach 3: two-pointer on char array
public static String reverseTwoPointer(String str) {
    char[] chars = str.toCharArray();
    int left = 0, right = chars.length - 1;
    while (left < right) {
        char temp   = chars[left];   // swap
        chars[left]  = chars[right];
        chars[right] = temp;
        left++; right--;
    }
    return new String(chars);
}

A palindrome is a string that reads the same forwards and backwards. Examples: "racecar", "madam", "level", "A man a plan a canal Panama" (ignoring spaces and case).

Approach 1 (reverse and compare): Reverse the string and compare with the original. If equal, it's a palindrome.

Approach 2 (two-pointer — more efficient): Use two pointers — one at the start, one at the end. Compare the characters they point to. If they match, move both inward. If they don't match at any point, not a palindrome. O(n/2) comparisons — optimal.

java — palindrome check
// Approach 1: reverse and compare
public static boolean isPalindromeSimple(String str) {
    if (str == null) return false;
    String reversed = new StringBuilder(str).reverse().toString();
    return str.equals(reversed);
}

// Approach 2: two-pointer (optimal)
public static boolean isPalindrome(String str) {
    if (str == null) return false;
    int left = 0, right = str.length() - 1;
    while (left < right) {
        if (str.charAt(left) != str.charAt(right))
            return false; // mismatch — not a palindrome
        left++;   // move inward
        right--;
    }
    return true; // all pairs matched
}

// Approach 3: case-insensitive, ignoring non-alphanumeric
public static boolean isPalindromeClean(String str) {
    if (str == null) return false;
    String clean = str.toLowerCase().replaceAll("[^a-z0-9]", "");
    int left = 0, right = clean.length() - 1;
    while (left < right) {
        if (clean.charAt(left) != clean.charAt(right)) return false;
        left++; right--;
    }
    return true;
}

Two strings are anagrams if they contain the same characters in the same quantities, just in a different order. Examples: "listen"/"silent", "race"/"care", "triangle"/"integral".

Approach 1 (sort and compare): Sort both strings' character arrays and compare. If equal, they're anagrams. O(n log n).

Approach 2 (frequency count — optimal): Count the frequency of each character in the first string (increment). Then process the second string (decrement). If all counts are zero at the end, they're anagrams. O(n) time, O(1) space (only 26 letter slots).

java — check if two strings are anagrams
// Approach 1: sort both, compare — simple and clear
public static boolean isAnagramSort(String s1, String s2) {
    if (s1 == null || s2 == null) return false;
    if (s1.length() != s2.length())   return false; // quick fail

    char[] arr1 = s1.toLowerCase().toCharArray();
    char[] arr2 = s2.toLowerCase().toCharArray();
    Arrays.sort(arr1);
    Arrays.sort(arr2);
    return Arrays.equals(arr1, arr2);
}

// Approach 2: frequency count — optimal O(n)
public static boolean isAnagram(String s1, String s2) {
    if (s1 == null || s2 == null) return false;
    if (s1.length() != s2.length())   return false;

    int[] freq = new int[26]; // one slot per letter a-z

    for (int i = 0; i < s1.length(); i++) {
        freq[Character.toLowerCase(s1.charAt(i)) - 'a']++;  // count up
        freq[Character.toLowerCase(s2.charAt(i)) - 'a']--;  // count down
    }

    for (int count : freq) {
        if (count != 0) return false; // mismatch in frequency
    }
    return true;
}

The factorial of n (written n!) is the product of all positive integers from 1 to n. 0! = 1 by definition. 5! = 5 × 4 × 3 × 2 × 1 = 120.

Two approaches: iterative (loop — preferred, no stack overflow risk) and recursive (elegant but risky for large n).

java — factorial: iterative and recursive
// Iterative — preferred (no stack overflow risk)
public static long factorial(int n) {
    if (n < 0)  throw new IllegalArgumentException("n cannot be negative");
    long result = 1;
    for (int i = 2; i <= n; i++) {
        result *= i; // multiply running product
    }
    return result; // 0! and 1! both return 1 correctly
}

// Recursive — elegant but limited by call stack
public static long factorialRecursive(int n) {
    if (n < 0) throw new IllegalArgumentException("n cannot be negative");
    if (n == 0 || n == 1) return 1;   // base case
    return n * factorialRecursive(n - 1); // recursive call
}

Both ++i (pre-increment) and i++ (post-increment) increase the variable by 1. The difference is when the incremented value is used in an expression.

  • ++i (Pre-increment): Increments FIRST, then uses the new value in the expression.
  • i++ (Post-increment): Uses the current value in the expression FIRST, then increments.

When used alone on its own line (i++; or ++i;), there is NO difference — both simply add 1 to i. The difference only matters inside expressions.

System.out.println("x=" + x + ", y=" + y); // x=6, y=5

int q = ++p;  // p becomes 6 FIRST, then q gets 6
System.out.println("p=" + p + ", q=" + q); // p=6, q=6

// Common use — in for loops (no difference here)

for (int i = 0; i < 5; ++i) { } // ++i — identical in for loop

A prime number is a number greater than 1 that has no divisors other than 1 and itself. 2, 3, 5, 7, 11, 13 are prime. 1 is NOT prime by definition.

Optimised approach: Check divisors only up to √n. If n has a factor larger than √n, its corresponding factor must be less than √n — so we'd have already found it. This reduces checks from O(n) to O(√n).

java — prime number check
// Optimised: check up to sqrt(n) — O(√n)
public static boolean isPrime(int n) {
    if (n <= 1)  return false; // 0, 1, negatives are NOT prime
    if (n <= 3)  return true;  // 2 and 3 are prime
    if (n % 2 == 0 || n % 3 == 0) return false; // even or divisible by 3

    // Check from 5 upward, step by 6 (all primes >3 are 6k±1)
    for (int i = 5; i * i <= n; i += 6) {
        if (n % i == 0 || n % (i + 2) == 0) return false;
    }
    return true;
}

    if (n <= 1) return false;
    for (int i = 2; i * i <= n; i++) { // check up to √n
        if (n % i == 0) return false; // found a divisor
    }
    return true;
}

The GCD (Greatest Common Divisor) of two numbers is the largest number that divides both without a remainder. GCD(12, 8) = 4. The most efficient algorithm is Euclidean Algorithm: GCD(a, b) = GCD(b, a % b), stopping when b becomes 0.

java — GCD using Euclidean algorithm
// Euclidean Algorithm — iterative (most efficient)
public static int gcd(int a, int b) {
    a = Math.abs(a); b = Math.abs(b); // handle negatives
    while (b != 0) {
        int temp = b;
        b = a % b; // remainder
        a = temp;  // old b becomes new a
    }
    return a; // when b=0, a holds the GCD
}

// Recursive version
public static int gcdRecursive(int a, int b) {
    return b == 0 ? Math.abs(a) : gcdRecursive(b, a % b);
}

// LCM using GCD: LCM(a,b) = (a * b) / GCD(a,b)
public static int lcm(int a, int b) {
    return Math.abs(a) / gcd(a, b) * Math.abs(b); // divide first to avoid overflow
}

Bubble Sort repeatedly steps through the array, compares adjacent elements, and swaps them if they are in the wrong order. After each full pass, the largest unsorted element "bubbles up" to its correct position at the end. Repeat for n-1 passes.

Time Complexity: O(n²) worst/average case. O(n) best case with the early-exit optimisation (already sorted). Space: O(1) — sorts in-place. Not efficient for large datasets — use Arrays.sort() in production — but important to understand as a foundational algorithm.

java — bubble sort
public static void bubbleSort(int[] arr) {
    int n = arr.length;
    for (int pass = 0; pass < n - 1; pass++) {      // n-1 passes needed
        boolean swapped = false;                      // optimisation flag

        for (int j = 0; j < n - 1 - pass; j++) {  // last 'pass' elements are sorted
            if (arr[j] > arr[j + 1]) {             // wrong order — swap
                int temp  = arr[j];
                arr[j]    = arr[j + 1];
                arr[j + 1] = temp;
                swapped = true;
            }
        }
        if (!swapped) break; // no swaps in this pass → already sorted → done
    }
}

Binary Search finds an element in a sorted array in O(log n) time by repeatedly halving the search space. Compare the target with the middle element — if equal, found. If target is smaller, search the left half. If larger, search the right half.

Prerequisite: The array MUST be sorted first. If unsorted, either sort first (then binary search), or use linear search O(n).

java — binary search (iterative and recursive)
// Iterative binary search — preferred
public static int binarySearch(int[] arr, int target) {
    int left = 0, right = arr.length - 1;

    while (left <= right) {
        int mid = left + (right - left) / 2; // safe mid (avoids int overflow vs (l+r)/2)

        if      (arr[mid] == target) return mid;      // found!
        else if (arr[mid] <  target) left  = mid + 1; // target in right half
        else                         right = mid - 1; // target in left half
    }
    return -1; // not found
}

// Recursive binary search
public static int binarySearchRecursive(int[] arr, int target, int left, int right) {
    if (left > right) return -1;
    int mid = left + (right - left) / 2;
    if      (arr[mid] == target) return mid;
    else if (arr[mid] <  target) return binarySearchRecursive(arr, target, mid+1, right);
    else                         return binarySearchRecursive(arr, target, left, mid-1);
}

int[] sorted = {2, 5, 8, 12, 16, 23, 38, 56, 72, 91};
System.out.println(binarySearch(sorted, 23)); // 5 (index)
System.out.println(binarySearch(sorted, 10)); // -1 (not found)

// Built-in: Arrays.binarySearch() — returns index or negative if not found
System.out.println(java.util.Arrays.binarySearch(sorted, 23)); // 5
