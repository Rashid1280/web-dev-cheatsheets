Q.01 What is JavaScript? What is the role of a JavaScript engine? ⭐ Key Concept ▼

JavaScript is a lightweight, interpreted, high-level programming language primarily used to make web pages interactive. It runs in the browser (client-side) and also on the server (via Node.js). It is single-threaded and supports event-driven, functional, and object-oriented programming styles.

Originally built only for browsers, JavaScript today is one of the most versatile languages — you can build frontends, backends, mobile apps, and even desktop applications with it.

The JavaScript Engine is the program that reads your JavaScript code and executes it. Browsers each have their own engine:

  • V8 — used in Chrome and Node.js
  • SpiderMonkey — used in Firefox
  • JavaScriptCore — used in Safari

The engine does several things under the hood: it parses your code into an Abstract Syntax Tree (AST), compiles it (modern engines use JIT — Just In Time compilation for speed), and then executes it. It also manages memory allocation and garbage collection.

javascript 
// This simple line is parsed, compiled and executed by the JS engine
console.log("Hello, JavaScript!");
Q.02 What are client-side and server-side? ▼

Client-side refers to anything that runs on the user's browser (the client). When you visit a website, the browser downloads HTML, CSS, and JavaScript files and executes them locally on your device. This means actions like form validation, animations, or DOM manipulation happen instantly without contacting the server.

Server-side refers to code that runs on a remote server. When you submit a login form, the server processes your credentials, talks to a database, and sends back a response. The user never sees this code.

  • Client-side JS example: Changing a button color on click — no server needed.
  • Server-side JS example (Node.js): Fetching a user's profile from a database and returning it as JSON.
javascript 
// Client-side: runs in browser
document.getElementById("btn").style.color = "red";

// Server-side (Node.js): runs on server
const http = require("http");
http.createServer((req, res) => {
  res.end("Hello from server!");
}).listen(3000);
Q.03 What are variables? What is the difference between var, let, and const? ⭐ Key Concept ▼

A variable is a named container that stores a value in memory. You can think of it as a labelled box — you put something in it and refer to it by name later.

JavaScript has three ways to declare variables — var, let, and const — and they differ in scope, hoisting, and mutability:

  • var — Function-scoped (not block-scoped). Gets hoisted to the top of its function. Can be re-declared and updated. Avoid in modern code — it leads to bugs.
  • let — Block-scoped (lives only inside {}). Not re-declarable, but can be updated. Preferred for values that change.
  • const — Block-scoped. Cannot be reassigned after declaration. Use for values that should not change. Note: objects/arrays declared with const can still have their contents mutated.
javascript 
// var — function scoped, hoisted
var name = "Arun";
var name = "Raj"; // ✅ re-declaration allowed (bad practice)

// let — block scoped
let age = 25;
age = 26;        // ✅ update allowed

// const — block scoped, no reassignment
const PI = 3.14;
// PI = 3.15;   ❌ TypeError

// const with object — contents CAN change
const user = { name: "Arun" };
user.name = "Raj"; // ✅ allowed
Q.04 What are some important string operations in JS? ▼

Strings in JavaScript are sequences of characters. They are immutable — you can't change individual characters in place, but you can create new strings using various built-in methods.

javascript 
const str = "Hello, World!";

str.length           // 13 — number of characters
str.toUpperCase()    // "HELLO, WORLD!"
str.toLowerCase()    // "hello, world!"
str.trim()           // removes whitespace from both ends
str.includes("World") // true — checks if substring exists
str.startsWith("He") // true
str.endsWith("!")    // true
str.indexOf("o")     // 4 — first occurrence index
str.slice(0, 5)      // "Hello" — extract part of string
str.replace("World", "JS") // "Hello, JS!"
str.split(", ")     // ["Hello", "World!"] — split into array
str.charAt(0)        // "H" — character at index
str.repeat(2)        // "Hello, World!Hello, World!"

// Template literals — modern way to build strings
const name = "Arun";
const greeting = `Hello, ${name}!`; // "Hello, Arun!"
Q.05 What is the DOM? What is the difference between HTML and DOM? ⭐ Key Concept ▼

The DOM (Document Object Model) is a programming interface that the browser creates when it loads an HTML page. It represents the page as a tree of objects (nodes), where each HTML element becomes a node that JavaScript can read and manipulate.

HTML vs DOM:

  • HTML is a static text file — the raw source code you write. It's just markup.
  • DOM is the browser's live, in-memory representation of that HTML. It can be changed dynamically by JavaScript without reloading the page.

For example: if JavaScript adds a new <div> to the page, the DOM changes — but if you "View Source", you still see the original HTML unchanged. The DOM is what the browser is actually rendering.

javascript 
// The browser parses this HTML into a DOM tree:
// <body> → <h1> → "Hello"

// JS can then access and modify those DOM nodes:
const heading = document.querySelector("h1");
heading.textContent = "Hello, DOM!"; // live update, HTML file unchanged
Q.06 What are selectors in JS? ▼

Selectors are methods used to find and grab HTML elements from the DOM so you can read or manipulate them with JavaScript. They work similarly to CSS selectors.

The two most modern and flexible selectors are querySelector and querySelectorAll, which accept any valid CSS selector string. The older methods target specific attributes like ID, class, or tag name.

javascript 
// Modern selectors (preferred)
document.querySelector(".btn")       // first element matching CSS selector
document.querySelectorAll(".card")   // NodeList of ALL matching elements

// Classic selectors
document.getElementById("header")           // by ID (returns single element)
document.getElementsByClassName("menu")    // by class (HTMLCollection)
document.getElementsByTagName("p")         // by tag (HTMLCollection)

// Chaining selectors
document.querySelector(".card h2")  // h2 inside .card
Q.07 What is the difference between getElementById, getElementsByClassName and getElementsByTagName? ▼

All three are DOM selector methods but differ in what they target and what they return:

  • getElementById("id") — Targets a single element by its unique id attribute. Since IDs are unique, it always returns one element (or null). Fastest selector.
  • getElementsByClassName("class") — Targets all elements with a given class name. Returns a live HTMLCollection (updates automatically if DOM changes). Can return multiple elements.
  • getElementsByTagName("tag") — Targets all elements of a given HTML tag (e.g., all <p> tags). Also returns a live HTMLCollection.
javascript 
// Returns ONE element (or null)
const header = document.getElementById("main-header");
header.style.color = "red";

// Returns HTMLCollection of ALL elements with class "card"
const cards = document.getElementsByClassName("card");
Array.from(cards).forEach(card => card.style.border = "1px solid red");

// Returns HTMLCollection of ALL <p> tags
const paras = document.getElementsByTagName("p");
console.log(paras.length);

Key difference: getElementById returns a single Element. The other two return an HTMLCollection (array-like, but not an actual array — you need to convert it with Array.from() to use array methods).

Primitive types (stored by value, immutable):

Q.09 What are operators? What are the types of operators in JS? ▼

Operators are symbols that perform operations on values (operands). JavaScript has several types:

  • Arithmetic: + - * / % ** — math operations
  • Assignment: = += -= *= /= — assign values
  • Comparison: == === != !== > < >= <= — compare values, return boolean
  • Logical: && || ! — AND, OR, NOT
  • Ternary: condition ? valueIfTrue : valueIfFalse
  • typeof: returns the type of a value
  • Nullish Coalescing ??: returns right side if left is null/undefined
  • Optional Chaining ?.: safely access nested properties
javascript 
// == vs === (important!)
5 == "5"    // true  (loose equality, type coercion)
5 === "5"   // false (strict equality, no coercion) ← prefer this

// Ternary
const status = age >= 18 ? "adult" : "minor";

// Nullish coalescing
const name = user.name ?? "Guest"; // "Guest" if name is null/undefined

// Optional chaining
const city = user?.address?.city; // won't throw if address is undefined
Q.10 What are the types of conditional statements in JS? ▼

Conditional statements let your code make decisions and run different blocks based on whether a condition is true or false. JS has four main types:

  • if / else if / else — the most common. Checks conditions top to bottom.
  • switch — cleaner alternative when comparing one value against many possible cases.
  • Ternary operator — a one-line shorthand for simple if/else.
  • Nullish / logical short-circuit — ?? and || used as inline conditionals.
javascript 
// if / else if / else
if (score >= 90) {
  console.log("A grade");
} else if (score >= 70) {
  console.log("B grade");
} else {
  console.log("Try again");
}

// switch
switch (day) {
  case "Mon": console.log("Start of week"); break;
  case "Fri": console.log("Weekend!"); break;
  default: console.log("Midweek");
}

// Ternary
const msg = isLoggedIn ? "Welcome" : "Please login";
Q.11 What is a loop? What are the types of loops in JS? ▼

A loop is a control structure that repeats a block of code multiple times — either a fixed number of times or until a condition is met. Loops save you from writing repetitive code.

  • for — best when you know the number of iterations
  • while — runs while a condition is true (check before each iteration)
  • do...while — runs at least once, then checks the condition
  • for...of — iterates over values of an iterable (array, string)
  • for...in — iterates over keys of an object
  • forEach — array method, runs a callback for each element
javascript 
// for loop
for (let i = 0; i < 5; i++) { console.log(i); }

// while loop
let count = 0;
while (count < 3) { console.log(count); count++; }

// for...of (arrays)
const fruits = ["apple", "banana", "cherry"];
for (const fruit of fruits) { console.log(fruit); }

// for...in (objects)
const user = { name: "Arun", age: 25 };
for (const key in user) { console.log(key, user[key]); }

// forEach
fruits.forEach(fruit => console.log(fruit));
Q.12 What are Functions in JS? What are the types of functions? ⭐ Key Concept ▼

A function is a reusable block of code that performs a specific task. You define it once and call it whenever you need it. Functions can accept inputs (parameters) and return outputs.

  • Function Declaration — classic way, gets hoisted so it can be called before it's defined.
  • Function Expression — assigned to a variable, NOT hoisted.
  • Arrow Function — concise ES6 syntax (covered in next Q).
  • IIFE (Immediately Invoked Function Expression) — runs immediately after being defined.
  • Callback Function — a function passed as an argument to another function.
  • Higher-Order Function — a function that takes or returns another function.
javascript 
// Function Declaration (hoisted)
function greet(name) {
  return `Hello, ${name}!`;
}

// Function Expression (not hoisted)
const greet = function(name) {
  return `Hello, ${name}!`;
};

// IIFE — runs immediately
(function() {
  console.log("I run right away!");
})();

// Callback
function runCallback(cb) { cb(); }
runCallback(() => console.log("Callback ran!"));
Q.13 What are Arrow Functions in JS? What is their use? ⭐ Key Concept ▼

Arrow functions (introduced in ES6) are a shorter syntax for writing functions. Beyond being concise, they have one critical behavioral difference: they do not have their own this — they inherit this from their surrounding (lexical) scope.

This makes them ideal for callbacks and array methods, where you don't want this to be rebound. However, they are NOT suitable as object methods or constructors because of this this behaviour.

javascript 
// Regular function
const add = function(a, b) { return a + b; };

// Arrow function — same thing, shorter
const add = (a, b) => a + b;

// Single param — no parentheses needed
const double = n => n * 2;

// No params — empty parentheses required
const greet = () => console.log("Hello!");

// Arrow functions in array methods (very common)
const nums = [1, 2, 3];
const doubled = nums.map(n => n * 2); // [2, 4, 6]

// 'this' difference — arrow inherits outer 'this'
function Timer() {
  this.seconds = 0;
  setInterval(() => {
    this.seconds++; // ✅ 'this' refers to Timer instance
  }, 1000);
}
Q.14 What are Arrays in JS? How to get, add & remove elements from arrays? ⭐ Key Concept ▼

An Array is an ordered list of values stored in a single variable. Arrays in JS are zero-indexed (first element is at index 0) and can hold any mix of data types.

javascript 
const fruits = ["apple", "banana", "cherry"];

// ── GET ──────────────────────────────────────
fruits[0]           // "apple" (by index)
fruits.at(-1)        // "cherry" (last element, ES2022)
fruits.length        // 3
fruits.indexOf("banana") // 1
fruits.includes("apple") // true
fruits.slice(1, 3)   // ["banana", "cherry"] (no mutation)

// ── ADD ──────────────────────────────────────
fruits.push("mango")     // add to END → ["apple","banana","cherry","mango"]
fruits.unshift("kiwi")  // add to START → ["kiwi","apple",...]
fruits.splice(1, 0, "grape") // insert at index 1

// ── REMOVE ───────────────────────────────────
fruits.pop()           // remove last element
fruits.shift()         // remove first element
fruits.splice(1, 1)    // remove 1 element at index 1
fruits.filter(f => f !== "banana") // remove by value (returns new array)
Q.15 What are Objects in JS? ⭐ Key Concept ▼

An Object is a collection of key-value pairs where keys (also called properties) are strings (or Symbols) and values can be anything — strings, numbers, arrays, functions, or even other objects. Objects are the core building block of JavaScript and almost everything in JS is an object (or behaves like one).

javascript 
// Creating an object
const user = {
  name: "Arun",
  age: 25,
  isAdmin: false,
  address: { city: "Mumbai", zip: "400001" },
  greet() { return `Hi, I'm ${this.name}`; } // method
};

// Access properties
user.name           // "Arun" (dot notation)
user["age"]          // 25 (bracket notation — useful for dynamic keys)
user.address.city   // "Mumbai" (nested)

// Add / update / delete
user.email = "a@b.com";   // add new key
user.age = 26;            // update
delete user.isAdmin;       // remove key

// Useful Object methods
Object.keys(user)    // ["name", "age", ...]
Object.values(user)  // ["Arun", 25, ...]
Object.entries(user) // [["name","Arun"], ["age",25], ...]

// Destructuring (very common in interviews)
const { name, age } = user;  // name="Arun", age=25

// Spread to clone
const clone = { ...user };
Q.16 What is Scope in JavaScript? ⭐ Key Concept ▼

Scope determines where a variable is accessible in your code. If a variable is "in scope", you can read and use it. If it's "out of scope", accessing it throws an error.

  • Global Scope — declared outside any function or block. Accessible everywhere in the program.
  • Function Scope — declared inside a function with var, let, or const. Only accessible inside that function.
  • Block Scope — declared inside {} (if, for, while) with let or const. Only accessible inside that block. var ignores block scope.
  • Lexical Scope — inner functions can access variables from their outer (parent) functions. This is the foundation of closures.
javascript 
let globalVar = "I'm global"; // accessible anywhere

function outer() {
  let outerVar = "I'm in outer";

  function inner() {
    let innerVar = "I'm in inner";
    console.log(outerVar);  // ✅ lexical scope — can see parent
    console.log(globalVar); // ✅ can see global
  }
  // console.log(innerVar); ❌ ReferenceError — out of scope
}

// Block scope with let vs var
if (true) {
  let blockLet = "block";
  var blockVar = "function";
}
// console.log(blockLet); ❌ ReferenceError
console.log(blockVar);   // ✅ var ignores block scope
Q.17 What is Hoisting in JavaScript? ⭐ Key Concept ▼

Hoisting is JavaScript's behaviour of moving declarations to the top of their scope before the code executes. It's not physical movement — the JS engine processes declarations first during the compilation phase, before actually running the code.

What gets hoisted:

  • Function declarations — fully hoisted. You can call them before they appear in code.
  • var declarations — hoisted but initialized as undefined. The declaration moves up, not the value.
  • let and const — hoisted but NOT initialized. They exist in a "Temporal Dead Zone (TDZ)" — accessing them before declaration throws a ReferenceError.
javascript 
// Function declaration — hoisted fully ✅
greet(); // "Hello!" — works fine
function greet() { console.log("Hello!"); }

// var — hoisted as undefined
console.log(x); // undefined (not an error, but not the value either)
var x = 5;
console.log(x); // 5

// let/const — TDZ, throws error
console.log(y); // ❌ ReferenceError: Cannot access 'y' before initialization
let y = 10;
Q.18 What is Error Handling in JS? ▼

Error handling is the process of gracefully managing runtime errors so your application doesn't crash entirely. JavaScript provides the try...catch...finally block for this purpose.

  • try — wrap code that might throw an error here.
  • catch — runs if an error is thrown. Receives the error object.
  • finally — runs no matter what — whether an error occurred or not. Used for cleanup tasks.
  • throw — manually trigger a custom error.
javascript 
// Basic try/catch
try {
  const result = JSON.parse("invalid json"); // throws SyntaxError
} catch (err) {
  console.log(err.name);    // "SyntaxError"
  console.log(err.message); // details of the error
} finally {
  console.log("This always runs"); // cleanup, close DB connections, etc.
}

// Custom error with throw
function divide(a, b) {
  if (b === 0) throw new Error("Cannot divide by zero");
  return a / b;
}

try {
  divide(10, 0);
} catch (err) {
  console.log(err.message); // "Cannot divide by zero"
}
Q.19 What is JSON? ▼

JSON (JavaScript Object Notation) is a lightweight, text-based data format used to store and transmit structured data. It's the standard format for APIs — when a server sends data to your frontend, it almost always sends it as JSON.

Despite its name, JSON is language-independent and is used in Python, Java, and virtually every other language. JSON looks like a JavaScript object but has stricter rules: all keys must be in double quotes, and values can only be strings, numbers, booleans, arrays, objects, or null (no functions, no undefined).

javascript 
// JSON string (what comes from an API)
const jsonString = '{"name":"Arun","age":25,"skills":["JS","React"]}';

// Parse JSON string → JS object (so you can use it)
const obj = JSON.parse(jsonString);
console.log(obj.name); // "Arun"

// Stringify JS object → JSON string (to send to API)
const user = { name: "Arun", age: 25 };
const json = JSON.stringify(user);
// '{"name":"Arun","age":25}'

// Pretty print (readable format)
console.log(JSON.stringify(user, null, 2));
Q.20 What is asynchronous programming in JS? What is its use? ⭐ Key Concept ▼

JavaScript is single-threaded — it can only do one thing at a time. Asynchronous programming allows JavaScript to start a long-running task (like fetching data from an API or reading a file) and move on to other tasks instead of waiting and blocking everything.

Without async programming, a slow network request would freeze your entire UI. With it, JS delegates the task, continues running other code, and comes back to handle the result when it's ready.

Three main ways to handle async in JS:

  • Callbacks — old way; pass a function to run when the task completes. Leads to "callback hell" when nested.
  • Promises — cleaner; represents a future value. Has .then() and .catch() handlers.
  • async/await — modern, most readable way; built on top of Promises but looks like synchronous code.
javascript 
// Callbacks (old way)
setTimeout(() => console.log("Done after 1s"), 1000);

// Promises
fetch("https://api.example.com/user")
  .then(res => res.json())
  .then(data => console.log(data))
  .catch(err => console.log(err));

// async/await (modern, preferred)
async function getUser() {
  try {
    const res = await fetch("https://api.example.com/user");
    const data = await res.json();
    console.log(data);
  } catch (err) {
    console.log("Error:", err);
  }
}

Q.01 What are variables? What is the difference between var, let, and const? ⭐ Key Concept ▼

A variable is a named container used to store data in memory. You give it a name, assign a value, and refer to that name throughout your code. In JavaScript, you declare variables using one of three keywords: var, let, or const — and they each behave differently in terms of scope, hoisting, and reassignment.

  • var — The old way. Function-scoped (not block-scoped), meaning it leaks out of if and for blocks. Gets hoisted to the top of its function as undefined. Can be re-declared and updated. Avoid in modern code — it causes unpredictable bugs.
  • let — Block-scoped. Stays within the {} it was declared in. Cannot be re-declared in the same scope but can be updated. Use this for values that change.
  • const — Block-scoped. Must be assigned a value when declared. Cannot be reassigned. However, if the value is an object or array, its contents (properties/elements) can still be mutated — only the reference is locked.
javascript 
// var — function scoped, hoisted
var city = "Mumbai";
var city = "Delhi";  // ✅ re-declaration allowed (bad practice)

// let — block scoped, reassignable
let score = 10;
score = 20;          // ✅ update fine
// let score = 30;   ❌ SyntaxError: already declared

// const — block scoped, no reassignment
const PI = 3.14159;
// PI = 3.2;         ❌ TypeError: Assignment to constant variable

// const with object — reference locked, contents mutable
const user = { name: "Arun" };
user.name = "Raj";   // ✅ mutating property is allowed
// user = {};        ❌ reassigning the variable is not

// var leaks out of blocks — classic gotcha
if (true) {
  var leaked = "I escape the block!";
  let safe    = "I stay inside";
}
console.log(leaked); // ✅ works (bad)
// console.log(safe); ❌ ReferenceError
Q.02 What are data types in JS? ⭐ Key Concept ▼

A data type tells JavaScript what kind of value a variable holds and what operations are valid on it. JS is dynamically typed — you don't declare types explicitly; the engine figures it out at runtime. There are 8 data types in total, split into two categories:

Primitive Types (7) — stored directly by value, immutable:

  • String — text, wrapped in quotes: "hello" or 'world' or `template`
  • Number — all numeric values (integers AND decimals): 42, 3.14, -7, Infinity, NaN
  • Boolean — only two values: true or false
  • Undefined — a variable declared but not assigned a value yet
  • Null — intentionally empty / no value (assigned deliberately)
  • Symbol — a guaranteed unique identifier, used for object keys (ES6+)
  • BigInt — for integers larger than Number.MAX_SAFE_INTEGER: 9007199254740991n

Non-Primitive (Reference) Type (1):

  • Object — a collection of key-value pairs. This umbrella type includes plain objects {}, Arrays [], Functions, Dates, Maps, Sets, etc.
javascript 
let str   = "hello";           // String
let num   = 42;               // Number
let bool  = true;             // Boolean
let undef;                    // Undefined (no assignment)
let empty = null;             // Null
let sym   = Symbol("id");     // Symbol
let big   = 9999999999999999n; // BigInt
let obj   = { name: "Arun" }; // Object
let arr   = [1, 2, 3];        // Array (type of Object)
let fn    = () => {};         // Function (type of Object)

// JS is dynamically typed — same variable can change type
let x = 5;       // Number
x = "five";     // Now a String — perfectly valid in JS
Q.03 What is the difference between primitive and non-primitive data types? ⭐ Key Concept ▼

The most fundamental difference lies in how they are stored and copied in memory:

Primitive types are stored by value. When you copy a primitive into another variable, you get a completely independent copy. Changing one does not affect the other.

Non-primitive (reference) types are stored by reference. When you assign an object or array to a variable, the variable holds a memory address pointing to where the data lives — not the actual data. Copying that variable just copies the pointer. Both variables then point to the same object in memory, so changes through one variable affect the other.

javascript 
// PRIMITIVE — copied by value
let a = 10;
let b = a;   // b gets its own independent copy
b = 20;
console.log(a); // 10 — unchanged ✅
console.log(b); // 20

// NON-PRIMITIVE — copied by reference
let obj1 = { name: "Arun" };
let obj2 = obj1; // obj2 points to the SAME object
obj2.name = "Raj";
console.log(obj1.name); // "Raj" — obj1 was also changed! ⚠️

// To truly clone an object (avoid shared reference):
let obj3 = { ...obj1 };       // shallow clone via spread
let obj4 = JSON.parse(JSON.stringify(obj1)); // deep clone

// Comparing — primitives compare by value, objects by reference
console.log(5 === 5);                        // true
console.log({} === {});                      // false (different references)
const x = {}; console.log(x === x);         // true (same reference)
Q.04 What is the difference between null and undefined in JS? ⭐ Key Concept ▼

Both null and undefined represent the absence of a value, but they signal that absence in different ways and come from different sources:

  • undefined — JavaScript sets this automatically. It means a variable has been declared but not yet assigned a value. It also appears when you access a function parameter that wasn't passed, or a property that doesn't exist on an object. JS itself produces undefined.
  • null — You set this deliberately. It's an intentional assignment that means "this variable exists but currently has no value." It's a way of explicitly clearing or resetting a variable. Developers produce null.
javascript 
// undefined — JS assigns it automatically
let x;
console.log(x);           // undefined

function greet(name) {
  console.log(name);       // undefined if no arg passed
}
greet();

const obj = { a: 1 };
console.log(obj.b);       // undefined — property doesn't exist

// null — developer assigns it deliberately
let user = null;           // intentionally empty for now
user = { name: "Arun" };  // assigned later

// Type comparison — a famous JS quirk
console.log(typeof undefined); // "undefined"
console.log(typeof null);      // "object" ← historical JS bug, not fixed to avoid breaking old code

// Loose vs strict equality
console.log(null == undefined);  // true  (loose — they're treated as equal)
console.log(null === undefined); // false (strict — different types)

Q.06 What is type coercion in JS? ⭐ Key Concept ▼

Type coercion is JavaScript's automatic (or manual) conversion of a value from one data type to another. Q.01 What are operators? What are the types of operators in JS? ⭐ Key Concept ▼

Operators are special symbols or keywords that perform operations on one or more values (called operands) and produce a result. Think of them as the verbs of JavaScript — they tell the engine what to do with your data.

JavaScript has the following categories:

  • Arithmetic — perform math: + - * / % ** (exponentiation), ++ --

  • typeof / instanceof — type-checking operators

  • Spread / Rest ... — expand or collect values

Q.03 What is short-circuit evaluation in JS? ⭐ Key Concept ▼

Short-circuit evaluation means JavaScript stops evaluating a logical expression the moment the final result is determined — it doesn't bother processing the rest of the expression. This pattern is used constantly in real-world JS and React code.

  • && (AND) — if the left side is falsy, it immediately returns that left value and skips the right side. If left is truthy, it evaluates and returns the right side.
  • || (OR) — if the left side is truthy, it immediately returns the left value and skips the right side. If left is falsy, it returns the right side.
  • ?? (Nullish Coalescing) — like || but only short-circuits for null/undefined — not for 0, "", or false.
javascript 
// && : left falsy → skip right
false && console.log("never runs"); // nothing printed
true  && console.log("this runs");  // "this runs"

// Common pattern — call function only if condition met
isLoggedIn && showDashboard();

// || : left truthy → skip right (used for fallbacks)
const name = "" || "Guest";      // "Guest" (empty string is falsy)
const port = userPort || 3000;   // 3000 if userPort is falsy

// Problem with || : valid falsy values get replaced
const count  = 0 || 10;          // 10 — but 0 was a valid value!

// ?? solves this — only null/undefined triggers fallback
const count2 = 0 ?? 10;          // 0  ✅ zero preserved
const count3 = null ?? 10;       // 10 ✅ null triggers fallback
const count4 = "" ?? 10;         // ""  ✅ empty string preserved
Q.04 What is operator precedence? ▼

Operator precedence determines the order in which operators are evaluated in a complex expression — similar to BODMAS/PEMDAS in math. Higher precedence operators are evaluated first. When two operators share the same precedence, associativity (left-to-right or right-to-left) decides the order.

Common precedence order (high → low):

  • Highest: () grouping → . / [] member access → function call ()
  • Unary: ! ++ -- typeof
  • Arithmetic: ** → * / % → + -
  • Comparison: > < >= <= → == === != !==
  • Logical: && → || → ??
  • Lowest: ternary ? : → assignment =
javascript 
// * before + (like math)
console.log(2 + 3 * 4);      // 14, not 20
console.log((2 + 3) * 4);    // 20  (parentheses override)

// ** is right-to-left associative
console.log(2 ** 3 ** 2);    // 512 (3**2=9 first, then 2**9)

// && binds tighter than ||
console.log(false || true && false);
// → false || (true && false) → false || false → false

// Comparison before logical
console.log(1 + 2 === 3 && 4 > 2);
// → (3 === 3) && (4 > 2) → true && true → true

// Best practice: use parentheses to be explicit
const result = (a > b) && (c !== d);

Conditional statements let your program make decisions — running different code paths depending on whether a condition is true or false. JS provides four main forms:

  • if / else if / else — the most versatile. Checks conditions top-to-bottom and runs the first matching block. The else block is the catch-all fallback.
  • switch — tests one value against many exact cases. Cleaner than long else if chains when you're comparing one variable to many discrete values. Always add break to prevent fall-through.
  • Ternary ? : — a one-liner for simple if/else assignments. Best for single-expression decisions.
  • Logical short-circuit (&&, ||, ??) — inline conditionals for guard clauses, fallbacks, and conditional rendering in React.
javascript 
// if / else if / else

else                  { console.log("C"); }

// Short-circuit (React JSX pattern)
{isLoggedIn && <Dashboard />}
const username = user?.name ?? "Guest";

Choosing the right conditional keeps your code readable and maintainable. Here is a practical guide:

  • Use if/else — when conditions are complex, involve multiple variables, ranges, or compound logic. Also when each branch has multiple statements or side effects.
  • Use switch — when comparing one variable against many exact, discrete values. Classic use cases: routing, user roles, Redux action types, day/month names.
  • Use ternary ? : — for simple, single-expression decisions, especially when assigning a value or returning inline. Avoid nesting deeper than one level.
  • Use && — "only do this if true". Perfect for conditional rendering in React, or calling a function only when a value exists.
  • Use || — for fallback defaults when any falsy value should trigger the fallback. Use ?? instead when 0, "", or false are valid values you want to keep.

Q.07 What is the difference between == and ===? ⭐ Key Concept ▼

One of the most classic JavaScript interview questions. Both check equality, but with a critical difference:

  • == Loose Equality — compares values after performing type coercion. JS tries to convert both sides to a common type first. Results can be surprising and buggy.
  • === Strict Equality — compares both value AND type with no conversion. If types differ, it returns false immediately. This is predictable, safe, and what you should always use.

Always use === in production code. Most linters (ESLint) enforce this by default.

javascript 
// == (loose) — type coercion happens behind the scenes
console.log(5   ==  "5");        // true  (string coerced to number)
console.log(0   ==  false);      // true  (false coerced to 0)
console.log(""  ==  false);      // true  (both become 0)
console.log(null == undefined);  // true  (special JS rule)
console.log([1] ==  1);          // true  (array coerced to primitive)

// === (strict) — no coercion, type AND value must match
console.log(5   === "5");        // false (number ≠ string)
console.log(0   === false);      // false (number ≠ boolean)
console.log(null === undefined); // false (different types)
console.log(5   === 5);          // true  ✅
console.log("hi" === "hi");      // true  ✅

// Same for != vs !==
console.log(5 !=  "5");   // false (coercion makes them equal)
console.log(5 !== "5");   // true  ✅ (different types)
Q.08 What is the difference between Spread and Rest operator in JS? ⭐ Key Concept ▼

Both Spread and Rest use the same three-dot syntax (...), but they do the exact opposite things depending on context:

  • Spread ... — expands an iterable (array, object) into individual elements. Used in function calls, array literals, or object literals. Think of it as unpacking a box.
  • Rest ... — collects multiple individual values into a single array. Used in function definitions and destructuring. Think of it as packing things into a box.

Quick rule: In a function definition or destructuring → it's Rest. In a function call or array/object literal → it's Spread.

javascript 
// ── SPREAD — expands iterables ──────────────────────────

// Spread in function call
const nums = [1, 2, 3];
console.log(Math.max(...nums));   // 3 (same as Math.max(1, 2, 3))

// Spread to merge arrays
const a = [1, 2], b = [3, 4];
const merged = [...a, ...b];      // [1, 2, 3, 4]

// Spread to clone / merge objects
const user    = { name: "Arun", age: 25 };
const updated = { ...user, age: 26, city: "Mumbai" };
// { name: "Arun", age: 26, city: "Mumbai" }

// ── REST — collects remaining values ────────────────────

// Rest in function parameters
function sum(...numbers) {         // all args collected into array
  return numbers.reduce((acc, n) => acc + n, 0);
}
sum(1, 2, 3, 4);                // 10

// Rest must always be the LAST parameter
function log(first, ...rest) {
  console.log(first);  // "a"
  console.log(rest);   // ["b", "c", "d"]
}
log("a", "b", "c", "d");

// Rest in array destructuring
const [head, ...tail] = [1, 2, 3, 4];
// head = 1,  tail = [2, 3, 4]

// Rest in object destructuring
const { name, ...details } = user;
// name = "Arun",  details = { age: 25 }

Q.05 What is the difference between pop() and shift() methods of an Array? ▼

javascript 
// Sorting strings (default works fine)

console.log(nums); // [10, 2, 30, 5] → [10, 2, 30, 5] WRONG ❌
// Default converts to strings: "10" < "2" alphabetically!

// Correct way — use comparator function
nums.sort((a, b) => a - b);  // ascending  → [2, 5, 10, 30]
nums.sort((a, b) => b - a);  // descending → [30, 10, 5, 2]

// How comparator works:
// if (a - b) < 0  → a comes first
// if (a - b) > 0  → b comes first
// if (a - b) === 0 → order unchanged

// Sorting objects by property

  { name: "Raj", age: 30 },
  { name: "Arun", age: 25 },

Q.11 What are array-like objects in JS? ▼

// Array.from() — most versatile

Q.01 What is a loop? What are the types of loops in JS? ⭐ Key Concept ▼

A loop is a control structure that repeats a block of code multiple times — either a set number of times, or until a condition becomes false. Loops are fundamental to programming because they let you automate repetitive tasks without writing the same code over and over.

JavaScript provides the following loop types:

  • for — classic counter-based loop. Best when you know exactly how many iterations you need.
  • while — runs as long as a condition is true. Best when the number of iterations is unknown upfront.
  • do...while — like while, but the body executes at least once before the condition is checked.
  • for...of — iterates over the values of any iterable (arrays, strings, Sets, Maps). Clean and modern.
  • for...in — iterates over the keys of an object. Also works on arrays (gives indexes as strings — avoid this).
  • forEach() — an array method that runs a callback for each element. Cannot be broken out of early.
javascript 
// for — know the count upfront
for (let i = 0; i < 5; i++) {
  console.log(i); // 0, 1, 2, 3, 4
}

// while — condition-driven
let count = 0;
while (count < 3) { console.log(count++); } // 0, 1, 2

// do...while — runs at least once
do { console.log("runs once even if false"); } while (false);

// for...of — values of an iterable
for (const fruit of ["apple", "banana"]) {
  console.log(fruit);
}

// for...in — keys of an object
const user = { name: "Arun", age: 25 };
for (const key in user) {
  console.log(key, user[key]); // "name Arun", "age 25"
}

// forEach — array method with callback
[1, 2, 3].forEach((n, i) => console.log(i, n));
Q.02 What is the difference between while and for loops? ▼

Both for and while repeat a block of code, but they are designed for different scenarios and differ in structure:

  • for loop — packages the initialisation, condition check, and update expression all in one line. Best used when you know the number of iterations upfront — like looping through an array by index. Everything is visible at a glance, making it harder to accidentally forget the increment and cause an infinite loop.
  • while loop — only has a condition. The initialisation happens before the loop, and the update happens inside the body. Best used when the number of iterations is unknown — like reading data until a certain event occurs, polling until a value changes, or waiting for user input.

Under the hood they are equivalent — a for loop can always be rewritten as a while loop and vice versa. The choice is about readability and intent.

javascript 
// for — all control in one line, iteration count known
for (let i = 0; i < 5; i++) {
  console.log(i);
}

// Same logic written as while
let i = 0;            // initialisation before loop
while (i < 5) {       // condition only
  console.log(i);
  i++;                 // update inside body
}

// while shines when count is unpredictable
let input = "";
while (input !== "quit") {
  input = getUserInput(); // keep asking until user types "quit"
}

// Another real example: retry logic
let attempts = 0;
while (!isConnected() && attempts < 5) {
  connect();
  attempts++;
}
Q.03 What is the difference between while and do...while loops? ▼

The only but very important difference is when the condition is evaluated:

  • while — checks the condition before executing the body. If the condition is false from the start, the body never runs — not even once.
  • do...while — executes the body first, then checks the condition. This guarantees the body runs at least once, regardless of the condition.

Use do...while when you need to perform an action before you can even evaluate whether to continue — the most classic example is showing a prompt or menu to a user at least once before deciding to repeat it.

javascript 
// while — condition checked FIRST
let x = 10;
while (x < 5) {
  console.log("runs"); // ❌ never runs — 10 < 5 is false from start
}

// do...while — body runs FIRST, then condition checked
let y = 10;
do {
  console.log("runs once!"); // ✅ executes before checking
} while (y < 5);             // condition is false, loop stops after 1 run

// Real-world use — show menu at least once
let choice;
do {
  choice = showMenuAndGetInput(); // always shown at least once
} while (choice !== "exit");

// Validation loop — ask until valid input received
let age;
do {
  age = parseInt(prompt("Enter your age:"));
} while (isNaN(age) || age <= 0); // keep asking until valid
Q.04 What is the difference between break and continue statements? ⭐ Key Concept ▼

Both break and continue are loop control statements that change the normal flow of execution, but they do very different things:

  • break — immediately exits the entire loop and moves on to the code after the loop. Use it when you've found what you were looking for and there's no point continuing.
  • continue — skips the rest of the current iteration and jumps to the next one. The loop itself keeps running — only the current cycle is cut short. Use it to skip over elements that don't meet a condition.

Both work with for, while, do...while, and for...of. They do not work inside forEach() — another key reason to sometimes prefer for...of over forEach().

javascript 
// break — stop the loop entirely
for (let i = 0; i < 10; i++) {
  if (i === 5) break;  // exits loop when i is 5
  console.log(i);       // prints 0, 1, 2, 3, 4
}
console.log("After loop"); // ← execution resumes here

// continue — skip this iteration, keep looping
for (let i = 0; i < 6; i++) {
  if (i % 2 === 0) continue; // skip even numbers
  console.log(i);               // prints 1, 3, 5
}

// Real-world: find first matching user and stop
let foundUser;
for (const user of users) {
  if (user.id === targetId) {
    foundUser = user;
    break;  // no need to keep searching
  }
}

// Real-world: process only valid items, skip invalid
for (const item of items) {
  if (!item.isValid) continue; // skip invalid
  processItem(item);           // only processes valid ones
}

// ⚠️ break/continue do NOT work inside forEach!
[1, 2, 3].forEach(n => {
  if (n === 2) break; // ❌ SyntaxError — use for...of instead
});
Q.05 What is the difference between for and for...of loop in JS? ⭐ Key Concept ▼

Both loop over arrays but differ significantly in syntax, readability, and capability:

  • for loop — counter-based. You manually manage the index variable, the condition, and the increment. Gives you full control: you can loop backwards, skip indices, or jump by 2s. Works on anything that has numeric indices.
  • for...of loop — value-based. It directly gives you each value without worrying about indices. Cleaner syntax, less error-prone. Works on any iterable — arrays, strings, Sets, Maps, NodeLists, and generator functions. Does NOT work on plain objects.

Use for...of by default for iterables. Fall back to a classic for loop when you need the index or need fine-grained control over iteration.

javascript 
const fruits = ["apple", "banana", "cherry"];

// Classic for — verbose but full control
for (let i = 0; i < fruits.length; i++) {
  console.log(i, fruits[i]);       // index AND value available
}

// for...of — clean, direct access to values
for (const fruit of fruits) {
  console.log(fruit);              // "apple", "banana", "cherry"
}

// Need index in for...of? Use entries()
for (const [index, fruit] of fruits.entries()) {
  console.log(index, fruit);       // 0 "apple", 1 "banana"...
}

// for...of works on strings, Sets, Maps too
for (const char of "hello") {
  console.log(char);               // h, e, l, l, o
}

for (const [key, val] of new Map([["a", 1], ["b", 2]])) {
  console.log(key, val);           // "a" 1, "b" 2
}

// break/continue work in both
for (const fruit of fruits) {
  if (fruit === "banana") break;  // ✅ works fine
}
Q.06 What is the difference between for...of and for...in loop? ⭐ Key Concept ▼

The names are similar but they iterate over fundamentally different things, and are designed for different data structures:

  • for...of — iterates over the values of an iterable (arrays, strings, Sets, Maps). This is the one you reach for when you want to loop through items in a collection.
  • for...in — iterates over the enumerable keys (property names) of an object. Primarily designed for plain objects. When used on an array it gives the indices as strings — which is almost always undesirable and considered bad practice.

Quick rule: Use for...of for arrays and iterables. Use for...in for objects. Never use for...in on arrays.

javascript 
// for...of — gives VALUES of iterable
const colors = ["red", "green", "blue"];
for (const color of colors) {
  console.log(color);  // "red", "green", "blue"
}

// for...in — gives KEYS of object
const user = { name: "Arun", age: 25, city: "Mumbai" };
for (const key in user) {
  console.log(key, user[key]);
  // "name Arun", "age 25", "city Mumbai"
}

// ⚠️ for...in on an array — BAD PRACTICE
for (const i in colors) {
  console.log(i);     // "0", "1", "2" — strings, not numbers!
}
// Danger: for...in also picks up inherited/prototype properties

// for...in does NOT work on Sets/Maps
const mySet = new Set([1, 2, 3]);
for (const v of mySet) console.log(v);  // ✅ 1, 2, 3
// for (const k in mySet) — gives nothing useful

// for...of does NOT work on plain objects
// for (const v of user) ❌ TypeError: user is not iterable
// Workaround: iterate Object.entries()
for (const [key, val] of Object.entries(user)) {
  console.log(key, val); // "name Arun", "age 25"...
}

It is designed purely for side effects — it always returns undefined and cannot be exited early with break or continue.

javascript 
const nums = [10, 20, 30];

// forEach — callback receives (value, index, array)
nums.forEach((value, index, arr) => {
  console.log(index, value, arr);
});

// for...of — cleaner, also gives values
for (const num of nums) {
  console.log(num);
}

// for...in — gives keys (string indices for arrays)
for (const key in nums) {
  console.log(key);  // "0", "1", "2" — not recommended for arrays
}

// Comparison table (in comments):
// ┌──────────────┬──────────────┬────────────┬────────────┐
// │              │ forEach      │ for...of   │ for...in   │
// ├──────────────┼──────────────┼────────────┼────────────┤

Q.08 When to use for...of loop and when to use forEach method in applications? ▼

Both for...of and forEach iterate over arrays and run code for each element. The practical choice comes down to four factors:

  • Use forEach when: you want clean, functional-style code and you're only performing simple side effects (logging, DOM updates, firing events). The intent is clear — "do something for each item".
  • Use for...of when: you need to use break or continue to exit or skip early. forEach does not support these at all.
  • Use for...of when: you are working with async/await. Inside a forEach callback, await does not pause the outer loop — all iterations fire at once. for...of correctly awaits each iteration.
  • Use for...of when: you're iterating over non-array iterables (Sets, Maps, strings, NodeLists). forEach is only available on arrays.
javascript 
// ✅ forEach — simple side effects, no early exit needed
users.forEach(user => {
  renderUserCard(user);
});

// ✅ for...of — need to break early
for (const user of users) {
  if (user.isBanned) break; // stop as soon as banned user found
  processUser(user);
}

// ✅ for...of — async/await in sequence
async function processAll(items) {
  for (const item of items) {
    await uploadToServer(item); // waits for each upload to finish
  }
}

// ❌ forEach with async — does NOT work as expected
async function badUpload(items) {
  items.forEach(async item => {
    await uploadToServer(item); // fires all simultaneously, order lost
  });
  // function returns before any upload completes!
}

// ✅ for...of — iterating a Set (forEach not available on Set)
const uniqueTags = new Set(["js", "react", "node"]);
for (const tag of uniqueTags) {
  console.log(tag);
}

Q.01 What are Functions in JS? What are the types of functions? ⭐ Key Concept ▼

A function is a reusable, named block of code that performs a specific task. You define it once and call it whenever you need it. Functions can accept inputs (parameters), do something with them, and optionally return an output. They are the primary building blocks for organising and reusing logic in JavaScript.

JavaScript supports several types of functions:

  • Function Declaration — the classic syntax. Gets fully hoisted, so it can be called before it is defined in code.
  • Function Expression — a function assigned to a variable. Not hoisted — must be defined before it is called.
  • Arrow Function — a concise ES6 shorthand. Does not have its own this — inherits it from the surrounding scope.
  • Anonymous Function — a function with no name, usually used inline as a callback.
  • Named Function Expression — a function expression that also has an internal name (useful for recursion and debugging).
  • IIFE (Immediately Invoked Function Expression) — defined and called at the same time. Used to create private scope.
  • Callback Function — a function passed as an argument to another function, to be called later.
  • Higher-Order Function — a function that accepts a function as an argument or returns a function.
  • Generator Function — uses function* and yield to produce values lazily, one at a time.
  • Async Function — uses async/await to handle asynchronous operations in a synchronous style.
javascript 
// Function Declaration — hoisted
function greet(name) {
  return `Hello, ${name}!`;
}

// Function Expression — not hoisted
const greet = function(name) { return `Hello, ${name}!`; };

// Arrow Function
const greet = name => `Hello, ${name}!`;

// IIFE
(function() { console.log("Runs immediately!"); })();

// Async function
async function fetchData() {
  const res = await fetch("/api/data");
  return res.json();
}

// Generator function
function* counter() {
  yield 1; yield 2; yield 3;
}
const gen = counter();
gen.next(); // { value: 1, done: false }
Q.02 What is the difference between named and anonymous functions? When to use which? ▼

The distinction is simply whether the function has an identifier (name) attached to it:

  • Named function — has an explicit name. The name appears in stack traces when an error occurs, making debugging much easier. It can also call itself recursively by name. Preferred for any standalone, reusable logic.
  • Anonymous function — has no name. When an error occurs inside it, the stack trace shows "anonymous", making it harder to debug. They are commonly used as short-lived inline callbacks where the logic is simple and doesn't need to be reused.

When to use which: Use named functions for anything reusable, complex, or that you might want to debug. Use anonymous functions for short, one-time callbacks passed to methods like map(), setTimeout(), or event listeners, where naming them adds no real value.

// Anonymous function — no name, harder to debug
const tax = function(income) { return income * 0.3; };

// Named function expression — has an internal name for recursion
const factorial = function fact(n) {

};

// Anonymous used as short callback — acceptable
[1, 2, 3].map(n => n * 2);          // arrow = anonymous, fine here

// Named callback — easier to reuse and debug

Q.03 What is a function expression in JS? ▼

A function expression is when you create a function and assign it to a variable, rather than declaring it with the function keyword as a standalone statement. The function becomes a value — just like a string or number — and the variable holds a reference to it.

The critical behavioural difference from a function declaration is that function expressions are not hoisted. The variable gets hoisted (as undefined with var, or into the TDZ with let/const), but the function value itself is not assigned until that line executes.

javascript 
// Function Declaration — hoisted, callable anywhere

sayHello(); // ❌ ReferenceError (with const) or TypeError (with var)
const sayHello = function() { console.log("Hello!"); };

// Function expressions can be anonymous or named
const double = function(n) { return n * 2; };       // anonymous
const triple = function tripleIt(n) { return n * 3; }; // named expression

// Arrow functions are always expressions
const square = n => n * n;

// Expressions are useful when you need conditional function assignment
const processPayment = isUPI

Q.04 What are Arrow Functions in JS? What is their use? ⭐ Key Concept ▼

Arrow functions (ES6+) are a shorter, more concise syntax for writing function expressions. But beyond syntax, they have an important behavioural difference: they do not have their own this. They inherit this from the enclosing lexical scope — the context where they were defined. This is called lexical this.

Arrow functions also do not have their own arguments object, cannot be used as constructors (no new), and have no prototype property.

  • Use arrow functions for callbacks, array methods, and anywhere you need this to refer to the outer context.
  • Avoid arrow functions for object methods, event handlers where you need this to refer to the element, and constructors.
javascript 
// Syntax variations
const add     = (a, b) => a + b;          // multiple params
const double  = n => n * 2;              // single param, no parens needed
const greet   = () => "Hello!";          // no params
const getUser = () => ({ name: "Arun" }); // return object — wrap in ()
const log     = msg => { console.log(msg); return msg; }; // block body

// 'this' — the key difference
function Timer() {
  this.count = 0;

  // Regular function — 'this' is rebound, refers to window/undefined
  setInterval(function() {
    this.count++; // ❌ 'this' is NOT the Timer instance
  }, 1000);

  // Arrow function — 'this' stays as Timer instance
  setInterval(() => {
    this.count++; // ✅ correctly refers to Timer
  }, 1000);
}

// Arrow functions in array methods (very common)
const prices  = [100, 200, 300];
const taxed    = prices.map(p => p * 1.18);
const above200 = prices.filter(p => p > 200);
const total    = prices.reduce((sum, p) => sum + p, 0);
Q.05 What are Callback Functions? What is their use? ⭐ Key Concept ▼

A callback function is a function that is passed as an argument to another function, and is intended to be called later — either after some operation completes, on a certain event, or at a specific time. The word "callback" literally means "call me back when you're done."

Callbacks are fundamental to JavaScript's asynchronous model. Since JS is single-threaded, it uses callbacks to defer work — "start this task, and when it's done, call this function." They're also used heavily in array methods, event listeners, and timers.

Drawback: Nesting multiple callbacks creates deeply indented code known as "Callback Hell" — which is why Promises and async/await were introduced as cleaner alternatives.

javascript 
// Basic callback — function passed as argument
function greet(name, callback) {
  console.log("Hello, " + name);
  callback(); // call it when ready
}
greet("Arun", () => console.log("Callback fired!"));

// Callbacks in built-in methods
[1, 2, 3].forEach(n => console.log(n));  // arrow fn is a callback
setTimeout(() => console.log("1 second later"), 1000);

// Async callback — classic Node.js style
readFile("data.txt", (err, data) => {
  if (err) { console.log("Error!"); return; }
  console.log(data);
});

// Callback Hell — nested async operations (hard to read)
login(user, () => {
  getProfile(user.id, (profile) => {
    getPosts(profile.id, (posts) => {
      renderPosts(posts); // deeply nested ← "callback hell"
    });
  });
});

// Solution: Promises / async-await flatten this
const profile = await getProfile(user.id);
const posts   = await getPosts(profile.id);
renderPosts(posts);
Q.06 What is a Higher-Order Function in JS? ⭐ Key Concept ▼

A Higher-Order Function (HOF) is a function that does at least one of these two things:

  • Accepts a function as an argument (takes a callback)
  • Returns a function as its result

This is possible in JavaScript because functions are first-class citizens — they can be stored in variables, passed around, and returned just like any other value. Higher-order functions are the foundation of functional programming patterns in JS. Every built-in array method that takes a callback — map, filter, reduce, forEach, sort — is a higher-order function.

javascript 
// HOF that TAKES a function
function applyTwice(fn, value) {
  return fn(fn(value));
}
const double = n => n * 2;
applyTwice(double, 3); // 12 (3 → 6 → 12)

// HOF that RETURNS a function
function multiplier(factor) {
  return n => n * factor; // returns a new function
}
const triple = multiplier(3);
const tenX   = multiplier(10);
triple(5); // 15
tenX(5);   // 50

// All built-in array methods are HOFs
const nums = [1, 2, 3, 4, 5];
nums.map(n => n ** 2);                   // [1, 4, 9, 16, 25]
nums.filter(n => n % 2 === 0);          // [2, 4]
nums.reduce((acc, n) => acc + n, 0);    // 15

// Real-world: HOF for logging wrapper
function withLogging(fn) {
  return function(...args) {
    console.log(`Calling with`, args);
    const result = fn(...args);
    console.log(`Result:`, result);
    return result;
  };
}
const loggedAdd = withLogging((a, b) => a + b);

Q.07 What is the difference between arguments and parameters? ▼

These two terms are often used interchangeably in conversation, but they refer to different things in code:

  • Parameters — the named placeholders (variables) listed in a function's definition. They are like labels for expected inputs. They only exist inside the function body.
  • Arguments — the actual values you pass into a function when you call it. They are the real data that gets assigned to the parameters.

Think of it this way: parameters are the recipe slots ("flour", "sugar"), and arguments are the actual ingredients you put in (500g flour, 200g sugar).

javascript 
// Parameters: 'name' and 'age' are parameters (in definition)

  return `I'm ${name}, ${age} years old.`;

introduce("Arun", 25);

// Mismatch — JS won't throw, it handles gracefully

introduce("Arun", 25, "extra"); // extra arg ignored

function sum() {
  console.log(arguments);       // array-like object of all passed args

Q.08 In how many ways can you pass arguments to a function? ▼

JavaScript gives you several flexible ways to pass data into a function:

  • Positional arguments — the standard way: pass values in order, matched to parameters by position.
  • Default parameters — define fallback values in the function signature so you don't need to always pass every argument.
  • Rest parameters (...args) — collect any number of extra arguments into a real array.
  • Spread operator — expand an array into individual arguments at the call site.
  • Object destructuring (named args) — pass an object and destructure it in the signature. Great for functions with many optional parameters — order doesn't matter.
javascript 
// 1. Positional arguments
function add(a, b) { return a + b; }
add(3, 4); // 7

// 2. Default parameters
function greet(name = "Guest") { return `Hello ${name}`; }

greet("Arun");     // "Hello Arun"

// 3. Rest parameters — variable number of args

sum(1, 2, 3, 4); // 10

// 4. Spread at call site — expand array into args
const nums = [5, 10, 15];
sum(...nums);      // 30
Math.max(...nums); // 15

// 5. Object destructuring — named args, order-independent
function createUser({ name, age = 18, role = "user" }) {

// Basic default parameter

Q.11 What are First-Class Functions in JS? ⭐ Key Concept ▼

JavaScript is said to have first-class functions, which means functions are treated as values — just like strings, numbers, or objects. This is a language design concept that enables powerful patterns like higher-order functions, callbacks, and closures.

Specifically, first-class functions mean you can:

  • Assign a function to a variable
  • Pass a function as an argument to another function
  • Return a function from another function
  • Store a function in a data structure (array, object)

This is what makes callbacks, higher-order functions, closures, and functional programming possible in JavaScript.

javascript 
// 1. Assigned to a variable
const sayHi = function() { return "Hi!"; };

// 2. Passed as an argument
function run(fn) { return fn(); }
run(sayHi); // "Hi!"

// 3. Returned from a function
function makeAdder(x) {
  return (y) => x + y; // returns a function
}
const add5 = makeAdder(5);
add5(3); // 8

// 4. Stored in an array
const pipeline = [
  x => x * 2,
  x => x + 10,
  x => x / 2,
];
const result = pipeline.reduce((val, fn) => fn(val), 4);
// 4 → 8 → 18 → 9

// 5. Stored in an object (method)
const utils = {
  double: n => n * 2,
  square: n => n ** 2,
};

// First-class + higher-order = expressive functional code

Q.12 What are Pure and Impure Functions in JS? ⭐ Key Concept ▼

This is a functional programming concept. The distinction matters a lot for writing predictable, testable code:

  • Pure Function — given the same inputs, it always returns the same output, and it has no side effects (it doesn't modify anything outside itself — no mutating global state, no console.log, no API calls, no DOM changes). Pure functions are easy to test, debug, and reason about.
  • Impure Function — either depends on or modifies something outside its own scope (external state, global variables, DOM, APIs, random numbers, current time). The same inputs may produce different outputs depending on external state.

In React, components should be pure functions — given the same props, they should always render the same output. Side effects belong in useEffect.

javascript 
// PURE — same input always gives same output, no side effects
function add(a, b) { return a + b; }          // ✅ pure
function double(arr) { return arr.map(x => x * 2); } // ✅ pure (new array)

// IMPURE — modifies external state
let total = 0;
function addToTotal(n) {
  total += n;  // ❌ modifies external variable — impure
}

// IMPURE — depends on external state (different output each call)
function getRandomGreeting() {
  return Math.random() > 0.5 ? "Hi!" : "Hello!"; // ❌ impure
}

// IMPURE — side effects (DOM, API, console)
function showAlert(msg) {
  document.getElementById("msg").textContent = msg; // ❌ DOM mutation
}

// Refactoring impure → pure: separate logic from side effects
// Impure: mixes calculation and DOM update
function calculateAndShow(a, b) {
  const result = a + b;
  document.getElementById("result").textContent = result; // side effect
}

// Pure: just calculates
function calculate(a, b) { return a + b; }      // ✅ pure
// Side effect handled separately:
document.getElementById("result").textContent = calculate(3, 4);
Q.13 What is Function Currying in JS? ⭐ Key Concept ▼

Currying is a functional programming technique where a function that takes multiple arguments is transformed into a sequence of functions, each taking a single argument. Instead of calling fn(a, b, c), you call fn(a)(b)(c).

Each call returns a new function that "remembers" the previous argument via closure, until all arguments are collected and the final result is produced. Currying is built on top of closures and first-class functions.

Why use it? It enables partial application — you can pre-fill some arguments and create specialised reusable functions without repeating yourself.

javascript 
// Normal function
function add(a, b, c) { return a + b + c; }
add(1, 2, 3); // 6

// Curried version — one arg at a time
function curriedAdd(a) {
  return (b) => {
    return (c) => a + b + c;
  };
}
curriedAdd(1)(2)(3); // 6

// Partial application — pre-fill some args
const add1   = curriedAdd(1);    // fixed first arg
const add1And2 = add1(2);        // fixed second arg
add1And2(10); // 13
add1And2(20); // 23 — reuse with different last arg

// Real-world: curried logger
const log = level => message =>
  console.log(`[${level.toUpperCase()}]: ${message}`);

const info  = log("info");   // pre-filled level
const error = log("error");

info("Server started");       // [INFO]: Server started
error("Something broke");     // [ERROR]: Something broke

// Arrow function currying — concise form
const multiply = a => b => a * b;
const double   = multiply(2);
const triple   = multiply(3);
double(5); // 10
triple(5); // 15
Q.14 What are call(), apply() and bind() methods in JS? ⭐ Key Concept ▼

All three are methods available on every function. They all let you explicitly set the value of this when calling a function — instead of letting JavaScript determine it based on how the function is invoked. The difference is in how and when they call the function:

  • call(thisArg, arg1, arg2, ...) — immediately invokes the function with a specified this context and individual arguments.
  • apply(thisArg, [arg1, arg2, ...]) — same as call, but arguments are passed as an array. Useful when your args are already in an array.
  • bind(thisArg, arg1, arg2, ...) — does not invoke the function immediately. Instead, it returns a new function with this permanently bound to the given value. That new function can be called later.

Memory hook: call = comma, apply = array, bind = borrow (later).

javascript 
function introduce(city, country) {
  return `I'm ${this.name} from ${city}, ${country}`;
}

const user1 = { name: "Arun" };
const user2 = { name: "Raj"  };

// call() — invoke immediately, args as comma-separated values
introduce.call(user1, "Mumbai", "India");
// "I'm Arun from Mumbai, India"

// apply() — invoke immediately, args as array
const args = ["Delhi", "India"];
introduce.apply(user2, args);
// "I'm Raj from Delhi, India"

// bind() — does NOT call immediately, returns new function
const introduceArun = introduce.bind(user1, "Pune", "India");
introduceArun(); // "I'm Arun from Pune, India" — called later

// Real-world: bind in event handlers to fix 'this'
class Button {
  constructor() { this.count = 0; }
  handleClick() { this.count++; console.log(this.count); }
}
const btn = new Button();
// Without bind, 'this' inside handleClick won't be the Button
document.addEventListener("click", btn.handleClick.bind(btn));

// apply trick — spread array into Math.max (before spread operator)
const nums = [5, 2, 8, 1];
Math.max.apply(null, nums); // 8 (null = no 'this' needed)
// Modern equivalent: Math.max(...nums)

Q.02 What are template literals and string interpolation? ⭐ Key Concept ▼

Template literals (introduced in ES6) are strings wrapped in backticks (` `) instead of quotes. They are a powerful upgrade over regular strings with three key features:

  • String interpolation — embed expressions and variables directly inside the string using ${expression}. No more messy + concatenation.

  • Tagged templates — an advanced feature where a function processes the template before the string is produced (used in styled-components, GraphQL queries, etc.).

javascript

console.log(`2 + 2 = ${2 + 2}`);                     // "2 + 2 = 4"
console.log(`Score: ${score >= 90 ? "A" : "B"}`);   // "Score: B"
console.log(`Upper: ${name.toUpperCase()}`);        // "Upper: ARUN"

Q.05 What is string immutability? ⭐ Key Concept ▼

String immutability means that once a string is created in JavaScript, its characters cannot be changed in place. You cannot go into a string and modify one character directly — the existing string always stays the same. Any string operation that appears to "modify" a string is actually creating and returning a brand new string, leaving the original untouched.

This is different from arrays, which are mutable — you can directly change an element in an array at a given index. With strings, you can't.

Why does this matter? It means string operations never have side effects on the original string. You must always capture the return value of string methods — otherwise your work is thrown away.

javascript 
let str = "hello";

// Attempt to change a character by index — silently fails!
str[0] = "H";
console.log(str); // "hello" — unchanged! ❌ (no error in non-strict mode)

// In strict mode it throws a TypeError
// "Cannot assign to read only property '0' of string"

// String methods return NEW strings — original untouched
const upper = str.toUpperCase();
console.log(str);   // "hello" — still original
console.log(upper); // "HELLO" — new string

// Common mistake — forgetting to capture the return value
str.toUpperCase();           // ❌ result discarded, str unchanged
str = str.toUpperCase();     // ✅ reassign variable to updated string

// Reassigning the VARIABLE is fine — it's not the same as mutating
let name = "arun";
name = "Arun";  // ✅ variable now points to a new string — original "arun" is gone

// Compare with arrays — arrays ARE mutable
const arr = ["h", "e", "l", "l", "o"];
arr[0] = "H";   // ✅ works — arrays can be mutated

  • += assignment — appends to an existing string variable, building it up iteratively.
  • Template literals — the modern, preferred way. Clean, readable, supports expressions inline.

  • Array.join() — an indirect but powerful approach: put strings in an array and join with a separator. Great for building lists or CSV-like outputs.

Q.01 What is DOM? What is the difference between HTML and DOM? ⭐ Key Concept ▼

The DOM (Document Object Model) is a programming interface created by the browser when it loads an HTML page. It represents the page as a live, in-memory tree of objects (nodes) — where every HTML element, attribute, and piece of text becomes a node that JavaScript can access, read, and manipulate.

HTML vs DOM — key differences:

  • HTML is a static text file — the raw source markup you write. It is just a string of characters sitting on disk.
  • DOM is the browser's live, structured, in-memory interpretation of that HTML. It can be changed dynamically by JavaScript without touching or reloading the original HTML file.
  • If you add a <div> via JavaScript, the DOM changes — but "View Source" still shows the original HTML.
  • The browser may also silently correct invalid HTML (missing closing tags, etc.) when building the DOM, so the DOM can differ from the raw HTML source.
javascript 
// The browser turns this HTML...
// <body>
//   <h1 id="title">Hello</h1>
// </body>

// ...into a DOM tree:
// Document
// └── html
//     └── body
//         └── h1#title
//             └── "Hello" (text node)

// JavaScript can then interact with these nodes:
const title = document.getElementById("title");
title.textContent = "Hello, DOM!"; // live update — HTML file unchanged
title.style.color = "gold";

// JS can also ADD nodes that never existed in the HTML
const p = document.createElement("p");
p.textContent = "Added by JS";
document.body.appendChild(p); // DOM changes, source HTML unchanged

Q.05 What is the difference between querySelector() and querySelectorAll()? ⭐ Key Concept ▼

Q.07 What is the difference between innerHTML and textContent? ⭐ Key Concept ▼

Both are properties used to read or set the content of a DOM element, but they handle HTML tags very differently:

  • innerHTML — gets or sets the content including HTML tags. When you set it, the browser parses the string as HTML and renders it. Powerful, but carries a serious security risk: if you insert user-supplied content via innerHTML, a malicious user can inject script tags — this is called XSS (Cross-Site Scripting).
  • textContent — gets or sets the content as plain text only. Any HTML tags in the string are treated as literal text characters, not parsed. Safe from XSS. Also slightly faster than innerHTML since no HTML parsing is needed.
  • innerText — similar to textContent but is aware of CSS styles (respects display:none, line breaks, etc.). Slower because it triggers a layout reflow.

Rule: Use textContent for plain text. Use innerHTML only for trusted, developer-controlled HTML — never for user input.

javascript 
const div = document.querySelector("div");

// innerHTML — reads/writes HTML markup
div.innerHTML = "<strong>Bold text</strong>";
// → renders actual bold text in the browser ✅

div.innerHTML = "<img src=x onerror='alert(1)'>"; // ❌ XSS attack!

// textContent — reads/writes plain text, no parsing
div.textContent = "<strong>Bold text</strong>";
// → renders literally: <strong>Bold text</strong> as text ✅ (safe)

// Reading — innerHTML includes tags, textContent strips them
// <div><p>Hello <b>World</b></p></div>
div.innerHTML    // "<p>Hello <b>World</b></p>"
div.textContent  // "Hello World" — tags stripped, just text

// Safe way to insert user content
const userInput = "<script>alert('hack')</script>";
div.textContent = userInput; // ✅ rendered as text, not executed

// innerText vs textContent
// <div style="display:none">Hidden</div>
div.textContent  // "Hidden" — ignores CSS, gets all text
div.innerText    // ""       — respects display:none (CSS-aware)

Q.11 What is the difference between createElement() and createTextNode()? ▼

Q.01 What is Error Handling in JS? ⭐ Key Concept ▼

Error handling is the practice of anticipating, detecting, and gracefully recovering from runtime errors so that your application does not crash or leave the user with a broken experience. In JavaScript, errors that are not handled will propagate up the call stack and eventually crash the program (or in the browser, log an uncaught error to the console and halt further script execution on that call stack).

The primary mechanism for error handling in JS is the try...catch...finally block:

  • try — wrap code that might throw an error. If an error occurs anywhere inside, execution immediately jumps to catch.
  • catch(err) — receives the error object and lets you handle it gracefully — log it, show a user-friendly message, retry, etc.
  • finally — runs always, regardless of whether an error occurred. Used for cleanup — closing connections, hiding spinners, releasing resources.
javascript 
// Basic structure
try {
  // code that might fail
  const data = JSON.parse("invalid json"); // throws SyntaxError
  console.log(data);                        // never reached
} catch (err) {
  // handle the error
  console.log(err.name);     // "SyntaxError"
  console.log(err.message);  // "Unexpected token i in JSON..."
  console.log(err.stack);    // full stack trace
} finally {
  console.log("Always runs — do cleanup here");
}

// Without error handling — crashes program
JSON.parse("bad"); // ❌ Uncaught SyntaxError: halts execution
console.log("never runs");

// Async error handling — use try/catch with async/await
async function fetchUser(id) {
  try {
    const res  = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP error: ${res.status}`);
    return await res.json();
  } catch (err) {
    console.error("Failed to fetch user:", err.message);
    return null; // return safe fallback instead of crashing
  }
}

// Error object properties
try { undefinedFn(); } catch(e) {
  e.name;     // "ReferenceError"
  e.message;  // "undefinedFn is not defined"
  e.stack;    // full call stack string
}
Q.02 What is the role of the finally block in JS? ⭐ Key Concept ▼

The finally block is the guaranteed cleanup zone. It runs after the try and catch blocks — regardless of what happened. It does not matter whether the try succeeded, the catch handled an error, or even if a return or throw statement was encountered — finally still executes.

It is also optional — you can write try...catch without finally, or even try...finally without catch.

Common use cases for finally:

  • Hiding a loading spinner after an API call (success or failure)
  • Closing a database connection or file handle
  • Releasing a lock or mutex
  • Resetting form state after submission
javascript 
// finally runs regardless of outcome
function riskyOp(fail) {
  try {
    if (fail) throw new Error("Oops!");
    console.log("Try succeeded");
    return "success"; // even return doesn't skip finally
  } catch (err) {
    console.log("Caught:", err.message);
  } finally {
    console.log("Finally always runs!"); // ✅ always
  }
}
riskyOp(false); // "Try succeeded" → "Finally always runs!"
riskyOp(true);  // "Caught: Oops!" → "Finally always runs!"

// Real-world: API call with loading state
async function loadData() {
  showSpinner();          // show loading UI
  try {
    const data = await fetchFromAPI();
    renderData(data);
  } catch (err) {
    showError(err.message);
  } finally {
    hideSpinner();          // ✅ always hidden, success or failure
  }
}

// try...finally without catch — error propagates, cleanup still runs
function withCleanup() {
  try {
    doSomethingRisky();
  } finally {
    releaseResource(); // runs even if error propagates upward
  }
}

// finally overrides return value if it also returns
function test() {
  try     { return "try";     }
  finally { return "finally"; } // ⚠️ overrides — returns "finally"
}
test(); // "finally" — avoid returning from finally!
Q.03 What is the purpose of the throw statement in JS? ⭐ Key Concept ▼

The throw statement lets you manually trigger an error from anywhere in your code. When throw is executed, it immediately stops the current function, unwinds the call stack, and looks for the nearest enclosing catch block to handle it — exactly like a runtime error would.

You can throw literally anything in JavaScript — a string, number, boolean, or object — but the best practice is always to throw an Error object (or a subclass of it), because Error objects carry a message, a name, and a stack trace — making debugging vastly easier.

javascript 
// throw stops execution and jumps to nearest catch
function divide(a, b) {
  if (b === 0) throw new Error("Division by zero is not allowed");
  return a / b;
}

try {
  console.log(divide(10, 0));
} catch (err) {
  console.log(err.message); // "Division by zero is not allowed"
}

// You CAN throw anything (but shouldn't)
throw "something went wrong";  // ❌ string — no stack trace
throw 42;                       // ❌ number — useless for debugging
throw { code: 404 };           // ❌ plain object — missing stack
throw new Error("message");    // ✅ best practice

// Custom Error classes — extend Error for domain-specific errors
class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name  = "ValidationError";
    this.field = field;
  }
}

class NotFoundError extends Error {
  constructor(resource) {
    super(`${resource} not found`);
    this.name       = "NotFoundError";
    this.statusCode = 404;
  }
}

// Throw and catch custom errors by type
function validateAge(age) {
  if (typeof age !== "number") throw new ValidationError("Age must be a number", "age");
  if (age < 0 || age > 120)   throw new ValidationError("Age out of range", "age");
  return true;
}

try {
  validateAge("twenty");
} catch (err) {
  if (err instanceof ValidationError) {
    console.log(`Field [${err.field}]: ${err.message}`);
  } else {
    throw err; // re-throw errors you don't know how to handle
  }
}
Q.04 What is Error Propagation in JS? ⭐ Key Concept ▼

Error propagation describes how an unhandled error travels (bubbles) up the call stack — from the function where it was thrown, to its caller, to that caller's caller, and so on — until it either hits a catch block somewhere or reaches the top level and becomes an uncaught error that crashes the program (or logs to the browser console).

This means you don't have to handle every error at every level. You can let errors propagate naturally and catch them at a higher level where you have enough context to respond appropriately — like at a router boundary in a server, or a top-level error boundary in a React app.

javascript 
// Error propagates up the call stack
function level3() {
  throw new Error("Error originated here"); // thrown in deepest fn
}

function level2() {
  level3(); // no try/catch — error propagates upward
}

function level1() {
  level2(); // no try/catch — keeps propagating
}

try {
  level1(); // caught here at the top
} catch (err) {
  console.log("Caught at top level:", err.message);
  // stack trace will show: level3 → level2 → level1
}

// Selective catching — catch what you can handle, re-throw the rest
function processData(data) {
  try {
    return JSON.parse(data);
  } catch (err) {
    if (err instanceof SyntaxError) {
      console.warn("Bad JSON, returning null");
      return null;        // handle SyntaxError here
    }
    throw err;             // re-throw unknown errors upward
  }
}

// Async error propagation — rejected promises propagate too
async function step3() { throw new Error("Async error"); }
async function step2() { await step3(); }
async function step1() { await step2(); }

async function main() {
  try {
    await step1(); // error from step3 propagates through step2 → step1
  } catch (err) {
    console.log("Caught async error:", err.message);
  }
}

// Global fallback — catch anything that slipped through
window.addEventListener("error", (event) => {
  console.error("Global uncaught error:", event.error);
});
window.addEventListener("unhandledrejection", (event) => {
  console.error("Unhandled promise rejection:", event.reason);
});
Q.05 What are the best practices for error handling? ⭐ Key Concept ▼

Good error handling is what separates production-grade code from buggy prototypes. These are the most important practices:

  • Always throw Error objects — not strings or plain objects. Error objects carry a stack trace that is essential for debugging.
  • Use custom error classes — extend Error for domain-specific errors like ValidationError, NotFoundError, AuthError. This lets callers use instanceof to handle different error types differently.
  • Never silently swallow errors — an empty catch(err) {} block hides bugs. At minimum, log the error.
  • Catch at the right level — handle errors where you have enough context to respond meaningfully. Let them propagate if you can't handle them properly at that level.
  • Re-throw errors you can't handle — don't swallow unexpected errors. Catch what you expect, re-throw the rest.
  • Always handle async errors — unhandled promise rejections can silently fail. Use try/catch with async/await or .catch() on promises.
  • Use finally for guaranteed cleanup — spinners, connections, timers, and locks should always be released in finally.
  • Show user-friendly messages — log the technical error internally but show a clean, non-technical message to the user.
  • Set up global error fallbacks — use window.onerror / unhandledrejection as a last safety net.

// ❌ BAD — swallowing errors silently

  doSomething();
} catch (e) {} // silent catch — bugs disappear, very dangerous

// ❌ BAD — throwing a string

// ✅ GOOD — catch specific error types, re-throw unknown

Q.06 What are the different types of errors in JS? ⭐ Key Concept ▼

JavaScript has a built-in Error base class and six native error subtypes, each representing a different category of problem. Knowing these helps you identify bugs faster and write more targeted catch logic.

  • SyntaxError — invalid JavaScript syntax that the parser cannot understand. Caught at parse time (before code runs). Examples: missing brackets, invalid JSON, unexpected tokens.
  • ReferenceError — accessing a variable that does not exist in the current scope. Very common mistake — typos, accessing before declaration (TDZ).
  • TypeError — an operation is performed on a value of the wrong type. Extremely common — calling something that isn't a function, accessing a property on null or undefined.
  • RangeError — a value is outside the allowed range. Examples: invalid array length, too many recursive calls (stack overflow), invalid number formatting.
  • URIError — invalid use of URI-related functions like decodeURIComponent().
  • EvalError — related to the eval() function. Rarely seen in modern JS.
  • AggregateError — wraps multiple errors into one. Used by Promise.any() when all promises reject.
javascript 
// SyntaxError — invalid syntax (caught at parse time)
// JSON.parse("bad json")  ← SyntaxError at runtime
// let x = ;               ← SyntaxError at parse time (can't catch)
try { JSON.parse("{bad}"); } catch(e) { console.log(e.name); } // SyntaxError

// ReferenceError — variable doesn't exist
try { console.log(undeclaredVar); } catch(e) { console.log(e.name); } // ReferenceError
try { console.log(x); let x = 1; } catch(e) { console.log(e.name); } // ReferenceError (TDZ)

// TypeError — wrong type operation
try { null.toString(); }   catch(e) { console.log(e.name); } // TypeError
try { (42)(); }             catch(e) { console.log(e.name); } // TypeError: 42 is not a function
try { undefined.x; }       catch(e) { console.log(e.name); } // TypeError

// RangeError — value out of allowed range
try { new Array(-1); }      catch(e) { console.log(e.name); } // RangeError
try { (1.2).toFixed(200); } catch(e) { console.log(e.name); } // RangeError
function inf() { return inf(); } // RangeError: Maximum call stack exceeded

// URIError
try { decodeURIComponent("%"); } catch(e) { console.log(e.name); } // URIError

// AggregateError — from Promise.any() when all reject
Promise.any([
  Promise.reject("err1"),
  Promise.reject("err2"),
]).catch(err => {
  console.log(err.name);    // "AggregateError"
  console.log(err.errors);  // ["err1", "err2"]
});

// Check error type in catch with instanceof
try {
  riskyCode();
} catch (err) {
  if      (err instanceof TypeError)      console.log("Type problem");
  else if (err instanceof ReferenceError)  console.log("Missing variable");
  else if (err instanceof SyntaxError)    console.log("Bad syntax");
  else                                      throw err; // re-throw unknown
}

  • in operator — checks if a property exists on the object or anywhere in its prototype chain. Returns a boolean.

  • Object.hasOwn(obj, key) — modern ES2022 replacement for hasOwnProperty. Safer because it can't be overridden.
  • Checking for undefined — accessing a non-existent key returns undefined, but this approach has a gotcha: the property might exist but genuinely hold the value undefined.
javascript

user.hasOwnProperty("name")      // true
user.hasOwnProperty("toString")  // false — inherited, not own
user.hasOwnProperty("email")     // false — doesn't exist
user.hasOwnProperty("score")     // true  — exists even if undefined

// 3. Object.hasOwn() — modern, preferred (ES2022)

"score" in user          // true  — correct

Q.08 How do you clone or copy an object? ⭐ Key Concept ▼

Since objects are reference types, simple assignment (const b = a) just copies the reference — both variables point to the same object. To get an independent copy, you need to explicitly clone it. There are two categories: shallow clone and deep clone.

  • Shallow clone — copies the top-level properties. If a property value is itself an object/array, only the reference is copied (both share the nested object).
  • Deep clone — recursively copies everything, creating fully independent nested objects with no shared references.
javascript 
const user = { name: "Arun", age: 25, address: { city: "Mumbai" } };

// ── SHALLOW CLONE (top-level only) ───────────────────

// Spread operator — most common
const copy1 = { ...user };
copy1.name = "Raj";          // ✅ doesn't affect original
copy1.address.city = "Delhi"; // ⚠️ DOES affect original (shared ref)

// Object.assign()
const copy2 = Object.assign({}, user);
// same shallow behaviour as spread

// ── DEEP CLONE (fully independent) ───────────────────

// JSON round-trip — most common, simple
const deep1 = JSON.parse(JSON.stringify(user));
deep1.address.city = "Pune"; // ✅ original city stays "Mumbai"

// JSON limitations — loses: functions, undefined, Date, Symbol, Infinity, NaN
const obj2 = { fn: () => {}, d: new Date(), x: undefined };
JSON.parse(JSON.stringify(obj2));
// { d: "2024-01-01T..." } — fn and x are dropped, Date becomes string

// structuredClone() — modern (Node 17+, all modern browsers)
const deep2 = structuredClone(user); // ✅ handles Date, RegExp, Map, Set
deep2.address.city = "Goa";           // original unaffected ✅
// Still doesn't clone: functions, DOM nodes, class instances

// Quick summary of options:
// { ...obj }               → shallow, misses prototype
// Object.assign({}, obj)   → shallow, same behaviour as spread
// JSON.parse(JSON.stringify) → deep, loses functions/Date/Symbol
// structuredClone()         → deep, handles most types ✅ preferred
Q.09 What is the difference between deep copy and shallow copy in JS? ⭐ Key Concept ▼

This is one of the most frequently misunderstood concepts in JavaScript, and a very common interview question. The difference comes down to how nested objects and arrays are handled:

  • Shallow Copy — creates a new object and copies the top-level property values. For primitive values (string, number, boolean), actual values are copied. But for nested objects/arrays, only the reference is copied — so the copy and original still share the same nested structure.
  • Deep Copy — creates an entirely new object tree. Every level of nesting is recursively cloned. The copy and original have zero shared references — modifying any part of one has absolutely no effect on the other.
javascript 
const original = {

  scores: [90, 85, 95],     // nested array
  address: { city: "Mumbai" } // nested object

shallow.scores.push(100);     // ⚠️ AFFECTS original.scores too!
shallow.address.city = "Delhi";// ⚠️ AFFECTS original.address.city too!

console.log(original.scores);        // [90,85,95,100] — affected ⚠️
console.log(original.address.city);  // "Delhi" — affected ⚠️

// DEEP COPY via structuredClone (modern)
const deep = structuredClone(original);

deep.name = "Raj";
deep.scores.push(100);
deep.address.city = "Pune";

// Reset original for the demo
const orig2 = { name: "Arun", scores: [90], address: { city: "Mumbai" } };
const deep2  = structuredClone(orig2);
deep2.scores.push(100);
deep2.address.city = "Pune";

console.log(orig2.scores);       // [90]      — unaffected ✅
console.log(orig2.address.city); // "Mumbai"  — unaffected ✅

// Flat objects with only primitives → shallow copy is fine
// Objects with nested arrays/objects → always use deep copy
Q.10 What is Set Object in JS? ⭐ Key Concept ▼

Q.12 What is the difference between Map and Object in JS? ⭐ Key Concept ▼

Both Map and Object store key-value pairs, but they have important differences that make each better suited for different situations:

  • Key types — Object keys are always strings (or Symbols). Map keys can be any type — objects, functions, primitives.
  • Key order — Map preserves insertion order strictly. Object mostly preserves it for string keys but integer-like keys are sorted first.
  • Size — Map has a built-in .size property. Objects need Object.keys(obj).length.
  • Performance — Map is optimised for frequent additions/deletions. Objects are better for static structures with known keys.
  • Prototype — Plain objects have inherited prototype properties (toString, hasOwnProperty, etc.) that can cause naming collisions. Maps don't.
  • Iteration — Maps are directly iterable with for...of. Objects need Object.keys/values/entries.
  • JSON — Objects serialize to JSON natively. Maps do not — you must convert them first.
javascript 
// KEY TYPES — object keys become strings, Map keys stay as-is
const obj = {};
obj[1]    = "a";      // key is coerced to string "1"
obj[true] = "b";      // key is coerced to string "true"
console.log(Object.keys(obj)); // ["1", "true"] — strings!

const map = new Map();
map.set(1, "a");       // key stays as number 1
map.set(true, "b");    // key stays as boolean true
map.get(1);            // "a" — 1 (not "1")

// SIZE
Object.keys(obj).length  // manual count
map.size                 // built-in, always accurate

// PROTOTYPE COLLISION — Object has inherited keys, Map doesn't
const obj2 = {};
"toString" in obj2    // true — inherited! Can be problematic
map.has("toString")   // false — Map is clean

// WHEN TO USE EACH
// Use Object when:
// - Representing structured data / entities (user, config, response)
// - JSON serialization required
// - Keys are always strings and known at design time

// Use Map when:
// - Keys are not strings (objects, functions, numbers as keys)
// - You need to count or track size easily
// - Frequent additions/deletions
// - Need strict insertion-order iteration
// - Building a cache/dictionary/lookup table

// Example: cache with object keys (only possible with Map)
const cache = new Map();
function memoize(fn) {
  return function(arg) {
    if (cache.has(arg)) return cache.get(arg);
    const result = fn(arg);
    cache.set(arg, result);
    return result;
  };
}

Q.01 What are Events? How are events triggered? ⭐ Key Concept ▼

Events are signals fired by the browser (or by code) to indicate that something has happened — a user clicked a button, pressed a key, moved the mouse, submitted a form, or a page finished loading. Events are the foundation of interactive web pages. Without events, JavaScript would run once and stop — it's events that keep it listening and responding.

How events are triggered:

  • User interaction — clicks, keypresses, mouse movements, touch events, form submissions.
  • Browser lifecycle — page load, DOM ready, resource loaded, window resize, scroll.
  • Timer-based — setTimeout, setInterval completing.
  • Programmatic dispatch — code manually fires a custom event using dispatchEvent().

To respond to events you attach an event listener — a callback function that runs whenever the specified event fires on a given element.

javascript 
// addEventListener — the modern, preferred way to attach events
// Syntax: element.addEventListener(event, handler, options)

  console.log("Button clicked!", event);

// Multiple listeners on same element — all fire
btn.addEventListener("click", () => console.log("First listener"));
btn.addEventListener("click", () => console.log("Second listener"));

// Old inline style — avoid (mixes HTML & JS, only one handler allowed)
// <button onclick="handleClick()">Click</button>

// Old property style — avoid (only one handler possible)

Q.04 What is Event Delegation in JS? ⭐ Key Concept ▼

Event Delegation is a pattern where instead of attaching event listeners to each child element individually, you attach a single listener to a parent element and use event bubbling to catch events from all its children. When a child is clicked, the event bubbles up to the parent, and you use event.target to figure out which child triggered it.

Why it's important:

  • Performance — one listener instead of hundreds. A list with 1000 items only needs one listener on the <ul>.
  • Dynamic elements — works automatically for elements added to the DOM after the listener was attached. Individual listeners wouldn't cover new elements.
  • Memory — fewer listeners means less memory usage and easier cleanup.
javascript 
// ❌ Without delegation — one listener per button (bad)
document.querySelectorAll(".btn").forEach(btn => {
  btn.addEventListener("click", handleClick);
  // New buttons added later won't have this listener!
});

// ✅ With delegation — one listener on parent (good)
document.getElementById("btn-container").addEventListener("click", (e) => {
  // Check if the clicked element is what we care about
  if (e.target.matches(".btn")) {
    handleClick(e.target);
  }
});

// Real-world: todo list with delete buttons
// HTML: <ul id="todo-list"><li data-id="1">Task <button class="del">X</button></li></ul>
document.getElementById("todo-list").addEventListener("click", (e) => {
  if (e.target.matches(".del")) {
    const li = e.target.closest("li");    // traverse up to find the parent li
    const id = li.dataset.id;
    deleteTask(id);
    li.remove();
  }
});

// Works for dynamically added items too!
function addTask(text) {
  const li = document.createElement("li");
  li.innerHTML = `${text} <button class="del">X</button>`;
  document.getElementById("todo-list").appendChild(li);
  // No need to add a new listener — delegation handles it ✅
}
Q.05 What is Event Bubbling in JS? ⭐ Key Concept ▼

Event Bubbling is the default behaviour where an event fired on a child element propagates upward through its ancestor elements in the DOM tree — from the innermost element all the way up to document and then window. At each ancestor, if there is a listener for that event type, it will fire.

Think of it like dropping a pebble in water — the ripple (event) starts at the point of impact (the clicked element) and spreads outward (upward through ancestors).

Most events bubble by default. Notable exceptions that do not bubble: focus, blur, load, mouseenter, mouseleave.

javascript 
// HTML structure:
// <div id="grandparent">
//   <div id="parent">
//     <button id="child">Click me</button>
//   </div>
// </div>

document.getElementById("grandparent").addEventListener("click", () =>
  console.log("3. Grandparent fired")
);
document.getElementById("parent").addEventListener("click", () =>
  console.log("2. Parent fired")
);
document.getElementById("child").addEventListener("click", () =>
  console.log("1. Child fired")
);

// When you click the button, output order is:
// "1. Child fired"      ← event starts at the target
// "2. Parent fired"     ← bubbles up
// "3. Grandparent fired"← keeps going up

// event.target vs event.currentTarget during bubbling
document.getElementById("parent").addEventListener("click", (e) => {
  e.target        // always the #child (where event ORIGINATED)
  e.currentTarget // #parent (where THIS listener is attached)
});
Q.06 How can you stop event propagation or event bubbling in JS? ⭐ Key Concept ▼

Sometimes you need to prevent an event from bubbling up to parent elements — for example, when clicking inside a dropdown should not also trigger the document-level "click outside to close" handler. JavaScript provides two methods on the event object for this:

  • event.stopPropagation() — stops the event from bubbling further up the DOM. Listeners on ancestor elements will NOT fire. However, if there are multiple listeners on the same element for the same event, they all still fire.
  • event.stopImmediatePropagation() — stops propagation AND prevents any other listeners on the same element from firing. The most complete stop.

Use with caution — stopping propagation can break features like event delegation, analytics, or accessibility libraries that rely on events bubbling to the top.

javascript 
// stopPropagation — stops bubbling, other listeners on same el still fire
document.getElementById("child").addEventListener("click", (e) => {
  e.stopPropagation(); // parent and grandparent listeners won't fire
  console.log("Child clicked — stopped here");
});

// Parent listener — never fires after stopPropagation
document.getElementById("parent").addEventListener("click", () => {
  console.log("This never runs"); // blocked ✋
});

// stopImmediatePropagation — stops EVERYTHING
const btn = document.querySelector("#btn");
btn.addEventListener("click", (e) => {
  e.stopImmediatePropagation();
  console.log("First listener — stops everything");
});
btn.addEventListener("click", () => {
  console.log("Second listener — never runs"); // also blocked
});

// Real-world: dropdown stays open when clicking inside it
const dropdown = document.querySelector(".dropdown");
dropdown.addEventListener("click", (e) => {
  e.stopPropagation(); // prevent document click from closing dropdown
});
document.addEventListener("click", () => {
  dropdown.classList.remove("open"); // close dropdown when clicking outside
});

// Difference summary:
// stopPropagation()          → stops bubbling to PARENTS only
// stopImmediatePropagation() → stops bubbling + other listeners on SAME element
// preventDefault()           → does NOT stop bubbling — only stops default action
Q.07 What is Event Capturing in JS? ▼

Event Capturing (also called the capture phase) is the opposite of bubbling. While bubbling travels upward from child to parent, capturing travels downward from the root (window) toward the target element.

The full lifecycle of any DOM event has three phases:

  • 1. Capture phase — event travels DOWN from window → document → html → body → ... → parent → target.
  • 2. Target phase — event reaches the actual element that was clicked/interacted with.
  • 3. Bubble phase — event travels back UP from target → parent → ... → window.

By default, addEventListener listens in the bubble phase. To listen in the capture phase, pass true (or { capture: true }) as the third argument. Capture listeners fire before bubble listeners.

javascript 
// addEventListener third argument: true = capture, false = bubble (default)

document.getElementById("grandparent").addEventListener("click", () =>
  console.log("Grandparent CAPTURE"), true   // capture phase
);
document.getElementById("parent").addEventListener("click", () =>
  console.log("Parent CAPTURE"), true
);
document.getElementById("child").addEventListener("click", () =>
  console.log("Child TARGET")       // default = bubble/target phase
);
document.getElementById("parent").addEventListener("click", () =>
  console.log("Parent BUBBLE")       // default = bubble phase
);

// Clicking #child fires in this order:
// 1. "Grandparent CAPTURE" ← capture goes DOWN
// 2. "Parent CAPTURE"      ← capture goes DOWN
// 3. "Child TARGET"        ← at the target
// 4. "Parent BUBBLE"       ← bubble goes UP

// Options object form (more readable)
el.addEventListener("click", handler, { capture: true });
el.addEventListener("click", handler, { capture: false }); // default

// stopPropagation in capture phase stops event before reaching target
document.getElementById("parent").addEventListener("click", (e) => {
  e.stopPropagation(); // child never receives the event
}, true);
Q.08 What is the purpose of event.preventDefault() in JS? ⭐ Key Concept ▼

event.preventDefault() tells the browser: "do not perform your default built-in action for this event." It does not stop the event from bubbling — it only cancels what the browser would normally do automatically.

Every interactive HTML element has default behaviours built into the browser:

  • A form's submit event → page reload / navigation
  • An anchor <a> click → navigate to the href
  • A checkbox click → check/uncheck toggle
  • keydown on a form input → typing characters in the field
  • Right-click → show context menu

Calling preventDefault() suppresses that default and lets your JavaScript take full control.

javascript 
// 1. Prevent form page reload — handle submission in JS
const form = document.querySelector("form");
form.addEventListener("submit", (e) => {
  e.preventDefault(); // stop page from reloading
  const data = new FormData(form);
  sendToAPI(data);     // handle it with JS instead
});

// 2. Prevent link navigation — handle with JS router
document.querySelector("a").addEventListener("click", (e) => {
  e.preventDefault();  // stop navigating to href
  navigateTo(e.target.href); // use SPA router instead
});

// 3. Prevent right-click context menu
document.addEventListener("contextmenu", (e) => {
  e.preventDefault();  // hide browser menu
  showCustomMenu(e.clientX, e.clientY); // show custom one
});

// 4. Prevent specific keys in an input
input.addEventListener("keydown", (e) => {
  if (e.key === "e") e.preventDefault(); // prevent 'e' in number field
});

// 5. Prevent drag default to enable custom drop zones
dropzone.addEventListener("dragover", (e) => {
  e.preventDefault(); // REQUIRED to allow dropping
});

// Check if an event can be prevented
event.cancelable // true/false — only call preventDefault if true

// Important distinction:
// preventDefault()  → cancels browser's default action, bubbling continues
// stopPropagation() → stops bubbling, default action still happens
// Both together     → cancels default AND stops bubbling
Q.09 What is the use of "this" keyword in the context of event handling? ⭐ Key Concept ▼

Inside an event handler, the value of this depends on how the handler function was defined:

  • Regular function handler — this refers to the element the event listener is attached to (event.currentTarget). This is the classic, intuitive behaviour.
  • Arrow function handler — arrow functions do NOT have their own this. They inherit it from the surrounding lexical scope (usually window or a class instance). Inside an arrow handler, this is NOT the element.

Use event.target or event.currentTarget as the safe, explicit alternative to this in event handlers — they always refer to the expected elements regardless of function type.

javascript 
const btn = document.querySelector("button");

// Regular function — 'this' is the element (currentTarget)
btn.addEventListener("click", function(e) {
  console.log(this);           // <button> element ✅
  console.log(this === e.currentTarget); // true
  this.style.color = "red";   // works — styles the button
  this.textContent = "Clicked!";// works
});

// Arrow function — 'this' is NOT the element
btn.addEventListener("click", (e) => {
  console.log(this);           // window (or undefined in strict mode) ⚠️
  this.style.color = "red";   // TypeError — 'this' is not the button!
  e.currentTarget.style.color = "red"; // ✅ use event object instead
});

// In a class — arrow function or bind needed to keep 'this' as instance
class Counter {
  constructor() {
    this.count = 0;
    const btn = document.querySelector("#counter-btn");

    // Regular function loses 'this' — broken
    // btn.addEventListener("click", function() { this.count++; }); ❌

    // Fix 1: bind 'this' explicitly
    btn.addEventListener("click", this.increment.bind(this)); // ✅

    // Fix 2: arrow function inherits class 'this'
    btn.addEventListener("click", () => this.increment()); // ✅
  }
  increment() { this.count++; console.log(this.count); }
}
Q.10 How to remove an event handler from an element in JS? ▼

You remove an event listener using removeEventListener(event, handler). The critical requirement is that the handler reference must be exactly the same function that was passed to addEventListener. This is why anonymous functions and inline arrow functions cannot be removed — every time you write () => {}, it creates a new function object with a different reference.

For situations where you want a listener to fire only once, the { once: true } option is a cleaner solution than manually removing it.

javascript 
const btn = document.querySelector("button");

// ✅ Named function — CAN be removed
function handleClick() {
  console.log("clicked");
}
btn.addEventListener("click", handleClick);
btn.removeEventListener("click", handleClick); // ✅ same reference

// ❌ Anonymous function — CANNOT be removed
btn.addEventListener("click", () => console.log("clicked"));
btn.removeEventListener("click", () => console.log("clicked"));
// ❌ Different arrow function instance — remove does nothing!

// ✅ Store reference to arrow function if you need to remove it
const handler = () => console.log("clicked");
btn.addEventListener("click", handler);
btn.removeEventListener("click", handler); // ✅ same reference

// ✅ { once: true } — auto-removes after firing once (cleanest)
btn.addEventListener("click", () => console.log("fires only once!"), {
  once: true
});

// { once: true } is equivalent to:
function onceHandler() {
  console.log("fires only once!");
  btn.removeEventListener("click", onceHandler); // self-remove
}
btn.addEventListener("click", onceHandler);

// Full options object
btn.addEventListener("click", handler, {
  once:    true,  // remove after first fire
  capture: false, // bubble phase (default)
  passive: true,  // won't call preventDefault (perf hint for scroll/touch)
});

// Note: capture mode must match when removing
btn.addEventListener("click", handler, true);  // capture
btn.removeEventListener("click", handler, true); // must pass true here too

Q.01 Explain the concept of Lexical Scoping. ⭐ Key Concept ▼

Lexical scoping means that the scope of a variable is determined by where it is written in the source code — specifically, which function or block it is declared in. The word "lexical" refers to the text of the code itself. This is also called static scoping because the scope is decided at write-time, not at run-time.

The rule is simple: a function can access variables from its own scope, and from any outer scope that wraps it — all the way up to the global scope. This creates a chain of scopes called the scope chain. When JavaScript looks up a variable, it starts in the current scope and walks outward up the chain until it finds it (or throws a ReferenceError).

Lexical scoping is the foundation upon which closures are built. Because scope is determined by where functions are defined — not where they are called — inner functions retain access to outer variables even when executed in a completely different context.

javascript 
// Lexical scope — determined by POSITION in code
const globalVar = "I'm global";

function outer() {
  const outerVar = "I'm in outer";

  function middle() {
    const middleVar = "I'm in middle";

    function inner() {
      const innerVar = "I'm in inner";

      // inner can access ALL outer scopes — lexical chain
      console.log(innerVar);  // ✅ own scope
      console.log(middleVar); // ✅ parent scope
      console.log(outerVar);  // ✅ grandparent scope
      console.log(globalVar); // ✅ global scope
    }
    inner();

    // middle CANNOT access inner's variables
    // console.log(innerVar); ❌ ReferenceError
  }
  middle();
}
outer();

// Scope is set at DEFINITION — not where the function is CALLED
const city = "Mumbai";

function printCity() {
  console.log(city); // uses 'city' from where it was DEFINED
}

function runInDifferentContext() {
  const city = "Delhi"; // local city — does NOT affect printCity
  printCity();          // still logs "Mumbai" — lexical, not dynamic
}

runInDifferentContext(); // "Mumbai" — not "Delhi"
Q.02 What is a Closure? ⭐ Key Concept ▼

A closure is a function that remembers and retains access to variables from its outer (enclosing) scope even after that outer function has finished executing and its execution context has been removed from the call stack.

In other words — normally, when a function returns, its local variables are garbage collected. But if an inner function was defined inside it and that inner function is still referenced somewhere, the inner function forms a closure over the outer variables. Those variables are kept alive in memory as long as the closure exists.

Every function in JavaScript forms a closure over the scope in which it was created. Closures are not a special syntax — they are a natural consequence of lexical scoping + first-class functions working together.

javascript 
// Classic closure example
function makeCounter() {
  let count = 0; // local to makeCounter — normally would be discarded

  return function increment() {
    count++;          // inner function closes over 'count'
    return count;
  };
}

const counter = makeCounter(); // makeCounter has returned, but count lives on
counter(); // 1
counter(); // 2 — count is remembered between calls!
counter(); // 3

// Each call to makeCounter creates a NEW independent closure
const counterA = makeCounter();
const counterB = makeCounter();
counterA(); // 1 — counterA's own count
counterA(); // 2
counterB(); // 1 — counterB has its own separate count
// They do not share state ✅

// Another example: closure over a parameter
function greetWith(greeting) {
  return (name) => `${greeting}, ${name}!`; // closes over 'greeting'
}

const sayHello   = greetWith("Hello");
const sayNamaste = greetWith("Namaste");

sayHello("Arun");    // "Hello, Arun!"
sayNamaste("Raj");  // "Namaste, Raj!"
// 'greeting' was the parameter — closed over and remembered
Q.03 What are the benefits of Closures? ⭐ Key Concept ▼

Closures are one of the most powerful features in JavaScript and enable several important patterns used in real-world applications:

  • Data privacy / Encapsulation — variables inside a closure are not accessible from outside. This simulates private state without needing classes.
  • Stateful functions — closures can remember state between calls without using global variables. Counters, accumulators, caches all benefit from this.
  • Partial application / Currying — pre-fill some arguments and return a specialised function. Closures make this possible.
  • Module pattern — closures are the foundation of the module pattern — keeping implementation details private while exposing a clean public API.
  • Callbacks and async code — closures let callbacks access variables from the scope they were created in, even when called later.
  • Memoization — closures can cache expensive computation results in a private map for fast re-use.
javascript 
// 1. DATA PRIVACY — no one outside can touch 'balance'
function createBankAccount(initialBalance) {
  let balance = initialBalance; // private — inaccessible from outside

  return {
    deposit(amount)    { balance += amount; return balance; },
    withdraw(amount)   { balance -= amount; return balance; },
    getBalance()       { return balance; }
  };
}
const account = createBankAccount(1000);
account.deposit(500);   // 1500
account.balance;         // undefined — truly private ✅

// 2. STATEFUL FUNCTIONS — counter remembers its value
const counter = (() => {
  let n = 0;
  return { inc: () => ++n, dec: () => --n, val: () => n };
})(); // IIFE — runs immediately
counter.inc(); counter.inc(); counter.dec();
counter.val(); // 1

// 3. PARTIAL APPLICATION — pre-fill arguments
function multiply(factor) {
  return (n) => n * factor; // closes over 'factor'
}
const double = multiply(2);
const triple = multiply(3);
double(5); // 10,  triple(5) = 15

// 4. MEMOIZATION — cache expensive results
function memoize(fn) {
  const cache = {}; // private to the closure
  return function(...args) {
    const key = JSON.stringify(args);
    if (cache[key]) return cache[key]; // return cached result
    cache[key] = fn(...args);          // compute and cache
    return cache[key];
  };
}
const slowSquare = (n) => { /* expensive */ return n * n; };
const fastSquare = memoize(slowSquare);
fastSquare(10); // computed
fastSquare(10); // returned from cache instantly
Q.04 What is the concept of Encapsulation in the context of closures? ⭐ Key Concept ▼

Encapsulation is the OOP principle of bundling data and the functions that operate on it together, while hiding the internal details from the outside world. In class-based languages like Java, this is done with private keywords. In JavaScript, closures achieve the same thing naturally — variables inside a closure are simply inaccessible from outside, making them effectively private.

The Module Pattern is the classic closure-based encapsulation technique: you use an IIFE to create a private scope, define your private state and helper functions inside it, and return only the public API you want to expose. This gives you full control over what is visible and what is hidden.

javascript 
// Module Pattern — closure-based encapsulation
const ShoppingCart = (() => {
  // PRIVATE — not accessible outside
  let items = [];
  let discountCode = null;

  function _calculateTotal() { // private helper (convention: _ prefix)
    return items.reduce((sum, item) => sum + item.price * item.qty, 0);
  }

  // PUBLIC API — only what we choose to expose
  return {
    addItem(item) {
      items.push(item);
      console.log(`Added: ${item.name}`);
    },
    removeItem(name) {
      items = items.filter(i => i.name !== name);
    },
    applyDiscount(code) {
      discountCode = code; // modifies private state
    },
    getTotal() {
      let total = _calculateTotal(); // calls private helper
      return discountCode === "SAVE10" ? total * 0.9 : total;
    },
    getItems() { return [...items]; } // return copy, not reference
  };
})();

ShoppingCart.addItem({ name: "Shirt", price: 500, qty: 2 });
ShoppingCart.applyDiscount("SAVE10");
ShoppingCart.getTotal();   // 900 (10% off)

// Private state — truly inaccessible from outside
ShoppingCart.items;          // undefined ✅ private
ShoppingCart.discountCode;   // undefined ✅ private
ShoppingCart._calculateTotal; // undefined ✅ private
Q.05 What are the disadvantages or limitations of Closures? ▼

While closures are extremely useful, they come with trade-offs that can cause real problems if you are not careful:

  • Memory consumption — closed-over variables are kept alive in memory for as long as the closure exists. If many closures are created (e.g., inside a loop adding event listeners), all those scopes stay in memory simultaneously.
  • Memory leaks — if a closure inadvertently holds a reference to a large object or a DOM element, the garbage collector cannot reclaim that memory even if you no longer need it. This is a common source of memory leaks in long-running SPAs.
  • Performance overhead — accessing variables through the scope chain takes slightly longer than accessing local variables, because the engine has to walk up the chain.
  • The loop-with-var bug — one of the most classic JS mistakes. Using var in a loop with closures results in all closures sharing the same final value of the loop variable, not the value at each iteration.
  • Debugging difficulty — closures can make it harder to trace where a variable comes from, especially in deeply nested functions.
javascript 
// ── THE CLASSIC LOOP BUG ─────────────────────────────────
// var is function-scoped — all callbacks share the SAME 'i'
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Output: 3, 3, 3  — all print 3 (loop is done by the time callbacks run)

// Fix 1: use let (block-scoped — each iteration gets its own 'i')
for (let i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 100);
}
// Output: 0, 1, 2  ✅

// Fix 2: IIFE to create a new scope for each iteration (pre-ES6 way)
for (var i = 0; i < 3; i++) {
  ((j) => { setTimeout(() => console.log(j), 100); })(i);
}
// Output: 0, 1, 2  ✅ (j captures the current i at each iteration)

// ── MEMORY LEAK — closure holds DOM reference ─────────────
function setupHandler() {
  const heavyElement = document.getElementById("bigDiv"); // large DOM node
  heavyElement.addEventListener("click", function() {
    console.log(heavyElement.id); // closure keeps heavyElement alive!
  });
  // Even if bigDiv is removed from DOM, it stays in memory
  // because the event handler closure still references it
}

// ── MEMORY — many closures accumulate ────────────────────
const handlers = [];
for (let i = 0; i < 10000; i++) {
  handlers.push(function() { return i * i; }); // 10,000 closures in memory
}
// Each closure holds its own scope — significant memory overhead
Q.06 How can you release variable references or closures from memory? ▼

JavaScript has automatic garbage collection — memory is reclaimed when a value has no more references pointing to it. To release memory held by a closure, you need to ensure nothing references the closure anymore. Here are the practical techniques:

  • Set the variable to null — this removes the reference from the variable. If no other references exist, the closure and its captured variables become eligible for garbage collection.
  • Remove event listeners — event listeners are common sources of closure memory leaks. Always call removeEventListener when a component is destroyed (or use { once: true }).
  • Clear timers — setInterval callbacks hold closure references until the interval is cleared. Always store the ID and call clearInterval.
  • Avoid capturing large objects unnecessarily — only close over what you actually need. Destructure to extract just the properties you need, rather than closing over the whole large object.
  • Use WeakRef / WeakMap — for caches that hold object references, use WeakMap so entries are automatically removed when the object key is garbage collected.
javascript 
// 1. Set variable to null — breaks the closure reference
let counter = (() => {
  let count = 0;
  return () => ++count;
})();

counter(); // 1
counter(); // 2
counter = null; // release — closure and 'count' can now be GC'd

// 2. Remove event listeners — avoids DOM memory leaks
function handleClick() { console.log("clicked"); }
btn.addEventListener("click", handleClick);
// When component unmounts or button is removed:
btn.removeEventListener("click", handleClick); // ✅ release

// 3. Clear intervals — prevent indefinite closure retention
const intervalId = setInterval(() => {
  fetchLatestData(); // closure holds fetchLatestData reference
}, 5000);
// When page unloads or component destroys:
clearInterval(intervalId); // ✅ stop the interval, release closure

// 4. Avoid capturing large objects — destructure what you need
function processUser(user) {
  // ❌ BAD — entire user object kept alive by closure
  return () => console.log(user);

  // ✅ GOOD — only the name string is captured
  const { name } = user;
  return () => console.log(name);
}

// 5. WeakMap — cache that doesn't prevent garbage collection
const cache = new WeakMap();

function computeExpensive(obj) {
  if (cache.has(obj)) return cache.get(obj);
  const result = /* expensive work */ obj.value * 2;
  cache.set(obj, result);  // when obj is GC'd, cache entry disappears too ✅
  return result;
}
Q.07 What is the difference between a Regular Function and a Closure? ⭐ Key Concept ▼

Technically, every function in JavaScript is a closure — because every function captures the scope it was defined in. But in practice, when we talk about "a closure" we mean a function that actively uses variables from an outer function's scope that has already returned. The distinction is about whether the outer scope is still alive or not.

  • Regular function — accesses only its own local variables and global variables. When it returns, its local variables are immediately discarded by the garbage collector. No retained state.
  • Closure — an inner function that accesses variables from an outer function's scope. Even after the outer function has returned, those variables persist in memory — kept alive by the inner function's reference. The closure has retained, private state.
javascript 
// REGULAR FUNCTION — no outer scope dependency
function add(a, b) {
  return a + b; // uses only its own params, no outer vars
}
add(2, 3); // 5 — a and b are gone after this call
add(2, 3); // 5 — starts fresh, no memory of previous call

// CLOSURE — inner function holds outer scope alive
function makeAdder(a) {    // outer function — defines 'a'
  return (b) => a + b;    // inner function — closes over 'a'
}
// makeAdder has returned — but 'a' is NOT gone
const add5 = makeAdder(5); // add5 remembers a=5
add5(3);  // 8  — 'a' is still 5
add5(10); // 15 — 'a' is still 5 — state persists

// Side-by-side: stateless vs stateful
function regularCounter() {
  let count = 0;
  count++;
  return count;
}
regularCounter(); // 1
regularCounter(); // 1 — always 1! count resets each call
regularCounter(); // 1

function makeClosureCounter() {
  let count = 0;            // defined once, kept alive in closure
  return () => ++count;
}
const cCounter = makeClosureCounter();
cCounter(); // 1
cCounter(); // 2 — count persists between calls ✅
cCounter(); // 3

// │ Outer scope access │ No (or global only)│ Yes — retains it     │

Q.01 What is asynchronous programming in JS? What is its use? ⭐ Key Concept ▼

JavaScript is a single-threaded language — it has one call stack and can only execute one piece of code at a time. If a task takes a long time (like fetching data from a server, reading a file, or waiting for a timer), a synchronous approach would block everything else — the UI would freeze, buttons would stop responding, and the browser would appear to hang.

Asynchronous programming is the solution. It lets JavaScript start a long-running task, then move on to other work while waiting for it to complete. When the task finishes, a callback (or Promise resolution) is scheduled to handle the result — without ever blocking the main thread.

Real-world uses:

  • Fetching data from APIs (most common)
  • Reading/writing files (Node.js)
  • Accessing databases
  • Timers and delays (setTimeout, setInterval)
  • Loading images, scripts, or resources
  • Handling user events over time
javascript 
// PROBLEM: synchronous code — blocks everything
function slowTask() {
  const start = Date.now();
  while (Date.now() - start < 3000) {} // blocks for 3 seconds!
  console.log("Done");
}
slowTask();
console.log("This waits 3 seconds"); // only runs after slowTask finishes

// SOLUTION: asynchronous code — non-blocking
setTimeout(() => {
  console.log("Done after 3s"); // runs later, when ready
}, 3000);

console.log("This runs immediately!"); // doesn't wait ✅

// Real-world: async API call — page stays responsive
async function loadUser() {
  console.log("Fetching...");
  const user = await fetch("/api/user").then(r => r.json());
  console.log("Got user:", user.name);
}
loadUser();
console.log("UI stays interactive while fetch runs"); // runs before user arrives
Q.02 What is the difference between synchronous and asynchronous programming? ⭐ Key Concept ▼

The fundamental difference is how code handles waiting:

  • Synchronous — code executes line by line, in order. Each line must fully complete before the next one starts. If a line takes 5 seconds, everything after it waits 5 seconds. Simple to understand but catastrophic for performance when dealing with slow operations like network requests.
  • Asynchronous — slow tasks are kicked off and execution immediately continues to the next line. When the slow task finishes, its callback or Promise handler runs. The main thread is never blocked. More complex to reason about, but essential for responsive applications.

The underlying mechanism that makes this possible in JavaScript is the Event Loop — it continuously checks if the call stack is empty and, if so, moves queued callbacks into the stack to execute.

javascript 
// SYNCHRONOUS — top-to-bottom, waits for each line
console.log("Step 1");
console.log("Step 2"); // always after Step 1
console.log("Step 3"); // always after Step 2
// Output: Step 1 → Step 2 → Step 3 (guaranteed order)

// ASYNCHRONOUS — kicks off task, moves on immediately
console.log("Step 1");
setTimeout(() => console.log("Step 2 (async)"), 0); // even 0ms is async!
console.log("Step 3");
// Output: Step 1 → Step 3 → Step 2 (async runs after call stack clears)

// Why even setTimeout(0) is async:
// The callback goes to the Web API, then the task queue,
// and only runs after the current synchronous code finishes.

// Real-world comparison

// Synchronous restaurant (bad): customer A orders → chef cooks → serves → then customer B can order

// Asynchronous restaurant (good): customer A orders → chef starts cooking →
// customer B can also order → food ready → serve A → serve B

// Event Loop — the mechanism behind async
// 1. Call Stack    — runs currently executing code
// 2. Web APIs      — browser handles async tasks (timers, fetch, events)
// 3. Task Queue    — completed callbacks wait here
// 4. Microtask Q   — Promise callbacks (higher priority than task queue)
// 5. Event Loop    — moves from queue to stack when stack is empty

console.log("1 sync");
setTimeout(() => console.log("3 macro-task"), 0);
Promise.resolve().then(() => console.log("2 micro-task"));
console.log("1 sync end");
// Output: "1 sync" → "1 sync end" → "2 micro-task" → "3 macro-task"
Q.03 What are the techniques for achieving asynchronous operations in JS? ⭐ Key Concept ▼

JavaScript has evolved through three generations of async techniques, each improving on the previous one's readability and error-handling:

  • Callbacks — the original approach. Pass a function to be called when the async task completes. Works, but leads to "callback hell" when nested.
  • Promises — introduced in ES6. A Promise represents a future value — it's either pending, fulfilled, or rejected. Chained with .then() and .catch(). Much cleaner than nested callbacks.
  • async / await — introduced in ES2017. Built on top of Promises but written to look synchronous. The most readable and preferred approach today.
  • setTimeout / setInterval — for delay-based or repeating async tasks.
  • Event listeners — respond asynchronously to user/browser events.
  • Generators — advanced, used with libraries like Redux-Saga for complex async flows.

  .then(data  => renderUser(data))
  .catch(err  => handleError(err))
  .finally(() => hideSpinner());

  try {
    const data = await getUserData(userId);
    renderUser(data);
  } catch (err) {
    handleError(err);
  } finally {
    hideSpinner();
  }

setTimeout(() => showToast("Saved!"), 500);
const id = setInterval(() => pollServer(), 5000);

// 5. Promise combinators — run multiple async tasks

Q.04 What is setTimeout()? How is it used to handle asynchronous operations? ⭐ Key Concept ▼

setTimeout(callback, delay) schedules a callback function to run once after a specified delay in milliseconds. It is a Web API (provided by the browser or Node.js), not part of the JavaScript language itself. The callback does not block the main thread — it is handed off to the runtime, which fires it after the delay when the call stack is free.

Important nuance: the delay is the minimum wait time, not a guaranteed exact time. If the call stack is busy when the timer fires, the callback waits in the task queue until the stack is empty.

It returns a timer ID you can use to cancel the timeout before it fires with clearTimeout(id).

const timerId = setTimeout(() => console.log("never runs"), 5000);
clearTimeout(timerId); // cancelled before 5s

// setTimeout(fn, 0) — defer to end of current call stack
console.log("1");

console.log("2");
// Output: 1 → 2 → 3

Q.05 What is setInterval()? How is it used to handle asynchronous operations? ⭐ Key Concept ▼

setInterval(callback, delay) schedules a callback to run repeatedly, every delay milliseconds, until explicitly stopped. Like setTimeout, it is non-blocking — the intervals fire in the background and the callback runs each time the call stack is clear. It also returns an ID for cancellation via clearInterval(id).

Key difference from setTimeout: setTimeout runs once. setInterval runs forever until you stop it. Always store the ID and call clearInterval when done — failing to do so is a common memory leak and performance problem.

Important caveat: if the callback takes longer than the interval delay, intervals can overlap or pile up. For such cases, use a recursive setTimeout pattern instead, which guarantees a gap between executions.

javascript

  console.log("Ticks every second");

// Stop after 5 seconds
setTimeout(() => clearInterval(id), 5000);

    display.textContent = remaining;
    remaining--;
    if (remaining < 0) {
      clearInterval(intervalId); // ✅ always stop when done
      display.textContent = "Time's up!";
    }
  }, 1000);
}

Q.06 What is the role of callbacks in fetching API data asynchronously? ⭐ Key Concept ▼

Before Promises and async/await, callbacks were the only way to handle asynchronous results like API responses. Instead, you'd pass a callback function that the runtime would call when the data arrived.

The error-first callback convention became the Node.js standard: the first argument is always the error (or null if successful), and the second is the result data. This forces you to check for errors before using the data.

    }

  xhr.onerror = () => callback(new Error("Network error"), null);
  xhr.send();
}

Q.07 What is callback hell? How can it be avoided? ⭐ Key Concept ▼

Callback hell (also called the Pyramid of Doom) is the pattern that emerges when you have multiple sequential asynchronous operations, each depending on the result of the previous one, all written as nested callbacks. The code indents further and further to the right, becoming extremely difficult to read, maintain, and debug.

It has three main problems: deeply nested code that is hard to follow, error handling that must be repeated at every level, and steps that are tightly coupled and cannot be easily reused or tested in isolation.

javascript 
// ❌ CALLBACK HELL — Pyramid of Doom
loginUser(credentials, (err, user) => {
  if (err) { handleError(err); return; }
  getProfile(user.id, (err, profile) => {
    if (err) { handleError(err); return; }
    getFriends(profile.id, (err, friends) => {
      if (err) { handleError(err); return; }
      getPosts(friends[0].id, (err, posts) => {
        if (err) { handleError(err); return; }
        renderPosts(posts); // finally!
        // ← deeply nested, error handling repeated, hard to debug
      });
    });
  });
});

// ✅ FIX 1: Promises + .then() chaining — flat structure
loginUser(credentials)
  .then(user    => getProfile(user.id))
  .then(profile => getFriends(profile.id))
  .then(friends => getPosts(friends[0].id))
  .then(posts   => renderPosts(posts))
  .catch(err   => handleError(err)); // ONE catch for all steps ✅

// ✅ FIX 2: async/await — reads like synchronous code (best)
async function loadDashboard(credentials) {
  try {
    const user    = await loginUser(credentials);
    const profile = await getProfile(user.id);
    const friends = await getFriends(profile.id);
    const posts   = await getPosts(friends[0].id);
    renderPosts(posts);
  } catch (err) {
    handleError(err); // ONE catch for all steps ✅
  }
}

// ✅ FIX 3: Named functions — break callbacks into separate named functions
function onLoginSuccess(user) { getProfile(user.id, onProfileSuccess); }
function onProfileSuccess(profile) { getFriends(profile.id, onFriendsSuccess); }
function onFriendsSuccess(friends) { renderFriends(friends); }
loginUser(credentials, onLoginSuccess); // flat, but more verbose

Q.01 What are Promises in JavaScript? ⭐ Key Concept ▼

A Promise is an object that represents the eventual completion (or failure) of an asynchronous operation and its resulting value. Think of it like a receipt you get at a restaurant — you don't have your food yet, but you have a promise that it will arrive. You can specify what to do when it arrives (success) or if it never comes (failure), and go do other things in the meantime.

A Promise is always in one of three states:

  • Pending — the initial state. The async operation has started but hasn't completed yet. Neither fulfilled nor rejected.
  • Fulfilled (Resolved) — the operation completed successfully. The Promise now has a value, and any .then() handlers are called.
  • Rejected — the operation failed. The Promise has a reason (error), and any .catch() handlers are called.

Once a Promise settles (fulfilled or rejected), it is immutable — its state and value can never change again.

javascript 
// A Promise has three states
// pending → fulfilled OR pending → rejected

// Creating a Promise
const myPromise = new Promise((resolve, reject) => {
  // executor function runs immediately and synchronously
  const success = true;

  if (success) {
    resolve("Operation succeeded!"); // fulfills the promise
  } else {
    reject(new Error("Something went wrong")); // rejects the promise
  }
});

// Consuming the Promise
myPromise
  .then(result  => console.log("Success:", result))  // fulfilled handler
  .catch(error  => console.log("Error:", error.message)) // rejection handler
  .finally(()  => console.log("Always runs"));          // always runs

// Promise state is immutable — once settled, cannot change
const settled = new Promise((resolve, reject) => {
  resolve("first");  // ← this one counts
  resolve("second"); // ignored — already settled
  reject("error");   // ignored — already settled
});
settled.then(v => console.log(v)); // "first" — only ever resolves once

// Quick static Promise helpers
Promise.resolve("immediate value"); // already-fulfilled promise
Promise.reject(new Error("fail"));   // already-rejected promise
Q.02 How to implement Promises in JavaScript? ⭐ Key Concept ▼

Implementing Promises involves two steps: creating them (producer side) and consuming them (consumer side). You create a Promise by passing an executor function to new Promise(). You consume it with .then(), .catch(), .finally(), or await.

Key implementation details:

  • .then(onFulfilled, onRejected) — takes up to two callbacks. The first runs on success, the second on rejection. Usually you only pass the first and use a separate .catch().
  • Chaining — .then() always returns a new Promise, enabling flat chaining instead of nesting. The value returned from a .then() callback becomes the resolved value of the next Promise in the chain.
  • .catch() is shorthand for .then(null, onRejected). It also catches errors thrown inside any preceding .then().
  • .finally() runs regardless of outcome. It receives no value and its return value is ignored.
javascript 
// ── CREATING a Promise ────────────────────────────────────

  return new Promise((resolve, reject) => {
    setTimeout(() => { // simulate network delay
      if (id > 0) {
        resolve({ id, name: "Arun", role: "admin" });
      } else {
        reject(new Error("Invalid user ID"));
      }
    }, 1000);
  });
}

// ── CONSUMING with .then() / .catch() / .finally() ───────
fetchUser(1)
  .then(user => {
    console.log("Got user:", user.name);
    return user.role;           // pass value to next .then()
  })
  .then(role => {
    console.log("Role is:", role); // receives "admin"
  })
  .catch(err => {
    console.error("Failed:", err.message); // catches any error above
  })
  .finally(() => {
    hideSpinner(); // always runs
  });

// ── CONSUMING with async/await (preferred) ────────────────
async function loadUser() {
  try {
    const user = await fetchUser(1);
    console.log("Got user:", user.name);
  } catch (err) {
    console.error("Failed:", err.message);
  } finally {
    hideSpinner();
  }
}

  .then(posts   => fetchComments(posts[0].id)) // chained

Q.03 When to use Promises in real applications? ⭐ Key Concept ▼

Promises are the right tool whenever your code needs to wait for something that takes time, without blocking the rest of the application. In modern JavaScript, you'll encounter and use Promises constantly in real applications. Here are the most common scenarios:

  • API / HTTP requests — every fetch() call returns a Promise. This is the most frequent use case — getting data from a backend server.
  • Sequential async steps — when step B depends on the result of step A (login → get profile → get posts).
  • Parallel async operations — when you want to run multiple independent async tasks simultaneously and wait for all of them with Promise.all().
  • Database operations — in Node.js, any query to a database returns a Promise.
  • File system operations — reading/writing files in Node.js (fs.promises).
  • Timeouts as Promises — wrapping setTimeout in a Promise to use with await.
  • Any callback-based API — you can "promisify" old callback APIs to work with modern Promise/async patterns.

Q.04 What is the use of Promise.all() method? ⭐ Key Concept ▼

Promise.all(iterable) takes an array (or iterable) of Promises and returns a single new Promise that:

  • Resolves when all of the input Promises have resolved. The resolved value is an array of all the individual results, in the same order as the input (regardless of which resolved first).
  • Rejects immediately (fail-fast) if any one of the input Promises rejects. The rejection reason is the first error encountered. Other Promises continue running but their results are discarded.

Why it matters for performance: running async tasks one by one with await in sequence means they run serially — each waits for the previous one. Promise.all() runs them all in parallel, cutting total time to roughly the slowest single task.

javascript 
// Basic usage
const p1 = fetch("/api/users").then(r => r.json());
const p2 = fetch("/api/products").then(r => r.json());
const p3 = fetch("/api/orders").then(r => r.json());

const [users, products, orders] = await Promise.all([p1, p2, p3]);
// All 3 run in parallel — total time ≈ slowest single request

// vs sequential await — much slower
const usersSeq    = await fetch("/api/users").then(r => r.json());    // 1s
const productsSeq = await fetch("/api/products").then(r => r.json()); // +1s
const ordersSeq   = await fetch("/api/orders").then(r => r.json());   // +1s
// Total: 3 seconds. Promise.all would take ~1 second ✅

// Fail-fast behaviour — one rejection fails ALL
const safe1 = Promise.resolve("ok");
const fails = Promise.reject(new Error("Network error"));
const safe2 = Promise.resolve("also ok");

try {
  await Promise.all([safe1, fails, safe2]);
} catch (err) {
  console.log(err.message); // "Network error" — entire all() rejects
}

// Results keep the same ORDER as input, regardless of resolution order
const slow = new Promise(res => setTimeout(() => res("slow"), 2000));
const fast = new Promise(res => setTimeout(() => res("fast"), 500));

const [r1, r2] = await Promise.all([slow, fast]);
console.log(r1, r2); // "slow", "fast" — order matches INPUT not resolution time

// When ALL should succeed — use Promise.all
// When some can fail — use Promise.allSettled instead
const results = await Promise.allSettled([p1, p2, p3]);
results.forEach(r => {
  if (r.status === "fulfilled") console.log("✅", r.value);
  else                           console.log("❌", r.reason);
});
Q.05 What is the use of Promise.race() method? ⭐ Key Concept ▼

Promise.race(iterable) takes an array of Promises and returns a new Promise that settles (resolves or rejects) as soon as the first Promise in the array settles — whichever comes first wins the "race". The result of the other Promises is completely ignored.

This is useful when you want the fastest result, need to implement timeouts on async operations, or want to retry a request across multiple sources and use whichever responds first.

Unlike Promise.any(), race() settles on the first result whether it's a resolve or a rejection — so if the first to settle is a rejection, the race Promise also rejects.

javascript 
// Basic race — first to settle wins
const slow = new Promise(res => setTimeout(() => res("slow"), 3000));
const fast = new Promise(res => setTimeout(() => res("fast"), 500));
const mid  = new Promise(res => setTimeout(() => res("mid"),  1500));

const winner = await Promise.race([slow, fast, mid]);
console.log(winner); // "fast" — resolved at 500ms first

// Most important use case: TIMEOUT pattern
// If fetch takes longer than 5s, reject with a timeout error
function withTimeout(promise, ms) {
  const timeout = new Promise((_, reject) =>
    setTimeout(() => reject(new Error(`Timed out after ${ms}ms`)), ms)
  );
  return Promise.race([promise, timeout]);
}

try {
  const data = await withTimeout(
    fetch("/api/heavy-data").then(r => r.json()),
    5000  // 5 second limit
  );
  render(data);
} catch (err) {
  console.log(err.message); // "Timed out after 5000ms" if fetch was slow
}

// Race with rejection — FIRST to settle, even if it's a rejection
const willFail   = new Promise((_, rej) => setTimeout(() => rej("error"), 100));
const willSucceed = new Promise(res => setTimeout(() => res("ok"), 500));

try {
  await Promise.race([willFail, willSucceed]);
} catch (err) {
  console.log("Race rejected:", err); // "error" — rejection won the race
}
// Use Promise.any() if you want to ignore rejections and wait for first SUCCESS
Q.06 What is the difference between Promise.all() and Promise.race()? ⭐ Key Concept ▼

Both Promise.all() and Promise.race() accept an array of Promises and return a single Promise, but they differ completely in when they settle and what they resolve to:

  • Promise.all() — waits for all to finish. Resolves with an array of all results. Rejects immediately if any one fails (fail-fast). Use when every result is needed and all must succeed.
  • Promise.race() — settles as soon as any one settles, whether it resolves or rejects. The result of the others is ignored. Use for timeouts or taking the fastest result.

It's also worth knowing the full family of Promise combinators in JS:

  • Promise.all() — wait for all, fail on any rejection.
  • Promise.allSettled() — wait for all, never fails, gives each result status.
  • Promise.race() — first to settle (resolve OR reject) wins.
  • Promise.any() — first to resolve wins. Rejects only if ALL reject.
javascript 
const p1 = new Promise(res => setTimeout(() => res("A"), 1000));
const p2 = new Promise(res => setTimeout(() => res("B"), 2000));
const p3 = new Promise(res => setTimeout(() => res("C"), 500));

// Promise.all — resolves at ~2000ms (waits for all)
const allResult = await Promise.all([p1, p2, p3]);
console.log(allResult); // ["A", "B", "C"] — ALL results, in input order

// Promise.race — resolves at ~500ms (first wins)
const raceResult = await Promise.race([p1, p2, p3]);
console.log(raceResult); // "C" — only first result, p1 & p2 ignored

// Fail-fast comparison
const good = Promise.resolve("ok");
const bad  = Promise.reject(new Error("fail"));

// Promise.all — rejects immediately because bad rejected
await Promise.all([good, bad]); // ❌ throws "fail"

// Promise.race — rejects because bad settled first (if it's fastest)
await Promise.race([bad, good]); // ❌ throws "fail" (bad was first in array)

// ── Full combinator cheat sheet ────────────────────────────────────────────
// Promise.all([...])        → ✅ all resolve → array | ❌ any rejects → fail-fast
// Promise.allSettled([...]) → ✅/❌ all settle → [{status, value/reason}]
// Promise.race([...])       → first to settle (✅ or ❌) wins
// Promise.any([...])        → first to ✅ resolve wins | ❌ all reject → AggregateError

// When to use which:
// "I need everything to work"          → Promise.all()
// "I want everything but some may fail"→ Promise.allSettled()
// "I want the fastest, even if fail"   → Promise.race()
// "I want the fastest SUCCESS"         → Promise.any()

Q.01 What is the purpose of async/await? Compare it with Promises. ⭐ Key Concept ▼

async/await is syntactic sugar built on top of Promises. It does not replace Promises — it makes working with them far more readable by letting asynchronous code look and behave like synchronous code. Under the hood, every async function returns a Promise, and await simply pauses the function's execution until the awaited Promise settles.

Why async/await over raw Promises:

  • Readability — sequential async steps read top-to-bottom like normal code, instead of a chain of .then() callbacks.
  • Error handling — use familiar try/catch/finally instead of .catch(), and the same block covers both synchronous throws and async rejections.
  • Debugging — stack traces are cleaner and more meaningful. Breakpoints work intuitively inside async functions.
  • Conditionals and loops — writing if/else or loops around async operations is straightforward with await; doing the same with .then() chains is awkward.
javascript 
// Same logic — three styles side by side

// 1. Callbacks (old) — deeply nested, error-prone

  if (err) return handleError(err);
  getProfile(user.id, (err, profile) => {
    if (err) return handleError(err);
    render(profile);
  });
});

// 2. Promises — flat but still verbose
loginUser(creds)
  .then(user    => getProfile(user.id))
  .then(profile => render(profile))
  .catch(err   => handleError(err));

// 3. async/await — reads like sync, handles errors naturally ✅
async function loadDashboard(creds) {
  try {
    const user    = await loginUser(creds);
    const profile = await getProfile(user.id);
    render(profile);
  } catch (err) {
    handleError(err);
  }
}

// async/await with conditionals — impossible to express cleanly with .then()
async function loadContent(userId) {
  const user = await fetchUser(userId);
  let data;
  if (user.role === "admin") {
    data = await fetchAdminData();  // conditional await
  } else {
    data = await fetchUserData(userId);
  }
  return render(data);
}

// async fn ALWAYS returns a Promise — even if you return a plain value
async function getNumber() { return 42; }
getNumber().then(n => console.log(n)); // 42 — wrapped in Promise automatically
Q.02 Explain the use of async and await keywords in JS. ⭐ Key Concept ▼

The two keywords work together but have distinct roles:

  • async — placed before a function declaration or expression. It does two things: (1) it guarantees that the function always returns a Promise, and (2) it enables the use of await inside that function. If the function returns a plain value, it's automatically wrapped in Promise.resolve(). If it throws, the returned Promise rejects with that error.
  • await — placed before a Promise (or any thenable). It pauses the execution of the async function until that Promise settles. When the Promise resolves, await unwraps its value. If it rejects, await throws the rejection reason as an error. Importantly, while the function is "paused", the JavaScript event loop is free to handle other tasks — it is not a real block.
javascript 
// async — makes function return a Promise
async function greet() {
  return "Hello!"; // plain value auto-wrapped in Promise.resolve()
}
const result = await greet(); // "Hello!" — unwrapped by await
greet() instanceof Promise;  // true — always returns a Promise

// Works on any function syntax
const arrowAsync   = async () => "arrow";
const exprAsync    = async function() { return "expression"; };
class MyClass { async method() { return "class method"; } }

// await — pauses function until Promise settles
async function fetchData() {
  console.log("Before await");

  const response = await fetch("/api/data"); // pauses here
  // Event loop handles other tasks while waiting

  console.log("After await");  // resumes when fetch resolves
  const data = await response.json(); // await again for body parsing
  return data;
}

// await can be used on anything — even non-Promises (no-op)

const y = await Promise.resolve("hi"); // "hi"

const b = await fetch("/api/b").then(r => r.json()); // waits another 1s

  fetch("/api/b").then(r => r.json()),

// Total: ~1s ✅ both run simultaneously

// Top-level await (ES2022) — usable at module top level without async wrapper

Q.03 Can we use async keyword without await and vice versa? ⭐ Key Concept ▼

These are two separate but related questions, and the answers are different:

  • async without await — allowed, but usually pointless for async operations. The function still returns a Promise (wrapping whatever it returns), but there is no pausing — everything inside runs synchronously in order. You'd typically only do this when you want to ensure a function's return type is always a Promise, or when using top-level async wrappers.

Q.04 How do you handle errors in async/await functions? ⭐ Key Concept ▼

When a Promise that is being awaited rejects, it throws the rejection as an error inside the async function — exactly like a throw statement. This means you can handle it with the same try/catch/finally you already know. There are several patterns for async error handling, each useful in different situations:

  • try/catch/finally — the most common. Wraps one or more await calls. Single catch for all errors in the block. finally always runs for cleanup.
  • Per-await .catch() — attach .catch() directly to each awaited Promise. Useful when different steps need different error handling, or when you want to provide a fallback value for a specific step without failing the whole function.
  • Higher-order wrapper — wrap the async function in a utility that separates error and data into a two-element array (Go-style). Eliminates repetitive try/catch boilerplate.
javascript 
// ── PATTERN 1: try/catch — standard approach ─────────────
async function loadUser(id) {
  try {
    const res  = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}: ${res.statusText}`);
    const user = await res.json();
    renderUser(user);
  } catch (err) {
    // catches: network failure, non-ok status, JSON parse error, renderUser error
    console.error("loadUser failed:", err.message);
    showError("Could not load user. Please try again.");
  } finally {
    hideSpinner(); // ✅ always runs
  }
}

// ── PATTERN 2: per-step .catch() — granular control ──────
async function loadWithFallback(id) {
  // If user fetch fails, default to guest user — don't stop the function
  const user = await fetchUser(id).catch(() => ({ name: "Guest", role: "viewer" }));
  // If prefs fetch fails, use empty object as fallback
  const prefs = await fetchPrefs(id).catch(() => ({}));
  renderDashboard(user, prefs);
}

// ── PATTERN 3: error boundary wrapper (Go-style) ─────────
async function safe(promise) {
  try {
    const data = await promise;
    return [null, data];   // [error, result]
  } catch (err) {
    return [err, null];
  }
}

// Usage — clean, no try/catch scattered everywhere
async function handleSubmit() {
  const [err, user] = await safe(fetchUser(1));
  if (err) { showError(err.message); return; }

  const [saveErr] = await safe(saveUser(user));
  if (saveErr) { showError("Save failed"); return; }

  showSuccess("Saved!");
}

// ── Unhandled rejection — always handle! ─────────────────
async function unhandled() {
  await Promise.reject(new Error("whoops")); // no try/catch
}
unhandled(); // ❌ UnhandledPromiseRejection warning in Node / console error in browser

  console.error("Unhandled:", event.reason);

Q.03 What is Web Storage and its use? How many types are there? ⭐ Key Concept ▼

Web Storage is a Browser API that allows websites to store key-value data directly in the user's browser — persistently or for the duration of a session — without sending it to the server on every request (unlike cookies). It is much simpler and has a larger capacity than cookies.

Q.07 How much data can be stored in localStorage and sessionStorage? ▼

Both localStorage and sessionStorage have a storage limit of approximately 5 MB per origin in most browsers (some allow up to 10 MB). This limit is per origin (domain + protocol + port) — not per key. The limit applies to the total combined size of all keys and values stored.

Since only strings are stored, sizes can add up quickly when storing large JSON objects. The exact limit varies by browser and can also change based on user settings. If you exceed the limit, the browser throws a QuotaExceededError.

javascript 
// Storage limits (approximate, vary by browser)
// Chrome/Edge:  ~10 MB per origin for localStorage
// Firefox:      ~10 MB per origin
// Safari:       ~5 MB per origin (stricter)
// All browsers: ~5-10 MB for sessionStorage

// Safely write with QuotaExceededError handling
function safeStore(key, value) {
  try {
    localStorage.setItem(key, JSON.stringify(value));
    return true;
  } catch (err) {
    if (err.name === "QuotaExceededError") {
      console.warn("Storage full! Clearing old data...");
      localStorage.clear(); // or be selective about what to remove
      return false;
    }
    throw err;
  }
}

// Estimate how much space an item takes (approximate)
function getItemSize(key) {
  const val = localStorage.getItem(key) ?? "";
  return (key.length + val.length) * 2; // UTF-16: 2 bytes per char
}

// Estimate total localStorage usage
function getTotalStorageUsed() {
  let total = 0;
  for (let i = 0; i < localStorage.length; i++) {
    const key = localStorage.key(i);
    total += getItemSize(key);
  }
  return (total / 1024).toFixed(2) + " KB";
}
console.log(getTotalStorageUsed());

// IndexedDB can store hundreds of MB and supports binary data
Q.08 What are cookies? How do you create and read cookies? ⭐ Key Concept ▼

Cookies are small pieces of text data stored in the browser and automatically sent to the server with every HTTP request to the matching domain. This is the key difference from Web Storage — cookies travel with requests, making them the only client-side storage mechanism the server can also access.

Cookies support expiry dates, domain scoping, path restriction, and security flags (Secure, HttpOnly, SameSite). HttpOnly cookies are particularly important — they are inaccessible from JavaScript, protecting them from XSS attacks, making them the recommended way to store session tokens.

javascript 
// ── CREATE a cookie ──────────────────────────────────────
// Cookies are set via document.cookie — one at a time
document.cookie = "username=Arun";

// With expiry date (otherwise it's a session cookie)
document.cookie = "username=Arun; expires=Fri, 31 Dec 2025 23:59:59 GMT";

// Max-age in seconds (preferred over expires)
document.cookie = "username=Arun; max-age=86400"; // 1 day

// With path (default is current path)
document.cookie = "theme=dark; path=/"; // accessible site-wide

// Secure (HTTPS only) + SameSite (CSRF protection)
document.cookie = "token=abc123; secure; samesite=strict; path=/";

// ── READ cookies ─────────────────────────────────────────
// document.cookie returns ALL cookies as one string
console.log(document.cookie); // "username=Arun; theme=dark; token=abc"

// Parse a specific cookie by name
function getCookie(name) {
  const cookies = document.cookie.split("; ");
  const found   = cookies.find(c => c.startsWith(name + "="));
  return found ? found.split("=")[1] : null;
}
getCookie("username"); // "Arun"
getCookie("missing");  // null

// ── DELETE a cookie ──────────────────────────────────────
// Set max-age to 0 or expires to a past date
document.cookie = "username=; max-age=0; path=/";

// HttpOnly cookies — set by server, invisible to JS
// Response header: Set-Cookie: sessionId=abc; HttpOnly; Secure
// document.cookie → does NOT show sessionId (protected from XSS) ✅
Q.09 What is the difference between cookies and web storage? ⭐ Key Concept ▼

The most critical difference is that cookies are automatically sent to the server with every HTTP request, while web storage (localStorage/sessionStorage) stays purely in the browser and is never sent automatically. This shapes when and why you'd use each.

  • Capacity — cookies are limited to ~4 KB per cookie. Web storage holds ~5–10 MB per origin.
  • Server access — cookies are visible to the server (via request headers). Web storage is client-only.
  • Expiry — cookies can have explicit expiry dates and max-age. localStorage never expires. sessionStorage expires on tab close.
  • Security — cookies can be flagged HttpOnly (hidden from JS) and Secure. Web storage is always accessible from JS.
  • API — web storage has a clean, modern API. Cookies are managed via the awkward document.cookie string.
  • Sent automatically — cookies go with every request (including images, CSS); web storage data must be manually included when needed.
javascript 
// ┌────────────────────┬──────────────┬──────────────┬──────────────┐
// │                    │ Cookies      │ localStorage │sessionStorage│
// ├────────────────────┼──────────────┼──────────────┼──────────────┤
// │ Capacity           │ ~4 KB        │ ~5-10 MB     │ ~5-10 MB     │
// │ Sent with requests │ ✅ Auto      │ ❌ No        │ ❌ No        │
// │ Server readable    │ ✅ Yes       │ ❌ No        │ ❌ No        │
// │ JS accessible      │ ✅ (unless   │ ✅ Yes       │ ✅ Yes       │
// │                    │  HttpOnly)   │              │              │
// │ Expiry control     │ ✅ Fine-grain│ ❌ Never     │ Tab session  │
// │ HttpOnly option    │ ✅ Yes       │ ❌ No        │ ❌ No        │
// │ API quality        │ 👎 Awkward  │ 👍 Clean     │ 👍 Clean     │
// └────────────────────┴──────────────┴──────────────┴──────────────┘

// Example of automatic request sending (cookies only)
// Browser sends this automatically with every request to example.com:
// GET /api/profile HTTP/1.1
// Cookie: sessionId=abc123; theme=dark

// Web storage — must manually attach if needed
const token = localStorage.getItem("authToken");
fetch("/api/profile", {
  headers: { "Authorization": `Bearer ${token}` } // manual inclusion
});
Q.10 When to use cookies and when to use web storage? ⭐ Key Concept ▼

The decision comes down to who needs the data (server vs client) and how sensitive it is:

  • Use Cookies when: the server needs to see the data on every request — session IDs, auth tokens, CSRF tokens. Also use for data that needs an expiry time, or when you need HttpOnly protection from XSS. Cookies are the right choice for session management in server-rendered apps.
  • Use localStorage when: data only needs to be available to the client (browser) and should persist long-term across sessions — user preferences (theme, language), saved drafts, cached API responses, non-sensitive user settings. Any data a SPA manages client-side.
  • Use sessionStorage when: data should only live for one tab session and reset when the user closes the tab — multi-step form data, wizard state, temporary tokens valid for one session, scroll position restoration within a single visit.
javascript 
// ✅ COOKIES — server needs it, security matters, has expiry
// Set by server: Set-Cookie: sessionId=abc123; HttpOnly; Secure; SameSite=Strict
// Use for: session IDs, auth tokens (HttpOnly), CSRF tokens, tracking

// ✅ localStorage — long-lived, client-only data
localStorage.setItem("theme",       "dark");    // user preference
localStorage.setItem("language",    "en");      // locale setting
localStorage.setItem("sidebarOpen", "true");   // UI state
localStorage.setItem("cartItems",   JSON.stringify(cart)); // shopping cart
// ⚠️ Don't store sensitive data (passwords, full credit cards) in localStorage
// JS-accessible = XSS-vulnerable

// ✅ sessionStorage — temporary, per-tab, single-visit data
sessionStorage.setItem("step",       "2");      // checkout wizard step
sessionStorage.setItem("formData",   JSON.stringify(form)); // draft form
sessionStorage.setItem("scrollPos",  "450");    // restore scroll on back
sessionStorage.setItem("tempToken",  otp);      // one-time token

// Decision checklist:
// → Does the server need this data?         → Cookie
// → Should data survive tab close?          → localStorage
// → Should data reset on tab close?         → sessionStorage
// → Is it a session ID / auth token?        → HttpOnly Cookie
// → Is it > 4KB?                            → Web Storage or IndexedDB
// → Is it non-sensitive user preference?    → localStorage
