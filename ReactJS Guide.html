React is an open-source JavaScript library created and maintained by Meta (Facebook), first released in 2013. It is used to build fast, interactive, and reusable User Interfaces (UI), primarily for single-page web applications.

It is important to note that React is a library, not a full framework — it focuses solely on the View layer of an application (the "V" in MVC). For routing, state management, and data fetching, you pair it with other libraries like React Router, Redux, or React Query.

Role of React in Software Development:

  • Component-Based Architecture: React lets you break the UI into small, isolated, reusable pieces called components. This makes large applications easier to build and maintain.
  • Efficient UI Rendering: React uses a Virtual DOM to minimize expensive real DOM manipulations, making UI updates fast and smooth.
  • Cross-Platform Reach: With React Native, the same React concepts can be used to build native mobile apps for iOS and Android.
  • Ecosystem Hub: React sits at the center of a massive ecosystem — Next.js for SSR, Remix, Gatsby, and many more frameworks are built on top of it.
jsx — simplest react component
function Greeting() {
  return <h1>Hello, World!</h1>;
}

// React renders this component into the real DOM
ReactDOM.render(<Greeting />, document.getElementById('root'));

React comes packed with several powerful features that make it stand out for building modern UIs:

  • Component-Based Architecture: Everything in React is a component — a self-contained unit of UI logic and markup. Components can be composed together to build complex UIs from simple building blocks.
  • Virtual DOM: React maintains a lightweight copy of the real DOM in memory. When state changes, React diffs the virtual DOM against the previous version and only updates the parts of the real DOM that actually changed — making it highly efficient.
  • JSX (JavaScript XML): JSX lets you write HTML-like syntax directly inside JavaScript. It makes the code more readable and expressive. Browsers don't understand JSX directly — it gets compiled to React.createElement() calls by tools like Babel.
  • Unidirectional Data Flow: Data flows in one direction — from parent to child via props. This makes data flow predictable and debugging easier.
  • Hooks: Introduced in React 16.8, Hooks like useState, useEffect, and useContext allow functional components to manage state and side effects — previously only possible in class components.
  • Rich Ecosystem: React integrates well with libraries for routing (React Router), state management (Redux, Zustand), server-side rendering (Next.js), and testing (Jest, React Testing Library).

// 3. useState Hook

DOM (Document Object Model) is a programming interface provided by the browser. When a browser loads an HTML page, it parses the HTML and creates a tree-like in-memory representation of all the elements — this tree is the DOM. JavaScript can then interact with this tree to dynamically read and manipulate the page content, structure, and styles.

Difference between HTML and DOM:

  • HTML is the raw source code — a static text file you write. It's what you see in .html files. The browser reads it once and uses it as the blueprint.
  • DOM is the live, structured object the browser builds from that HTML. It is dynamic — JavaScript can add, remove, or update nodes in the DOM without changing the original HTML file.
  • HTML is a string. The DOM is a tree of JavaScript objects (nodes) that the browser creates and maintains in memory.
  • Inspecting your page in DevTools shows the DOM, not the original HTML — you may notice differences if JavaScript has modified the page after load.
javascript — dom manipulation
// HTML: <p id="msg">Hello</p>

// DOM: JavaScript accessing and changing the live tree
const el = document.getElementById('msg');
el.textContent = 'Hello, DOM!'; // HTML file unchanged, DOM updated

The Virtual DOM (VDOM) is a lightweight, in-memory JavaScript object that is a copy/representation of the real DOM. React keeps this virtual tree in memory and uses it as an intermediary before touching the actual browser DOM.

How it works (3-step process):

  1. Render: When your component's state or props change, React creates a new Virtual DOM tree.
  2. Diff: React compares (diffs) the new VDOM tree with the previous one using a diffing algorithm to find exactly what changed.
  3. Patch: Only the changed nodes are updated in the real DOM — this targeted update is called reconciliation.

DOM vs Virtual DOM:

  • Real DOM updates are slow — any change causes reflow and repaint of the whole page or large sections of it.
  • Virtual DOM updates are fast — changes happen in JavaScript memory first, and only the minimal real DOM changes are applied.
  • The Real DOM is the browser's native tree. The Virtual DOM is React's JS object clone of that tree, living entirely in memory.
  • Directly manipulating the Real DOM (like with jQuery) can lead to performance bottlenecks in large apps. React's VDOM approach avoids this.
jsx — react's vdom in action
// Every time count changes, React:
// 1. Creates a new VDOM tree
// 2. Diffs it with the old VDOM
// 3. Updates ONLY the <p> tag in real DOM

function Counter() {
  const [count, setCount] = useState(0);
  return (
    <div>
      <p>Count: {count}</p>         // only this updates
      <button onClick={() => setCount(count + 1)}>+</button>
    </div>
  );
}

A React Component is a reusable, independent piece of UI. Think of it like a custom HTML element — you define it once and reuse it anywhere. Every React application is essentially a tree of components.

There are two types of components:

  • Functional Components — plain JavaScript functions that return JSX. This is the modern, preferred approach.
  • Class Components — ES6 classes extending React.Component. These are older and mostly seen in legacy codebases.

Main Elements of a React Component:

  • Props (Properties): Read-only data passed from a parent component to a child. Props make components dynamic and reusable.
  • State: Data that is managed within the component itself. When state changes, React re-renders the component to reflect the new UI.
  • JSX (Return value): The markup the component renders — a mix of HTML-like syntax and JavaScript expressions.
  • Hooks: Functions like useState and useEffect that let functional components tap into React features like state, lifecycle, and context.
  • Event Handlers: Functions that respond to user interactions (clicks, inputs, form submissions, etc.).
jsx — anatomy of a functional component
// Props passed in from parent
function UserCard({ name, age }) {
  // State — local data
  const [liked, setLiked] = useState(false);

  // Event handler
  const handleLike = () => setLiked(!liked);

  // JSX — what gets rendered
  return (
    <div>
      <h2>{name} — Age: {age}</h2>
      <button onClick={handleLike}>
        {liked ? '❤️ Liked' : '🤍 Like'}
      </button>
    </div>
  );
}

A Single Page Application (SPA) is a web application that loads a single HTML page and dynamically updates the content without doing a full page reload when the user navigates. Instead of the browser requesting a new HTML page from the server for every route, JavaScript intercepts navigation and swaps out only the necessary parts of the UI.

How a traditional (Multi-Page) app works: Each navigation triggers a full request-response cycle — the browser fetches a brand new HTML page from the server, and the whole page reloads.

How an SPA works: The browser downloads the app once (HTML + JS bundle). After that, all routing, rendering, and UI updates happen on the client side using JavaScript. The server is only contacted for data (via API calls).

Key characteristics:

  • Fast, app-like navigation after the initial load — no full page refreshes.
  • Client-side routing is handled by libraries like React Router.
  • Data is fetched from APIs (REST or GraphQL) and rendered dynamically.
  • Downside: Larger initial JavaScript bundle, SEO challenges (mitigated by SSR with Next.js), and the initial load can be slower.
jsx — client-side routing in an SPA
import { BrowserRouter, Route, Routes } from 'react-router-dom';

// No page reload — React Router swaps components
function App() {
  return (
    <BrowserRouter>
      <Routes>
        <Route path="/"        element={<Home />} />
        <Route path="/about"  element={<About />} />
        <Route path="/profile" element={<Profile />} />
      </Routes>
    </BrowserRouter>
  );
}

Here are 5 key advantages that make React a go-to choice for building modern UIs:

  • 1. Reusable Components: React's component model lets you build encapsulated UI pieces that manage their own logic and appearance. Once built, components like a <Button>, <Modal>, or <Card> can be reused across the entire application or even across different projects, saving time and reducing code duplication.
  • 2. Fast Rendering with Virtual DOM: React doesn't touch the real DOM directly for every change. Its diffing algorithm ensures only the elements that actually changed get updated. This results in buttery-smooth UIs even in data-heavy, frequently-updating apps.
  • 3. Strong Ecosystem & Community: React has one of the largest developer communities in the world. There's a library for almost everything — Next.js, Redux, React Query, Framer Motion, Storybook — and thousands of open-source components available via npm.
  • 4. React Native for Mobile: The knowledge you build in React translates directly to building cross-platform mobile apps with React Native. You can share business logic and sometimes even UI components between your web and mobile apps.
  • 5. Developer Tools & Debugging: The official React DevTools browser extension lets you inspect the component tree, view props/state in real time, and profile rendering performance — making debugging significantly easier.

React is powerful, but it comes with some trade-offs worth knowing — especially in interviews where balanced answers stand out:

  • It's a Library, Not a Full Framework: React only handles the UI layer. You have to independently choose and integrate solutions for routing, state management, form handling, data fetching, etc. This flexibility is great but can lead to "decision fatigue" for teams.
  • Steep Learning Curve for Beginners: Concepts like JSX, the Virtual DOM, hooks, state lifting, context, and the component lifecycle can be overwhelming when starting out. The ecosystem evolves rapidly too, making it hard to keep up.
  • SEO Challenges (with pure CSR): Since React renders on the client side by default, search engine crawlers may not index content properly. This is largely solved by frameworks like Next.js (which provides SSR/SSG), but it adds complexity.
  • Large Bundle Size: React apps can produce large JavaScript bundles that slow down the initial page load, especially on low-powered devices or slow networks. Code splitting and lazy loading help, but require extra effort.
  • Boilerplate & Over-Engineering: For simple applications, setting up React with all its tooling (Webpack/Vite, Babel, ESLint, Redux, etc.) can feel like overkill compared to plain HTML/JS or a simpler framework.

JSX (JavaScript XML) is a syntax extension for JavaScript that lets you write HTML-like markup directly inside your JS code. It is one of React's most distinctive features. Browsers cannot run JSX directly — it is compiled to plain JavaScript by tools like Babel at build time.

3 Key Roles of JSX in React:

  • 1. Readable UI Description: JSX makes it natural to describe what the UI should look like by combining markup and logic in one place. Without JSX, you'd have to write verbose React.createElement() calls for every element, which quickly becomes unreadable.
  • 2. Embedding JavaScript Expressions: Inside JSX, you can use curly braces {} to embed any valid JavaScript expression — variables, function calls, ternary operators, array maps, etc. This makes dynamic UIs very expressive.
  • 3. Compile-Time Error Detection: Since JSX is compiled by Babel, many errors (like typos in tag names or missing closing tags) are caught at build time rather than at runtime — improving developer experience and code reliability.
jsx vs compiled javascript
// What you write (JSX):
const el = <h1 className="title">Hello, {name}!</h1>;

// What Babel compiles it to:
const el = React.createElement(
  'h1',
  { className: 'title' },
  'Hello, ', name, '!'
);

// Embedding JS in JSX:
const items = ['Apple', 'Banana'];
return <ul>{items.map(i => <li key={i}>{i}</li>)}</ul>;

This is a fundamental programming concept and React is specifically designed around the declarative paradigm.

  • Imperative Syntax — You tell the computer how to do something, step by step. You describe the exact sequence of operations to reach the desired result. Traditional DOM manipulation with vanilla JavaScript is imperative.
  • Declarative Syntax — You tell the computer what you want the result to look like, and let the system figure out how to get there. React is declarative — you describe what the UI should look like for a given state, and React handles all the DOM updates.

Think of it this way: Imperative is like giving someone turn-by-turn directions. Declarative is like giving them the destination address and letting GPS figure out the route.

imperative vs declarative
// ❌ IMPERATIVE — Vanilla JS (how to do it)
const btn = document.createElement('button');
btn.textContent = 'Click me';
btn.style.color = 'white';
btn.style.background = 'blue';
document.body.appendChild(btn);
btn.addEventListener('click', () => {
  btn.textContent = 'Clicked!';
});

// ✅ DECLARATIVE — React (what it should look like)
function App() {
  const [clicked, setClicked] = useState(false);
  return (
    <button onClick={() => setClicked(true)}>
      {clicked ? 'Clicked!' : 'Click me'}
    </button>
  );
  // React figures out the DOM changes automatically
}

An Arrow Function Expression is a shorter, modern syntax for writing functions in JavaScript (introduced in ES6). In React and JSX, arrow functions are used extensively — for defining components, writing event handlers, and inline callbacks inside JSX.

The key differences from regular functions:

  • Shorter syntax: No need for the function keyword. For single-expression returns, even the return keyword and curly braces can be omitted.
  • Lexical this: Arrow functions don't have their own this binding — they inherit this from the surrounding scope. This was critical in class components where regular functions caused this binding issues in event handlers.
  • Inline event handlers in JSX: Arrow functions are commonly used directly inside JSX props to pass short callbacks without defining a separate named function.
jsx — arrow functions in react
// Regular function component
function Greet() { return <h1>Hello</h1>; }

// Arrow function component (same result)
const Greet = () => <h1>Hello</h1>;

// Arrow function with params (implicit return)
const Add = (a, b) => a + b;

// Inline arrow function as event handler in JSX
const App = () => {
  const [val, setVal] = useState('');
  return (
    <input
      value={val}
      onChange={(e) => setVal(e.target.value)} // inline arrow fn
    />
  );
};

// Arrow fn with block body (needs explicit return)
const double = (n) => {
  const result = n * 2;
  return result;
};

Understanding the boot sequence of a React app is important for interviews. Here's the step-by-step process of how React loads and displays your UI:

  1. Browser loads index.html: The browser fetches the single index.html file. This file is mostly empty — it has a <div id="root"></div> and a <script> tag pointing to the bundled JavaScript file.
  2. JavaScript bundle is executed: The browser downloads and runs the JS bundle (built by Vite or Webpack). This bundle contains all your React components, third-party libraries, and application logic.
  3. ReactDOM mounts the App: The entry file (main.jsx) calls ReactDOM.createRoot() targeting the #root div, then calls .render(<App />). This tells React to take over that DOM node.
  4. React builds the Virtual DOM: React calls your App component (and all child components it returns), constructing a complete Virtual DOM tree representing the entire UI.
  5. React writes to the real DOM: React compares the Virtual DOM against the (empty) real DOM and injects all the generated HTML elements into the #root div. The user now sees the UI.
  6. Subsequent updates via state/props: When state or props change, React re-renders affected components, diffs the new VDOM against the old one, and surgically updates only the changed parts of the real DOM.
html + jsx — the full boot chain
<!-- index.html -->
<body>
  <div id="root"></div>   <!-- React mounts here -->
  <script type="module" src="/src/main.jsx"></script>
</body>

// main.jsx — entry point
ReactDOM.createRoot(document.getElementById('root'))
        .render(<App />);

// App.jsx — root component tree
function App() {
  return (
    <div>
      <Header />   // child component
      <Main />     // child component
      <Footer />   // child component
    </div>
  );
}

React and Angular are both popular tools for building modern web UIs, but they are fundamentally different in nature and philosophy.

  • Type: React is a UI library — it only handles the view layer. Angular is a full-fledged MVC framework — it includes built-in solutions for routing, HTTP, forms, animations, dependency injection, and more, all out of the box.
  • Language: React uses plain JavaScript (or JSX). Angular is built on and requires TypeScript — a superset of JavaScript with static types.
  • Data Binding: React uses one-way data binding — data flows from parent to child via props, making data flow predictable. Angular supports two-way data binding — the UI and the model stay in sync automatically (changes in the input update the model and vice versa).
  • Learning Curve: React's API surface is smaller and easier to get started with. Angular has a steeper learning curve due to its many concepts: decorators, modules, services, directives, dependency injection, etc.
  • Performance: React uses a Virtual DOM. Angular uses real DOM with change detection (Zone.js) — both are performant, but in different ways. Angular Ivy (their renderer) is highly optimized.
  • Flexibility: React gives you freedom to choose your own libraries. Angular is opinionated and provides everything as a cohesive, integrated system.
  • Maintained by: React → Meta (Facebook). Angular → Google.
comparison — counter in react vs angular
// ⚛ React (JSX + useState hook)
function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(count+1)}>{count}</button>;
}

// 🅰 Angular (TypeScript + template)
// component.ts
@Component({ selector: 'app-counter', template: `
  <button (click)="increment()">{{count}}</button>`
})
export class CounterComponent {
  count = 0;
  increment() { this.count++; }
}

The JavaScript ecosystem has a rich variety of UI frameworks and libraries. Here are 5 notable ones beyond React:

  • 1. Angular (by Google): A full MVC framework built with TypeScript. It's opinionated and batteries-included — ideal for large enterprise applications that need a consistent structure. Uses two-way data binding and a powerful CLI.
  • 2. Vue.js (by Evan You): A progressive framework that is often considered the easiest to learn. It blends the best ideas from React (component model) and Angular (two-way binding). Great for both small and large apps. Uses .vue Single File Components.
  • 3. Svelte (by Rich Harris): A radical departure — Svelte is a compiler, not a runtime library. It compiles your components to highly optimized vanilla JavaScript at build time, resulting in zero framework overhead and very fast runtime performance.
  • 4. Next.js (by Vercel): Built on top of React, Next.js is a full-stack framework that adds Server-Side Rendering (SSR), Static Site Generation (SSG), file-based routing, API routes, and image optimization. Widely used in production.
  • 5. SolidJS (by Ryan Carniato): A newer, high-performance library with a React-like syntax but a completely different reactivity model. Instead of a Virtual DOM, Solid uses fine-grained reactivity to update only the exact DOM nodes that need to change — making it one of the fastest frameworks available.

React is a Library — specifically, a UI library. It is not a framework. This distinction is commonly asked in interviews.

The core difference lies in a concept called "Inversion of Control":

  • Library: You are in control. You call the library's functions when you need them. The library does a specific job (like building UI components) and stays out of everything else. You wire things together yourself. React is this.
  • Framework: The framework is in control. It provides the overall structure and calls your code at the right time (known as the "Hollywood Principle" — "Don't call us, we'll call you"). A framework dictates how your entire application is organized, including routing, data fetching, forms, etc. Angular and Next.js are closer to this.

What React does vs what it leaves to you:

  • ✅ React handles: Rendering UI components, managing component state and lifecycle, and efficient DOM updates via the Virtual DOM.
  • 🔧 You choose separately: Routing (React Router), global state (Redux/Zustand/Context), HTTP requests (Axios/Fetch/React Query), form validation (React Hook Form), styling (Tailwind/CSS Modules/styled-components), etc.
analogy in code
// LIBRARY — you call it when you need it
import { useState } from 'react';   // you choose when to use this
import axios from 'axios';           // you pick HTTP library yourself
import { BrowserRouter } from 'react-router-dom'; // you pick router

// FRAMEWORK (Angular) — it calls your code
// Angular modules, dependency injection, lifecycle hooks
// are all dictated BY the framework structure
@NgModule({ imports: [RouterModule, HttpClientModule] })
export class AppModule {} // framework controls the flow

Two of React's biggest design goals are Reusability and Composition — they work hand in hand and are core to how React apps are architected.

Reusability means building a component once and using it many times with different data (via props). Instead of duplicating UI code, you create a generic component and configure it wherever you need it.

Composition means assembling complex UIs by combining smaller, simpler components together — like LEGO bricks. A <Card> can contain a <Badge>, an <Avatar>, and a <Button>. None of them need to know about each other — they're composed at a higher level.

Key techniques React provides for this:

  • Props: Pass different data into the same component to produce different outputs — the foundation of reusability.
  • children prop: Lets a parent component inject arbitrary JSX into a child, enabling highly flexible "slot-like" composition.
  • Custom Hooks: Extract and reuse stateful logic across multiple components without duplicating code.
  • Higher-Order Components (HOC) / Render Props: Older patterns for sharing behaviour between components (largely replaced by hooks).
jsx — reusability and composition
// Reusable Button component
const Button = ({ label, onClick, variant = 'primary' }) => (
  <button className={variant} onClick={onClick}>{label}</button>
);

// Reuse the same Button with different props
<Button label="Save"   onClick={handleSave} />
<Button label="Delete" onClick={handleDelete} variant="danger" />

// Composition using children prop
const Card = ({ children }) => (
  <div className="card">{children}</div>
);

// Compose complex UI from simple parts
<Card>
  <Avatar src="user.jpg" />
  <h2>Jane Doe</h2>
  <Button label="Follow" onClick={handleFollow} />
</Card>

These four terms are fundamental to understanding how data flows and behaves in React applications.

  • State: State is any data that can change over time and directly affects what a component renders. When state changes, React automatically re-renders the component to reflect the new data. Examples: a counter value, a list of fetched users, whether a modal is open, a form field's current value.
  • Stateless Component: A component that does not manage any state of its own. It receives data exclusively via props and simply renders UI based on those props. Stateless components are also called pure components or dumb components — they always render the same output for the same props.
  • Stateful Component: A component that owns and manages its own state using useState (or useReducer). It controls its own data, responds to user interactions, and triggers re-renders when its state changes. Also called smart components or container components.
  • State Management: As apps grow, multiple components need to share and sync state. Passing state through props across many levels becomes messy ("prop drilling"). State management refers to patterns and tools that solve this — providing a global store of state accessible anywhere in the app. Tools include React Context API, Redux, Zustand, Jotai, and Recoil.
jsx — stateless vs stateful
// Stateless — just renders props, no internal state
const Label = ({ text }) => <span>{text}</span>;

// Stateful — owns and manages data
const Toggle = () => {
  const [on, setOn] = useState(false); // local state
  return (
    <button onClick={() => setOn(!on)}>
      {on ? 'ON' : 'OFF'}
    </button>
  );
};

// Global state management with Context (simple example)
const ThemeContext = React.createContext();
// Any nested component can consume ThemeContext
// without prop-drilling through every layer

Props (short for Properties) are the mechanism React uses to pass data from a parent component to a child component. They are the primary way components communicate and share data in React.

Key characteristics of Props:

  • Read-only / Immutable: A child component must never modify its own props. Props are owned by the parent — if data needs to change, the parent updates its state and passes new props down. This enforces React's one-way data flow.
  • Any data type: Props can be strings, numbers, booleans, arrays, objects, functions, or even other React components (JSX).
  • Destructuring: Props are received as a single object in the function parameter. Destructuring them directly in the signature is the most common and clean pattern.
  • Default values: You can define default prop values using JavaScript default parameters or the defaultProps property (legacy).
  • children prop: A special built-in prop — anything you place between a component's opening and closing tags in JSX is available as props.children.
jsx — props in action
// Parent passes props to child
function App() {
  return (
    <UserCard
      name="Riya"
      age={25}
      isAdmin={true}
      onClick={() => alert('clicked')}
    />
  );
}

// Child receives via destructured props
function UserCard({ name, age, isAdmin, onClick }) {
  return (
    <div onClick={onClick}>
      <h2>{name} (Age: {age})</h2>
      {isAdmin && <span>🔑 Admin</span>}
    </div>
  );
}

// children prop — content between tags
function Box({ children }) {
  return <div className="box">{children}</div>;
}
// Usage:
<Box><p>This is children content</p></Box>

// Default prop value
function Badge({ label = 'Default' }) {
  return <span>{label}</span>;
}

NPM (Node Package Manager) is the default package manager that comes bundled with Node.js. It serves two main purposes: it is a command-line tool you use to install, update, and manage project dependencies, and it is also the world's largest software registry — a massive public database of open-source JavaScript packages (libraries and tools) that developers share and reuse.

What NPM does in a React project:

  • Installs libraries like React, React Router, Axios, etc. into your project.
  • Manages versioning — ensures your project always uses the correct versions of its dependencies via package.json and package-lock.json.
  • Runs scripts defined in package.json — like npm start, npm run build, npm test.

node_modules folder: When you run npm install, NPM reads your package.json and downloads all listed dependencies (and their own dependencies) into the node_modules folder. This folder contains the actual source code of every library your project uses.

  • It can contain hundreds of sub-folders (even for small projects) because every package has its own dependencies.
  • Never commit it to Git — it is always listed in .gitignore. Anyone cloning the repo simply runs npm install to regenerate it.
  • It can be safely deleted and recreated at any time using npm install.
terminal — common npm commands
# Install all dependencies from package.json
npm install

# Install a specific package and save to dependencies
npm install axios

# Install a dev-only package (not needed in production)
npm install --save-dev eslint

# Run a script defined in package.json
npm run dev
npm run build

# Remove a package
npm uninstall axios

The public/ folder contains static assets that are served directly to the browser without being processed or transformed by the build tool (Webpack/Vite). Files placed here bypass the module bundler entirely — they are copied as-is to the root of the final build output.

What typically lives in the public folder:

  • index.html — The single HTML page that bootstraps the entire React SPA. It contains the <div id="root"> where React mounts. This is the most critical file in the folder.
  • favicon.ico — The small icon shown in the browser tab.
  • manifest.json — Metadata for Progressive Web App (PWA) support (app name, icons, theme color).
  • robots.txt — Instructions for web crawlers and search engines.
  • Static images or fonts that need a fixed, predictable public URL.

Key distinction: Assets in public/ are referenced by absolute paths (e.g., /logo.png) and are not optimized by the bundler. Assets imported inside src/ (e.g., import logo from './logo.png') are processed, hashed, and optimized by the bundler — this is the preferred approach for most images and assets used in components.

<!-- In index.html — direct public URL, no import needed -->

<img src="/logo.png" alt="Logo" />

// In a component — src/ import (bundler optimizes this)
import logo from './assets/logo.png';
const Header = () => <img src={logo} alt="Logo" />;

Unlike the public/ folder, all files inside src/ are processed, bundled, and optimized by the build tool (Vite/Webpack) when you run a build.

The index.html file (located in the public/ folder in CRA, or the project root in Vite) is the single HTML page that the browser loads for the entire React application. Since React builds SPAs, there is only ever one HTML file — every "page" the user navigates to is actually rendered by JavaScript swapping components in and out.

Key responsibilities of index.html:

  • The mount point: It contains a <div id="root"></div> — a completely empty div that React uses as its container. React injects all of its generated HTML into this single div at runtime.
  • Meta information: It holds the page <title>, character set, viewport settings, and any global meta tags for SEO or PWA support.

index.js (or main.jsx in Vite) is the JavaScript entry point of a React application — it is the very first JS file executed when the app loads. Its sole job is to connect the React component tree to the real browser DOM, and it does this using ReactDOM.

What is ReactDOM? ReactDOM is a separate package (react-dom) that acts as the bridge between React's component world and the browser's DOM. React itself is platform-agnostic (it also works on mobile via React Native), so the DOM-specific rendering logic lives in ReactDOM, not in React core.

Key ReactDOM methods:

  • ReactDOM.createRoot(container) — (React 18+) Creates a root DOM node for React to manage. The container is the #root div from index.html. This enables React 18's concurrent features.
  • .render(<App />) — Tells React to render the <App /> component (the entire component tree) into the root container. From this point, React owns that DOM node.
  • ReactDOM.unmountComponentAtNode() — Removes a mounted React tree from the DOM (used for cleanup).
jsx — index.js / main.jsx
import React from 'react';
import ReactDOM from 'react-dom/client';  // React 18+
import App from './App';
import './index.css';  // global styles

// 1. Find the #root div in index.html
const rootElement = document.getElementById('root');

// 2. Create a React root (React 18 API)
const root = ReactDOM.createRoot(rootElement);

// 3. Render the entire App into the root
root.render(
  <React.StrictMode>   // helps catch bugs in dev
    <App />
  </React.StrictMode>
);

App.js (or App.jsx) is the root component of a React application — the top of the entire component tree. It is the component that index.js renders into the DOM, and every other component in your app is a descendant of it, directly or indirectly.

Typical responsibilities of App.js:

  • Application-level layout: It usually renders persistent layout components like <Navbar />, <Sidebar />, and <Footer /> that appear on every page.
  • Routing setup: In apps using React Router, App.js is where the <Routes> and <Route> components are defined, mapping URL paths to page components.
  • Global providers: Context providers (for auth, theme, language, etc.) are often wrapped at the App level so all child components can access them.
  • Top-level state: State that needs to be shared across many parts of the app may live here or in a context provider set up here.

In App.js, the function defines the component itself, and the return statement inside it tells React exactly what UI to render for that component. These two pieces are fundamental to how every functional component in React works.

The function:

  • A React functional component is just a regular JavaScript function whose name starts with a capital letter.
  • React calls this function every time the component needs to render or re-render (due to state or prop changes).
  • The function can contain logic — hooks, variable declarations, event handlers, calculations — anything you need to prepare the data before rendering.

The return statement:

  • The return must output JSX — the declarative description of what the UI should look like.
  • You can only return one root element from a component. If you need sibling elements, wrap them in a <div> or a React Fragment (<>...</>).
  • For multi-line JSX, wrap the return value in parentheses () to avoid JavaScript's automatic semicolon insertion (ASI) issues.
  • The return value can be null if you want the component to render nothing.
jsx — function and return explained
function App() {
  // ── Logic area (before return) ──
  const userName = 'Riya';
  const greet = () => alert('Hi!');

  // ── Return: what React renders ──
  return (                       // parentheses for multi-line
    <>                           // Fragment — one root rule
      <h1>Hello, {userName}</h1>
      <button onClick={greet}>Say Hi</button>
    </>
  );
}

// Returning null renders nothing
function Hidden({ show }) {
  if (!show) return null;
  return <p>Now you see me!</p>;
}

Technically yes, a JavaScript function can exist without a return statement, but in the context of a React component, a function without a return (or one that returns undefined) will cause an error or render nothing — with an important nuance.

  • No return at all: JavaScript functions without a return implicitly return undefined. React will throw an error: "Nothing was returned from render". A valid React component must return either JSX or null — never undefined.
  • Returning null: This is perfectly valid and intentional. Returning null tells React to render absolutely nothing for that component — no DOM output. This is useful for conditional rendering (hide a component entirely under certain conditions).
  • Helper functions inside the component: Not every function inside a component file needs to return JSX. You can have utility functions, event handlers, and data-processing functions that return normal values or nothing — only the component function itself must return JSX or null.
jsx — return scenarios
// ❌ No return — React throws an error
function BadComponent() {
  const x = 1 + 1;
  // forgot to return JSX → "Nothing was returned from render"
}

// ✅ Return null — renders nothing, no error
function ConditionalBanner({ show }) {
  if (!show) return null;   // valid, renders nothing
  return <div className="banner">Important notice!</div>;
}

// ✅ Helper function inside component — no JSX needed
function App() {
  const formatDate = (date) => date.toLocaleDateString(); // no JSX
  return <p>Today: {formatDate(new Date())}</p>;
}

In JavaScript (ES6 modules), export default is the syntax used to expose a value — a function, class, or object — from a module (file) so that other files can import and use it. In the context of React components, every component file typically exports its component so it can be used (composed) in other files.

Two types of exports in JavaScript:

  • Default Export (export default): A file can have only one default export. When importing, you can give it any name you like — you're not bound to the original name. This is the most common pattern for React components.
  • Named Export (export { name }): A file can have multiple named exports. When importing, you must use the exact same name (or alias it with as). Commonly used for utility functions, constants, or when exporting multiple things from one file.

For App.js specifically, export default App makes the App component available to index.js, which imports it to mount the entire application.

jsx — export default vs named export
// ── Default export (one per file) ──
function App() {
  return <h1>Hello</h1>;
}
export default App;

// OR inline default export
export default function App() {
  return <h1>Hello</h1>;
}

// Importing default — any name works
import App from './App';         // ✅
import MyApp from './App';       // ✅ also works

// ── Named exports (multiple per file) ──
export const add = (a, b) => a + b;
export const subtract = (a, b) => a - b;

// Importing named — must match exact name
import { add, subtract } from './utils';

No, it is not technically required by React or JavaScript for the file name and the component name to match. JavaScript modules don't enforce this. However, matching the filename to the component name is an extremely strong convention in the React community — and for good reason.

Why they should match (convention):

  • Readability & Discoverability: When you see import Navbar from './Navbar', you instantly know to look for a file named Navbar.jsx. If they didn't match, navigating large codebases would be confusing and error-prone.
  • Tooling support: Many editors, linters (ESLint rules), and code generation tools assume filename === component name. Deviating from this can break auto-imports, refactoring tools, and certain ESLint rules.
  • Team consistency: In team projects, predictable naming conventions reduce cognitive overhead and onboarding time.

The one real rule: React component names must start with a capital letter. This is a strict requirement — lowercase names are treated as HTML elements, not components. The filename casing is a convention, not enforced by React itself.

// ✅ Works but BAD practice — name mismatch

JSX (JavaScript XML) is a syntax extension for JavaScript that allows you to write HTML-like markup directly within your JS code. It is one of React's most distinctive and important features. Browsers cannot execute JSX natively — it must first be compiled into plain JavaScript by a tool like Babel.

3 Key Roles of JSX in React:

  • 1. Descriptive UI Definition: JSX gives you an expressive, readable way to describe what the UI should look like. Instead of constructing the UI with verbose React.createElement() calls, JSX lets you write familiar HTML-like syntax right next to your JavaScript logic, making the intent of every component immediately clear.
  • 2. Embedding JavaScript Expressions: JSX lets you break out of the markup and inject any JavaScript expression using curly braces {}. This includes variables, function calls, arithmetic, ternary operators, and array methods — making it trivial to produce dynamic, data-driven UIs without switching between templating languages.
  • 3. Build-Time Safety & Optimization: Because JSX is compiled at build time by Babel, structural errors (unclosed tags, invalid nesting) are caught early — before the code ever runs in the browser. The compiler also transforms JSX into optimized React.createElement() calls, ensuring efficient Virtual DOM construction at runtime.
jsx — the three roles demonstrated
// Role 1: Readable UI description
const card = (
  <div className="card">
    <h2>Profile</h2>
  </div>
);

// Role 2: Embedding JS expressions with {}
const name = 'Riya';
const isAdmin = true;
const ui = (
  <p>
    Hello, {name}! Score: {10 * 5}. {isAdmin ? 'Admin' : 'User'}
  </p>
);

// Role 3: Compiled to createElement at build time
// JSX:  <h1 className="title">Hi</h1>
// Output: React.createElement('h1', {className:'title'}, 'Hi')

JSX isn't just syntactic sugar — it brings real, tangible benefits to the development experience and the quality of your code:

  • 1. Developer-Friendly Readability: JSX closely resembles HTML, which most web developers already know. Having markup and logic co-located in the same file — rather than split across separate HTML templates and JS files — makes components easier to read, understand, and reason about at a glance.
  • 2. Full Power of JavaScript in UI: Unlike HTML templates in frameworks like Angular or Vue (which have their own limited templating syntax), JSX lets you use the full expressiveness of JavaScript. Any JS expression — array methods, destructuring, spread, ternary, logical operators — works natively inside JSX.
  • 3. Early Error Detection: JSX is compiled at build time. Syntax errors, unclosed tags, or incorrect nesting are flagged immediately by the compiler — before you even open the browser. This dramatically speeds up the debugging cycle compared to runtime errors.
  • 4. Excellent Tooling Support: Because JSX is a well-defined syntax, editors like VS Code can provide accurate syntax highlighting, auto-completion, IntelliSense, and refactoring support for JSX. ESLint also has JSX-specific rules to enforce best practices.
  • 5. Prevents XSS (Cross-Site Scripting) Attacks: React automatically escapes any values embedded in JSX before rendering them. If you inject user-provided data into JSX, React converts it to a plain string — it never executes arbitrary HTML or scripts. This makes React apps inherently safer against XSS vulnerabilities by default.
jsx — xss protection example
// User-provided malicious input
const userInput = '<script>alert("hacked!")</script>';

// ✅ React escapes it — renders as plain text, NOT as HTML
return <p>{userInput}</p>;
// Browser shows: <script>alert("hacked!")</script>  (safe text)

// ⚠️ Only dangerouslySetInnerHTML bypasses this — use with caution
return <p dangerouslySetInnerHTML={{ __html: userInput }} />; // unsafe!

Babel is a popular open-source JavaScript transpiler (source-to-source compiler). Its primary job is to transform modern JavaScript and JSX code into a version that older browsers can understand and execute. In a React project, Babel is a critical part of the build toolchain — it runs before your code reaches the browser.

What Babel does in a React project:

  • Transforms JSX → JavaScript: Babel converts every JSX expression into equivalent React.createElement() calls that browsers can execute natively. This is its most important role in React.
  • Transpiles modern JS (ES6+): Babel converts modern syntax — arrow functions, destructuring, optional chaining, async/await, classes — into equivalent ES5 code for maximum browser compatibility.
  • Plugin & Preset System: Babel is highly configurable via plugins and presets. @babel/preset-react handles JSX transformation, and @babel/preset-env handles modern JS transpilation based on your target browser list.

In modern React setups using Vite, Babel is replaced by esbuild and SWC for JSX transformation — both are significantly faster — but the concept of JSX-to-JS transformation remains identical.

jsx — babel transformation
// ── BEFORE Babel (what you write) ──
const Greet = ({ name }) => (
  <h1 className="title">Hello, {name}!</h1>
);

// ── AFTER Babel (what the browser runs) ──
const Greet = ({ name }) =>
  React.createElement(
    "h1",
    { className: "title" },
    "Hello, ", name, "!"
  );

// babel.config.js — typical React config
module.exports = {
  presets: [
    '@babel/preset-env',    // modern JS → ES5
    '@babel/preset-react',  // JSX → createElement
  ]
};

A React component's return can only return one root element. If you have multiple sibling elements to render, you need to wrap them in something. The naive solution is wrapping in a <div> — but this pollutes the DOM with unnecessary wrapper nodes, which can break CSS layouts (flexbox/grid), add unwanted styling, and bloat the DOM tree.

Fragment solves this elegantly. A <React.Fragment> (or its shorthand <>...</>) groups multiple elements without adding any actual DOM node. It's an invisible wrapper that exists only in the React Virtual DOM — it leaves zero trace in the real browser DOM.

Two ways to use Fragments:

  • Shorthand syntax <></>: The most common and concise form. Use this most of the time.
  • Explicit <React.Fragment>: Used when you need to pass a key prop to the fragment — for example, when rendering a list of fragments. The shorthand syntax does not support the key prop.
jsx — fragment vs div wrapper
// ❌ Unnecessary div wrapper — pollutes the DOM
return (
  <div>
    <h1>Title</h1>
    <p>Paragraph</p>
  </div>
);

// ✅ Fragment shorthand — no extra DOM node
return (
  <>
    <h1>Title</h1>
    <p>Paragraph</p>
  </>
);

// ✅ Explicit Fragment with key (required for lists)
{items.map(item => (
  <React.Fragment key={item.id}>
    <dt>{item.term}</dt>
    <dd>{item.definition}</dd>
  </React.Fragment>
))}

The spread operator (...) is a JavaScript ES6 feature. In JSX, it is used to spread all key-value pairs of an object as individual props onto a component or HTML element — instead of passing each prop manually one by one. This is very useful when you want to forward a collection of props from a parent to a child without having to destructure and re-list each one.

Common use cases in JSX:

  • Forwarding props: A wrapper component can pass all its received props down to a child component without knowing their names.
  • Merging prop objects: You can combine a base config object with specific overrides using spread.
  • Avoiding repetition: Instead of writing name={user.name} age={user.age} email={user.email}, you simply write {...user}.

Caution: Overusing the spread operator to blindly forward all props can pass unnecessary or invalid HTML attributes to DOM elements, causing React warnings. Always be intentional about what you spread.

jsx — spread operator usage
// Without spread — verbose prop passing
const user = { name: 'Riya', age: 25, role: 'admin' };
<UserCard name={user.name} age={user.age} role={user.role} />

// ✅ With spread — clean and concise
<UserCard {...user} />

// Forwarding props in a wrapper component
function FancyInput(props) {
  return <input className="fancy" {...props} />;
  // passes all props (placeholder, value, onChange…) to input
}

// Merging with spread — overrides take precedence
const defaults = { type: 'text', disabled: false };
<input {...defaults} type="email" />;
// Result: type="email" (override), disabled=false (from defaults)

Conditional rendering in React means showing or hiding UI elements based on some condition — a state value, a prop, a user's auth status, etc. Since JSX is JavaScript, you have several idiomatic ways to achieve this, each suited to different scenarios.

  • 1. if / else statement: The most straightforward approach. Write a regular JavaScript if/else block outside the return statement and return different JSX from each branch. Best for complex logic or when the two outputs are very different.
  • 2. Ternary Operator (condition ? a : b): Works inside JSX (within {}). Perfect for choosing between two outputs inline. Keeps the return statement concise for simple conditions.
  • 3. Logical AND (condition && element): Renders the element only when the condition is true. If the condition is false, nothing is rendered. Use this when you either show something or show nothing at all (no "else" branch).
  • 4. Nullish / returning null: A component can return null entirely to render nothing. Useful for completely hiding a component based on a prop.
  • 5. Switch statement / Object map: For multiple conditions (more than 2 states), a switch block or an object lookup map keeps the logic clean instead of nested ternaries.
jsx — all conditional rendering types
// 1. if/else (outside return)
function Auth({ isLoggedIn }) {
  if (isLoggedIn) return <Dashboard />;
  return <Login />;
}

// 2. Ternary (inline, inside return)
return <p>{score >= 50 ? 'Pass ✅' : 'Fail ❌'}</p>;

// 3. Logical AND (show or nothing)
return (
  <div>
    {hasNotifications && <NotificationBadge />}
  </div>
);

// 4. Return null (render nothing)
function Tooltip({ show, text }) {
  if (!show) return null;
  return <span className="tooltip">{text}</span>;
}

// 5. Object map (for multiple states)
const statusMap = {
  loading: <Spinner />,
  error:   <ErrorMsg />,
  success: <DataTable />,
};
return statusMap[status] || <p>Unknown state</p>;

In JSX, the standard way to render a list of elements from an array is using JavaScript's built-in Array.map() method. Since JSX is just JavaScript expressions inside {}, map() fits perfectly — it transforms each item in the array into a JSX element and returns a new array of those elements, which React knows how to render.

What is map()? Array.map() is a JavaScript array method that iterates over every element of an array, applies a callback function to each element, and returns a new array of the transformed results. It does not mutate the original array.

The key prop (very important!): When rendering lists, React requires each element to have a unique key prop. React uses keys to efficiently identify which items in the list have changed, been added, or removed during re-renders. Keys should be stable, unique identifiers — typically an ID from your data. Avoid using array indices as keys if the list can be reordered.

const fruits = [

];

    <ul>
      {fruits.map((fruit) => (
        <li key={fruit.id}>        // ← key is required!
          {fruit.name}
        </li>

    </ul>
  );
}

// map() with a component per item

Both compilers and transpilers are tools that transform source code — but they operate at different levels of abstraction.

  • Compiler: A compiler translates source code from a high-level language into a fundamentally different, lower-level language — typically machine code or bytecode that the operating system or hardware can execute directly. The input and output are at very different abstraction levels. Examples: GCC (C → machine code), Java compiler (Java → JVM bytecode).
  • Transpiler (Source-to-Source Compiler): A transpiler translates source code from one language (or version) into another language at the same level of abstraction — typically one high-level language to another. The output is still human-readable source code, just in a different syntax. Examples: Babel (JSX/ES6 → ES5), TypeScript compiler (TS → JS), Sass (SCSS → CSS).

In the React ecosystem: Babel is the most common transpiler. It transforms JSX (which browsers can't read) into standard ES5 JavaScript (which all browsers can run). It also transforms modern ES6+ features (arrow functions, classes, optional chaining) into their ES5 equivalents for older browser support.

Components — Functional & Class
9 Qs

A React Component is a self-contained, reusable building block of UI. It is essentially a JavaScript function (or class) that accepts inputs called props and returns JSX describing what should appear on the screen. Every React application is a tree of components — from the root <App /> down to the smallest button or icon.

Think of components like custom HTML elements you design yourself: once defined, <UserCard /> or <Navbar /> can be used anywhere in your app just like <div> or <p>.

Main Elements of a React Component:

  • Props: Read-only data passed into the component from its parent. Props make a component dynamic and reusable — the same component can render differently based on the props it receives. A component must never modify its own props.
  • State: Data that the component owns and manages internally. Unlike props, state can change over time (via useState or this.setState). Every state change triggers a re-render of the component, updating the UI to reflect the new data.
  • JSX (Return / Render output): The markup the component returns — describing what the UI looks like for the current props and state. It is the visual output of the component.
  • Lifecycle / Hooks: Mechanisms that let a component respond to events in its existence — mounting (appearing in the DOM), updating (re-rendering due to state/prop change), and unmounting (being removed). In functional components, useEffect handles this; in class components, lifecycle methods like componentDidMount are used.
  • Event Handlers: Functions defined inside (or passed as props to) the component that respond to user interactions — clicks, form inputs, keyboard events, etc.
  • Context (optional): A way to consume globally shared data (like a theme or auth status) without receiving it through props at every level.
jsx — anatomy of a complete component
function ProductCard({ name, price }) {   // ← Props
  const [qty, setQty] = useState(1);       // ← State

  const handleAdd = () => setQty(qty + 1); // ← Event handler

  // ← JSX return (render output)
  return (
    <div className="card">
      <h2>{name}</h2>
      <p>₹{price} × {qty} = ₹{price * qty}</p>
      <button onClick={handleAdd}>Add More</button>
    </div>
  );
}

React has two main types of components — Functional and Class. There are also some sub-categorisations based on their role and behaviour.

Types of React Components:

  • Functional Components — Plain JavaScript functions that return JSX. The modern, preferred standard since the introduction of Hooks in React 16.8.
  • Class Components — ES6 classes extending React.Component. The older approach, still found in legacy codebases.
  • Pure Components — A class component variant (React.PureComponent) or functional component wrapped in React.memo() that performs a shallow prop comparison and skips re-rendering if props haven't changed.
  • Higher-Order Components (HOC) — A function that takes a component and returns a new enhanced component. A pattern for reusing component logic.

Functional Components in depth:

A Functional Component is simply a JavaScript function that accepts a single argument — props — and returns JSX. Before hooks, these were called "stateless" components because they couldn't manage state or side effects. With hooks (React 16.8+), functional components can now do everything class components can — and with far less boilerplate.

Characteristics of Functional Components:

  • Simpler, more concise syntax — no class, constructor, this, or render() needed.
  • Use Hooks (useState, useEffect, useContext, etc.) for state and lifecycle.
  • Easier to read, test, and reason about — functions are pure by default.
  • Better performance characteristics and easier to optimise with React.memo.
  • They are the recommended approach in all modern React code.
jsx — functional component (modern)
// Simplest form — no state
const Hello = ({ name }) => <h1>Hello, {name}!</h1>;

// With state and effect hooks
function Timer() {
  const [seconds, setSeconds] = useState(0);

  useEffect(() => {
    const id = setInterval(() => setSeconds(s => s + 1), 1000);
    return () => clearInterval(id); // cleanup
  }, []);

  return <p>Elapsed: {seconds}s</p>;
}

In React, data flows in one direction — from parent to child. The primary mechanism for this is props. However, there are several patterns depending on which direction you need data to travel.

  • 1. Parent → Child (via Props): The most common pattern. The parent component passes data as attributes on the child component's JSX tag. The child receives them as its props argument.
  • 2. Child → Parent (via Callback Functions): Since props are one-way, a child cannot directly send data up to a parent. Instead, the parent passes a callback function as a prop to the child. The child calls that function with the data it wants to send — effectively "lifting state up" to the parent.
  • 3. Sibling → Sibling (via Shared Parent — "Lifting State Up"): Two sibling components can't communicate directly. The solution is to move the shared state to their closest common ancestor (parent), and then pass it down to both siblings via props.
  • 4. Globally (via Context API or State Management): For data that many components across the tree need, the Context API (useContext) or a state management library (Redux, Zustand) is used — avoiding the need to pass props through every intermediate layer.
jsx — parent → child and child → parent
// Parent Component
function Parent() {
  const [msg, setMsg] = useState('Hello from Parent');

  // Callback to receive data FROM child
  const handleChildData = (data) => {
    console.log('Child sent:', data);
  };

  return (
    <Child
      message={msg}                     // ← Parent → Child (prop)
      onSendData={handleChildData}      // ← callback passed down
    />
  );
}

// Child Component
function Child({ message, onSendData }) {
  return (
    <div>
      <p>{message}</p>                    // uses parent's data
      <button onClick={() => onSendData('Hi Parent!')}>
        Send to Parent                   // ← Child → Parent (callback)
      </button>
    </div>
  );
}

Prop Drilling (also called "threading props") is the situation where you need to pass data through several intermediate components that don't actually need or use that data themselves — just so it can reach a deeply nested component that does need it.

Since React's data flow is strictly one-directional (parent → child via props), if a component deep in the tree needs data from a distant ancestor, every component in between must accept and re-pass the prop — even if they have no use for it. This creates tight coupling, makes components less reusable, and makes the code harder to maintain and refactor.

Example scenario: A user authentication object lives in <App />. It's needed in <UserAvatar />, which is nested inside <Header /> → <Navbar />. Both <Header /> and <Navbar /> must accept and forward the user prop even though they don't use it — this is prop drilling.

jsx — prop drilling illustrated
// user lives in App but is needed deep in UserAvatar
function App() {
  const user = { name: 'Riya', avatar: 'riya.jpg' };
  return <Header user={user} />;   // drills down
}

function Header({ user }) {           // doesn't USE user
  return <Navbar user={user} />;   // just passes it on
}

function Navbar({ user }) {           // doesn't USE user either
  return <UserAvatar user={user} />; // passes it on again
}

function UserAvatar({ user }) {       // finally uses it!
  return <img src={user.avatar} alt={user.name} />;
}
// Header and Navbar are unnecessarily coupled to `user`

Why avoid it:

  • Tight coupling: Intermediate components become unnecessarily aware of data they don't need, making them harder to reuse in other contexts.
  • Maintenance nightmare: If the shape of the data changes, you must update every component in the chain — even those that don't use it.
  • Readability suffers: It becomes harder to trace where data comes from and where it's actually consumed when it's passed through many layers.
  • Refactoring is risky: Moving a component to a different level in the tree may break the prop chain entirely.

4 Ways to Avoid Prop Drilling:

  • 1. React Context API: The built-in solution. Create a Context with React.createContext(), wrap the tree in a Provider with the value, and any descendant can consume it with useContext() — no intermediate passing required. Best for low-to-medium frequency updates (theme, locale, auth).
  • 2. State Management Libraries (Redux / Zustand / Jotai): A global store holds shared state outside the component tree. Any component can read from or write to the store directly via hooks — completely bypassing the hierarchy. Best for complex, frequently-changing global state.
  • 3. Component Composition (children / render props): Instead of passing data through intermediate components, restructure the tree so the component that needs the data is rendered directly by the component that has it — passing it as children or a render prop. No drilling needed.
  • 4. Custom Hooks: Extract shared logic (including data fetching or state) into a custom hook. Any component that needs the data calls the hook directly, avoiding the need to receive it as a prop from above.
jsx — context api solution to prop drilling
// 1. Create context
const UserContext = React.createContext();

// 2. Provide value at the top — no drilling needed
function App() {
  const user = { name: 'Riya', avatar: 'riya.jpg' };
  return (
    <UserContext.Provider value={user}>
      <Header />  // ← no user prop needed!
    </UserContext.Provider>
  );
}

// 3. Header and Navbar don't touch user at all
function Header()  { return <Navbar />; }
function Navbar()  { return <UserAvatar />; }

// 4. Consumer grabs it directly
function UserAvatar() {
  const user = useContext(UserContext); // direct access!
  return <img src={user.avatar} alt={user.name} />;
}

Class Components are the older way of defining React components using ES6 JavaScript classes. Before the introduction of Hooks in React 16.8 (2019), class components were the only way to use state and lifecycle methods in React. You still encounter them heavily in legacy codebases and older tutorials.

A class component must extend React.Component and must implement a render() method that returns JSX.

Key characteristics of Class Components:

  • Extends React.Component: Inherits built-in React component behaviour — state management, lifecycle methods, and the forceUpdate method.
  • constructor + super(props): The constructor initialises the component's local state and must call super(props) to properly initialise the parent React.Component class. Without this, this.props would be undefined inside the constructor.
  • State via this.state: State is initialised as an object in the constructor and updated exclusively via this.setState() — never by mutating this.state directly.
  • Lifecycle methods: componentDidMount (after first render), componentDidUpdate (after re-render), componentWillUnmount (before removal) are methods you override to handle side effects.
  • Props via this.props: Unlike functional components where props are a function argument, class components access props through this.props.
jsx — class component structure
import React, { Component } from 'react';

class Counter extends Component {
  // Initialise state in constructor
  constructor(props) {
    super(props);              // required!
    this.state = { count: 0 };
  }

  // Lifecycle — runs after first render
  componentDidMount() {
    console.log('Component mounted!');
  }

  // Lifecycle — runs before removal
  componentWillUnmount() {
    console.log('Component unmounted!');
  }

  // Event handler
  increment = () => {
    this.setState({ count: this.state.count + 1 });
  };

  // Required render method — returns JSX
  render() {
    return (
      <div>
        <p>Count: {this.state.count}</p>
        <p>Name: {this.props.name}</p>
        <button onClick={this.increment}>+</button>
      </div>
    );
  }
}

export default Counter;

Data passing between class components follows the same unidirectional flow as functional components — parent to child via props, and child to parent via callback functions. The difference is purely in syntax — class components use this.props and this.state instead of destructured arguments and useState.

  • Parent → Child: The parent's render() method passes props as JSX attributes on the child component tag. The child accesses them via this.props.propName.
  • Child → Parent: The parent passes a callback method (bound to this) as a prop to the child. When the child needs to send data up, it calls this.props.callbackName(data). The parent's callback receives the data and can update its own state via this.setState().
  • Shared state (sibling communication): State is "lifted up" to the common parent, which holds the state and distributes it to both siblings as props while providing setter callbacks.
jsx — data passing in class components
// Parent class component
class ParentComp extends Component {
  constructor(props) {
    super(props);
    this.state = { parentMsg: 'Hello from Parent', childReply: '' };
    this.handleChildData = this.handleChildData.bind(this); // bind!
  }

  handleChildData(data) {
    this.setState({ childReply: data }); // receive from child
  }

  render() {
    return (
      <div>
        <p>Child said: {this.state.childReply}</p>
        <ChildComp
          message={this.state.parentMsg}       // prop down
          onReply={this.handleChildData}        // callback down
        />
      </div>
    );
  }
}

// Child class component
class ChildComp extends Component {
  render() {
    return (
      <div>
        <p>{this.props.message}</p>           // use parent prop
        <button onClick={() => this.props.onReply('Hi Parent!')}>
          Reply to Parent
        </button>
      </div>
    );
  }
}

In JavaScript classes, this refers to the current instance of the class. In React class components, this is how you access everything that belongs to the component — its props, state, and custom methods. It is the central mechanism tying the component together, and one of the most common sources of bugs for beginners.

What this is used for in class components:

  • this.props: Access the props passed in from the parent component.
  • this.state: Read the component's current local state object.
  • this.setState(): Trigger a state update and re-render. This is the only correct way to update state — never mutate this.state directly.
  • this.methodName(): Call other methods defined on the class instance.

The this binding problem: In regular JavaScript methods, this depends on how the function is called, not where it's defined. When you pass a class method as an event handler (e.g., onClick={this.handleClick}), the method loses its this binding and becomes undefined in strict mode — causing errors. There are three standard solutions:

  • Bind in the constructor: this.handleClick = this.handleClick.bind(this)
  • Use an arrow function class property (auto-binds): handleClick = () => {}
  • Use an inline arrow function in JSX: onClick={() => this.handleClick()} (creates a new function on every render — least preferred)
jsx — this keyword and binding solutions
class MyComp extends Component {
  constructor(props) {
    super(props);
    this.state = { clicked: false };

    // Solution 1: bind in constructor
    this.handleClick = this.handleClick.bind(this);
  }

  // Solution 2: arrow function class property (recommended)
  handleClick = () => {
    this.setState({ clicked: true }); // this works correctly
  };

  render() {
    return (
      <div>
        <p>Prop: {this.props.title}</p>
        <p>State: {this.state.clicked ? 'Yes' : 'No'}</p>
        <button onClick={this.handleClick}>Click Me</button>
      </div>
    );
  }
}

This is one of the most frequently asked React interview questions. Here are the 5 key differences:

  • 1. Syntax & Boilerplate: A Functional component is a plain JavaScript function — clean, concise, no ceremony. A Class component requires extending React.Component, writing a constructor with super(props), and implementing a render() method. Functional components are significantly less verbose.
  • 2. State Management: In functional components, state is managed using the useState hook — you can have multiple independent state variables, each with a clean getter and setter. In class components, all state lives in a single this.state object and is updated with this.setState(), which merges the update shallowly.
  • 3. Lifecycle & Side Effects: Functional components use the useEffect hook to handle all lifecycle events (mount, update, unmount) in one unified API. Class components use separate, named lifecycle methods: componentDidMount, componentDidUpdate, componentWillUnmount — the logic for a single concern can be split across multiple methods.
  • 4. The this keyword: Class components rely heavily on this for accessing props (this.props), state (this.state), and methods — and this binding issues are a common source of bugs. Functional components have no this at all — props are plain function arguments and there are zero binding concerns.
  • 5. Performance & Modern Support: Functional components are easier for React to optimise internally. They work seamlessly with React's concurrent features (Suspense, transitions), can be memoised easily with React.memo, and are the only type that works with all modern hooks. Class components don't support hooks, and some newer React APIs (like use()) are functional-only.

// ✅ Functional Component (modern)
function Counter({ title }) {
  const [count, setCount] = useState(0);

  return <button onClick={() => setCount(count + 1)}>{title}: {count}</button>;
}

// 🏛 Class Component (legacy)
class Counter extends Component {
  constructor(props) {
    super(props);
    this.state = { count: 0 };
  }
  componentDidUpdate() {
    document.title = `Count: ${this.state.count}`;
  }
  render() {
    return (
      <button onClick={() => this.setState({ count: this.state.count + 1 })}>
        {this.props.title}: {this.state.count}

  }
}
Routing — Introduction
6 Qs

Routing in React refers to the mechanism that maps a specific URL path (e.g., /home, /about, /user/42) to a specific component that should be rendered when the user visits that URL — all without triggering a full page reload from the server. This is what makes React a true Single Page Application (SPA) framework in practice.

Because React itself is only a UI library, it has no built-in routing. Routing is handled by a separate library — React Router (react-router-dom), which is the de facto standard and most widely used routing solution in the React ecosystem.

How traditional vs React routing differs:

  • In a traditional multi-page app, clicking a link sends a request to the server, which returns a new HTML page — the whole browser reloads.
  • In a React SPA, the browser never requests a new page. React Router intercepts navigation, reads the URL, matches it against defined routes, and swaps out the rendered component — all client-side, near-instantly.

What is a Router? A Router is the top-level context provider from React Router that enables routing for your entire application. It listens to the browser's URL bar, makes the current location available to all child components, and re-renders the correct component whenever the URL changes. There are two main types:

  • BrowserRouter: Uses the HTML5 History API (pushState, replaceState) to keep the UI in sync with the URL. Produces clean URLs like /about. This is the standard choice for modern apps. Requires server-side configuration to handle direct URL access properly.
  • HashRouter: Uses the URL hash (#) to simulate navigation — e.g., /#/about. Works without server configuration because the hash is never sent to the server. Mostly used for static hosting environments.
  • MemoryRouter: Keeps routing state entirely in memory — no URL bar changes. Used for testing and non-browser environments like React Native.

# Install the latest React Router (v6+)
npm install react-router-dom
jsx — wrapping the app in a router
import { BrowserRouter } from 'react-router-dom';

// BrowserRouter wraps the entire app in index.js / main.jsx

Implementing routing in a React app involves 4 steps: installing React Router, wrapping the app in a Router, defining routes with <Routes> and <Route>, and using <Link> for navigation.

React Router has two major versions in common use:

  • v5 and below — Uses <Switch> and the exact prop. Component-based route matching.
  • v6+ (current standard) — Replaced <Switch> with <Routes>, dropped exact (now default behaviour), introduced <Outlet> for nested routing, and redesigned hooks like useNavigate. This is what all new projects should use.

Step-by-step implementation (React Router v6):

  • Step 1: Install — npm install react-router-dom
  • Step 2: Wrap the entire app in <BrowserRouter> in index.js or main.jsx.
  • Step 3: In App.jsx, define your route mappings using <Routes> (the container) and individual <Route> entries (each mapping a path to a component).
  • Step 4: Use <Link to="..."> or <NavLink> instead of <a href="..."> for in-app navigation to prevent full page reloads.
jsx — complete routing setup (react router v6)
// main.jsx — wrap app in BrowserRouter

root.render(<BrowserRouter><App /></BrowserRouter>);

// App.jsx — define routes
import { Routes, Route, Link } from 'react-router-dom';
import Home    from './pages/Home';
import About   from './pages/About';
import Contact from './pages/Contact';
import NotFound from './pages/NotFound';

function App() {
  return (
    <>
      // Navigation — Link prevents full page reload
      <nav>
        <Link to="/">Home</Link>
        <Link to="/about">About</Link>
        <Link to="/contact">Contact</Link>
      </nav>

      // Route definitions
      <Routes>
        <Route path="/"        element={<Home />}    />

  );

Both <Link> and <NavLink> are React Router components used for client-side navigation. They render as <a> anchor tags in the DOM but intercept the click event to prevent the browser from sending a server request — keeping navigation entirely within the React app.

Why not use a regular <a href>? A standard HTML anchor tag causes a full browser reload — the browser makes a new request to the server, downloads the page fresh, and React loses all its current state. <Link> prevents this, updating the URL and swapping components without any reload.

<Link> — Basic Navigation Component:

  • The fundamental navigation component. Renders a standard-looking anchor tag that navigates to the specified to path on click without a page reload.
  • Use this for any navigation that doesn't need to know whether it's currently the "active" route — footer links, breadcrumbs, cards, buttons, etc.
  • Props: to (path string or object), replace (replaces history instead of pushing), state (pass invisible state data to the destination route).

<NavLink> — Navigation with Active State Awareness:

  • <NavLink> is a special version of <Link> that knows when its route is currently active (i.e., the current URL matches its to prop).
  • When active, it automatically adds an active CSS class to the element — letting you style the currently selected navigation item differently (e.g., highlight the active tab in a navbar).
  • You can customise the active class name or use a function to apply styles and classes dynamically based on the isActive flag.
  • Use <NavLink> for navigation menus, tabs, sidebars — anywhere you want visual feedback of the current route.

Route Parameters (also called dynamic segments) are variable parts of a URL path that capture a specific value. Instead of defining a separate route for every possible ID or slug, you define a single route with a parameter placeholder — React Router fills in the value at runtime based on the actual URL the user visits.

You define a route parameter by prefixing a segment with a colon: :paramName. For example, /users/:id matches /users/1, /users/42, /users/riya — any value in that position.

Accessing route parameters: In React Router v6, you read dynamic URL parameters inside a component using the useParams() hook. It returns an object where the keys are the parameter names you defined in the route path, and the values are whatever was in the actual URL.

Query parameters (?key=value) are different — they are the key-value pairs after the ? in a URL. These are accessed using the useSearchParams() hook, not useParams().

jsx — route parameters with useParams()
import { Routes, Route, Link, useParams } from 'react-router-dom';

// Define a dynamic route with :id parameter
function App() {
  return (
    <Routes>
      <Route path="/"            element={<UserList />}   />
      <Route path="/users/:id"   element={<UserDetail />} />
      // matches /users/1, /users/42, /users/riya …
    </Routes>
  );
}

// Link to dynamic routes
function UserList() {
  const users = [{ id: 1, name: 'Riya' }, { id: 2, name: 'Arjun' }];

        <li key={u.id}>
          <Link to={`/users/${u.id}`}>{u.name}</Link>
        </li>
      ))}
    </ul>
  );
}

// Read the :id param in the destination component
function UserDetail() {
  const { id } = useParams(); // { id: "42" } — always a string
  return <h2>Viewing profile for user ID: {id}</h2>;
}

Note: <Switch> was a React Router v5 concept and has been replaced by <Routes> in React Router v6. Understanding both is important for interviews, as many existing codebases still use v5.

<Switch> in React Router v5:

In React Router v5, without a <Switch>, the router renders every <Route> whose path matches the current URL — potentially rendering multiple components at once. For example, if the URL is /about, both a route for / and /about might match.

<Switch> solves this by rendering only the first child <Route> that matches the current URL — exclusive matching. It iterates through its children from top to bottom and stops at the first match, rendering only that component.

Key behaviours of Switch (v5):

  • Exclusive matching: Only one route renders at a time — the first one that matches.
  • Order matters: More specific routes must be placed before more general ones. Without the exact prop, / would match everything and always win if listed first.
  • 404 fallback: A <Route> with no path at the end of a <Switch> acts as a catch-all for unmatched URLs — perfect for a "Page Not Found" component.

<Routes> in React Router v6 — the replacement:

<Routes> was introduced in v6 and improves upon <Switch> significantly. It always performs exclusive matching (no need for exact), uses smarter path-ranking to find the best match regardless of order, and enables nested routing with <Outlet>.

jsx — switch (v5) vs routes (v6)
// ── React Router v5 with <Switch> ──
import { Switch, Route } from 'react-router-dom'; // v5

<Switch>
  <Route exact path="/"       component={Home}    />
  <Route path="/about"        component={About}   />
  <Route path="/users/:id"   component={UserDetail} />
  <Route                       component={NotFound} />
  // ↑ no path = catch-all 404
</Switch>

// ── React Router v6 with <Routes> ──
import { Routes, Route } from 'react-router-dom'; // v6

<Routes>
  <Route path="/"          element={<Home />}      />
  <Route path="/about"    element={<About />}     />
  <Route path="/users/:id" element={<UserDetail />} />
  <Route path="*"          element={<NotFound />}  />
  // path="*" = catch-all in v6. No "exact" needed.
</Routes>

The exact prop was a React Router v5 feature used on <Route> components to enforce exact path matching. It has been removed in React Router v6 (where exact matching is the default behaviour). Understanding it is important for legacy codebases.

The problem it solved (v5): In React Router v5, path matching was prefix-based by default. A route with path="/" would match any URL because every URL starts with / — so the home route would render on /about, /users, and every other page too, unless told otherwise.

What exact does: When exact is added to a <Route>, it tells React Router to only render that component when the URL matches the path exactly — not just as a prefix.

  • Without exact: path="/" matches /, /about, /users/42 — everything.
  • With exact: path="/" matches only /.

In React Router v6: The exact prop was completely removed because v6 always performs exact matching by default. If you want to match a path and all its children in v6, you explicitly use a wildcard: path="/dashboard/*".

jsx — exact prop (v5) explained
// ── React Router v5 — WITHOUT exact ──
<Switch>
  <Route path="/"     component={Home}  />   // ⚠️ matches ALL routes!
  <Route path="/about" component={About} />   // never reached
</Switch>

// ── React Router v5 — WITH exact ──
<Switch>
  <Route exact path="/"  component={Home}  />  // ✅ only matches "/"
  <Route path="/about"   component={About} />  // ✅ now reachable
</Switch>

// ── React Router v6 — exact is gone, always exact by default ──
<Routes>
  <Route path="/"      element={<Home />}  />  // exact by default ✅
  <Route path="/about" element={<About />} />  // works fine ✅

  // To match /dashboard AND /dashboard/settings etc. in v6:
  <Route path="/dashboard/*" element={<Dashboard />} />
</Routes>

React Hooks are special built-in functions introduced in React 16.8 (2019) that allow functional components to "hook into" React features that were previously exclusive to class components — namely state, lifecycle methods, context, and more. Before hooks, if you needed state or side effects, you had to write a class component. Hooks eliminated that need entirely.

Rules of Hooks (must follow — they're enforced by a linter rule):

  • Only call hooks at the top level — never inside loops, conditions, or nested functions. React relies on hook call order being consistent between renders.
  • Only call hooks inside React functional components (or other custom hooks) — not in plain JavaScript functions.

Top React Hooks and what they do:

  • useState — Adds local state to a functional component. Returns a state value and a setter function. Every state update triggers a re-render.
  • useEffect — Runs side effects (data fetching, subscriptions, DOM manipulation, timers) after render. Replaces componentDidMount, componentDidUpdate, and componentWillUnmount in one API.
  • useContext — Subscribes to a React Context and returns its current value. Eliminates the need for consumer wrapper components.
  • useRef — Returns a mutable ref object that persists across renders without causing re-renders. Used to access DOM nodes directly or store mutable values (like timers or previous state).
  • useMemo — Memoises the result of an expensive computation. Re-computes only when its dependencies change. Prevents unnecessary recalculations on every render.
  • useCallback — Memoises a function definition so it isn't recreated on every render. Essential for passing stable callbacks to child components wrapped in React.memo.
  • useReducer — An alternative to useState for complex state logic. Works like Redux — you dispatch actions and a reducer function returns the new state. Ideal when state transitions have multiple cases or depend on the previous state.
  • useLayoutEffect — Like useEffect but fires synchronously after all DOM mutations and before the browser paints. Used for measuring DOM elements or making synchronous DOM changes.
jsx — hooks at a glance
import {
  useState, useEffect, useContext,
  useRef, useMemo, useCallback, useReducer
} from 'react';

function Demo() {
  const [count, setCount]       = useState(0);          // state
  const theme                   = useContext(ThemeCtx);  // context
  const inputRef                = useRef(null);          // DOM ref
  const doubled = useMemo(() => count * 2, [count]);    // memo
  const handleClick = useCallback(() => {}, []);        // callback

  useEffect(() => {                                       // side effect
    document.title = `Count: ${count}`;

}

These four terms are foundational vocabulary for any React developer. Understanding how they relate to hooks deepens your grasp of React's data model.

  • State: State is any data that a component owns and that can change over time — and whose change should cause the UI to update (re-render). Examples: a form input's current value, whether a dropdown is open, a list of users fetched from an API, a counter number. In functional components, state is created and managed with the useState hook (or useReducer for complex state). State is local to the component by default — other components can only know about it if you pass it via props or share it via Context/global store.
  • Stateless Component: A component that owns no state of its own. It is a pure function of its props — given the same props, it always renders the same output. These are also called presentational or dumb components. They are simple, predictable, easy to test, and highly reusable. With hooks, even "stateless" components technically can have state — but the term still refers to components designed to avoid owning state.
  • Stateful Component: A component that manages its own state using useState, useReducer, or (in class components) this.state. Also called smart or container components. They contain the business logic — fetching data, handling user interactions, and coordinating what gets passed down to child components as props.
  • State Management: As an application grows, multiple distant components need access to the same shared data. Passing state through props becomes impractical ("prop drilling"). State management refers to the patterns and tools used to organise, share, and synchronise state across the component tree. Options range from the lightweight React Context API (built-in, no extra library) to full-featured external libraries like Redux (predictable, centralised store with actions/reducers), Zustand (minimal, hook-based), Jotai (atomic state), and Recoil.
jsx — stateless vs stateful with hooks
// ── Stateless — pure, no state, only renders props ──
const Badge = ({ label, color }) => (

);

// ── Stateful — owns and manages data with useState ──
function LikeButton() {

      ❤️ {likes} Likes
    </button>
  );
}

useState is the most fundamental React hook. It lets you add reactive local state to a functional component. "Reactive" means: when the state value changes, React automatically re-renders the component with the new value — you never manually touch the DOM.

Syntax:

jsx — useState syntax
const [stateValue, setterFunction] = useState(initialValue);

How it works — step by step:

  • Initialisation: You call useState(initialValue). The initialValue can be any JavaScript value — a number, string, boolean, array, object, or even null. On the very first render, this value is used as the initial state. On subsequent re-renders, React ignores the initial value and returns the current state instead.
  • Array Destructuring: useState returns a 2-element array: [currentStateValue, setterFunction]. You destructure it to give both meaningful names. The convention is [thing, setThing].
  • Reading state: Simply use the state variable (e.g., count) anywhere in your JSX or component logic. React keeps track of its current value between renders.
  • Updating state: Call the setter function (e.g., setCount(newValue)). This does two things — it updates the stored state value and schedules a re-render of the component. React then re-executes your function, and the new state value is returned by useState.
  • Functional update form: When the new state depends on the previous state, always use the callback form setCount(prev => prev + 1) instead of setCount(count + 1). This prevents stale closure bugs when multiple updates happen in rapid succession.
  • State is immutable — never mutate directly: Never do count = count + 1 or user.name = 'new'. Always use the setter. Mutating state directly doesn't trigger a re-render.
jsx — useState in depth
function Counter() {
  // number state
  const [count, setCount] = useState(0);

  // object state — always spread to avoid mutation
  const [user, setUser] = useState({ name: 'Riya', age: 25 });

  // array state
  const [items, setItems] = useState(['Apple', 'Banana']);

  // ✅ Functional update — safe when new state depends on prev
  const increment = () => setCount(prev => prev + 1);

  // ✅ Updating an object — spread to keep other fields
  const birthday = () => setUser(prev => ({ ...prev, age: prev.age + 1 }));

  // ✅ Adding to an array — don't mutate, create new array
  const addItem  = () => setItems(prev => [...prev, 'Mango']);

  return (
    <div>
      <p>Count: {count}</p>
      <p>{user.name}, Age: {user.age}</p>
      <button onClick={increment}>+1</button>
      <button onClick={birthday}>Birthday</button>
      <button onClick={addItem}>Add Fruit</button>
    </div>
  );
}

useEffect is the hook for handling side effects in functional components. A side effect is any operation that reaches outside the component's own render cycle — something that interacts with the world beyond just returning JSX.

Common use cases for useEffect:

  • Fetching data from an API after the component mounts.
  • Setting up or tearing down subscriptions (WebSockets, event listeners).
  • Directly manipulating the DOM (e.g., setting document title, focusing an input).
  • Starting and clearing timers (setTimeout, setInterval).
  • Logging or analytics tracking.

How useEffect works:

  • Timing: useEffect runs after the component renders and the browser has painted the screen. It does not block the render — making it non-blocking and safe for performance.
  • Syntax: It accepts two arguments — a callback function (the effect) and an optional dependency array (controls when the effect re-runs).
  • Cleanup function: The callback can optionally return a cleanup function. React calls this cleanup before the effect runs again (on re-runs) and when the component unmounts. This is critical for preventing memory leaks — clearing intervals, unsubscribing, aborting fetch requests, etc.
  • Replaces 3 class lifecycle methods: componentDidMount (run once on mount), componentDidUpdate (run on updates), and componentWillUnmount (cleanup) are all unified into a single useEffect call.
jsx — useEffect patterns
// ── Pattern 1: Runs AFTER EVERY render (no dep array) ──
useEffect(() => {
  console.log('Ran after every render');
});

// ── Pattern 2: Runs ONCE on mount (empty dep array) ──
useEffect(() => {
  console.log('Component mounted — fetch initial data here');
  fetchUsers();
}, []);  // ← empty array = run only once

// ── Pattern 3: Runs when dependency changes ──
useEffect(() => {
  document.title = `You clicked ${count} times`;
}, [count]);  // ← re-runs only when `count` changes

// ── Pattern 4: With cleanup (prevents memory leaks) ──
useEffect(() => {
  const timer = setInterval(() => {
    console.log('tick');
  }, 1000);

  // Cleanup — clears interval when component unmounts
  return () => clearInterval(timer);
}, []);

// ── Pattern 5: Data fetching with cleanup (abort) ──
useEffect(() => {
  const controller = new AbortController();
  fetch('/api/users', { signal: controller.signal })
    .then(r => r.json())
    .then(data => setUsers(data));
  return () => controller.abort(); // cancel if unmounted early
}, []);

The dependency array is the optional second argument you pass to useEffect(callback, [deps]). It is an array of values that React watches — the effect re-runs only when one or more of these values has changed since the last render. It is the mechanism that gives you precise, intentional control over when your side effect executes.

How React checks dependencies: React compares each dependency value using Object.is() (essentially === strict equality). If any value is different from the previous render, the effect runs again. If they're all the same, React skips the effect entirely for that render.

Three behaviours based on the dependency array:

  • No array provided: The effect runs after every single render — both on mount and on every re-render. Use this rarely — usually only when you genuinely want to react to every possible change.
  • Empty array []: The effect runs only once — after the initial mount. It never runs again, even if state or props change. Perfect for one-time setup: API calls, event listener registration, initialising third-party libraries.
  • Array with values [a, b]: The effect runs after mount and then again whenever a or b changes. This is the most precise and most commonly used form — it ties the effect to specific pieces of data.

What should go in the dependency array? The React team's rule: include every reactive value used inside the effect — props, state variables, context values, and variables derived from them. The eslint-plugin-react-hooks exhaustive-deps rule enforces this and warns you when dependencies are missing.

jsx — dependency array behaviour
const [userId, setUserId] = useState(1);
const [theme, setTheme]   = useState('dark');

// Runs after EVERY render (dangerous — often causes infinite loops)
useEffect(() => { console.log('every render'); });

// Runs ONCE — on mount only (like componentDidMount)
useEffect(() => {
  fetchUser(userId);
}, []);  // ← stale! userId might change but effect won't re-run

// ✅ Correct — re-fetches whenever userId changes
useEffect(() => {
  fetchUser(userId);
}, [userId]);  // re-runs when userId changes

// Multiple dependencies — re-runs when EITHER changes
useEffect(() => {
  applyTheme(theme);
  fetchUserData(userId);
}, [userId, theme]); // ← watches both

// ⚠️ Objects/arrays as deps — risky! New reference each render
// const options = { id: userId }; ← new object every render
// useEffect(() => { ... }, [options]); ← runs every render!

Passing an empty array [] as the second argument to useEffect tells React: "This effect has no dependencies — there is nothing to watch for changes, so run this effect only once, after the very first render, and never again."

This makes useEffect(() => { ... }, []) the functional component equivalent of the class lifecycle method componentDidMount — it runs exactly once when the component appears in the DOM.

When to use []:

  • Initial data fetching: Load data from an API when the component first appears — you don't want this to re-run on every re-render.
  • One-time subscriptions: Set up a WebSocket connection, add a global event listener (e.g., window.addEventListener), or start a timer — things that should happen once and be cleaned up on unmount.
  • Third-party library initialisation: Initialise a chart library, a map, or any external widget that should set up once and persist.

Important nuance — Strict Mode double-invocation: In React 18 with <React.StrictMode> enabled, effects with [] actually run twice in development (mount → unmount → remount) to help you spot missing cleanup logic. This is a development-only behaviour — it does not happen in production builds.

Don't confuse these three:

  • useEffect(fn) — no array → runs after every render.
  • useEffect(fn, []) — empty array → runs once on mount.
  • useEffect(fn, [a, b]) — with values → runs on mount and whenever a or b changes.
jsx — empty array [] use cases
// ✅ Fetch data once on mount
useEffect(() => {
  fetch('/api/products')
    .then(res => res.json())
    .then(data => setProducts(data));
}, []);  // ← runs once — no deps to watch

// ✅ Set up an event listener once, clean up on unmount
useEffect(() => {
  const handleResize = () => setWidth(window.innerWidth);
  window.addEventListener('resize', handleResize);

  // Cleanup — removes listener when component unmounts
  return () => window.removeEventListener('resize', handleResize);
}, []);  // ← setup once, teardown once

// Equivalent class lifecycle methods:
// componentDidMount()      → useEffect(() => {...}, [])
// componentWillUnmount()   → return () => {...} inside useEffect([], [])
// componentDidUpdate()     → useEffect(() => {...}, [dep1, dep2])

useContext is a React hook that lets a functional component subscribe to and read data from a React Context without needing to nest Consumer components. It is the clean, modern way to consume shared global data — like the currently logged-in user, the app's colour theme, the active locale, or feature flags — anywhere in the component tree, without prop drilling.

What problem it solves: Normally, to share data between components that aren't in a direct parent-child relationship, you'd have to pass props down through every intermediate component — even those that don't need the data. Context + useContext creates a direct pipeline from a provider at the top of the tree to any consumer anywhere below it, skipping all the layers in between.

How useContext works:

  • You pass a Context object (created by React.createContext()) as the argument to useContext.
  • It returns the current value of that context — whatever value the nearest <Context.Provider> above in the tree has provided.
  • When the Provider's value changes, every component that calls useContext with that context automatically re-renders with the new value — no manual subscription needed.
  • If there is no Provider above in the tree, useContext returns the default value set when the context was created with createContext(defaultValue).
jsx — useContext hook usage
import { createContext, useContext, useState } from 'react';

// 1. Create the context
const ThemeContext = createContext('light'); // default = 'light'

// 2. Provide it at the top of the tree
function App() {
  const [theme, setTheme] = useState('dark');
  return (
    <ThemeContext.Provider value={theme}>
      <Toolbar />
      <button onClick={() => setTheme(t => t === 'dark' ? 'light' : 'dark')}>
        Toggle Theme
      </button>
    </ThemeContext.Provider>
  );
}

// 3. Intermediate component — no props needed
function Toolbar() { return <ThemedButton />; }

// 4. Deep consumer — reads context directly via useContext
function ThemedButton() {
  const theme = useContext(ThemeContext); // ← direct access!
  return (
    <button style={{ background: theme === 'dark' ? '#333' : '#fff' }}>
      Current theme: {theme}
    </button>
  );
}

React.createContext() is the function used to create a Context object. This object is the container that holds the shared data and acts as the communication channel between a Provider (the data source) and any Consumer (the data reader) anywhere in the component tree.

createContext(defaultValue):

  • Accepts an optional defaultValue — this is only used when a consuming component does not have a matching Provider above it in the tree.
  • Returns a Context object that has two important properties: .Provider and .Consumer.
  • The Context object itself does not hold the value — the Provider does. The Context object is just the key that connects providers and consumers.

Provider (<Context.Provider value={...}>):

  • A React component that supplies the context value to all descendant components.
  • The value prop is what gets distributed. It can be anything: a primitive, an object, a function, or even an object containing both state and setter functions.
  • Every component that reads from this context will re-render whenever the Provider's value changes.
  • Providers can be nested — the nearest Provider wins for any given Consumer.

Consumer (<Context.Consumer>):

  • The older, pre-hooks way of reading context. It uses a render prop pattern — you provide a function as its child, and it calls that function with the current context value.
  • In modern React, useContext() hook completely replaces the Consumer component — it's cleaner and much easier to read. Consumer is still technically valid but rarely used in new code.
jsx — createContext, provider and consumer
import { createContext, useContext, useState } from 'react';

// createContext — creates the channel
const UserContext = createContext(null); // null = default if no Provider

// Provider — wraps the tree and supplies the value
function UserProvider({ children }) {
  const [user, setUser] = useState({ name: 'Riya', role: 'admin' });
  return (
    <UserContext.Provider value={{ user, setUser }}>
      {children}  // all children can now access { user, setUser }
    </UserContext.Provider>
  );
}

// ✅ Modern way — useContext hook (recommended)
function Profile() {
  const { user } = useContext(UserContext);
  return <p>Hello, {user.name}!</p>;
}

// 🏛 Legacy way — Consumer render prop (avoid in new code)
function ProfileLegacy() {
  return (
    <UserContext.Consumer>
      {({ user }) => <p>Hello, {user.name}!</p>}
    </UserContext.Consumer>
  );
}

Choosing between props and Context is about scope and distance. Props are the right tool for most situations — they are explicit, easy to trace, and make data flow obvious. Context is for specific scenarios where props become impractical.

Use props when:

  • The data only needs to travel one or two levels deep — parent to immediate child or grandchild.
  • The data is component-specific and not truly "global" to the app.
  • You want the data flow to be explicit and easy to trace while reading the code.

Use useContext when:

  • Deeply nested components need shared data: If you're passing a prop through 3+ levels of components that don't use it themselves (prop drilling), Context is the solution. Classic examples: a user authentication object needed deep in a profile avatar, or the app theme needed in individual button components.
  • Truly global / app-wide data: Data that is relevant to the entire application or large subsections of it — the logged-in user, the active language/locale, the colour theme (light/dark), the shopping cart contents, feature flags, or notification state.
  • Multiple unrelated components need the same data: When the same piece of data is needed in components that are far apart in the tree with no common parent nearby — like a sidebar, a header, and a page component all needing the current user.
  • Avoiding re-renders via component structure isn't possible: When you've already tried component composition to solve prop drilling but the architecture makes it impractical.

Important caveat: Context is not a replacement for all state management. It's best for low-to-medium frequency updates (theme, auth, language). For high-frequency updates (like a real-time counter that updates every second), Context re-renders all consumers on every change, which can hurt performance. For those cases, consider Zustand, Redux, or Jotai.

jsx — real-world useContext examples
// ── Auth Context — used app-wide ──
const AuthContext = createContext();
function Navbar() {
  const { user, logout } = useContext(AuthContext);
  return <p>Logged in as: {user?.name} <button onClick={logout}>Logout</button></p>;
}

// ── Theme Context — used across many UI components ──
const ThemeContext = createContext('light');
function Card() {
  const theme = useContext(ThemeContext);
  return <div className={`card card--${theme}`}>...</div>;
}

// ── Custom hook pattern — encapsulate context access ──
function useAuth() {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth must be used inside AuthProvider');
  return ctx;
}
// Components now call: const { user } = useAuth();

Both useState and useReducer are React hooks for managing local component state. They share the same fundamental purpose and behaviour at a high level — only their internal mechanics and ideal use cases differ.

Key similarities:

  • Both manage local state: Both hooks store state that belongs to a single component (or can be lifted via Context). Neither is inherently global — they both hold state locally unless you explicitly share them via Context or prop.
  • Both trigger re-renders on state change: When the state managed by either hook changes, React schedules a re-render of the component to reflect the new state in the UI. This is identical behaviour.
  • Both return the current state value: useState returns [state, setState]; useReducer returns [state, dispatch]. In both cases, the first element is the current state value you read and display.
  • Both accept an initial state: useState(initialValue) and useReducer(reducer, initialState) both take an initial state that is used only on the first render.
  • Both support the functional update pattern (implicitly): Both hooks guarantee that state updates are based on the most recent state, not a stale closure value — useState via its setter callback setState(prev => ...), and useReducer because the reducer always receives the current state as its first argument.
  • Both can replace each other: Technically, anything you can do with useState you can do with useReducer and vice versa. They are two APIs solving the same problem at different scales of complexity.
jsx — same counter with useState and useReducer
// With useState
const [count, setCount] = useState(0);
const increment = () => setCount(prev => prev + 1);

// With useReducer — same end result
const reducer = (state, action) => {
  if (action.type === 'INCREMENT') return state + 1;
  return state;
};
const [count, dispatch] = useReducer(reducer, 0);
const increment = () => dispatch({ type: 'INCREMENT' });

// Both: clicking increment → count goes 0 → 1 → 2 → ...
// Both: trigger a re-render on each update
// Both: return current count as first array element

useReducer is a React hook that manages state through a reducer function — a pure function that takes the current state and an action, and returns the next state. It is inspired by the Redux pattern and is the built-in alternative to useState for more complex state logic.

Syntax:

jsx — useReducer signature
const [state, dispatch] = useReducer(reducerFn, initialState);

When to use useState:

  • The state is a simple primitive — a number, string, boolean, or a small, flat object.
  • The state updates are independent — each piece of state is updated separately with no relationship to other state values.
  • There are only a couple of ways to update the state (e.g., toggle on/off, increment/decrement).
  • You're building something small and the logic is easy to follow inline.

When to use useReducer:

  • The state is a complex object with multiple sub-values that update together (e.g., a form with 10 fields, a shopping cart item).
  • The next state depends on the previous state in non-trivial ways — multiple fields change simultaneously based on one action.
  • There are many different update operations on the same state (add item, remove item, update quantity, apply discount, clear cart — all acting on the same cart state).
  • You want to co-locate state logic — keeping all the "how state can change" logic in one reducer function, rather than scattered across multiple setter calls in event handlers.
  • You want easier testing — a reducer is a pure function that's trivial to unit test independently of React.

The React team's rule of thumb: If you find yourself writing useState multiple times for related state, and the update logic is getting complex — reach for useReducer.

While both hooks manage local state, they differ significantly in structure, scale, and philosophy:

  • State update mechanism: useState updates state directly — you call the setter with the new value (setCount(5)) or a function of the previous value. useReducer updates state indirectly — you dispatch an action (a description of what happened) and the reducer function decides what the new state should be.
  • Logic location: With useState, the update logic is inline at the call site — inside event handlers or callbacks. With useReducer, all update logic lives in one centralised reducer function outside the component — making it easier to audit all possible state transitions in one place.
  • State shape: useState is ideal for simple, independent values. useReducer is designed for a single complex state object where multiple properties may change together in response to one action.
  • Readability at scale: For simple cases, useState is more concise and easier to follow. For complex cases with many update types, useReducer is cleaner — the reducer makes all state transitions explicit and named.
  • Testability: A reducer is a pure JavaScript function ((state, action) => newState) with no side effects and no React dependency — it can be unit tested in isolation with a simple test runner, without mounting any components. useState logic is harder to test in isolation since it lives inside React components.
  • Performance: For complex state objects, useReducer can offer marginally better performance because React can optimise dispatch calls more aggressively than multiple independent setState calls. In practice, the difference is negligible for most apps.
jsx — form state: useState vs useReducer
// ❌ useState for complex related state — gets messy fast
const [name, setName]       = useState('');
const [email, setEmail]     = useState('');
const [loading, setLoading] = useState(false);
const [error, setError]     = useState(null);

// ✅ useReducer — all related state in one object
const initialState = { name: '', email: '', loading: false, error: null };

function formReducer(state, action) {
  switch (action.type) {
    case 'SET_FIELD':
      return { ...state, [action.field]: action.value };
    case 'SUBMIT_START':
      return { ...state, loading: true, error: null };
    case 'SUBMIT_ERROR':
      return { ...state, loading: false, error: action.error };
    default: return state;
  }
}

const [formState, dispatch] = useReducer(formReducer, initialState);

These two are the core building blocks of the useReducer pattern. They work in tandem — one to describe what happened, the other to decide what changes.

The Reducer Function:

The reducer is a pure function you define that contains all the logic for how your state should change. It takes two arguments — the current state and an action — and must return the new state. It should never have side effects and should never mutate the state directly — always return a new object.

  • Signature: (state, action) => newState
  • Pure function: Same inputs always produce the same output. No API calls, no randomness, no direct mutations.
  • Action: A plain JavaScript object describing what event occurred. By convention it has a type string property (e.g., 'INCREMENT', 'ADD_TO_CART') and an optional payload property carrying any additional data needed to compute the new state.
  • A switch statement on action.type is the standard pattern inside a reducer, though if/else works too.

The Dispatch Function:

dispatch is the function returned by useReducer that you call from your component to trigger a state update. Instead of providing the new state value directly (like setState), you pass it an action object describing what happened. React then calls your reducer with the current state and this action, and uses the return value as the new state — triggering a re-render.

  • Think of dispatch as sending a message to the reducer: "This thing happened — figure out the new state."
  • dispatch is stable across renders (like setState), so it's safe to include in useEffect dependency arrays without causing infinite loops.
  • You call it from event handlers: dispatch({ type: 'ADD_ITEM', payload: newItem })
jsx — reducer and dispatch in a shopping cart
// ── The Reducer Function — pure, defines all state transitions ──
function cartReducer(state, action) {
  switch (action.type) {
    case 'ADD_ITEM':
      return { ...state, items: [...state.items, action.payload] };
    case 'REMOVE_ITEM':
      return { ...state, items: state.items.filter(i => i.id !== action.payload) };
    case 'CLEAR_CART':
      return { ...state, items: [] };
    default:
      return state; // always return state for unknown actions
  }
}

// ── Component — uses dispatch to trigger state changes ──
function Cart() {
  const [cart, dispatch] = useReducer(cartReducer, { items: [] });

  const addProduct    = (item) => dispatch({ type: 'ADD_ITEM',    payload: item });
  const removeProduct = (id)   => dispatch({ type: 'REMOVE_ITEM', payload: id });
  const clearAll      = ()    => dispatch({ type: 'CLEAR_CART' });

  return (
    <div>
      <p>Items in cart: {cart.items.length}</p>
      <button onClick={() => addProduct({ id: 1, name: 'Laptop' })}>Add Laptop</button>
      <button onClick={clearAll}>Clear Cart</button>
    </div>
  );
}

In useReducer, the initial state is almost always defined as a JavaScript object (rather than a primitive like a number or string). This is intentional and central to the entire purpose of using useReducer over useState.

Why an object?

  • Groups related state together: The whole point of useReducer is to manage multiple, interconnected pieces of state as a single cohesive unit. An object is the natural data structure for this — it holds all the related properties (e.g., { items: [], total: 0, discount: 0, isLoading: false, error: null }) in one place. If you only had a single primitive value, you'd just use useState.
  • Atomic updates: When the reducer returns a new state, it returns a whole new object. Using spread syntax ({ ...state, fieldToChange: newValue }), you can update one or more properties while keeping the rest unchanged — all in a single action dispatch. This is impossible with multiple independent useState calls without multiple setter calls.
  • Single source of truth: Having all related state in one object means there's one definitive place to look at the "shape" of your state. You can define const initialState = {...} separately and reuse it — for initialisation, for resetting the form, or for test fixtures.
  • Consistent state transitions: The reducer always receives and returns the complete state object — ensuring that state transitions are total and consistent. You can never accidentally have isLoading: true and error: 'Something went wrong' at the same time if your reducer handles them together.
  • Extractability and reusability: An initialState object defined outside the component can be exported, imported, and reused in tests, storybook stories, or as a reset action payload — making code more modular.
jsx — initial state as object in useReducer
// Defined outside component — reusable, exportable, testable
const initialState = {
  username:  '',
  email:     '',
  password:  '',
  isLoading: false,
  error:     null,
  success:   false,
};

function authReducer(state, action) {
  switch (action.type) {
    case 'SET_FIELD':
      // spread preserves all other fields — only changes one
      return { ...state, [action.field]: action.value };

    case 'SUBMIT':
      // atomic: both isLoading and error update together
      return { ...state, isLoading: true, error: null };

    case 'SUCCESS':
      return { ...state, isLoading: false, success: true };

    case 'ERROR':
      return { ...state, isLoading: false, error: action.payload };

    case 'RESET':
      return initialState; // reuse initialState to reset the form!

    default: return state;
  }
}

function RegisterForm() {
  const [state, dispatch] = useReducer(authReducer, initialState);

  return (
    <form>
      <input
        value={state.username}
        onChange={e => dispatch({ type: 'SET_FIELD', field: 'username', value: e.target.value })}
      />
      {state.error && <p className="error">{state.error}</p>}
      <button onClick={() => dispatch({ type: 'RESET' })}>Reset</button>
    </form>
  );
}
Hooks — useCallback / useMemo / useRef / useLayoutEffect
6 Qs

useCallback is a React hook that memoises (caches) a function definition between renders. It returns the same function instance across renders as long as its dependencies haven't changed — instead of creating a brand-new function object on every render.

Why this matters — the problem it solves:

In JavaScript, every time a component re-renders, every function defined inside it is recreated as a new object in memory — even if the function logic is identical. This is usually harmless, but it becomes a performance problem in two specific scenarios:

  • Passing callbacks to memoised child components (React.memo): React.memo skips re-rendering a child if its props haven't changed (using shallow equality). But if the parent passes a callback as a prop and that callback is recreated on every render, the child sees a "new" prop every time — breaking the memoisation and causing unnecessary re-renders. useCallback ensures the callback reference is stable, so React.memo can do its job.
  • Callbacks listed as useEffect dependencies: If a function is created inside a component and used inside a useEffect dependency array, a new function reference every render causes the effect to re-run on every render — potentially creating infinite loops. Wrapping the function in useCallback stabilises its reference.

Important note: Don't overuse useCallback. Memoisation itself has a cost — React must store the previous function and compare dependencies. Only use it when you have a measurable performance problem caused by unstable function references.

jsx — useCallback preventing unnecessary child re-renders
import { useState, useCallback, memo } from 'react';

// Child wrapped in React.memo — skips render if props unchanged
const ExpensiveList = memo(({ onItemClick }) => {
  console.log('ExpensiveList rendered');
  return <button onClick={onItemClick}>Click Item</button>;
});

function Parent() {
  const [count, setCount] = useState(0);

  // ❌ Without useCallback — new function every render
  // const handleClick = () => console.log('clicked');
  // → ExpensiveList re-renders every time count changes

  // ✅ With useCallback — same function reference between renders
  const handleClick = useCallback(() => {
    console.log('item clicked');
  }, []); // no deps — never recreated

  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>Count: {count}</button>
      <ExpensiveList onItemClick={handleClick} />
      // ↑ won't re-render when count changes — handleClick is stable
    </div>
  );
}

useCallback has a straightforward signature that mirrors useEffect — it takes a function and a dependency array, and gives back a memoised version of that function.

Parameters:

  • Parameter 1 — fn (the function to memoize): The function definition you want to cache. On the initial render, React stores this function and returns it. On subsequent renders, React checks the dependencies — if nothing changed, it returns the previously cached function. If any dependency changed, React stores and returns the new function you passed.
  • Parameter 2 — dependencies (array): An array of reactive values that the function depends on — props, state, context values, or variables derived from them. React uses this to decide when to create a fresh version of the function. The rules are the same as useEffect — if the array is empty [], the function is memoised forever (created only once on mount). If it contains values, the function is recreated when any of those values change.

Return value:

  • Returns a memoised function — specifically, the same function object from the previous render if dependencies haven't changed, or a newly created function if they have.
  • On the very first render, it always returns the function you passed in (there is no previous cached version yet).
  • The returned function is identical in every way to the one you passed — same logic, same closure variables — just potentially the same object reference as the previous render.
jsx — useCallback parameters and return
// Signature
const memoizedFn = useCallback(fn, dependencies);

// ── Example 1: No dependencies — created once, never changes ──
const logHello = useCallback(() => {
  console.log('Hello!');
}, []);  // ← stable forever

// ── Example 2: With dependency — recreated when userId changes ──
const [userId, setUserId] = useState(1);

const fetchUser = useCallback(() => {
  fetch(`/api/users/${userId}`)
    .then(r => r.json())
    .then(data => console.log(data));
}, [userId]);  // ← recreated only when userId changes

// ── Return value — the memoised function ──
// fetchUser is the SAME function object between renders
// as long as userId doesn't change.

// ── useCallback vs useMemo ──
const memoFn  = useCallback(() => doX(), [dep]); // returns the function
const memoVal = useMemo(() => doX(), [dep]);      // returns doX()'s RESULT
// useCallback(fn, deps) === useMemo(() => fn, deps)

useMemo is a React hook that memoises (caches) the result of a computation between renders. It takes a function and a dependency array — on the initial render it runs the function and caches the return value. On subsequent renders, it skips the computation and returns the cached value, unless one of the dependencies has changed.

While useCallback caches a function definition, useMemo caches a computed value — the result of calling a function. Think: useCallback = remember the recipe; useMemo = remember the dish that was cooked.

When useMemo is genuinely useful:

  • Expensive calculations: Sorting, filtering, or transforming a large dataset on every render would be wasteful. Wrapping it in useMemo ensures the computation only runs when its inputs (e.g., the raw data or the sort key) actually change — not on every unrelated state update.
  • Stable object/array references for child components: If you derive an object or array from state and pass it as a prop, a new object is created on every render — breaking React.memo on the child. useMemo ensures the same reference is returned when the inputs haven't changed.
  • Avoiding recalculation in complex render logic: Any derived value that involves non-trivial computation (e.g., calculating totals, building lookup maps, running regex matches on large strings) is a candidate for useMemo.

Golden rule: Profile first, optimise second. Don't wrap every calculation in useMemo — the overhead of memoisation (storing the previous result, comparing dependencies on every render) can actually hurt performance for cheap operations. Only apply it when you've confirmed there's a measurable bottleneck.

jsx — useMemo for expensive computation
import { useState, useMemo } from 'react';

function ProductList({ products }) {
  const [filterText, setFilterText] = useState('');
  const [sortBy, setSortBy]         = useState('name');
  const [theme, setTheme]           = useState('dark');

  // ✅ Expensive filter + sort — only recalculates when
  // products, filterText, or sortBy change.
  // Changing `theme` does NOT trigger this calculation.
  const processedProducts = useMemo(() => {
    console.log('Filtering and sorting...'); // expensive!
    return products
      .filter(p => p.name.toLowerCase().includes(filterText.toLowerCase()))
      .sort((a, b) => a[sortBy] > b[sortBy] ? 1 : -1);
  }, [products, filterText, sortBy]); // ← deps that affect the result

  // ✅ Stable object reference for a memoised child
  const config = useMemo(() => ({ theme, itemsPerPage: 20 }), [theme]);

  return (
    <div>
      <input value={filterText} onChange={e => setFilterText(e.target.value)} />
      <p>Showing {processedProducts.length} products</p>
    </div>
  );
}

useRef returns a mutable ref object that persists for the full lifetime of the component. It is a box that holds a value in its .current property. The critical difference from useState is that mutating ref.current does NOT trigger a re-render — React is completely unaware of the change.

useRef serves two distinct and important purposes:

Purpose 1 — Direct DOM access:

React is declarative and normally manages the DOM for you. But sometimes you genuinely need direct, imperative access to a DOM element — to focus an input programmatically, measure its dimensions, trigger a video play/pause, or integrate a third-party non-React library. You attach a ref to a JSX element via the ref attribute, and React sets ref.current to the actual DOM node after mounting.

Purpose 2 — Persisting mutable values without triggering re-renders:

Sometimes you need to store a value that changes over time but whose change should not cause the component to re-render — like a timer ID from setInterval, a previous render's value, a flag tracking whether the component is still mounted, or a counter that is read but not displayed. Using useState for these would cause unnecessary re-renders. useRef is the right tool — its value persists across renders but doesn't trigger them.

jsx — two uses of useRef
import { useRef, useEffect, useState } from 'react';

// ── Use 1: DOM reference — focus input on mount ──
function SearchBar() {
  const inputRef = useRef(null);

  useEffect(() => {
    inputRef.current.focus(); // direct DOM access after mount
  }, []);

  return <input ref={inputRef} placeholder="Search..." />;
}

// ── Use 2: Persist value without re-render — store timer ID ──
function Stopwatch() {
  const [seconds, setSeconds] = useState(0);
  const timerRef = useRef(null); // stores interval ID — no re-render on change

  const start = () => {
    timerRef.current = setInterval(() => setSeconds(s => s + 1), 1000);
  };

  const stop = () => clearInterval(timerRef.current);

  return (
    <div>
      <p>{seconds}s</p>
      <button onClick={start}>Start</button>
      <button onClick={stop}>Stop</button>
    </div>
  );
}

// ── Use 3: Track previous state value ──
function Counter() {
  const [count, setCount] = useState(0);
  const prevCount = useRef(0);

  useEffect(() => {
    prevCount.current = count; // update ref after render — no re-render
  });

  return <p>Now: {count}, Before: {prevCount.current}</p>;
}

useLayoutEffect is a version of useEffect that fires synchronously after all DOM mutations but before the browser paints the screen. It has the exact same signature as useEffect — a callback and a dependency array — but a fundamentally different execution timing.

Understanding the render pipeline helps here:

  1. React renders your component (runs the function, builds the Virtual DOM diff).
  2. React commits the changes to the real DOM.
  3. useLayoutEffect fires here — synchronously, before the browser paints.
  4. The browser paints the updated screen (what the user sees).
  5. useEffect fires here — asynchronously, after the paint.

Key comparison — useEffect vs useLayoutEffect:

  • Timing: useEffect runs after the browser paints — asynchronously. useLayoutEffect runs before the browser paints — synchronously after DOM mutations.
  • Blocking: useEffect is non-blocking — the browser can paint freely before the effect runs. useLayoutEffect is blocking — it runs synchronously, so it can delay the browser paint if it does heavy work.
  • Use for DOM reads/writes: If you need to measure a DOM element (its width, height, scroll position) and immediately update the DOM based on that measurement — before the user sees anything — use useLayoutEffect. If you used useEffect for this, the user would briefly see the un-adjusted layout before the update — a visible flicker.
  • SSR (Server-Side Rendering): useLayoutEffect does not run on the server (React prints a warning). useEffect also doesn't run on the server, but the warning only applies to useLayoutEffect because its synchronous nature is browser-specific.
  • Default choice: Always start with useEffect. Only switch to useLayoutEffect if you notice a visual flicker or need synchronous DOM measurement.
jsx — useEffect vs useLayoutEffect timing
// useEffect — fires AFTER browser paints (async, non-blocking)
useEffect(() => {
  // Runs after user already sees the new screen
  // Safe for: API calls, subscriptions, logging, timers
  fetch('/api/data').then(...);
}, []);

// useLayoutEffect — fires BEFORE browser paints (sync, blocking)
useLayoutEffect(() => {
  // Runs BEFORE user sees the screen
  // Safe for: DOM measurements, synchronous DOM mutations
  const { width, height } = divRef.current.getBoundingClientRect();
  setDimensions({ width, height }); // update before paint → no flicker
}, []);

// Execution order (both in same component):
// 1. React renders (Virtual DOM diff)
// 2. React commits to real DOM
// 3. useLayoutEffect fires ← synchronous, blocks paint
// 4. Browser paints ← user sees the screen
// 5. useEffect fires ← async, after paint

The rule of thumb: start with useEffect always. Reach for useLayoutEffect only in specific, well-defined scenarios where the asynchronous timing of useEffect causes visible problems — typically a flicker or layout jump that the user can see.

Specific scenarios where useLayoutEffect is the right choice:

  • DOM measurement that affects layout: When you need to read a DOM element's size or position (e.g., getBoundingClientRect(), offsetWidth) and then immediately adjust the DOM based on it — like positioning a tooltip, popover, or dropdown relative to a trigger element. If you used useEffect, the element would momentarily render in the wrong position, then jump — causing a visible flicker. useLayoutEffect makes the measurement and adjustment before the browser ever paints, so the user sees the correct position from the start.
  • Preventing visible layout flashes: Animations or style changes that should happen before the first paint — like setting an initial scroll position, applying a transition origin point, or ensuring a modal is centred before appearing.
  • Synchronising with third-party imperative libraries: When integrating a library that manipulates the DOM directly (chart libraries, maps, rich text editors like Quill, canvas-based tools) and needs to be initialised or updated in sync with React's DOM changes.
  • Reading layout properties to set state: When the initial state of a component depends on an actual DOM measurement — for example, clamping text to a container width or setting the height of a resizable panel.

What to avoid: Don't use useLayoutEffect for data fetching, subscriptions, logging, or anything that doesn't need to happen before the browser paints. Blocking the paint unnecessarily will make your app feel sluggish and slow down Time to First Paint (TFP) metrics.

jsx — useLayoutEffect: tooltip positioning without flicker
function Tooltip({ targetRef, children }) {
  const tooltipRef = useRef(null);
  const [pos, setPos] = useState({ top: 0, left: 0 });

  // ✅ useLayoutEffect — measure and position BEFORE browser paints
  // If we used useEffect, tooltip would flash in wrong position first
  useLayoutEffect(() => {
    const target  = targetRef.current.getBoundingClientRect();
    const tooltip = tooltipRef.current.getBoundingClientRect();
    setPos({
      top:  target.top - tooltip.height - 8,     // 8px gap above
      left: target.left + (target.width / 2) - (tooltip.width / 2),
    });
  }, []); // runs once after initial DOM commit

  return (
    <div
      ref={tooltipRef}
      style={{ position: 'fixed', top: pos.top, left: pos.left }}
    >
      {children}
    </div>
  );
}

// ── When NOT to use useLayoutEffect ──
useLayoutEffect(() => {
  fetch('/api/data'); // ❌ no reason to block paint for this — use useEffect
}, []);
Hooks — Best Practices & Short Answers
9 Qs

React enforces two official rules for hooks — these are not suggestions, they are requirements. Violating them causes bugs that are difficult to diagnose because React's hook system depends on stable call order. The eslint-plugin-react-hooks package enforces both rules automatically.

Official Rule 1 — Only call hooks at the top level:

Never call hooks inside loops, conditionals, nested functions, or event handlers. React tracks hook state by the order in which hooks are called on each render. If you put a hook inside an if block, its call might be skipped on some renders, changing the call order and causing React to associate the wrong state with the wrong hook — leading to unpredictable, hard-to-debug bugs.

jsx — rule 1: top level only
// ❌ Violation — hook inside a condition (call order may change)
function Bad({ isLoggedIn }) {
  if (isLoggedIn) {
    const [name, setName] = useState(''); // ← NEVER do this
  }
}

// ✅ Correct — hook at top level, condition applied to its value
function Good({ isLoggedIn }) {
  const [name, setName] = useState(''); // ← always called
  if (!isLoggedIn) return null;           // condition applied after
  return <p>Hello {name}</p>;
}

Official Rule 2 — Only call hooks from React functions:

Hooks must be called from React functional components or from custom hooks. Never call them from regular JavaScript utility functions, class methods, or event listeners outside of React. This ensures React can track the hook's state and lifecycle correctly.

jsx — rule 2: react functions only
// ❌ Violation — hook inside a plain JS function
function fetchData() {
  const [data, setData] = useState(null); // ← NEVER, not a React fn
}

// ✅ Correct — inside a functional component
function DataComponent() {
  const [data, setData] = useState(null); // ← valid
  return <p>{data}</p>;
}

// ✅ Correct — inside a custom hook (starts with "use")
function useFetchData(url) {
  const [data, setData] = useState(null); // ← valid in custom hook
  return data;
}

Additional best practices beyond the two rules:

  • Name custom hooks with the use prefix: e.g. useFetch, useAuth. This is what allows the linter to treat them as hooks and apply the rules.
  • Keep the dependency array accurate in useEffect / useCallback / useMemo: Missing dependencies cause stale closure bugs; unnecessary ones cause excessive re-runs. Use the exhaustive-deps ESLint rule to catch errors automatically.
  • Don't over-optimise with useMemo / useCallback: Only apply them when you've measured a real performance issue. Premature memoisation adds noise to the code and has its own overhead.
  • Clean up side effects in useEffect: Always return a cleanup function for subscriptions, timers, or event listeners to prevent memory leaks.
  • Prefer multiple small state variables over one large object with useState: Unless the values are tightly coupled and change together — in which case, use useReducer.

A Custom Hook is a regular JavaScript function whose name starts with use and that calls one or more built-in React hooks inside it. Custom hooks let you extract and reuse stateful logic across multiple components — without duplicating code, without changing the component hierarchy, and without using higher-order components or render props.

They are one of the most powerful patterns in modern React. Any logic that involves hooks (useState, useEffect, useRef, etc.) and is needed in more than one place is a great candidate to be extracted into a custom hook.

Key characteristics:

  • Must start with use: This naming convention tells React (and the ESLint plugin) that this function follows the rules of hooks. Without the use prefix, the linter can't apply hook rules to it.
  • Can call any hook: A custom hook can use useState, useEffect, useRef, other custom hooks, or any combination.
  • State is isolated per component: Each component that calls the same custom hook gets its own independent state. Custom hooks share logic, not state. Two components calling useCounter() get two completely separate counters.
  • Can return anything: A custom hook can return a value, an array, an object, or nothing — whatever makes sense for its use case.
jsx — custom hooks examples
// ── Custom Hook 1: useFetch — reusable data fetching ──
function useFetch(url) {
  const [data, setData]       = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError]     = useState(null);

  useEffect(() => {
    fetch(url)
      .then(r => r.json())
      .then(d  => { setData(d); setLoading(false); })
      .catch(e => { setError(e); setLoading(false); });
  }, [url]);

  return { data, loading, error };
}

// ── Custom Hook 2: useLocalStorage — persist state ──
function useLocalStorage(key, initialValue) {
  const [value, setValue] = useState(() => {
    const stored = localStorage.getItem(key);
    return stored ? JSON.parse(stored) : initialValue;
  });

  const setStoredValue = (newVal) => {
    setValue(newVal);
    localStorage.setItem(key, JSON.stringify(newVal));
  };
  return [value, setStoredValue];
}

// ── Using them in components ──
function UserProfile() {
  const { data, loading, error } = useFetch('/api/user/1');
  const [theme, setTheme]        = useLocalStorage('theme', 'light');

  if (loading) return <p>Loading...</p>;
  if (error)   return <p>Error!</p>;
  return <h1>{data.name}</h1>;
}

useState() returns an array with exactly two elements, which you destructure immediately using JavaScript array destructuring syntax.

  • Element 0 — the current state value: This is the current value of the state variable. On the very first render it is the initial value you passed to useState(initialValue). On subsequent renders it is the most recently set value. You use this variable in your JSX and component logic to display or compute based on the current state.
  • Element 1 — the setter function: A function that you call to update the state. When called, it schedules a re-render of the component with the new state value. You can call it with a new value directly (setState(newValue)), or with a callback that receives the previous state and returns the new state (setState(prev => newValue)). The setter function identity is stable across renders — it never changes.

The array (rather than object) return makes destructuring flexible — you name the two elements whatever you like, following the [value, setValue] convention.

jsx — what useState returns
// useState returns [currentValue, setterFunction]
const stateArray = useState(0);
// stateArray[0] → 0  (current value)
// stateArray[1] → ƒ  (setter function)

// Destructured (conventional way)
const [count, setCount] = useState(0);

// You choose the names — all of these are valid:
const [isOpen, setIsOpen]     = useState(false);
const [userName, setUserName] = useState('');
const [items, setItems]       = useState([]);
const [user, setUser]         = useState(null);

// Using both return values
return (
  <div>
    <p>Count: {count}</p>               // ← use current value
    <button onClick={() => setCount(count + 1)}>+</button>  // ← call setter
    <button onClick={() => setCount(prev => prev - 1)}>-</button> // ← functional update
  </div>
);

useEffect is the hook for side effects — operations that reach outside the component's pure render cycle. Here are 3 practical, real-world scenarios where it is the right tool:

  • Scenario 1 — Fetching data from an API on component mount: When a component needs to load data from a server when it first appears (e.g., a user profile page loading user details, a dashboard loading analytics), you perform the fetch inside a useEffect with an empty dependency array []. This ensures the fetch runs exactly once after the initial render — equivalent to componentDidMount in class components. You update state with the fetched data, triggering a re-render that displays it.
  • Scenario 2 — Synchronising with an external system (subscriptions / event listeners): When a component needs to subscribe to something outside React — a WebSocket for real-time messages, a window resize event, a keyboard shortcut listener, or a browser geolocation watch — you set up the subscription in useEffect and crucially return a cleanup function to unsubscribe when the component unmounts (preventing memory leaks and ghost listeners).
  • Scenario 3 — Reacting to state or prop changes: When a side effect should re-run whenever a specific value changes — like re-fetching data when a URL parameter changes, updating the document title when a counter changes, or logging analytics when a tab changes — you list those values in the dependency array. React re-runs the effect only when those specific values change, not on every render.
jsx — 3 scenarios
// Scenario 1: Fetch on mount (once)
useEffect(() => {
  fetch('/api/user/1').then(r => r.json()).then(d => setUser(d));
}, []);

// Scenario 2: Subscribe + cleanup (event listener)
useEffect(() => {
  const onResize = () => setWidth(window.innerWidth);
  window.addEventListener('resize', onResize);
  return () => window.removeEventListener('resize', onResize);
}, []);

// Scenario 3: React to dependency change
useEffect(() => {
  document.title = `Page ${pageId} | MyApp`;
  fetch(`/api/pages/${pageId}`).then(r => r.json()).then(d => setPage(d));
}, [pageId]); // ← re-runs whenever pageId changes

You cannot put useEffect itself inside an if statement (that would violate the rules of hooks). But you can conditionally run the logic inside the effect — using conditions within the effect callback itself, or by controlling the dependency array strategically.

3 approaches to conditional effect execution:

  • 1. Early return inside the effect: Check your condition at the start of the effect callback and return early if the condition isn't met. The effect still "runs" on every relevant render, but the actual logic is skipped when the condition is false. This is the most common and cleanest pattern.
  • 2. Dependency array drives the re-run: Include the value that drives the condition in the dependency array. The effect only re-runs when that value changes — making the dependencies themselves the condition for execution.
  • 3. Conditional mounting (via JSX): If an effect should only ever run when a certain feature is active, conditionally render the entire component that contains that effect. When the component unmounts, the effect's cleanup runs automatically. This is the most robust approach for complex cases.
jsx — conditional effects
// ── Approach 1: Guard clause inside the effect ──
const [userId, setUserId] = useState(null);

useEffect(() => {
  if (!userId) return; // ← condition: skip if no userId
  fetch(`/api/users/${userId}`)
    .then(r => r.json())
    .then(d => setUser(d));
}, [userId]); // runs when userId changes, but guards against null

// ── Approach 2: Dependency array as the condition ──
const [searchTerm, setSearchTerm] = useState('');

useEffect(() => {
  // Only runs when searchTerm is truthy AND changes
  if (!searchTerm.trim()) return;
  fetch(`/api/search?q=${searchTerm}`)
    .then(r => r.json())
    .then(d => setResults(d));
}, [searchTerm]);

// ── Approach 3: Conditional rendering = conditional effect ──
function App() {
  const [showMap, setShowMap] = useState(false);
  return showMap ? <MapComponent /> : null;
  // MapComponent's useEffect only runs when it's mounted (showMap=true)
  // and cleans up when unmounted (showMap=false)
}

React Hooks were introduced in React 16.8 to solve several long-standing pain points in the pre-hooks React development model. They fundamentally changed how React code is written and organised.

  • Problem 1 — Stateless functional components couldn't use state or lifecycle: Before hooks, if a component needed state or access to lifecycle events (componentDidMount, etc.), you had to convert it to a class component — even for simple cases. This forced unnecessary class boilerplate for nearly all non-trivial components. Hooks let functional components do everything classes could, with far less code.
  • Problem 2 — Logic reuse was awkward and complex: Sharing stateful logic between components required verbose, hard-to-follow patterns like Higher-Order Components (HOCs) or Render Props. These patterns wrapped components in multiple layers, creating confusing "wrapper hell" in the component tree that was hard to debug in React DevTools. Custom hooks enable the same logic reuse simply by calling a function — no extra components, no nesting.
  • Problem 3 — Related logic was split across lifecycle methods: In class components, logic for a single feature (e.g., setting up a data subscription and cleaning it up) was split across componentDidMount, componentDidUpdate, and componentWillUnmount — three separate methods. Unrelated logic was often grouped together in the same method. useEffect lets you co-locate related logic in one place, organising code by concern rather than lifecycle phase.
  • Problem 4 — this binding confusion in class components: The unpredictable behaviour of this in JavaScript caused common bugs in class component event handlers — requiring .bind(this) in constructors or arrow function class properties as workarounds. Functional components with hooks have no this at all.

Lifecycle methods in class components were functional but came with significant friction. Hooks provide a cleaner, more composable alternative.

  • Logic co-location: With lifecycle methods, related logic was spread across three separate methods (componentDidMount to set up, componentDidUpdate to react to changes, componentWillUnmount to clean up). With useEffect, all of this lives in one unified block — the setup, the re-run condition, and the cleanup are all together, making the code much easier to read and reason about.
  • Reusable stateful logic: You cannot extract lifecycle logic from a class component and share it with another class component — the lifecycle methods are baked into the class. With hooks, stateful logic (including effects) can be extracted into a custom hook and reused across any number of components with a single function call.
  • No this binding issues: Class lifecycle methods suffer from this binding bugs. Forgetting to .bind(this) or misunderstanding arrow function class properties caused hard-to-debug errors. Functional components with hooks have no this — it's simply not a concept you need to manage.
  • Smaller, more readable components: Hooks encourage composing a component from small, focused custom hooks rather than one large class with many lifecycle methods intermixed with business logic. This leads to smaller, more testable, and more maintainable code.
  • Better optimisation by React: The React team has stated that hooks work better with future React features like Concurrent Mode and React Server Components. Class components are in maintenance mode — no new features are planned for them.
jsx — lifecycle methods vs useEffect
// 🏛 Class — logic split across THREE methods
class ChatRoom extends Component {
  componentDidMount()              { subscribeToRoom(this.props.roomId); }
  componentDidUpdate(prevProps) {
    unsubscribeFromRoom(prevProps.roomId);
    subscribeToRoom(this.props.roomId);
  }
  componentWillUnmount()           { unsubscribeFromRoom(this.props.roomId); }
}

// ✅ Hooks — same logic, ONE unified place
function ChatRoom({ roomId }) {
  useEffect(() => {
    subscribeToRoom(roomId);               // setup
    return () => unsubscribeFromRoom(roomId); // cleanup
  }, [roomId]); // re-runs when roomId changes
}

The Context API is a built-in React feature (no extra library needed) that provides a way to share data across the component tree without passing props manually at every level. It creates a global broadcast channel — a Provider pumps data in at the top, and any Consumer anywhere below in the tree can tap into it directly.

The 3 core parts of the Context API:

  • React.createContext(defaultValue): Creates and returns a Context object. This object acts as the identifier — the Provider and all Consumers must reference this same object. The optional defaultValue is used only when a Consumer has no Provider above it in the tree.
  • <Context.Provider value={...}>: A component that wraps the subtree that needs access to the context value. The value prop is the data being shared. When the value changes, all components consuming this context automatically re-render.
  • useContext(Context) hook: The modern way for any functional component to read the context value. Pass the Context object and receive the current value from the nearest Provider above. Replaces the older <Context.Consumer> render-prop pattern.

Best use cases: Authentication state (current user), app theme (light/dark), locale/language settings, feature flags. Avoid for high-frequency updates (every keystroke) — all consumers re-render on every value change, which can hurt performance. For those cases, prefer Zustand or Redux.

jsx — full context api flow
import { createContext, useContext, useState } from 'react';

// Step 1: Create
const ThemeContext = createContext('light');

// Step 2: Provide (wrap your app or subtree)
export function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  const toggleTheme = () => setTheme(t => t === 'light' ? 'dark' : 'light');
  return (
    <ThemeContext.Provider value={{ theme, toggleTheme }}>
      {children}
    </ThemeContext.Provider>
  );
}

// Step 3: Consume from any nested component
function Header() {
  const { theme, toggleTheme } = useContext(ThemeContext);
  return (
    <header className={theme}>
      <button onClick={toggleTheme}>Toggle {theme} mode</button>
    </header>
  );
}

Here is a concise reference of all major built-in React hooks and their purpose — a great table to memorise for interviews:

  • useState(initialValue) — Adds reactive local state to a functional component. Returns [value, setter]. Every state change schedules a re-render.
  • useEffect(fn, deps) — Runs side effects (data fetching, subscriptions, DOM manipulation, timers) after render. Supports cleanup via return function. Replaces componentDidMount, componentDidUpdate, and componentWillUnmount.
  • useContext(Context) — Subscribes to a Context and returns its current value. Eliminates prop drilling for shared global data like auth, theme, and locale.
  • useReducer(reducer, initialState) — Alternative to useState for complex state logic with multiple sub-values or many update types. Uses a dispatch/action/reducer pattern inspired by Redux.
  • useRef(initialValue) — Returns a mutable { current: value } object that persists across renders without causing re-renders. Used for DOM access and storing mutable non-state values (timer IDs, previous values).
  • useMemo(fn, deps) — Caches the result of an expensive calculation. Recalculates only when dependencies change. Prevents unnecessary heavy computations on every render.
  • useCallback(fn, deps) — Caches a function definition. Returns the same function reference across renders unless dependencies change. Prevents child component re-renders when passing stable callbacks to React.memo children.
  • useLayoutEffect(fn, deps) — Same as useEffect but fires synchronously after DOM mutations and before the browser paints. Used for DOM measurements and preventing layout flicker.
  • useImperativeHandle(ref, fn, deps) — Customises the value exposed to a parent component when using forwardRef. Lets a parent call specific child methods imperatively (e.g., childRef.current.focus()).
  • useDebugValue(value) — Displays a label for a custom hook in React DevTools. Used purely for debugging custom hooks — no runtime effect.
  • useId() — (React 18+) Generates a unique, stable ID that is consistent between server and client renders. Used for accessibility attributes like htmlFor / id pairs.
  • useTransition() — (React 18+) Marks a state update as non-urgent, letting React keep the UI responsive while processing the update in the background. Returns [isPending, startTransition].
  • useDeferredValue(value) — (React 18+) Returns a deferred version of a value that "lags behind" the latest value during high-priority updates. Used to keep fast interactions responsive while deferring slow re-renders.

Every React component goes through a defined sequence of events from the moment it is created to the moment it is removed from the screen. These stages are collectively called the Component Lifecycle. Understanding these phases is fundamental — it determines when to fetch data, set up subscriptions, perform DOM operations, and clean up resources.

React defines three main lifecycle phases:

  • 1. Mounting Phase — "Birth": The component is being created and inserted into the DOM for the very first time. This is when the initial state and props are set, JSX is rendered, and the real DOM node is created. After mounting, the component is visible on the screen. This is the right time to fetch initial data, set up subscriptions, or initialise third-party libraries.
    Class methods: constructor() → render() → componentDidMount()
    Hook equivalent: useEffect(() => {...}, [])
  • 2. Updating Phase — "Growth": The component re-renders in response to changes in its state or props. React re-runs the render, diffs the Virtual DOM, and applies the minimal set of changes to the real DOM. This can happen many times throughout a component's life. This phase is where you react to data changes — re-fetching when a prop ID changes, syncing with external systems, etc.
    Class methods: render() → componentDidUpdate(prevProps, prevState)
    Hook equivalent: useEffect(() => {...}, [dep1, dep2])
  • 3. Unmounting Phase — "Death": The component is being removed from the DOM — because it's conditionally hidden, the user navigated away, or the parent stopped rendering it. This is the critical phase for cleanup — clearing timers, unsubscribing from WebSockets, removing event listeners — to prevent memory leaks and stale callbacks.
    Class method: componentWillUnmount()
    Hook equivalent: The cleanup function returned from useEffect
jsx — three lifecycle phases mapped to hooks
function MyComponent({ userId }) {
  const [user, setUser] = useState(null);

  useEffect(() => {
    // ── MOUNTING: runs once after first render ──
    console.log('Component mounted');
    subscribeToUpdates();

    return () => {
      // ── UNMOUNTING: cleanup before removal ──
      console.log('Component unmounted');
      unsubscribeFromUpdates();
    };
  }, []); // ← empty array = mount/unmount only

  useEffect(() => {
    // ── UPDATING: re-runs when userId changes ──
    console.log('userId changed, re-fetching...');
    fetch(`/api/users/${userId}`)
      .then(r => r.json())
      .then(d => setUser(d));
  }, [userId]); // ← runs on mount + every userId change

  return <p>{user?.name}</p>;
}

Lifecycle methods are special methods built into React class components that React automatically calls at specific points during a component's life. By overriding these methods in your class, you can run code at exactly the right moment — on first render, after updates, or before removal.

These are the most important lifecycle methods, organised by phase:

🟢 Mounting Phase:

  • constructor(props) — The first method called. Used to initialise this.state and bind event handlers. Must call super(props) first.
  • static getDerivedStateFromProps(props, state) — Called before every render. Returns an object to update state based on props, or null to make no changes. Rarely needed.
  • render() — The only required method. Returns JSX (the component's UI). Must be a pure function — no side effects.
  • componentDidMount() — Called immediately after the component is inserted into the DOM. The ideal place for API calls, subscriptions, and third-party library initialisation.

🟡 Updating Phase:

  • shouldComponentUpdate(nextProps, nextState) — Called before re-rendering when new props or state arrive. Return false to skip the re-render and optimise performance. (Replaced by React.PureComponent or React.memo in modern code.)
  • render() — Re-runs whenever state or props change.
  • getSnapshotBeforeUpdate(prevProps, prevState) — Called just before DOM changes are applied. Used to capture information (e.g., scroll position) before the update. Whatever it returns is passed to componentDidUpdate.
  • componentDidUpdate(prevProps, prevState, snapshot) — Called after re-rendering. Used to react to prop/state changes (e.g., re-fetch data when an ID prop changes). Always compare with previous props/state to avoid infinite loops.

🔴 Unmounting Phase:

  • componentWillUnmount() — Called immediately before the component is removed from the DOM. Used to cancel API requests, clear timers, and remove event listeners. You should not call setState here.

Error Handling:

  • componentDidCatch(error, info) and static getDerivedStateFromError(error) — Used to build Error Boundaries — class components that catch JavaScript errors in their child tree and display fallback UI instead of crashing the whole app.

  render()                    { return <p>{this.state.data}</p>; }
  componentDidMount()         { this.setState({ data: 'Loaded!' }); }

  // ── UPDATING ──
  componentDidUpdate(prev) {
    if (prev.id !== this.props.id) fetch(`/api/${this.props.id}`);
  }

  // ── UNMOUNTING ──
  componentWillUnmount()      { clearInterval(this.timerId); }
}

In a React class component, the constructor is a special method that runs first — before the component mounts, before render(), before anything else. It is a standard JavaScript ES6 class constructor, and React calls it automatically when the component is being instantiated.

The two specific, valid reasons to use a constructor in React:

  • 1. Initialising local state (this.state): The constructor is the only place where you should directly assign to this.state. Everywhere else in a class component, you must use this.setState(). In the constructor, you define the component's initial state as a plain object: this.state = { count: 0, name: '' }.
  • 2. Binding event handler methods: Regular class methods don't automatically bind this to the class instance. If you pass a method like this.handleClick as an event handler without binding, this will be undefined when it runs. Binding in the constructor (this.handleClick = this.handleClick.bind(this)) is the traditional fix. The modern alternative is using arrow function class properties — which avoids the need for binding altogether.

When NOT to use a constructor:

  • If you're not initialising state or binding methods, you don't need a constructor at all — React handles the base setup for you.
  • Never put API calls or side effects in the constructor — those belong in componentDidMount().
  • Never call setState() in the constructor — always use direct assignment to this.state.
  • In modern React with functional components and hooks, constructors are entirely replaced by useState(initialValue) — there is no constructor concept in functional components.
jsx — constructor: two valid uses
class LoginForm extends Component {
  constructor(props) {
    super(props); // ← ALWAYS first line — initialises React.Component

    // ✅ Use 1: Initialise state directly
    this.state = {
      email:    '',
      password: '',
      error:    null,
    };

    // ✅ Use 2: Bind event handlers
    this.handleSubmit = this.handleSubmit.bind(this);
  }

  handleSubmit(e) {
    e.preventDefault();
    console.log(this.state); // `this` works because of binding above
  }

  render() {
    return <form onSubmit={this.handleSubmit}>...</form>;
  }
}

// Modern alternative — arrow function class property avoids binding
class Modern extends Component {
  state = { count: 0 };  // ← no constructor needed for state
  handleClick = () => {   // ← arrow fn auto-binds `this`
    this.setState({ count: this.state.count + 1 });
  };
}

In JavaScript, super() inside a constructor calls the parent class's constructor. In a React class component, the parent class is React.Component. Calling super(props) is not optional — it is mandatory and must always be the very first line inside a class component constructor.

What super(props) specifically does:

  • Initialises React.Component: By calling super(props), you invoke the parent React.Component constructor, which sets up all the internal React machinery — the component's state management system, the update queue, and the internal fiber node. Without it, the component simply won't work as a React component — React will throw an error.
  • Makes this.props available inside the constructor: This is the subtle and commonly misunderstood part. If you call super() without passing props (just super()), then this.props will be undefined inside the constructor body — even though React sets this.props after the constructor returns. Passing props to super ensures this.props is immediately available throughout the constructor, including in any code that runs after super().
  • ES6 class rule: In any JavaScript class that extends another class, you cannot use this before calling super(). Attempting to do so throws a ReferenceError. This is a JavaScript language rule, not a React-specific one.

class MyComponent extends Component {

  // ❌ No super — JavaScript throws ReferenceError
  // constructor(props) {
  //   this.state = { x: 1 }; // Error: Must call super first
  // }

  // constructor(props) {
  //   super();
  //   console.log(this.props); // → undefined (bug!)
  // }

  // ✅ Correct — super(props) first, this.props available everywhere

    super(props);                    // ← 1st: initialises React.Component
    console.log(this.props.title);  // ← works! props available now
    this.state = {

The render() method is the only required method in a React class component and the most central part of the entire lifecycle. It is the bridge between your component's data (state and props) and the browser UI. React calls render() every time it needs to determine what the component's output should look like — both on first mount and on every subsequent state or props change.

What render() does:

  • Returns JSX (the component's UI description): The return value is a React element tree (JSX) that describes what should be rendered to the DOM. React takes this description and — via the Virtual DOM — efficiently updates only what has changed in the actual browser DOM.
  • Must be a pure function: This is critical. render() must not have side effects. It should not modify state, make API calls, start timers, or interact with the browser directly. Given the same this.state and this.props, it must always return the same JSX. Side effects belong in lifecycle methods like componentDidMount and componentDidUpdate.
  • Called during both mounting and updating: In mounting, it creates the initial virtual DOM tree. In updating (after setState or new props), it re-runs to produce the new virtual DOM — which React then diffs against the previous version to compute the minimal DOM changes.
  • Valid return values: JSX elements, arrays/fragments of elements, strings and numbers (rendered as text nodes), null (renders nothing), or a React Portal.

Equivalent in functional components: The entire function body is the render — every time a functional component re-renders, React calls the function from scratch. The return statement is the equivalent of render()'s return.

jsx — render() method in depth
class Dashboard extends Component {
  constructor(props) {
    super(props);
    this.state = { isAdmin: true, count: 0 };
  }

  render() {
    // ✅ Can read state and props
    const { title } = this.props;
    const { isAdmin, count } = this.state;

    // ✅ Can have conditional logic
    if (!isAdmin) return <p>Access Denied</p>;

    // ❌ NEVER do this in render — side effect!
    // fetch('/api/data'); // ← wrong place
    // this.setState({});  // ← infinite loop!

    // ✅ Returns JSX — the component's UI
    return (
      <div>
        <h1>{title}</h1>
        <p>Count: {count}</p>
        {count > 10 && <span>High count!</span>}
      </div>
    );
  }
}

In class components, state is maintained through the this.state object and updated exclusively via this.setState(). The state system in class components is more rigid and centralised compared to the multiple independent useState variables you use in functional components.

3-step state lifecycle in class components:

  • Step 1 — Initialise in constructor: State must be initialised as a plain JavaScript object assigned to this.state inside the constructor. This is the only place where you can directly assign to this.state — everywhere else, direct assignment is forbidden.
  • Step 2 — Read via this.state.propertyName: Access individual state properties using this.state.propertyName anywhere in the class — in render(), lifecycle methods, or event handlers.
  • Step 3 — Update via this.setState(): To update state, call this.setState() with either a partial object (React merges it with the existing state — unlike useState's setter which replaces it) or a callback function that receives the previous state and returns the updated partial object (safer when new state depends on previous state).

Critical rules for class component state:

  • Never mutate state directly: this.state.count = 5 is wrong — it bypasses React's update mechanism and won't trigger a re-render.
  • setState() is asynchronous: React may batch multiple setState calls together. Never read from this.state immediately after calling setState expecting the new value — use the callback form or componentDidUpdate.
  • setState() merges, not replaces: this.setState({ count: 1 }) only updates count — all other state properties remain untouched. This is different from useState's setter, which completely replaces the state value.
jsx — state in class components
class ShoppingCart extends Component {
  constructor(props) {
    super(props);
    // Step 1: Initialise state in constructor
    this.state = {
      items:    [],
      total:    0,
      isOpen:   false,
    };
  }

  addItem = (item) => {
    // Step 3a: Partial object update — only `items` changes, rest kept
    this.setState({ items: [...this.state.items, item] });
  };

  updateTotal = () => {
    // Step 3b: Functional form — safe when new state depends on previous
    this.setState(prevState => ({
      total: prevState.items.reduce((sum, i) => sum + i.price, 0)
    }));
  };

  render() {
    // Step 2: Read state
    const { items, total, isOpen } = this.state;
    return (
      <div>
        <p>{items.length} items — Total: ₹{total}</p>
        <button onClick={() => this.setState({ isOpen: !isOpen })}>
          {isOpen ? 'Close' : 'Open'} Cart
        </button>
      </div>
    );
  }
}

componentDidMount() is a lifecycle method that React calls immediately after a component has been inserted into the DOM for the first time. It runs once — and only once — per component instance. It is the most important and most commonly used lifecycle method in class components.

Why "after mounting" is the ideal moment for many operations:

Before componentDidMount, the component exists only in memory (virtual DOM). After it fires, the component is fully in the real browser DOM — which means you can safely interact with it programmatically, and the user can see the initial (often empty or loading) state on screen.

Primary uses of componentDidMount():

  • Fetching initial data from an API: This is the #1 use case. You make your API call here, and when the response arrives, call this.setState() with the data — triggering a re-render that shows the actual content. The component renders an initial loading state first, then updates with real data.
  • Setting up subscriptions and event listeners: WebSocket connections, global window event listeners, browser API subscriptions (geolocation watch, IntersectionObserver, etc.) should be initiated here — and always cleaned up in componentWillUnmount().
  • Directly accessing DOM nodes: If you need to read or manipulate a specific DOM element (e.g., measure its size, focus an input, initialise a third-party library that needs a DOM node), componentDidMount is the right place — the DOM is guaranteed to be available.
  • Initialising third-party libraries: Chart libraries, map SDKs, rich text editors, and other imperative JavaScript libraries often need a DOM container to attach to — initialise them here.

Hook equivalent: useEffect(() => { ... }, []) — an effect with an empty dependency array runs exactly once after the first render, equivalent to componentDidMount.

jsx — componentDidMount usage
class UserProfile extends Component {
  constructor(props) {
    super(props);
    this.state = { user: null, loading: true, error: null };
  }

  // Called once — immediately after component appears in DOM
  componentDidMount() {
    // ✅ Use 1: Fetch initial data
    fetch(`/api/users/${this.props.userId}`)
      .then(res => res.json())
      .then(data => this.setState({ user: data, loading: false }))
      .catch(err => this.setState({ error: err.message, loading: false }));

    // ✅ Use 2: Set up event listener
    window.addEventListener('resize', this.handleResize);

    // ✅ Use 3: Direct DOM access
    this.inputRef.current.focus();
  }

  // Always pair with cleanup in componentWillUnmount
  componentWillUnmount() {
    window.removeEventListener('resize', this.handleResize);
  }

  render() {
    const { user, loading, error } = this.state;
    if (loading) return <p>Loading...</p>;
    if (error)   return <p>Error: {error}</p>;
    return <h1>{user.name}</h1>;
  }
}

componentDidUpdate(prevProps, prevState, snapshot) is a lifecycle method that React calls immediately after every re-render — i.e., after the component updates due to new state or new props. It does not run after the initial mount (that's componentDidMount's job). It is the correct place to perform side effects that must happen in response to a change.

Parameters it receives:

  • prevProps: The props the component had before the re-render. Comparing prevProps with this.props lets you detect what specifically changed.
  • prevState: The state the component had before the update. Compare with this.state to detect state-driven changes.
  • snapshot: Whatever value was returned by getSnapshotBeforeUpdate(), if implemented. Used for things like restoring scroll position after a DOM update.

Primary use cases:

  • Re-fetching data when a prop changes: If a component receives a user ID as a prop and needs to load that user's data, you re-fetch whenever the ID prop changes. This is the most common pattern.
  • Syncing with external systems after a state change: Updating a chart library, posting an analytics event, or syncing with localStorage when a specific piece of state updates.
  • DOM manipulation after an update: Scrolling to the bottom of a chat list after new messages are added, adjusting scroll position, or measuring a newly-rendered element.

⚠️ The most common bug — infinite loop: If you call this.setState() inside componentDidUpdate without wrapping it in a comparison condition, it will trigger another re-render → another componentDidUpdate → another setState → infinite loop. Always compare prevProps/prevState before calling setState.

Hook equivalent: useEffect(() => { ... }, [dep]) — runs after mount and after every render where dep changed.

jsx — componentDidUpdate usage and infinite loop prevention
class PostDetail extends Component {
  state = { post: null, loading: true };

  componentDidMount() {

componentWillUnmount() is the last lifecycle method to run before a component is permanently removed from the DOM. React calls it right before tearing down the component. Once this method returns, the component is gone — no more renders, no more state updates.

Its singular purpose is cleanup. Any resources, connections, or listeners that were set up during the component's life (typically in componentDidMount) must be released here to prevent memory leaks, ghost event listeners, stale API responses updating unmounted components, and other hard-to-debug issues.

What you should always clean up here:

  • Timers: If you started a setInterval or setTimeout in componentDidMount, clear it here with clearInterval(this.timerId) or clearTimeout(this.timerId). Without this, the timer continues firing and may try to call setState on an unmounted component — causing the infamous React warning: "Can't perform a React state update on an unmounted component."
  • Event listeners: Remove any global event listeners added to window, document, or other DOM elements — resize, scroll, keydown, etc.
  • WebSocket / real-time subscriptions: Close WebSocket connections and unsubscribe from any pub/sub or observable subscriptions (Firebase, RxJS, Socket.io).
  • API request cancellation: Cancel any in-flight fetch requests using AbortController to prevent the response handler from calling setState on the now-dead component.
  • Third-party library teardown: Destroy chart instances, map widgets, or any other imperative library that requires explicit cleanup to release its DOM references and event bindings.

Important restriction: You must not call this.setState() inside componentWillUnmount(). The component is about to be destroyed — triggering a state update here will never result in a re-render and React will log a warning.

Hook equivalent: The cleanup function returned from useEffect — return () => { cleanup here }.

class LiveFeed extends Component {
  constructor(props) {
    super(props);

    this.socket     = null;
    this.controller = new AbortController();
    this.state      = { messages: [] };

  componentDidMount() {
    // Set up resources
    this.timerId = setInterval(() => this.ping(), 5000);
    this.socket  = new WebSocket('wss://example.com/feed');
    this.socket.onmessage = (e) => this.setState(prev => ({
      messages: [...prev.messages, e.data]
    }));

    fetch('/api/initial', { signal: this.controller.signal })
      .then(r => r.json())
      .then(d => this.setState({ messages: d }));

    window.addEventListener('focus', this.handleFocus);
  }

  componentWillUnmount() {
    // ✅ Clean up EVERYTHING set up in componentDidMount

    this.socket.close();                    // close websocket
    this.controller.abort();               // cancel fetch
    window.removeEventListener('focus', this.handleFocus); // remove listener

    // ❌ Never do this
    // this.setState({ messages: [] }); // pointless, causes warning
  }

  render() { return <ul>{this.state.messages.map(m => <li key={m}>{m}</li>)}</ul>; }

There are two valid ways to initialise state in a React class component. Both are widely used — the choice depends on whether you need access to props during initialisation.

Method 1 — Inside the constructor (traditional approach):

You define a constructor, call super(props), and then directly assign to this.state. This is the only place a direct assignment to this.state is allowed. Use this method when the initial state needs to be derived from this.props — since props are available inside the constructor via this.props after super(props).

Method 2 — Class field declaration (modern shorthand):

Using the ES2022 class fields syntax (supported by Babel/CRA/Vite out of the box), you can declare state = { ... } directly as a class property — no constructor needed. This is cleaner, less boilerplate, and is the preferred modern approach when you don't need to reference props in the initial state. It is syntactic sugar that Babel compiles to the constructor approach behind the scenes.

What you must never do: Never initialise or update state outside the constructor by directly assigning this.state = {...} in other methods. And never call this.setState() inside the constructor — use the direct assignment there instead.

      items:    [],
    };
  }

}

A component re-renders during the Updating Phase — the second of React's three lifecycle phases. Re-rendering is React's response to any change that requires the UI to reflect new data. The component is not removed and recreated — its internal state is preserved, and only the parts of the real DOM that actually changed are updated.

4 triggers that cause a re-render (all in the Updating Phase):

  • 1. State change (this.setState() / useState setter): The most common trigger. Whenever you call this.setState() in a class component, or a useState setter in a functional component, React schedules a re-render of that component and all its children. React may batch multiple setState calls in the same event handler into a single re-render for efficiency.
  • 2. New props from a parent: When a parent component re-renders, it re-passes props to its children. If the props received by a child have changed (or even if the parent just re-renders and passes the same props — unless the child is wrapped in React.memo), the child re-renders too.
  • 3. Context value change: If a component consumes a Context (via useContext or <Context.Consumer>) and the Provider's value changes, every consumer of that context re-renders — regardless of where they are in the component tree.
  • 4. Forced update (forceUpdate() — class only): A class component method that forces a re-render bypassing shouldComponentUpdate. Rarely used and generally a sign of design issues — avoid it.

When does a component NOT re-render?

  • When shouldComponentUpdate() returns false (class components).
  • When the component is wrapped in React.memo and receives the same props (shallow equality check).
  • When useMemo or useCallback ensure derived values and callbacks have stable references.
  • When the new state value is identical to the current state (React bails out — no re-render for primitive values).
jsx — what triggers and prevents re-renders
function Parent() {
  const [count, setCount] = useState(0);
  const [theme, setTheme] = useState('dark');

  return (
    <div>
      // Trigger 1 — state change re-renders Parent + all children
      <button onClick={() => setCount(c => c + 1)}>Count: {count}</button>

      // Child re-renders when Parent re-renders (new props ref)
      <Child theme={theme} />

      // Memoised child — only re-renders if theme changes, not when count changes
      <MemoChild theme={theme} />
    </div>
  );
}

// React.memo — prevents re-render if props haven't changed
const MemoChild = React.memo(({ theme }) => {
  console.log('MemoChild rendered'); // only logs when theme changes
  return <p>Theme: {theme}</p>;
});

Nothing breaks — a constructor is not required in a React class component. If you don't define one, JavaScript automatically calls the parent class constructor behind the scenes. The component still works perfectly as a React component with full access to this.props, lifecycle methods, and the render() method.

What JavaScript does without an explicit constructor:

ES6 classes have a default behaviour — if you don't define a constructor in a child class, JavaScript implicitly creates one that looks like this:

jsx — implicit constructor JavaScript creates
// JavaScript implicitly does this if you omit the constructor:
constructor(...args) {
  super(...args); // calls React.Component constructor with props automatically
}

This means super(props) is still called — just automatically. So this.props is still set up correctly by React, and all lifecycle methods work normally.

The two cases where you DO need an explicit constructor:

  • You need to initialise this.state using props: If initial state depends on props, you must write the constructor to access this.props after calling super(props).
  • You need to bind event handler methods: If you use the traditional binding pattern (not arrow function class properties), you need the constructor for this.handleClick = this.handleClick.bind(this).

Modern practice: With class field syntax, you rarely need a constructor at all. State is declared as a class field (state = {...}) and methods are defined as arrow functions — both without a constructor.

  }

Technically, in modern React (16.8+), you don't need class components anymore — functional components with hooks can do everything class components can, and generally do it better. However, understanding why class components still exist and when you might encounter them is important.

Reasons class components still matter:

  • Legacy codebase maintenance: Millions of real-world production applications were built with class components before hooks existed. As a developer joining any team, you will almost certainly encounter class components — in older open-source projects, at companies with established codebases, or in older npm packages. You need to be able to read, understand, and maintain them even if you write new code with functional components.
  • Error Boundaries — the one thing only classes can do: As of React 18, Error Boundaries can only be implemented as class components. They use the componentDidCatch and getDerivedStateFromError lifecycle methods, which have no hook equivalents yet. Error Boundaries catch JavaScript errors in child component trees and show fallback UI — a critical feature for production apps. So even in a fully hooks-based app, you may have one or two class components dedicated to error boundary logic.
  • Gradual migration: Teams don't rewrite everything at once. A healthy codebase may have a mix of class and functional components — it's perfectly valid for them to coexist and even render each other.
  • Interview preparation: Interviewers frequently ask about class components and lifecycle methods to assess the depth of your React knowledge. Understanding them shows you can work with any React codebase, not just greenfield projects.

In a new project started today: Write 100% functional components. Use class components only for Error Boundaries — or until React provides a hook-based Error Boundary API.

Out of all the lifecycle methods in a React class component, these 5 are the most important — the ones you'll encounter in virtually every codebase and every interview. They span all three lifecycle phases and cover the majority of real-world use cases.

  • 1. constructor(props) — Mounting Phase: The very first method called when a component is created. Its two jobs are initialising this.state and binding event handler methods. Must always start with super(props). In modern React, often replaced by class field declarations for state and arrow function methods, making a constructor unnecessary.
  • 2. render() — Mounting & Updating Phases: The only required lifecycle method. Called during every render cycle — both initial mount and every subsequent update. Returns JSX describing the current UI. Must be a pure function — no state mutations, no side effects, no API calls. It's React's window into what the component looks like.
  • 3. componentDidMount() — Mounting Phase: Called once, immediately after the component's first render and DOM insertion. The standard location for all setup side effects: API calls to load initial data, setting up WebSocket connections, registering event listeners, and initialising third-party DOM libraries. Equivalent to useEffect(() => {...}, []).
  • 4. componentDidUpdate(prevProps, prevState) — Updating Phase: Called after every re-render triggered by state or props changes. Used to react to changes — re-fetch data when a prop ID changes, sync with external systems, or update the DOM imperatively after a render. Must always compare prevProps / prevState before calling setState to prevent infinite re-render loops. Equivalent to useEffect(() => {...}, [dep]).
  • 5. componentWillUnmount() — Unmounting Phase: Called immediately before the component is removed from the DOM. Dedicated entirely to cleanup — clearing timers, closing WebSocket connections, removing event listeners, cancelling fetch requests. Prevents memory leaks and ghost callbacks. Never call setState here. Equivalent to the cleanup function returned from useEffect.
jsx — all 5 methods in one component
class FullLifecycle extends Component {

  // 1. constructor — first, initialise state
  constructor(props) {
    super(props);
    this.state   = { data: null, loading: true };
    this.timerId = null;
  }

  // 2. render — required, pure, no side effects
  render() {
    if (this.state.loading) return <p>Loading...</p>;
    return <h1>{this.state.data}</h1>;
  }

  // 3. componentDidMount — setup side effects after first render
  componentDidMount() {
    fetch(`/api/item/${this.props.id}`)
      .then(r => r.json())
      .then(d => this.setState({ data: d.name, loading: false }));
    this.timerId = setInterval(this.ping, 10000);
  }

  // 4. componentDidUpdate — react to prop/state changes
  componentDidUpdate(prevProps) {
    if (prevProps.id !== this.props.id) {        // guard to prevent loop
      fetch(`/api/item/${this.props.id}`)
        .then(r => r.json())
        .then(d => this.setState({ data: d.name }));
    }
  }

  // 5. componentWillUnmount — cleanup before removal
  componentWillUnmount() {
    clearInterval(this.timerId);               // prevent ghost timer
  }

  ping = () => console.log('heartbeat');
}
Controlled & Uncontrolled Components

A Controlled Component is a form element (input, textarea, select) whose value is driven entirely by React state. React is the single source of truth for the element's value at all times. The form field does not maintain its own internal value — it simply displays whatever React's state says, and every user keystroke goes through React first.

How it works — the two-way sync loop:

  1. The input's value prop is bound to a React state variable (e.g., value={name}).
  2. The user types a character.
  3. The browser fires the onChange event.
  4. Your event handler calls the state setter (setName(e.target.value)).
  5. React updates the state and re-renders the component with the new value.
  6. The input displays the new state value — completing the loop.

If you bind value without an onChange handler, React will make the input read-only and log a warning. The two must always go together in a controlled component.

This is the opposite of uncontrolled components where the DOM element maintains its own internal state and you read the value imperatively using a ref only when needed (like on form submit).

jsx — controlled component (the two-way sync)
function ControlledInput() {
  const [name, setName] = useState('');

  return (
    <div>
      <input
        type="text"
        value={name}                          // ← React owns the value
        onChange={(e) => setName(e.target.value)} // ← every keystroke updates state
        placeholder="Enter your name"
      />
      <p>Live preview: {name}</p>             // ← always in sync with input
      <button onClick={() => setName('')}>Clear</button>  // ← programmatic control
    </div>
  );
}

// Contrast: Uncontrolled — DOM owns the value
function UncontrolledInput() {
  const inputRef = useRef();
  const handleSubmit = () => console.log(inputRef.current.value); // read on demand
  return <input ref={inputRef} type="text" defaultValue="Initial" />;
}

The core difference is about who owns and controls the form element's value — React state or the DOM itself.

  • Source of Truth: In a controlled component, React state is the single source of truth. The DOM's displayed value is always a reflection of that state. In an uncontrolled component, the DOM element itself maintains its own internal state — it is the source of truth, and you reach into it only when needed.
  • How value is set: Controlled components use the value prop (always tied to state). Uncontrolled components use defaultValue (sets initial value once and lets the DOM take over after that) — or no value prop at all.
  • How value is read: In controlled, the current value is always in React state — you read it from state at any time. In uncontrolled, you read the value imperatively via a ref (ref.current.value) — typically only on form submit.
  • onChange handler: Controlled components require an onChange handler to update state on every keystroke. Uncontrolled components don't require any event handler during typing — the DOM manages updates silently.
  • Validation & dynamic behaviour: Controlled components make per-keystroke validation, conditional rendering, dependent fields, and disabling submit buttons trivially easy — you have the value in state at all times. Uncontrolled components make this harder — you'd need to add onChange listeners anyway, defeating the purpose.
  • Programmatic control: With controlled components, you can clear, pre-fill, or reset a field at any time just by setting state. With uncontrolled, you'd need to use ref.current.value = '' — an imperative DOM mutation.
  • Boilerplate: Controlled components require more code upfront (state + handler per field). Uncontrolled require less boilerplate and can be faster for simple, one-time-read forms.

  const [email, setEmail] = useState('');

    <input
      value={email}                          // ← bound to state
      onChange={e => setEmail(e.target.value)} // ← updates state on keystroke
    />
  );
}

// ── UNCONTROLLED ──

  const emailRef = useRef();
  const handleSubmit = () => {
    console.log(emailRef.current.value); // ← read from DOM on demand
  };
  return (
    <input
      ref={emailRef}                         // ← DOM owns the value
      defaultValue=""                        // ← sets initial value, not controlled
    />

Controlled components have a distinct set of defining characteristics that separate them from uncontrolled ones. Knowing these precisely is valuable for interviews:

  • React state is the single source of truth: The component's displayed value is always derived from state. There is never a moment where the DOM's value and React's state are out of sync — they are the same thing.
  • The value prop is always bound to state: The input's value attribute is explicitly set to a state variable (value={stateVar}). This makes the input a "slave" to React — it displays exactly what state says and nothing else.
  • Every change is captured via onChange: Every character the user types fires the onChange event. Your handler receives the event, extracts e.target.value, and immediately updates state. This makes the current value always available to your component logic.
  • Full programmatic control: Because React controls the value, you can reset, pre-fill, format, or transform the value at any time simply by changing state — without any direct DOM manipulation.
  • Enables per-keystroke validation and dynamic UI: Since you have the live value in state on every keystroke, you can validate input as the user types, show/hide error messages, enable/disable submit buttons, derive other state from the input, and build dependent fields (e.g., country → state → city) — all trivially.
  • Predictable and easy to debug: The state at any point in time is the complete picture of the form's data. You can log it, test it, and inspect it in React DevTools without needing to interact with the DOM.
  • More boilerplate than uncontrolled: Each field requires a state variable and an onChange handler. For large forms, this can be repetitive — which is why libraries like React Hook Form (which uses uncontrolled inputs internally for performance) or Formik exist to manage this boilerplate.
jsx — characteristics demonstrated
function ControlledEmail() {
  const [email, setEmail]   = useState('');      // ← source of truth
  const [error, setError]   = useState('');

  const handleChange = (e) => {
    const val = e.target.value;
    setEmail(val);                              // ← capture every keystroke
    // ← per-keystroke validation
    setError(val.includes('@') ? '' : 'Invalid email');
  };

  return (
    <div>
      <input
        type="email"
        value={email}           // ← always reflects state
        onChange={handleChange}
      />
      {error && <p style={{ color: 'red' }}>{error}</p>}
      // ← disable submit until valid (programmatic control)
      <button disabled={!!error || !email}>Submit</button>
      // ← programmatic reset
      <button onClick={() => setEmail('')}>Clear</button>
    </div>
  );
}

Controlled components are the recommended approach for forms in React for several compelling reasons — especially when building anything beyond a trivial form:

  • Instant validation on every keystroke: Since React has the live value in state after every character, you can validate the input in real time and show inline error messages instantly — without waiting for the user to submit. This dramatically improves the user experience for things like password strength indicators, email format checks, or character limit counters.
  • Conditionally disable the submit button: You can inspect the current state of all form fields at any moment and enable the submit button only when the entire form is valid — a pattern that's impossible without the live values in state.
  • Format and transform input on the fly: You can intercept the raw input and store a transformed version — like auto-converting a phone number to a formatted string, converting to uppercase, or stripping special characters — before it ever reaches state.
  • Dependent and cascading fields: When one field's value should affect another (e.g., selecting a country updates the available states in a dropdown), controlled components make this trivial — you have all values in state and can derive or reset dependent fields in the onChange handler.
  • Pre-filling and resetting forms: You can populate a form with existing data (for an edit page) or reset it to empty state purely by setting state — no direct DOM manipulation required.
  • Easier testing: Tests can set the form's values by simply updating state — no need to simulate DOM interactions. The state is the complete, testable representation of the form.
  • Predictable data flow: Consistent with React's overall philosophy — state is the source of truth, and UI is derived from state. There are no hidden, silent DOM-owned values to worry about.

Handling forms in React revolves around managing the form fields as controlled components — binding each field's value to state and handling changes via event handlers. The standard pattern involves three things: state for each field, onChange handlers to keep state in sync, and an onSubmit handler to process the data.

Key concepts for React form handling:

  • Prevent default browser submission: Always call e.preventDefault() in the submit handler. Without this, the form submission would cause a page reload — destroying your SPA's state.
  • Controlled inputs: Every form field (input, textarea, select) should have a value prop tied to state and an onChange handler that updates state. This ensures React is always in sync.
  • Reading submitted data: On submit, all the form data is already in React state — you can access it directly without reading from the DOM. Pass it to an API, validate it, or use it for client-side logic.
  • Resetting after submission: Clear the form by resetting state variables to their initial values — no DOM manipulation needed.
jsx — complete form handling pattern
function LoginForm() {
  const [email, setEmail]       = useState('');
  const [password, setPassword] = useState('');
  const [error, setError]       = useState('');

  const handleSubmit = (e) => {
    e.preventDefault();         // ← ALWAYS prevent default browser reload

    // Validate
    if (!email || !password) {
      setError('All fields are required.');
      return;
    }

    // Submit data — already in state, no DOM reading needed
    fetch('/api/login', {
      method:  'POST',
      headers: { 'Content-Type': 'application/json' },
      body:    JSON.stringify({ email, password }),
    }).then(res => { if (res.ok) setError(''); });

    // Reset form
    setEmail('');
    setPassword('');
  };

  return (
    <form onSubmit={handleSubmit}>
      {error && <p className="error">{error}</p>}

      <label>Email
        <input type="email" value={email}
               onChange={e => setEmail(e.target.value)} />
      </label>

      <label>Password
        <input type="password" value={password}
               onChange={e => setPassword(e.target.value)} />
      </label>

      <button type="submit">Log In</button>
    </form>
  );
}

When a form has many fields, creating a separate useState variable and onChange handler for each one leads to repetitive boilerplate. The cleaner, scalable approach is to store all form values in a single state object and use a single dynamic handler that reads the input's name attribute to know which field to update.

The pattern uses the computed property name syntax of ES6: [e.target.name]: e.target.value — this dynamically sets the property whose key matches the input's name attribute. The spread operator ...prev ensures all other fields retain their current values (since setState in functional components replaces, not merges — unlike class component this.setState).

Requirements for this to work:

  • Each input must have a name attribute that exactly matches its corresponding key in the state object.
  • The state object must be initialised with a key for every field.
  • A single handleChange function can serve all inputs — no duplication.
jsx — single handler for multiple fields
function RegistrationForm() {
  // One state object for all fields
  const [formData, setFormData] = useState({
    firstName: '',
    lastName:  '',
    email:     '',
    phone:     '',
    role:      'user', // for a select

Form validation in controlled components is natural and powerful because React always has the live values in state. You can validate at two points — on every change (per-keystroke) for immediate feedback, or on submit for batch validation before sending data.

Common validation approaches:

  • onChange validation: Validate inside the handleChange function as the user types. Show error messages in real time. Best for simple format checks (email, phone, min length). Can feel aggressive if errors show before the user finishes typing — consider validating only after the field has been "touched" (blurred at least once).
  • onBlur validation: Validate when the user leaves a field (onBlur event). Less intrusive — errors only appear after the user has finished entering the field. Track which fields have been "touched" in state.
  • onSubmit validation: Validate all fields at once when the form is submitted. Prevent submission if any field is invalid. Simple and sufficient for many use cases.

For production apps: Consider using React Hook Form or Formik + Yup — they handle touched state, error messages, schema validation, and performance optimisation (React Hook Form uses uncontrolled inputs for performance). These libraries dramatically reduce validation boilerplate.

jsx — form validation on submit + live feedback
function ValidatedForm() {
  const [form, setForm]     = useState({ email: '', password: '' });
  const [errors, setErrors] = useState({});
  const [touched, setTouched] = useState({});

  // Validation rules
  const validate = (data) => {
    const errs = {};
    if (!data.email.includes('@'))     errs.email    = 'Invalid email address';
    if (data.password.length < 8)      errs.password = 'Min 8 characters';
    if (!data.email)                     errs.email    = 'Email is required';
    return errs;
  };

  const handleChange = (e) => {
    const { name, value } = e.target;
    const updated = { ...form, [name]: value };
    setForm(updated);
    // Only show errors for fields already touched
    if (touched[name]) setErrors(validate(updated));
  };

  const handleBlur = (e) => {
    setTouched(prev => ({ ...prev, [e.target.name]: true }));
    setErrors(validate(form));
  };

  const handleSubmit = (e) => {
    e.preventDefault();
    const errs = validate(form);
    if (Object.keys(errs).length > 0) {
      setErrors(errs);
      setTouched({ email: true, password: true }); // show all errors
      return;
    }
    console.log('Form submitted!', form);
  };

  const isValid = Object.keys(validate(form)).length === 0;

  return (
    <form onSubmit={handleSubmit}>
      <input name="email" value={form.email}
             onChange={handleChange} onBlur={handleBlur} />
      {errors.email && <p className="error">{errors.email}</p>}

      <input type="password" name="password" value={form.password}
             onChange={handleChange} onBlur={handleBlur} />
      {errors.password && <p className="error">{errors.password}</p>}

      // Disable submit until form is valid
      <button type="submit" disabled={!isValid}>Submit</button>
    </form>
  );
}

While controlled components are recommended for most React form scenarios, there are specific situations where uncontrolled components — where the DOM manages the input's own state and you read it via a ref — are genuinely the better choice:

  • File inputs (<input type="file">): File inputs are inherently uncontrolled in React. You cannot set their value programmatically for security reasons — the browser restricts this. You always read the selected file using a ref: fileRef.current.files[0]. There is no controlled alternative for file inputs.
  • Integrating with non-React code or third-party libraries: When you're embedding a React component inside a legacy jQuery application, a CMS, or a non-React system, and the outer system controls the form or input values imperatively — uncontrolled inputs with refs let you bridge the two worlds without React fighting the DOM for control.
  • Very simple, one-off forms where you only need the value on submit: A single-field search bar or a minimal contact form where you don't need live validation, dependent fields, or programmatic control. Uncontrolled inputs require less code — just a ref and a read on submit. React Hook Form uses this pattern internally at scale for performance benefits.
  • Performance-critical forms with hundreds of fields: Every controlled input causes a React re-render on every keystroke. In forms with dozens or hundreds of fields (enterprise data grids, spreadsheet-like UIs), this can cause noticeable lag. Uncontrolled inputs bypass React's render cycle entirely — the DOM updates itself and React only processes data on submit.
  • Autofocus, scroll management, or other imperative DOM requirements: When you need to programmatically focus an input after mounting, scroll to a specific field, or integrate with browser autofill in a custom way — useRef and uncontrolled access give you the direct DOM handle you need.

// ── File input — MUST be uncontrolled ──
function FileUpload() {
  const fileRef = useRef();

  const handleUpload = () => {
    const file = fileRef.current.files[0];
    if (!file) return;
    const formData = new FormData();
    formData.append('file', file);
    fetch('/api/upload', { method: 'POST', body: formData });
  };

  return (
    <div>
      <input type="file" ref={fileRef} />  // ← always uncontrolled
      <button onClick={handleUpload}>Upload</button>
    </div>
  );
}

Code Splitting — Introduction
8 Qs

By default, when you build a React application, the bundler (Webpack or Vite) packages all your JavaScript code — every component, every library, every route — into a single large bundle.js file. The browser must download, parse, and execute this entire file before the user sees anything. As your app grows, this bundle can become very large, causing slow initial load times — especially on mobile or slow networks.

Code Splitting is the technique of breaking this single large bundle into multiple smaller chunks (separate JavaScript files). These chunks are then loaded on demand — the browser only downloads a chunk when it's actually needed, rather than loading everything upfront.

The core idea — lazy loading: Instead of shipping the code for the Settings page, the Admin dashboard, and the PDF export feature to every user on every visit, you split each into its own chunk and load them only when the user navigates to that page or triggers that feature. Users who never visit the Admin page never download that code.

What code splitting improves:

  • Initial page load speed: The browser downloads a smaller initial bundle → Time to First Contentful Paint (FCP) and Time to Interactive (TTI) improve significantly.
  • Bandwidth efficiency: Users only download code they actually use.
  • Perceived performance: The app feels faster because the critical path renders quickly, and non-critical code loads in the background or on demand.

In React, code splitting is built on top of the dynamic import() function (a JavaScript standard) and the React.lazy() API, with <Suspense> handling the loading state during chunk fetch.

concept — bundle without vs with code splitting
// ── Without Code Splitting ──
// Browser downloads ONE massive file on first load:
// bundle.js  →  2.4 MB  (Home + About + Dashboard + Admin + Charts…)

// ── With Code Splitting ──
// Browser downloads small initial chunk, rest on demand:
// main.js         →   180 KB  (downloaded immediately)
// home.chunk.js   →   45 KB   (downloaded when user visits /home)
// admin.chunk.js  →   320 KB  (downloaded only if user visits /admin)
// charts.chunk.js →   210 KB  (downloaded only when charts are needed)

// Result: 90% of users never download admin.chunk.js

React provides a first-class, built-in code splitting solution using two APIs: React.lazy() for dynamically importing components and <Suspense> for handling the loading state while the chunk downloads. This is the recommended approach for route-based code splitting.

Step-by-step implementation:

  • Step 1: Replace static imports with React.lazy() wrapping a dynamic import() call. This tells the bundler to put that component in a separate chunk.
  • Step 2: Wrap the lazy component (or your route switch) in a <Suspense> component with a fallback prop — the UI to show while the chunk is being downloaded.
  • Step 3: The bundler (Webpack/Vite) automatically creates a separate chunk file for each lazily imported module.

Route-based splitting is the highest-impact approach — splitting at the page/route level ensures each page's code is only loaded when the user navigates to it. Component-level splitting is used for heavy components like charts, maps, or rich text editors that aren't needed immediately.

jsx — code splitting with lazy and suspense
import React, { lazy, Suspense } from 'react';
import { BrowserRouter, Routes, Route } from 'react-router-dom';

// ❌ Static imports — ALL code in one bundle
// import Home from './pages/Home';
// import Dashboard from './pages/Dashboard';
// import AdminPanel from './pages/AdminPanel';

// ✅ Lazy imports — each gets its own chunk
const Home       = lazy(() => import('./pages/Home'));
const Dashboard  = lazy(() => import('./pages/Dashboard'));
const AdminPanel = lazy(() => import('./pages/AdminPanel'));

// Heavy component — loaded only when rendered
const ChartWidget = lazy(() => import('./components/ChartWidget'));

function App() {
  return (
    <BrowserRouter>
      // Suspense wraps all lazy routes — shows fallback while loading
      <Suspense fallback={<div className="loader">Loading page...</div>}>
        <Routes>
          <Route path="/"        element={<Home />}       />
          <Route path="/dashboard" element={<Dashboard />}  />
          <Route path="/admin"    element={<AdminPanel />} />
        </Routes>
      </Suspense>
    </BrowserRouter>
  );
}

These two APIs work as a pair — they are inseparable in React's code splitting story. React.lazy() creates the lazy component, and <Suspense> handles the waiting period.

React.lazy(fn) — Dynamic Component Loading:

  • Accepts a function that returns a dynamic import() promise — React.lazy(() => import('./MyComponent')).
  • Returns a special "lazy" React component. When React first tries to render this component, it fires the import, fetching the chunk from the server.
  • While the chunk is loading, React suspends rendering of that subtree — this is where Suspense takes over.
  • Once loaded, the chunk is cached in memory — subsequent renders of the same component are instant, no re-download.
  • Requirement: The dynamically imported module must have a default export that is a React component. Named exports are not supported directly with React.lazy.

<Suspense fallback={...}> — Loading State Handler:

  • Wraps one or more lazy components and defines what to render while their chunks are being fetched — the fallback prop.
  • The fallback can be any valid JSX: a spinner, a skeleton screen, a simple loading message, or even a full placeholder layout.
  • You can have multiple <Suspense> boundaries at different levels — a global one for route-level loading and more granular ones for individual heavy components within a page.
  • When the lazy component's chunk finishes loading, Suspense automatically replaces the fallback with the actual rendered component — no extra code needed.
  • In React 18, Suspense also works with data fetching frameworks (Next.js, Relay, React Query) — not just lazy loading — as part of Concurrent Mode.
jsx — lazy and suspense in detail
import { lazy, Suspense } from 'react';

// lazy() — fires import() on first render, caches forever after
const HeavyChart = lazy(() => import('./HeavyChart'));

// Named export? Re-export as default in HeavyChart.jsx or use this trick:
const PieChart = lazy(() =>
  import('./charts').then(m => ({ default: m.PieChart }))
);

// Suspense — nested boundaries for fine-grained loading UX
function AnalyticsPage() {
  return (
    <div>
      <h1>Analytics</h1>

      // Outer boundary — coarse fallback for the whole section
      <Suspense fallback={<p>Loading charts...</p>}>
        <HeavyChart data={chartData} />

        // Inner boundary — finer fallback for just the pie chart
        <Suspense fallback={<span>Loading pie...</span>}>
          <PieChart data={pieData} />
        </Suspense>
      </Suspense>
    </div>
  );
}

Code splitting is a powerful optimisation but — like all optimisations — comes with trade-offs. Knowing both sides shows technical maturity in interviews.

✅ Pros:

  • Faster initial load time: The browser downloads a smaller initial bundle, so the app becomes interactive much sooner. This directly improves Core Web Vitals metrics — Time to Interactive (TTI) and First Contentful Paint (FCP) — which also affects SEO rankings.
  • Reduced bandwidth usage: Users only download the JavaScript they actually need. Someone who only visits the home page never downloads the admin panel's code — saving data, especially on mobile.
  • Better cache efficiency: With multiple smaller chunks, when you update one part of the app, only that specific chunk's cache is invalidated — not the entire bundle. Unchanged chunks remain cached in the user's browser.
  • Scalability: As your application grows, code splitting keeps the initial load time manageable. Without it, every new feature added makes the app slower for all users on first load.
  • Improved perceived performance: Showing a skeleton or spinner while a secondary chunk loads often feels faster to users than staring at a blank screen while a huge bundle parses.

❌ Cons:

  • Increased complexity: You need to manage React.lazy(), <Suspense>, loading states, and potentially error boundaries. Simpler apps may not benefit enough to justify this overhead.
  • Loading delay on first navigation to a split chunk: When a user navigates to a route for the first time, there's a brief delay while the new chunk downloads. On slow networks, this can feel jarring without a good loading indicator.
  • More network requests: Instead of one bundle request, there are now many chunk requests. HTTP/2 mitigates this, but it's still more round trips compared to a single cached file.
  • Risk of over-splitting: Splitting too aggressively (every tiny component in its own chunk) can actually hurt performance — the overhead of many small network requests outweighs the savings. The sweet spot is splitting at the route or feature level.
  • SSR complexity: Code splitting in Server-Side Rendered apps (Next.js) requires extra care — libraries like loadable-components were historically needed for SSR-compatible splitting, though Next.js abstracts most of this.

The dynamic import() function is the JavaScript-standard foundation that makes all code splitting possible. It is not React-specific — it is a native ECMAScript feature (part of the ES2020 spec) that allows you to import a JavaScript module asynchronously and on demand, rather than statically at the top of the file.

Static import vs Dynamic import:

  • Static import (import X from './X') happens at parse time — the bundler sees it and includes the module in the current bundle. The browser downloads it eagerly with everything else.
  • Dynamic import (import('./X')) happens at runtime — the bundler creates a separate chunk for X. The browser downloads that chunk only when the import() call is actually executed in the running JavaScript.

How it works technically:

  • import() returns a Promise that resolves to the full module object — including all its exports. You use .then(m => m.default) or destructuring to get what you need.
  • When Webpack or Vite sees a dynamic import() during the build, they automatically extract the target module and all its dependencies into a separate chunk file, and generate the logic to fetch it on demand.
  • React.lazy() is simply a React-flavoured wrapper around dynamic import() — it standardises how React handles the Promise and integrates with Suspense.
jsx — dynamic import() in action
// ── Static import (eager — always in the main bundle) ──
import ChartLib from 'heavy-chart-library'; // always downloaded

// ── Dynamic import (lazy — separate chunk, loaded on demand) ──

// 1. Returns a Promise — use with .then()
const loadChart = () => {
  import('heavy-chart-library')
    .then(module => {
      const Chart = module.default;
      // use Chart here
    });
};

// 2. Async/await syntax — cleaner
const handleButtonClick = async () => {
  const { default: Chart } = await import('heavy-chart-library');
  // Chart is available here, downloaded on demand
};

// 3. With React.lazy — the React way for components
const LazyChart = React.lazy(() => import('./ChartComponent'));
// React.lazy wraps import() and connects it to Suspense

// 4. Webpack magic comment — name the chunk for debugging
const AdminPage = React.lazy(() =>
  import(/* webpackChunkName: "admin" */ './pages/AdminPage')
);
// Generates: admin.chunk.js — identifiable in DevTools

The fallback prop is a required prop on the <Suspense> component. It defines the UI that React renders in place of the lazy component while its JavaScript chunk is being downloaded from the server. Once the chunk finishes loading, the fallback is automatically replaced with the actual rendered component — no extra code or manual swapping required.

What the fallback prop accepts: Any valid React JSX — a component, a plain HTML element, or even null (to render nothing while loading). The fallback renders for the duration of the chunk download, which on a fast connection may be imperceptibly brief, but on a slow connection could be several seconds.

Designing good fallbacks matters for UX:

  • Skeleton screens — the most polished approach. Show a grey placeholder shaped like the real content — users perceive this as faster than a generic spinner because the layout is already in place.
  • Loading spinners — simpler and still effective for most cases. Use the app's existing loading indicator for consistency.
  • Keep the fallback lightweight: The fallback itself must never be a lazy component — it needs to be immediately available since it renders while the actual component is loading.
  • Match the dimensions of the lazy content where possible to prevent layout shift when the real component renders — a key factor in Core Web Vitals (Cumulative Layout Shift / CLS).

Multiple Suspense boundaries with different fallbacks: You can nest <Suspense> components at different levels — a coarse one around your entire router for page-level loading, and finer ones around individual heavy components for more granular feedback.

A Higher-Order Component (HOC) is a pattern in React — not an API — where a function takes a component as input and returns a new, enhanced component as output. It's a way to reuse component logic by wrapping components with shared behaviour, without modifying the original component.

The pattern borrows from functional programming's higher-order functions (functions that take or return other functions). In React, HOCs follow the signature: const EnhancedComponent = higherOrderComponent(WrappedComponent).

Common real-world uses of HOCs:

  • Authentication/authorisation: A withAuth(Component) HOC checks if the user is logged in — if not, redirects to login. Wrapping any page component with it adds protection without repeating the auth check.
  • Logging & analytics: A withLogger(Component) HOC logs every render or lifecycle event.
  • Data fetching: A withData(Component, url) HOC fetches data and passes it as props to the wrapped component.
  • Theme injection: Inject theme props before the Context API was common.

HOCs vs Custom Hooks: In modern React with hooks, custom hooks are preferred over HOCs for logic reuse — they're simpler, don't add components to the tree, and avoid "wrapper hell" (deeply nested component trees in DevTools). HOCs are still relevant in legacy codebases and some library APIs (React Redux's connect()).

jsx — HOC pattern: withAuth
// HOC — takes a component, returns an enhanced component
function withAuth(WrappedComponent) {
  // Returns a new component
  return function AuthenticatedComponent(props) {
    const isLoggedIn = checkAuth(); // auth check logic

    if (!isLoggedIn) {
      return <Navigate to="/login" />; // redirect if not authed
    }

    // Pass all props through to the wrapped component
    return <WrappedComponent {...props} />;
  };
}

// Usage — wrap any page that needs protection
const ProtectedDashboard = withAuth(Dashboard);
const ProtectedProfile   = withAuth(Profile);

// These now redirect to /login if user isn't authenticated
<Route path="/dashboard" element={<ProtectedDashboard />} />

// Modern equivalent with custom hook — simpler, no wrapper
function Dashboard() {
  useRequireAuth(); // custom hook — redirects if not logged in
  return <div>Dashboard content</div>;
}

React is flexible about styling — it supports multiple approaches, each with different trade-offs. Here are the 5 main methods:

  • 1. Inline Styles: Pass a JavaScript object directly to the style prop. CSS property names are camelCased (backgroundColor not background-color). Values are strings or numbers (numbers default to px for length properties). Scoped to the element — no class name conflicts. Best for truly dynamic, computed styles. Drawback: no pseudo-classes (:hover, :focus), no media queries, performance issues with many elements, styles aren't reusable.
  • 2. CSS Stylesheets (plain .css files): Traditional CSS files imported into components. Styles are global by default — class names can clash across the app. Simple and familiar but requires manual class name management.
  • 3. CSS Modules (.module.css): CSS files where each class name is automatically scoped locally to the component — the bundler generates a unique class name per module. Best of both worlds: full CSS syntax (pseudo-classes, media queries) with local scoping. The recommended approach for most React projects without a CSS-in-JS library.
  • 4. CSS-in-JS (styled-components, Emotion): Write CSS directly in JavaScript using tagged template literals or object syntax. Generates unique class names at runtime, supports full CSS with dynamic values, themes, and component-scoped styles. Powerful but adds bundle size and a runtime cost.
  • 5. Utility-First CSS (Tailwind CSS): Apply pre-built utility classes directly in JSX (className="flex bg-blue-500 text-white p-4"). No custom CSS written — compose styles from atomic classes. Extremely popular in modern React — fast to develop, consistent design system, and zero CSS bundle growth as you add components.

React and React Native share the same core philosophy and component model — but they target completely different platforms and use fundamentally different rendering engines.

  • Platform: React targets web browsers — it renders HTML and CSS into the browser's DOM. React Native targets mobile platforms (iOS and Android) — it renders to native platform UI components, not HTML.
  • What they render: React renders standard HTML elements (<div>, <p>, <button>) styled with CSS. React Native renders native mobile UI components (<View>, <Text>, <TouchableOpacity>) that the OS draws natively — resulting in truly native-feeling apps, not web views.
  • Styling: React uses CSS (stylesheets, CSS Modules, Tailwind, etc.). React Native uses a JavaScript styling system that resembles CSS but only supports a limited subset — no CSS selectors, no floats, no percentages for all properties, no display: grid. Layout is done with Flexbox only.
  • Navigation: React uses React Router (browser URL-based). React Native uses React Navigation or Expo Router (stack, tab, and drawer navigators).
  • API access: React Native can access native device APIs — camera, GPS, accelerometer, push notifications, biometrics — either through the framework or community libraries. React apps access browser APIs (localStorage, geolocation) and require permissions differently.
  • Performance: React Native apps feel like true native apps because they render native UI. Web apps rendered via React can also be fast but run in a browser runtime with browser-imposed constraints.
  • Code sharing: Core business logic, state management, and many custom hooks can be shared between a React web app and a React Native mobile app. UI components (JSX/layout) generally cannot — they use different primitives.

GraphQL is a query language for APIs and a runtime for fulfilling those queries, developed by Meta (Facebook) in 2012 and open-sourced in 2015. It is an alternative to REST APIs that gives clients precise control over what data they request — no more over-fetching (getting too much data) or under-fetching (needing multiple requests to get all the data you need).

Core concepts:

  • Single endpoint: Unlike REST which has multiple endpoints (/users, /posts, /comments), GraphQL has a single endpoint (e.g., /graphql). The query you send determines what data comes back.
  • Ask for exactly what you need: The client specifies precisely which fields it wants in its query. The server returns exactly that — no more, no less. This eliminates over-fetching (REST often returns 30 fields when you only need 3).
  • Types: Query, Mutation, Subscription: A Query reads data (like GET). A Mutation writes/updates data (like POST/PUT/DELETE). A Subscription sets up a real-time data stream (like WebSockets).
  • Strongly typed schema: The API is described by a type schema. This enables auto-documentation, introspection, and tooling like GraphiQL.

GraphQL in React: The most popular GraphQL clients for React are Apollo Client and React Query + graphql-request. Apollo provides hooks like useQuery and useMutation that integrate directly with React's component lifecycle and state model.

graphql + jsx — query and apollo hook
// ── GraphQL Query — ask for exactly what you need ──
const GET_USER = gql`
  query GetUser($id: ID!) {
    user(id: $id) {
      name          # only these 3 fields — not the whole user object
      email
      avatarUrl
    }
  }
`;

// ── React component using Apollo's useQuery hook ──
function UserProfile({ userId }) {
  const { loading, error, data } = useQuery(GET_USER, {
    variables: { id: userId },
  });

  if (loading) return <p>Loading...</p>;
  if (error)   return <p>Error!</p>;

  return (
    <div>
      <img src={data.user.avatarUrl} alt={data.user.name} />
      <h2>{data.user.name}</h2>
      <p>{data.user.email}</p>
    </div>
  );
}

As React applications scale, managing state that needs to be shared across many components becomes complex. Here are the top 3 approaches and clear guidelines on when to use each:

  • 1. React Context API + useState/useReducer (Built-in):
    The zero-dependency, built-in solution. Create a Context, wrap the app in a Provider, and any nested component can read the value via useContext. Best for low-frequency global data that doesn't change rapidly — authentication status, user preferences, app theme, locale/language, feature flags.
    Use when: Your state is simple, doesn't update very often, and you want no extra dependencies.
    Avoid when: State updates are frequent (every keystroke) — all Context consumers re-render on every change, which causes performance issues at scale.
  • 2. Zustand (Lightweight, Modern):
    A minimalist state management library (~1KB). Creates a simple hook-based store outside the React component tree. Components subscribe to only the slice of state they need — avoiding the "all consumers re-render" problem of Context. Very little boilerplate. Becoming the community favourite for medium-complexity apps.
    Use when: You need global state that updates frequently, want minimal boilerplate, and Context's performance limitations are a concern.
  • 3. Redux Toolkit (Enterprise Scale):
    The battle-tested, most feature-complete solution. Uses a centralised store, actions, and reducers. Redux Toolkit (RTK) eliminates Redux's traditional boilerplate — creating slices, immutable updates via Immer, and RTK Query for data fetching. Excellent DevTools (time-travel debugging). Best for large teams and complex apps with many interconnected state updates.
    Use when: The app is large, state logic is complex, you need a strong predictability guarantee, or you need excellent debugging capabilities and team standardisation.
jsx — zustand vs context vs redux toolkit

// ── 2. Zustand — minimal, performant ──
import { create } from 'zustand';
const useStore = create(set => ({
  count: 0,
  increment: () => set(s => ({ count: s.count + 1 })),
}));
function Counter() {
  const { count, increment } = useStore();
  return <button onClick={increment}>{count}</button>;
}

// ── 3. Redux Toolkit — enterprise scale ──
const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: { increment: state => { state.value += 1; } },
});
function Counter() {
  const count = useSelector(s => s.counter.value);
  const dispatch = useDispatch();
  return <button onClick={() => dispatch(counterSlice.actions.increment())}>{count}</button>;
}

Authentication in React is primarily a state management + routing challenge — you need to track whether a user is logged in, persist that status across page refreshes, and protect certain routes from unauthenticated access. The actual credential verification always happens on the server.

The standard JWT-based authentication flow:

  • Step 1 — Login: User submits credentials via a controlled form. A POST request is sent to the backend API. On success, the server returns a JWT (JSON Web Token).
  • Step 2 — Store the token: Store the JWT in localStorage (simple but vulnerable to XSS) or httpOnly cookies (more secure — the server sets it, JS can't read it). For most production apps, httpOnly cookies are preferred.
  • Step 3 — Share auth state globally: Store the user/token in React Context (or Zustand/Redux) so any component can know if the user is authenticated.
  • Step 4 — Send token with requests: Include the JWT in the Authorization: Bearer <token> header on all subsequent API calls that require authentication.
  • Step 5 — Protect routes: Create a <PrivateRoute> (or Protected Route) component that checks auth state — if not authenticated, redirects to the login page; if authenticated, renders the requested page.
  • Step 6 — Logout: Clear the token from storage and reset the auth context state — the user is redirected to login.
jsx — auth context + protected route pattern
// AuthContext.jsx — global auth state
const AuthContext = createContext();
export function AuthProvider({ children }) {
  const [user, setUser] = useState(() => {
    const saved = localStorage.getItem('user');
    return saved ? JSON.parse(saved) : null;
  });

  const login = async (email, password) => {
    const res  = await fetch('/api/login', {
      method: 'POST', body: JSON.stringify({ email, password })
    });
    const data = await res.json();
    setUser(data.user);
    localStorage.setItem('user', JSON.stringify(data.user));
  };

  const logout = () => { setUser(null); localStorage.removeItem('user'); };

  return (
    <AuthContext.Provider value={{ user, login, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

// ProtectedRoute.jsx — redirect if not authenticated
function ProtectedRoute({ children }) {
  const { user } = useContext(AuthContext);
  return user ? children : <Navigate to="/login" replace />;
}

// App.jsx — usage
<Routes>
  <Route path="/login"     element={<LoginPage />} />
  <Route path="/dashboard" element={<ProtectedRoute><Dashboard /></ProtectedRoute>} />
</Routes>

The React Profiler is a performance measurement tool that lets you identify which parts of your React application are rendering slowly, how often components re-render, and what the rendering cost of each component is. It comes in two forms:

  • React DevTools Profiler (browser extension): The most commonly used form. In the browser's React DevTools panel, the "Profiler" tab records rendering sessions. It shows a flame graph of every component render — how long each took, how many times it rendered, and which props/state change triggered each render. This is the go-to tool for identifying performance bottlenecks.
  • <Profiler> component (programmatic API): A React component you can add directly in JSX to programmatically measure the render performance of a specific part of the component tree. It calls a callback function after each render with detailed timing data — useful for logging performance metrics to an analytics service in production.

What the Profiler measures: Render duration (how long the commit phase took), interaction start time, which phase triggered the render (mount vs update), and the number of renders in a session.

jsx — programmatic profiler component
import { Profiler } from 'react';

function onRenderCallback(
  id,           // the "id" prop of the Profiler tree
  phase,        // "mount" or "update"
  actualDuration, // time spent rendering the committed update (ms)
  baseDuration,   // estimated time to render the whole subtree (ms)
  startTime,    // when React began rendering
  commitTime    // when React committed the update
) {
  console.log(`${id} [${phase}]: ${actualDuration.toFixed(2)}ms`);
  // Send to analytics: analytics.track('render', { id, phase, actualDuration })
}

// Wrap any component tree to measure it
function App() {
  return (
    <Profiler id="Dashboard" onRender={onRenderCallback}>
      <Dashboard />
    </Profiler>
  );
}

Both fetch and axios are used to make HTTP requests in React, but they differ significantly in their API design, features, and developer experience.

  • Built-in vs library: fetch is a native browser Web API — no installation needed. axios is a third-party library (npm install axios) that works in both browser and Node.js environments.
  • JSON parsing: fetch requires two await calls — one to get the response and a second await res.json() to parse the body. axios automatically parses JSON responses — the data is directly on response.data, no extra step needed.
  • Error handling: fetch only rejects on network failure — a 404 or 500 response is considered a successful fetch (you must manually check res.ok). axios automatically throws errors for any 4xx or 5xx response — your catch block handles all non-2xx responses.
  • Request/Response interceptors: axios has a powerful interceptors API — you can attach logic that runs on every request (e.g., add the auth token header globally) or every response (e.g., refresh token on 401) without repeating code everywhere. fetch has no interceptor concept — you'd need to write a wrapper function manually.
  • Request cancellation: fetch uses AbortController. axios also supports AbortController (since v0.22) and historically had its own CancelToken API.
  • Upload progress: axios supports upload and download progress tracking via onUploadProgress and onDownloadProgress callbacks. fetch doesn't have built-in progress events (you'd need to use the Response body reader API, which is complex).
jsx — fetch vs axios comparison
// ── fetch — native, requires manual JSON parse and error check ──
const getUser = async (id) => {
  const res = await fetch(`/api/users/${id}`, {
    headers: { Authorization: `Bearer ${token}` }, // ← add headers manually
  });
  if (!res.ok) throw new Error(`HTTP error: ${res.status}`); // ← manual error check
  return res.json();  // ← second await needed
};

// ── axios — auto JSON, auto error throw, interceptors ──
import axios from 'axios';

// Global interceptor — attach auth header to every request
axios.interceptors.request.use(config => {
  config.headers.Authorization = `Bearer ${token}`;
  return config;
});

const getUser = async (id) => {
  const { data } = await axios.get(`/api/users/${id}`);
  return data; // ← JSON auto-parsed, errors auto-thrown
};

Testing React applications typically involves a layered stack — a test runner, a rendering utility, and sometimes specialised tools for end-to-end tests. Here are the most widely used:

  • Jest (Test Runner): The de facto standard JavaScript test runner. Provides the test environment, assertion library (expect), mocking utilities (jest.fn(), jest.mock()), and code coverage reports. Works seamlessly with CRA and Vite. Most React testing setups use Jest as the engine.
  • React Testing Library (RTL): The community's strongly preferred library for testing React components. Philosophy: test components the way users actually use them — by querying the DOM for text, roles, and labels — not by testing implementation details like component state or method calls. RTL renders components into a real DOM (using jsdom) and provides queries like getByRole, getByText, findByLabelText, and utilities like fireEvent and userEvent.
  • Vitest: A fast, modern test runner built for Vite projects. API-compatible with Jest — you can often switch from Jest to Vitest with minimal changes. Significantly faster than Jest for Vite-based projects.
  • Cypress: An end-to-end (E2E) testing framework that runs your actual app in a real browser and simulates real user interactions — clicking, typing, navigating. Captures screenshots and videos on failure. Now also supports component testing for React.
  • Playwright: Another E2E tool from Microsoft. Supports Chromium, Firefox, and WebKit. Known for reliability, speed, and excellent cross-browser testing capabilities. Increasingly popular over Cypress for large-scale E2E suites.
  • MSW (Mock Service Worker): Intercepts API requests at the service worker level and returns mock responses. Used alongside RTL to test components that make real API calls — without actually hitting the server. Supports both browser and Node environments.

React is fast by default, but poorly architected code can cause unnecessary re-renders and slow components. Here are the key optimisation strategies:

  • Code Splitting with React.lazy + Suspense: The highest-impact optimisation. Splits the JS bundle into chunks loaded on demand — dramatically reduces initial load time.
  • React.memo for functional components: Wraps a component to prevent re-rendering if its props haven't changed (shallow comparison). Only apply to components that re-render often with the same props and have expensive renders.
  • useCallback for stable callbacks: Memoises function references so they don't change on every parent render — essential when passing callbacks to React.memo children.
  • useMemo for expensive calculations: Cache the result of heavy computations — filtering/sorting large lists, transforming data — so they don't run on every render.
  • Virtualisation for long lists: Libraries like react-window or react-virtual render only the visible rows of a large list — instead of rendering 10,000 DOM nodes for 10,000 list items.
  • Avoid anonymous functions and objects in JSX props: Writing onClick={() => doX()} or style={{ color: 'red' }} directly in JSX creates new references every render — breaking React.memo. Extract them with useCallback and useMemo.
  • Lazy load images: Use the native loading="lazy" attribute or a library like react-intersection-observer to defer off-screen image loading.
  • Debounce/Throttle event handlers: For search inputs, resize handlers, and scroll events — debounce with a library like Lodash to avoid firing expensive operations on every event.
  • Use React 18 Concurrent Features: useTransition marks non-urgent state updates so urgent ones (typing) stay responsive. useDeferredValue defers re-rendering of slow subtrees.
jsx — performance optimisation patterns
// React.memo — skip re-render if props unchanged
const ListItem = React.memo(({ item, onSelect }) => (
  <li onClick={() => onSelect(item.id)}>{item.name}</li>
));

// useCallback + useMemo — stable refs for memo'd child
function Parent({ items }) {
  const [filter, setFilter] = useState('');

  const handleSelect = useCallback((id) => {
    console.log('selected', id);
  }, []); // stable — won't break ListItem's memo

  const filteredItems = useMemo(() =>
    items.filter(i => i.name.includes(filter)),
    [items, filter]  // only re-filters when items or filter changes
  );

  return filteredItems.map(item =>
    <ListItem key={item.id} item={item} onSelect={handleSelect} />
  );
}

Reactive Programming is a programming paradigm centred around data streams and the propagation of change. Instead of imperatively pulling data when you need it, reactive systems push data to you as it changes — you declare how to react to those changes and the system handles the rest.

Think of it like a spreadsheet: when you change the value in cell A1, all cells that depend on it (A2 = A1 * 2, A3 = A1 + 5) update automatically. You don't manually recalculate them — the reactivity system does it. This is reactive programming.

Core concepts:

  • Observable/Stream: A source of data that emits values over time — mouse events, API responses, timers, user input.
  • Observer/Subscriber: A consumer that listens to a stream and reacts to each emitted value.
  • Operators: Functions that transform, filter, combine, or delay streams — like map, filter, debounceTime, mergeMap.

React itself is inherently reactive in the UI layer — when state changes, the UI automatically updates to reflect the new state. You declare what to show, not how to update the DOM. For more advanced reactive programming patterns (streams, event-based pipelines), libraries like RxJS are used.

jsx — reactive programming in react
// ── React's built-in reactivity ──
// State is the "stream" — UI reacts automatically to changes
function ReactiveCounter() {
  const [count, setCount] = useState(0);

  // doubled is DERIVED — reacts to count automatically
  const doubled = count * 2; // no manual recalculation needed

  return (
    <div>
      <button onClick={() => setCount(c => c + 1)}>Count: {count}</button>
      <p>Doubled: {doubled}</p> // updates reactively when count changes
    </div>
  );
}

Reactive patterns can be implemented at various levels in React — from the built-in primitives to external libraries. Here are the main approaches:

  • 1. React State & Derived Values (built-in): The simplest form — useState creates a reactive data source, and anything computed from it (derived values in JSX, useMemo) reacts automatically. The entire React rendering model is inherently reactive — state changes propagate to UI without manual DOM updates.
  • 2. useEffect as a Reactive Listener: useEffect with a dependency array is a reactive subscription — it fires a side effect in reaction to specific value changes. You declare what to watch, and the effect runs whenever those values emit a new value.
  • 3. React Context with useContext: A Publisher-Subscriber reactive pattern — the Provider publishes new values, and all Consumers automatically re-render in reaction to those changes.
  • 4. RxJS (Observables + Operators): The most complete reactive programming library for JavaScript. Integrates with React via useEffect and useState. Powerful for complex event streams — debounced search, real-time data, concurrent API calls, event buses.
  • 5. Reactive State Libraries (Zustand, Jotai, Recoil, MobX): All implement reactive patterns where component subscriptions update automatically when their subscribed state atoms/stores change. MobX is the most explicitly reactive — using observable decorators to make state fully reactive with automatic dependency tracking.
  • 6. React Query / SWR (Server State Reactivity): Reactive data fetching — components automatically re-render when server data is fetched, cached, refetched, or invalidated. The component subscribes to a cache key and reacts to its changes.

React's data flow is unidirectional — top-down. A child component cannot directly push data to its parent via props (props are read-only from the child's perspective). However, data can flow upward using a well-established callback function pattern — the parent passes a function down as a prop, and the child calls it with data when needed.

The pattern — 3 steps:

  • Step 1: The parent defines a function that accepts data and does something with it (updates state, logs, sends to API, etc.).
  • Step 2: The parent passes this function to the child as a prop (a callback prop, conventionally named onSomething).
  • Step 3: The child calls this function with its data at the appropriate time — e.g., when a button is clicked, a form is submitted, or a value changes.

Beyond direct parent-child: For components that aren't in a direct parent-child relationship:

  • "Lifting state up" — move shared state to the closest common ancestor. Both siblings can read and update it via props and callbacks.
  • Context API — the child updates Context state (if the Provider passed a setter via context value), which the parent or any other component can consume.
  • State management (Zustand, Redux) — the child updates a global store, the parent reads from the store.
jsx — child to parent data passing
// ── Pattern 1: Callback prop (most common) ──
function Parent() {
  const [childData, setChildData] = useState('');

  // Step 1: Define callback in parent
  const handleChildSubmit = (data) => {
    setChildData(data);
    console.log('Received from child:', data);
  };

  return (
    <div>
      <p>Child sent: {childData}</p>
      // Step 2: Pass callback down as prop
      <Child onSubmit={handleChildSubmit} />
    </div>
  );
}

function Child({ onSubmit }) {
  const [input, setInput] = useState('');

  // Step 3: Call the callback with data when needed
  return (
    <div>
      <input value={input} onChange={e => setInput(e.target.value)} />
      <button onClick={() => onSubmit(input)}>Send to Parent</button>
    </div>
  );
}

// ── Pattern 2: Lifting state up (siblings sharing data) ──
function CommonParent() {
  const [shared, setShared] = useState('');
  return (
    <>
      <SiblingA onChange={setShared} />   // pushes data up
      <SiblingB value={shared} />           // reads shared data
    </>
  );
}
