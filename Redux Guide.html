Redux — Component / Action / Store / Reducer
9 Qs

Redux is a predictable state management library for JavaScript applications. In the context of React, its role is to provide a single, centralised store for all application state — an approach called the single source of truth. Instead of scattering state across dozens of components and managing complex prop drilling or lifting, Redux holds everything in one place and any component in the tree can read from or write to it directly.

Why Redux became popular: As React applications grew beyond simple UIs, state management became painful. State that needed to be shared between many unrelated components required "lifting" it all the way up to a distant common ancestor and threading it back down through props. Redux solved this by decoupling state entirely from the component hierarchy.

Core principles Redux enforces:

  • Single Source of Truth: The entire app's state lives in one JavaScript object — the store. There is one definitive place to look at "what the app currently knows."
  • State is Read-Only: Components never mutate state directly. The only way to change state is to dispatch an action — a plain object describing what happened.
  • Changes are made with Pure Functions: State transitions are handled by reducers — pure functions that take the current state and an action and return a new state. No side effects, no mutation, always predictable.

Today's Redux landscape: The modern way to use Redux in React is with Redux Toolkit (RTK) — the official, opinionated toolset that eliminates the boilerplate of classic Redux. It wraps the same core concepts but makes them significantly more ergonomic.

This is one of the most practically important architectural decisions in React development. The answer lies in scope, complexity, and how widely the state is shared. There is no single right answer — they solve different problems and often coexist in the same app.

Use React Hooks (useState, useReducer, useContext) when:

  • State is local to a component or small subtree: Form input values, toggle states (open/closed), pagination page number, a component's own loading/error status — none of these need to be globally accessible. Keep them local with useState.
  • State only needs to be shared across a few levels: If a parent and its direct children need shared state, lifting state up with hooks is far simpler than setting up a Redux slice.
  • Low-to-medium complexity app: If your app has a handful of screens and the state is manageable, hooks + Context is cleaner, faster to build, and requires no extra dependencies.
  • You want zero extra bundle size: Hooks are built into React — no library to install.

Use Redux (Redux Toolkit) when:

  • State is needed by many unrelated components: The current logged-in user, a shopping cart, notification badges, feature flags — data that multiple disconnected parts of the UI need simultaneously. Redux's global store makes this effortless.
  • Complex, interdependent state logic: When state updates involve multiple related sub-values, action-based transitions, and complex derivations — the reducer pattern is cleaner and more predictable than multiple useState calls.
  • You need powerful debugging: Redux DevTools offers time-travel debugging — you can replay any sequence of actions, inspect the exact state at any point in history, and understand exactly why the UI changed. Invaluable in large apps.
  • Large team with many developers: Redux enforces a strict, predictable structure (actions → reducers → store) that scales well across large teams — everyone knows where state lives and how it's updated.
  • Server state management with RTK Query: Redux Toolkit includes RTK Query — a powerful data fetching and caching layer that handles API calls, caching, invalidation, and loading states automatically.

Redux enforces a strict unidirectional data flow. Data can only travel in one direction around a predictable cycle. This is what makes Redux apps so debuggable — given any action, you know exactly what will happen to the state.

The Redux data flow — 5 steps:

  1. User Interaction / Event: The user interacts with the UI — clicks a button, submits a form, changes a value. The React component responds to this event.
  2. Dispatch an Action: The component calls dispatch(action). An action is a plain JavaScript object with a type field (describing what happened) and an optional payload (the data associated with the event). The dispatch function is provided by useDispatch() hook from react-redux.
  3. Reducer Processes the Action: The Redux store sends the current state and the dispatched action to the relevant reducer function. The reducer is a pure function — it examines the action's type, applies the appropriate state transformation, and returns the new state. It never mutates the existing state directly.
  4. Store Updates: Redux replaces the old state in the store with the new state returned by the reducer. The store now holds the updated application state.
  5. UI Re-renders: All React components that are subscribed to the relevant part of the store (via useSelector()) are automatically notified of the state change. They re-render with the new data — the UI updates to reflect the new state.

jsx — full redux data flow traced
// Step 1 & 2: Component dispatches an action on button click
function AddToCartButton({ product }) {
  const dispatch = useDispatch();
  return (
    <button onClick={() => dispatch({
      type: 'cart/addItem',
      payload: { id: product.id, name: product.name, price: product.price }
    })}>
      Add to Cart
    </button>
  );
}

// Step 3: Reducer processes the action
function cartReducer(state = { items: [] }, action) {
  switch (action.type) {
    case 'cart/addItem':
      return { ...state, items: [...state.items, action.payload] };
    default: return state; // Step 4: new state returned to store
  }
}

// Step 5: Another component reads updated store & re-renders
function CartBadge() {
  const count = useSelector(state => state.cart.items.length);
  return <span>🛒 {count}</span>; // auto-updates when store changes
}

Modern Redux uses Redux Toolkit (RTK) — the official, opinionated way to write Redux code. It eliminates the classic Redux boilerplate and is the recommended starting point for all new Redux projects. You also need React Redux — the official binding library that connects Redux to React components.

Two packages you always install together:

  • @reduxjs/toolkit: The core Redux Toolkit library. Provides createSlice, createAsyncThunk, configureStore, and RTK Query.
  • react-redux: The React bindings layer. Provides the <Provider> component and the useSelector and useDispatch hooks that connect React components to the Redux store.

4-step setup process:

bash — step 1: install packages
npm install @reduxjs/toolkit react-redux

import { createSlice } from '@reduxjs/toolkit';

const counterSlice = createSlice({
  name:         'counter',
  initialState: { value: 0 },
  reducers: {
    increment:   state => { state.value += 1; },
    decrement:   state => { state.value -= 1; },
    incrementBy: (state, action) => { state.value += action.payload; },
  },
});

export const { increment, decrement, incrementBy } = counterSlice.actions;
export default counterSlice.reducer;

import counterReducer from './counterSlice';

An Action Creator is a function that creates and returns an action object. Instead of manually writing action objects every time you dispatch, you call the action creator function — which produces the correctly-structured action object for you. This promotes consistency, reduces typos, and centralises action object creation.

Why action creators exist:

  • Consistency: The action type string ('cart/addItem') is defined in one place — in the action creator. If you need to change the type string, you change it once, not everywhere it's dispatched.
  • Encapsulation: The shape of the payload is defined in the action creator. Callers don't need to know the exact structure — just call the function with the necessary data.
  • Testability: Action creators are pure functions that return plain objects — trivially easy to unit test.

In Redux Toolkit: When you define reducers inside createSlice(), RTK automatically generates action creators for each reducer function. You don't write them manually. The slice's .actions property exports these auto-generated action creators.

jsx — action creators: manual (classic) vs auto (RTK)
// ── Classic Redux — manual action creators ──
const addToCart = (product) => ({   // function returns action object
  type:    'cart/ADD_ITEM',
  payload: product,
});

const removeFromCart = (id) => ({
  type:    'cart/REMOVE_ITEM',
  payload: id,
});

// Usage: dispatch(addToCart({ id: 1, name: 'Laptop', price: 999 }))

// ── Redux Toolkit — action creators auto-generated by createSlice ──
const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [] },
  reducers: {
    addItem:    (state, action) => { state.items.push(action.payload); },
    removeItem: (state, action) => {
      state.items = state.items.filter(i => i.id !== action.payload);
    },
  },
});

// RTK auto-creates: cartSlice.actions.addItem and cartSlice.actions.removeItem
export const { addItem, removeItem } = cartSlice.actions;

// Usage in component — same dispatch syntax, just cleaner
const dispatch = useDispatch();
dispatch(addItem({ id: 1, name: 'Laptop', price: 999 }));
dispatch(removeItem(1));

// What addItem({ id: 1, name: 'Laptop' }) actually produces:
// { type: 'cart/addItem', payload: { id: 1, name: 'Laptop' } }

These three terms are related but distinct — confusing them is a common mistake. Here's a precise breakdown of each:

  • Action Type — the identifier string: A plain string constant that uniquely identifies a specific state change event. By convention it uses the format 'sliceName/actionName' (e.g., 'cart/addItem', 'auth/setUser'). The action type is the label — the reducer switches on this to decide which state transition to apply. In classic Redux, types were defined as separate constants (const ADD_ITEM = 'cart/ADD_ITEM') to avoid typos. In RTK, the type is automatically generated from the slice name and reducer name.
  • Action Object — the event description: A plain JavaScript object that describes what happened. It must have a type property (the action type string) and optionally a payload property containing any additional data needed to compute the new state. This is what you actually dispatch to the Redux store. The reducer receives this object as its second argument.
  • Action Creator — the factory function: A function that, when called with the necessary arguments, creates and returns the correctly-shaped action object. It abstracts the manual construction of the action object. In RTK, these are auto-generated. You call them and pass the result to dispatch.
jsx — the three concepts side by side
// ── 1. Action Type — just a string identifier ──
const ADD_ITEM_TYPE = 'cart/addItem'; // classic constant
// In RTK: cartSlice.actions.addItem.type → 'cart/addItem'

// ── 2. Action Object — plain object sent to the reducer ──
const actionObject = {
  type:    'cart/addItem',           // ← action type (required)
  payload: { id: 1, name: 'Laptop', price: 999 } // ← data (optional)
};

// ── 3. Action Creator — function that creates the action object ──
const addItem = (product) => ({  // called with data, returns action object
  type:    'cart/addItem',
  payload: product,
});

// ── How they fit together ──
const actionObj = addItem({ id: 1, name: 'Laptop', price: 999 });
// actionObj → { type: 'cart/addItem', payload: { id: 1, name: 'Laptop', price: 999 } }

dispatch(actionObj);         // dispatch the action object
// or inline:
dispatch(addItem({ id: 1, name: 'Laptop', price: 999 }));

When using Redux, the component architecture stays the same as any React app — the difference is how components read and update state. Instead of useState or props, components connect to the Redux store using two hooks from react-redux.

Two hooks that connect components to Redux:

  • useSelector(selectorFn) — read from store: Takes a selector function as argument. The selector receives the full Redux state tree and returns only the slice of state this component needs. Whenever that slice changes, the component re-renders. When the slice doesn't change, the component is skipped — this makes it efficient.
  • useDispatch() — write to store: Returns the store's dispatch function. You call this with an action object (or action creator call) whenever you want to trigger a state change. The dispatch goes to the reducer, which computes the new state.

The overall file/folder structure:

  • store/ folder: Contains all Redux-related code — slices, store configuration, selectors.
  • components/ folder: React components — they import action creators and use useSelector/useDispatch but contain no reducer logic.
  • main.jsx: Wraps the entire app in <Provider store={store}> — makes the store accessible to all components.
jsx — component structure with redux
// store/cartSlice.js — state + actions defined here
import { createSlice } from '@reduxjs/toolkit';
const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [], total: 0 },
  reducers: {
    addItem:    (state, { payload }) => { state.items.push(payload); },
    removeItem: (state, { payload }) => {
      state.items = state.items.filter(i => i.id !== payload);
    },
    clearCart:  state => { state.items = []; },
  },
});
export const { addItem, removeItem, clearCart } = cartSlice.actions;
export default cartSlice.reducer;

// store/store.js — single store for the app
import { configureStore } from '@reduxjs/toolkit';
import cartReducer from './cartSlice';
export const store = configureStore({ reducer: { cart: cartReducer } });

// components/CartBadge.jsx — reads from store (display component)
import { useSelector } from 'react-redux';
function CartBadge() {
  const itemCount = useSelector(state => state.cart.items.length);
  return <span>🛒 {itemCount}</span>;
}

// components/ProductCard.jsx — dispatches actions (interactive component)
import { useDispatch } from 'react-redux';
import { addItem }     from '../store/cartSlice';
function ProductCard({ product }) {
  const dispatch = useDispatch();
  return (
    <div>
      <h3>{product.name}</h3>
      <button onClick={() => dispatch(addItem(product))}>Add to Cart</button>
    </div>
  );
}

The Store is the central object in Redux — the single source of truth for the entire application's state. It is the container that holds the complete state tree, manages all state updates, and notifies subscribers when state changes.

The Store's 4 core responsibilities:

  • Holds the entire application state: The store maintains one JavaScript object that represents every piece of state your app has — user auth, shopping cart, UI flags, fetched data. There is only ever one store in a Redux app (unlike multiple useState calls scattered across components).
  • Receives dispatched actions: The only way to trigger a state change is to call store.dispatch(action). The store receives the action, passes it along with the current state to the appropriate reducer, and stores the result.
  • Runs reducers to compute new state: When an action is dispatched, the store passes it through all reducers (via combineReducers under the hood in configureStore). Each reducer handles its own slice of state. The results are combined into the new state tree.
  • Notifies subscribers of state changes: The store maintains a list of listener functions. When state updates, it notifies all listeners. In React, useSelector registers components as listeners — so they automatically re-render when their selected slice of state changes.

Creating the store with RTK (configureStore): RTK's configureStore wraps Redux's createStore, automatically sets up the Redux DevTools extension, adds the middleware for async thunks (redux-thunk), and combines reducers from the reducer config option.

jsx — store creation and key APIs
import { configureStore } from '@reduxjs/toolkit';
import authReducer  from './authSlice';
import cartReducer  from './cartSlice';
import notifReducer from './notifSlice';

export const store = configureStore({
  reducer: {
    auth:  authReducer,   // state.auth
    cart:  cartReducer,   // state.cart
    notif: notifReducer,  // state.notif
  },
  // devTools: true by default in development
  // middleware: auto-includes redux-thunk for async
});

// Store's built-in API (rarely used directly in React — hooks handle this)
store.getState();        // returns the current complete state tree
store.dispatch(action); // triggers a state update
store.subscribe(fn);   // register a listener (useSelector does this for you)

// Full state shape after configureStore:
// {
//   auth:  { user: null, token: null, isLoading: false },
//   cart:  { items: [], total: 0 },
//   notif: { messages: [], unread: 0 }
// }

A Reducer is a pure function that is the heart of Redux's state management logic. It is responsible for computing the new state in response to a dispatched action. Redux gets its name from the concept of the Array.reduce() method — a reducer accumulates state over a series of actions.

The reducer's signature:

jsx — reducer signature
(currentState, action) => newState

Rules a reducer must follow — it MUST be a pure function:

  • Same inputs → same output: Given the same current state and action, the reducer must always return the same new state. No randomness, no relying on external variables.
  • No side effects: A reducer must not make API calls, generate random numbers, read from Date.now(), modify files, log to the console, or do anything that interacts with the outside world. It computes state and that's it.
  • Never mutate the existing state: A reducer must treat state as immutable. It should always return a new state object — never modify the existing one in place. Exception: Redux Toolkit uses Immer internally, which lets you write seemingly mutating code (state.count++) that is safely converted to immutable updates behind the scenes.
  • Always return state for unknown actions: A reducer's default case must return the current state unchanged — this ensures other reducers' actions don't accidentally clear your state.

Slice reducers vs Root reducer: In a large app, you write individual reducers for each domain (cart, auth, notifications). configureStore combines them into a root reducer — each sub-reducer only manages its own slice of the state tree.

jsx — reducer: classic vs RTK (Immer)
// ── Classic Redux Reducer — manual immutable updates (verbose) ──
function cartReducer(state = { items: [], total: 0 }, action) {
  switch (action.type) {
    case 'cart/addItem':
      return {
        ...state,                              // spread to avoid mutation
        items: [...state.items, action.payload] // new array
      };
    case 'cart/removeItem':
      return {
        ...state,
        items: state.items.filter(i => i.id !== action.payload)
      };
    case 'cart/clearCart':
      return { ...state, items: [] };
    default:
      return state; // ← always return state for unknown actions
  }
}

// ── RTK createSlice Reducer — Immer allows "mutation" syntax ──
const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [], total: 0 },
  reducers: {
    // Looks like mutation but Immer makes it safe ✅
    addItem:    (state, { payload }) => { state.items.push(payload); },
    removeItem: (state, { payload }) => {
      state.items = state.items.filter(i => i.id !== payload);
    },
    clearCart:  state => { state.items = []; state.total = 0; },
  },
});

// Pure function check:
// ✅ No API calls    ✅ No Math.random()    ✅ No Date.now()
// ✅ Same in → same out    ✅ Default returns current state
Core Principles / Pros-Cons / Local & Redux State
4 Qs

Redux is built on three foundational principles. Every design decision in Redux flows from these — understanding them is the key to understanding why Redux works the way it does.

Principle 1 — Single Source of Truth:

The entire application state is stored in a single JavaScript object tree within a single store. No matter how many components or features your app has, there is one and only one authoritative source of data. This makes the application's state at any given moment completely transparent — you can log the store and see everything your app knows. It also enables powerful features like undo/redo, state persistence (saving the whole app state to localStorage), and server-side state hydration.

jsx — single source of truth
// The ENTIRE app state — one object, one store
{
  auth:  { user: { id: 1, name: 'Riya' }, token: 'jwt...' },
  cart:  { items: [{ id: 1, name: 'Laptop', price: 999 }], total: 999 },
  notif: { messages: [], unread: 0 },
  ui:    { sidebarOpen: false, theme: 'dark' },
}
// One store.getState() call gives you the complete picture — always

Principle 2 — State is Read-Only:

No component, no event handler, no API callback is allowed to directly modify the state in the store. The only legal way to change state is to dispatch an action — a plain JavaScript object that describes what happened. This enforces a deliberate, auditable update trail. Every state change is explicit, intentional, and goes through a controlled pipeline. With Redux DevTools, you can see every dispatched action and the state before and after — making bugs trivially traceable.

jsx — state is read-only
// ❌ Direct mutation — NEVER allowed in Redux
store.getState().cart.items.push(newItem);     // forbidden
store.getState().auth.user = null;             // forbidden

// ✅ Only way to change state — dispatch an action
dispatch({ type: 'cart/addItem', payload: newItem });
dispatch({ type: 'auth/logout' });
// Every change is explicit, traceable, and auditable

Principle 3 — Changes are Made with Pure Functions (Reducers):

To specify how the state changes in response to actions, you write pure reducer functions. A pure function has no side effects and always returns the same output for the same input. Reducers take the previous state and an action, and return the next state — never mutating the previous state in place. Because reducers are pure, they are completely predictable and trivial to test — you pass in a state and an action, and assert on the output, with no mocking required.

jsx — changes via pure reducer functions
// Pure reducer — same input always → same output, no side effects
function cartReducer(state = { items: [] }, action) {
  switch (action.type) {
    case 'cart/addItem':
      // Returns NEW state — never mutates existing state
      return { ...state, items: [...state.items, action.payload] };
    default: return state;
  }
}

// Trivially testable — no React, no mocks needed
const nextState = cartReducer(
  { items: [] },
  { type: 'cart/addItem', payload: { id: 1, name: 'Laptop' } }
);
// expect(nextState.items).toHaveLength(1); ← passes ✅

Redux isn't right for every project, but when it fits — it brings serious advantages:

  • 1. Predictable, Debuggable State — Time-Travel Debugging: Because state only changes via pure reducers and actions, the Redux DevTools browser extension can record every action dispatched during a session. You can replay the exact sequence of actions, jump back to any previous state, and see precisely which action caused a bug. This "time-travel debugging" capability is unmatched — it dramatically reduces the time spent hunting down hard-to-reproduce bugs in complex apps.
  • 2. Eliminates Prop Drilling — Global State Access: Any component, no matter how deeply nested in the component tree, can read from or write to the Redux store directly using useSelector and useDispatch. You never need to thread props down through 5 layers of components just to share data. This keeps intermediate components clean — they only receive the props they actually need.
  • 3. Consistent, Scalable Architecture: Redux enforces a strict, standard pattern — actions go in, reducers compute state, store holds it, selectors read it. Every developer on the team, regardless of their experience level, knows exactly where to find any piece of state and how to change it. This consistency is invaluable in large teams and makes onboarding new developers significantly faster.
  • 4. Centralised Business Logic: Reducer functions contain the state transition logic for the entire domain — independent of React components. This separation of concerns means your core business logic can be tested without rendering any components, shared between web and React Native apps, and refactored without touching UI code.
  • 5. Rich Ecosystem — Middleware, RTK Query: Redux's middleware system allows you to intercept dispatched actions and add cross-cutting concerns like logging, analytics tracking, error reporting, and async API calls. Redux Toolkit's RTK Query is a powerful data fetching and caching layer built into RTK — it handles API calls, request deduplication, cache invalidation, optimistic updates, and loading/error states automatically.
jsx — RTK Query: benefit 5 in action
import { createApi, fetchBaseQuery } from '@reduxjs/toolkit/query/react';

// Define API once — RTK Query handles caching, loading, errors
const api = createApi({
  baseQuery: fetchBaseQuery({ baseUrl: '/api' }),
  endpoints: (build) => ({
    getUser:  build.query({ query: id => `/users/${id}` }),
    getUsers: build.query({ query: () => '/users' }),
  }),
});

export const { useGetUserQuery, useGetUsersQuery } = api;

// Component — no useEffect, no fetch, no loading state needed
function UserProfile({ id }) {
  const { data, isLoading, error } = useGetUserQuery(id);
  if (isLoading) return <Spinner />;
  return <h1>{data?.name}</h1>;
}

Understanding when to use local state vs Redux state is the most important architectural decision in a Redux app. The two systems exist side-by-side in almost every real Redux application — they solve different problems and should both be used appropriately.

  • Ownership and Scope: Local state (useState, useReducer) is owned by a specific component. Only that component and its direct children (via props) can access it. When the component unmounts, the state is destroyed. Redux state lives in the global store — it is accessible by any component in the entire tree, survives component unmounting, and persists for the lifetime of the page session (unless explicitly reset).
  • Data type and usage: Local state is ideal for UI-specific, transient data — whether a modal is open, the current value of a text input, which tab is active, a component's own loading/error status. Redux state is for application-level, domain data — the logged-in user, a list of products fetched from the API, the contents of a shopping cart, global notification messages.
  • Update mechanism: Local state is updated by calling a setter function (setState) directly in the component. Redux state is updated by dispatching an action that flows through a reducer — a more structured but more verbose process.
  • Boilerplate: Local state requires almost zero setup — one line with useState. Redux state requires defining a slice (initial state + reducers), exporting action creators, registering the reducer in the store, and then using useSelector/useDispatch in components. The overhead is only justified when the state genuinely needs to be global.
  • Debugging: Local state changes are only visible inside the component — you need React DevTools to inspect them. Redux state changes are tracked in Redux DevTools with a full history of every action, the before/after state diff, and the ability to replay any sequence.
  • Performance: Updating local state only re-renders the component that owns it and its children. Updating Redux state notifies all useSelector subscribers — components that select the changed slice re-render. Well-written selectors with precise selections minimise unnecessary re-renders.
jsx — local vs redux state: side by side
// ── LOCAL STATE — UI-specific, transient, component-owned ──
function SearchBar() {
  const [query, setQuery]     = useState('');     // input value → local
  const [focused, setFocused] = useState(false); // focus state → local
  // Nobody else needs to know about these
  return <input value={query} onChange={e => setQuery(e.target.value)} />;
}

// ── REDUX STATE — app-level, shared, persists across components ──
// Needed in: Header (badge), CartPage, CheckoutPage, OrderSummary

Redux is a powerful tool, but it comes with real trade-offs. Knowing these shows balanced engineering judgment — and helps you make the right architectural decision for each project:

  • 1. Boilerplate and Ceremony (even with RTK): Even with Redux Toolkit reducing classic Redux's verbosity, adding a new piece of state still requires: creating or updating a slice file (initial state, reducers), exporting action creators, registering the reducer in the store, and then using useSelector/useDispatch in components. For simple state that could be two lines of useState, this feels like significant overhead. The upfront cost is justified at scale, but painful for small features.
  • 2. Steep Learning Curve: Redux introduces a lot of concepts that are non-trivial for beginners — the store, actions, action creators, action types, reducers, dispatch, selectors, middleware, thunks, slices, and the Provider pattern. React developers new to state management often find this overwhelming compared to the intuitive simplicity of useState.
  • 3. Over-Engineering Risk: Redux is overkill for many applications. Reaching for it in a small to medium app adds unnecessary complexity and bundle size when Context API + useState/useReducer or a lightweight library like Zustand would serve perfectly. The common anti-pattern: putting every piece of state — including ephemeral UI state like form values and toggle flags — into Redux, when it should stay local.
  • 4. Performance Pitfalls with Poorly Written Selectors: Every time any part of the Redux store updates, all useSelector calls re-evaluate their selector functions. If selectors are computing derived data on every call (like filtering or mapping an array) without memoisation, they trigger expensive recalculations on every unrelated state update. This requires using memoised selectors (via createSelector from Reselect, which RTK includes) — adding yet more code to write and maintain.
  • 5. Async Logic Complexity: Handling asynchronous operations (API calls) in Redux requires middleware. With RTK, createAsyncThunk handles the common case, but complex async flows (race conditions, polling, cancellation, request deduplication) still require careful thought and often significant boilerplate — pending/fulfilled/rejected action cases for every API call.
  • 6. Immutability Discipline (classic Redux): In classic Redux (without RTK/Immer), developers must manually write immutable update patterns — spread operators, array methods that return new arrays, never mutating nested objects directly. This is error-prone and verbose. RTK solves this with Immer, but it's still a concept developers must understand.

The <Provider> is a React component from the react-redux library. Its sole job is to make the Redux store available to every component in the application — without having to pass the store manually as a prop through every level of the component tree. It uses React's Context API internally to broadcast the store reference to all descendants.

How it works: You wrap your entire app (or the relevant subtree) in <Provider store={store}> — typically in main.jsx or index.js. Once wrapped, any component anywhere in the tree can subscribe to the store using the useSelector and useDispatch hooks from react-redux.

How components get state from the store — the two-step process:

  • Step 1 — useSelector(selectorFn): The component calls useSelector with a selector function. The selector receives the entire Redux state tree and returns only the specific slice that component needs. react-redux registers the component as a subscriber — whenever that selected slice changes, the component re-renders. When it doesn't change, the component is skipped — this is what makes Redux efficient.
  • Step 2 — useDispatch(): Returns the store's dispatch function. The component calls dispatch(action) to trigger state changes. The action flows through reducers, the store updates, and all affected useSelector subscribers re-render automatically.
jsx — provider setup and component state access
// main.jsx — Provider wraps entire app ONCE
import { Provider } from 'react-redux';
import { store }    from './store/store';

ReactDOM.createRoot(document.getElementById('root')).render(
  <Provider store={store}>   // ← makes store available to ALL descendants
    <App />
  </Provider>
);

// Any component — reads from store, no props needed
function CartBadge() {
  // useSelector — subscribes to cart.items.length
  // Only re-renders when that specific value changes
  const itemCount = useSelector(state => state.cart.items.length);
  return <span>🛒 {itemCount}</span>;
}

function AddButton({ product }) {
  const dispatch = useDispatch(); // ← get dispatch from Provider's store
  return (
    <button onClick={() => dispatch({ type: 'cart/addItem', payload: product })}>
      Add to Cart
    </button>
  );
}

connect() is the older, class-component-era API from react-redux for connecting a component to the Redux store. Before hooks were introduced (pre-React 16.8), connect() was the primary — and only — way to give a React component access to the Redux store. It is a Higher-Order Component (HOC) factory — it takes configuration functions and returns an HOC that wraps your component.

Signature:

jsx — connect signature
const ConnectedComponent = connect(mapStateToProps, mapDispatchToProps)(MyComponent);

What the two arguments do:

  • mapStateToProps(state): A function that receives the full Redux state tree and returns an object. The key-value pairs of this object become additional props on the wrapped component — giving it read access to the store slices it needs. Equivalent to useSelector in the modern hooks API.
  • mapDispatchToProps(dispatch): A function (or object) that receives the dispatch function and returns an object. The returned key-value pairs become additional props that are functions the component can call to dispatch actions. Equivalent to useDispatch in the modern hooks API.

Today's recommendation: In all new code, use useSelector and useDispatch hooks — they are simpler, require less code, and work only in functional components. connect() is still valid but considered legacy and you'll mainly encounter it in older codebases or when working with class components that cannot use hooks.

jsx — connect() vs modern hooks
// ── Legacy: connect() HOC approach ──
import { connect } from 'react-redux';

function CartButton({ itemCount, onAddItem }) { // ← props injected by connect
  return <button onClick={onAddItem}>Cart ({itemCount})</button>;
}

const mapStateToProps = state => ({
  itemCount: state.cart.items.length,  // ← becomes this.props.itemCount
});

const mapDispatchToProps = dispatch => ({
  onAddItem: item => dispatch({ type: 'cart/addItem', payload: item }),
});

export default connect(mapStateToProps, mapDispatchToProps)(CartButton);

// ── Modern: hooks (same result, half the code) ──
function CartButton() {
  const itemCount = useSelector(state => state.cart.items.length);
  const dispatch  = useDispatch();
  return (
    <button onClick={() => dispatch({ type: 'cart/addItem' })}>
      Cart ({itemCount})
    </button>
  );
}

A well-structured React-Redux project (using Redux Toolkit) typically organises its code around these four types of files. Each has a distinct responsibility in the Redux architecture:

  • 1. Slice File (e.g., store/cartSlice.js): Created using RTK's createSlice(). This is the most important file per feature — it defines the initial state, all reducer functions (state transitions), and auto-exports action creators. One slice per domain/feature: authSlice.js, cartSlice.js, notifSlice.js. This is where all state logic lives.
  • 2. Store Configuration File (store/store.js): Created using RTK's configureStore(). This file imports all slice reducers and combines them into the single root store. It is the central registry — the single place where all slices are registered. You import and export the store object from here.
  • 3. Entry Point File (main.jsx / index.js): The application bootstrap file that wraps the entire <App /> in the <Provider store={store}> component. This grants every component in the tree access to the Redux store. This file changes minimally — it just needs the Provider wrapper added once.
  • 4. Component File (e.g., components/CartPage.jsx): The React component that consumes Redux state. It imports action creators from the relevant slice and uses useSelector to read state and useDispatch to trigger updates. Components should contain zero reducer logic — they only call the store's API.

The recommended architecture for a Redux project is "Feature Slices" (also called the "Ducks" pattern or "Redux Toolkit's slice pattern") — organising your store code by feature/domain rather than by type of file. This keeps related logic co-located and scales naturally as the app grows.

Key structural guidelines:

  • One slice per domain: Each major feature (auth, cart, products, notifications, UI) gets its own slice file containing all the state, reducer, and action logic for that domain. This makes it easy to find everything related to one feature in one place.
  • Selectors close to the slice: Define reusable selector functions in the same slice file or a companion selectors.js file. This prevents state.cart.items.filter(...)` logic from being duplicated across every component that needs derived cart data.
  • Async thunks in the slice: Using createAsyncThunk, define API call thunks in the same slice file as the state they affect. The thunk's pending/fulfilled/rejected cases are handled in extraReducers.
  • Components only import from slices: Components import action creators and selectors from slice files — they never define their own Redux logic. This enforces separation of concerns.
jsx — feature-based slice with selectors and thunks
// store/productsSlice.js — all products logic in one file
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';

// Async thunk — API call lives with the slice
export const fetchProducts = createAsyncThunk('products/fetch', async () => {
  const res = await fetch('/api/products');
  return res.json();
});

const productsSlice = createSlice({
  name: 'products',
  initialState: { items: [], loading: false, error: null },
  reducers: {
    setFilter: (state, { payload }) => { state.filter = payload; },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchProducts.pending,   state => { state.loading = true; })
      .addCase(fetchProducts.fulfilled, (state, { payload }) => {
        state.loading = false; state.items = payload;
      })
      .addCase(fetchProducts.rejected,  (state, { error }) => {
        state.loading = false; state.error = error.message;
      });
  },
});

// Selectors defined in the same file — reusable across components
export const selectAllProducts  = state => state.products.items;
export const selectProductCount = state => state.products.items.length;
export const selectIsLoading    = state => state.products.loading;

export const { setFilter } = productsSlice.actions;
export default productsSlice.reducer;

// Component — imports everything from the slice
import { fetchProducts, selectAllProducts, selectIsLoading } from '../store/productsSlice';
function ProductList() {
  const dispatch  = useDispatch();
  const products  = useSelector(selectAllProducts);
  const isLoading = useSelector(selectIsLoading);
  useEffect(() => { dispatch(fetchProducts()); }, [dispatch]);
  if (isLoading) return <Spinner />;
  return products.map(p => <ProductCard key={p.id} product={p} />);
}

Immutability means that once a data structure is created, it cannot be modified in place. Instead of changing an existing object, you create a new object with the desired changes — the original remains untouched.

Why Redux requires immutable state updates:

  • Change detection: Redux (and React) detect state changes by reference equality — comparing whether oldState === newState. If you mutate the existing state object directly (state.items.push(x)), the reference stays the same — Redux thinks nothing changed and doesn't trigger re-renders. Returning a new object ensures the reference changes, reliably signalling an update.
  • Time-travel debugging: Redux DevTools stores a history of every previous state snapshot. If state were mutated in place, every historical snapshot would reflect the current (mutated) state — the history would be corrupted. Immutability guarantees that each saved snapshot is preserved exactly as it was.
  • Predictability and pure functions: Immutability enforces that reducers are truly pure — they take inputs and return a new value without side effects. This makes them trivially testable and guarantees the same input always produces the same output.

Redux Toolkit and Immer: RTK uses the Immer library internally. Immer wraps your reducer's state in a JavaScript Proxy — letting you write seemingly mutating code (state.count++) while Immer intercepts each "mutation" and produces a safe, new immutable state behind the scenes. This gives you the simplicity of mutable-style code with the safety of immutability.

jsx — immutability: wrong vs correct vs RTK immer
// ❌ WRONG — mutating state directly (same reference, no re-render)
case 'addItem':
  state.items.push(action.payload);  // mutates! reference unchanged
  return state;                        // Redux sees no change → no re-render

// ✅ CORRECT classic Redux — return a new object (new reference)
case 'addItem':
  return {
    ...state,                               // spread creates new object
    items: [...state.items, action.payload] // new array
  };

// ✅ RTK with Immer — write "mutations", Immer produces new state safely
const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [] },
  reducers: {
    addItem: (state, { payload }) => {
      state.items.push(payload); // Immer intercepts → safe immutable update ✅
    },
    updatePrice: (state, { payload: { id, price } }) => {
      const item = state.items.find(i => i.id === id);
      if (item) item.price = price; // Immer handles nested mutation safely ✅
    },
  },
});

An action object in Redux is a plain JavaScript object that describes what happened in the application. Redux and the Flux Standard Action (FSA) community have established a conventional structure for action objects to ensure consistency across codebases.

Properties of a standard action object:

  • type (required — string): The only mandatory property. A string identifier that uniquely names the event. By convention, it uses the format 'sliceName/actionName' (e.g., 'cart/addItem', 'auth/logout'). The reducer uses this to decide which state transformation to apply. In RTK, this is auto-generated from your slice name and reducer key.
  • payload (optional — any value): Any additional data required to perform the state update. This is the "content" of the action — the product to add to the cart, the user data returned from an API, the ID of the item to delete. By the FSA convention, all action data (beyond the type) goes in payload — keeping the action shape predictable. RTK enforces this convention automatically.
  • error (optional — boolean): If true, signals that the action represents a failure/error condition. When error: true, the payload should be an Error object. RTK's createAsyncThunk uses this convention for rejected actions.
  • meta (optional — any value): Extra information about the action that doesn't belong in the payload — things like timestamps, request IDs, or middleware-specific flags. Not commonly used in most apps.
jsx — action object anatomy
// Minimal action — only required property
{ type: 'auth/logout' }

// Standard action with payload
{
  type:    'cart/addItem',
  payload: { id: 1, name: 'MacBook Pro', price: 1999, qty: 1 }
}

// Error action (FSA convention)
{
  type:    'products/fetchFailed',
  payload: new Error('Network timeout'),
  error:   true
}

// Action with meta (uncommon but valid)
{
  type:    'analytics/pageView',
  payload: { page: '/dashboard' },
  meta:    { timestamp: Date.now(), sessionId: 'abc123' }
}

// RTK-generated action (from cartSlice.actions.addItem)
// calling addItem({ id: 1, name: 'Laptop' }) produces:

Both are configuration functions passed to the legacy connect() HOC, but they serve opposite purposes — one is for reading from the store, the other for writing to it.

  • mapStateToProps(state, ownProps) — Read (store → props): Receives the complete Redux state tree as its first argument (and optionally the component's own props as the second). It returns a plain object whose properties are merged into the wrapped component's props — giving the component read access to specific slices of Redux state. React-Redux calls this function every time the store state changes and checks whether the returned object has changed — if it has, the component re-renders. It is the connect()-era equivalent of useSelector.
  • mapDispatchToProps(dispatch, ownProps) — Write (actions → props): Receives the store's dispatch function as its first argument. It returns an object of functions — each one calls dispatch with a specific action. These functions are merged into the component's props as event-handler-style callbacks, allowing the component to trigger state changes without knowing about dispatch directly. It is the connect()-era equivalent of useDispatch. Can also be passed as a plain object of action creators — connect() automatically wraps them with dispatch.

Unidirectional Data Flow means data in a Redux application can only travel in one direction around a fixed, predictable cycle — never backwards, never sideways between components directly. This constraint is what makes Redux apps highly predictable and debuggable.

The one-way cycle is: Action → Reducer → Store → UI → Action → ...

  • UI triggers an action: A user interaction (click, submit, input change) causes the component to dispatch an action — a plain description of what happened.
  • Action flows to the reducer: The store passes the action to the reducer function. The reducer examines the action type and computes a new state. Data flows forward — never back to the component that dispatched it.
  • Reducer updates the store: The new state returned by the reducer replaces the old state in the store. The store now holds the authoritative, updated truth.
  • Store notifies subscribed components: Components subscribed via useSelector are notified. They re-render with the new state data — the UI updates to reflect the new store state.

Why unidirectional flow matters:

  • Predictability: Given any action, you know exactly where it goes (reducer), what it does (computed by a pure function), and what updates (all subscribed components). There are no surprises.
  • Debuggability: Because data only flows one way, you can intercept it at any point in the cycle. Redux DevTools shows you every action, the state before, and the state after — making it trivial to find the exact action that caused a bug.
  • No "data spaghetti": Unlike two-way data binding (Angular's older approach), there's no risk of cascading, circular updates where A updates B which updates A again.

Redux handles inter-component communication through the shared store — not through direct component-to-component messaging. Components never communicate with each other directly. Instead, they all communicate indirectly via the store, which acts as the central hub.

The pattern — any component can speak to any other through the store:

  • Component A (sender): Dispatches an action to the store — e.g., user adds an item to cart.
  • The store: Processes the action through the reducer, updates its state.
  • Component B (receiver): Has a useSelector subscribed to the cart state. When it changes, Component B re-renders automatically with the new data.

Component A and Component B don't know about each other. They're completely decoupled — they only interact with the store. This is what makes Redux so scalable: you can add new components that read or write to any part of the store without touching existing components.

Contrast with prop drilling: Without Redux, Component A would need to pass data up to a common ancestor, which passes it back down to Component B through props — potentially through many intermediate components. Redux eliminates this entirely.

jsx — decoupled communication via store
// Component A — ProductCard (anywhere in the tree)
// Dispatches to store — doesn't know about Header or CartPage
function ProductCard({ product }) {
  const dispatch = useDispatch();
  return (
    <button onClick={() => dispatch(addItem(product))}>
      Add to Cart
    </button>
  );
}

// Component B — Header (completely separate branch of the tree)
// Automatically updates when cart changes — doesn't know about ProductCard
function Header() {
  const count = useSelector(s => s.cart.items.length);
  return <span>🛒 {count} items</span>;
}

// Component C — CartPage (another branch)
// Also automatically updated — no connection to ProductCard
function CartPage() {
  const items = useSelector(s => s.cart.items);
  return <ul>{items.map(i => <li key={i.id}>{i.name}</li>)}</ul>;
}

The payload property is the conventional field in a Redux action object that carries the data associated with an action. While the type field answers "what happened?", the payload answers "with what data?".

Key characteristics:

  • Optional but standard: Not all actions need a payload — actions like 'auth/logout' or 'cart/clearCart' just need a type to trigger a state change. But whenever additional data is needed to compute the new state, it goes in payload.
  • Can be any JavaScript value: A primitive (number, string, boolean), an object, an array, or null. Whatever the reducer needs to do its job.
  • Convention from Flux Standard Actions (FSA): The payload key name is a widely adopted convention — it keeps action objects consistent and predictable. Before FSA, different projects used different keys (data, item, value) making codebases inconsistent.
  • RTK always uses payload: When you define reducers with createSlice, RTK automatically puts the argument you pass to an action creator into action.payload. This is automatic and consistent — you don't need to manually wrap data in a payload object.
jsx — payload in action objects and reducers
// Actions with different payload types
dispatch({ type: 'auth/logout' });                   // no payload needed
dispatch({ type: 'counter/setTo', payload: 42 });    // number payload
dispatch({ type: 'cart/addItem',  payload: { id: 1, name: 'Laptop' } }); // object payload
dispatch({ type: 'tags/setTags',  payload: ['react', 'redux'] }); // array payload

// Reducer reads payload from action
const counterSlice = createSlice({
  name: 'counter',
  initialState: { value: 0 },
  reducers: {
    incrementBy: (state, action) => {
      state.value += action.payload;  // ← reads from action.payload
    },
    // Destructured — cleaner
    setTo: (state, { payload }) => {
      state.value = payload;          // ← destructure payload directly
    },
  },
});

The fundamental difference is what they return — and therefore what kinds of operations they can handle.

Regular Action Creator — synchronous, returns an object:

A regular action creator is a plain function that returns a plain action object with a type and optional payload. It is synchronous — it does its job instantly and returns the result immediately. Redux's store can receive and process this object directly. Use regular action creators for any state change that doesn't require waiting for anything — UI updates, toggle flags, setting a value.

Thunk Action Creator — asynchronous, returns a function:

A thunk action creator returns a function instead of an object. This inner function receives dispatch and getState as arguments — allowing it to perform asynchronous operations (API calls, timeouts, etc.) and dispatch multiple actions at different points in time (e.g., dispatch "loading started", await the fetch, then dispatch "success" or "failure"). The redux-thunk middleware intercepts functions returned from dispatch calls and executes them — regular Redux only accepts plain objects.

In RTK, createAsyncThunk generates thunks automatically — it handles the pending/fulfilled/rejected action lifecycle for you.

jsx — regular vs thunk action creator
// ── Regular Action Creator — synchronous, returns object ──
const setUser = (user) => ({          // returns plain object
  type:    'auth/setUser',
  payload: user,
});
dispatch(setUser({ id: 1, name: 'Riya' })); // dispatches immediately

// ── Thunk Action Creator — async, returns a function ──
const loginUser = (credentials) => async (dispatch, getState) => {
  // 1. Dispatch loading state
  dispatch({ type: 'auth/setLoading', payload: true });

  try {
    // 2. Perform async API call
    const res  = await fetch('/api/login', {
      method: 'POST', body: JSON.stringify(credentials)
    });
    const data = await res.json();
    // 3. Dispatch success
    dispatch(setUser(data.user));
  } catch (error) {
    // 4. Dispatch failure
    dispatch({ type: 'auth/setError', payload: error.message });
  } finally {
    dispatch({ type: 'auth/setLoading', payload: false });
  }
};
dispatch(loginUser({ email: 'a@b.com', password: '...' })); // thunk-thunk

// ── RTK createAsyncThunk — auto-handles pending/fulfilled/rejected ──
export const loginUser = createAsyncThunk('auth/login', async (credentials) => {
  const res = await fetch('/api/login', { method: 'POST', body: JSON.stringify(credentials) });
  return res.json(); // becomes action.payload on fulfilled
});

Middleware in Redux is a way to extend the store's capabilities by inserting custom logic into the dispatch pipeline — the path an action travels from when you call dispatch(action) to when it reaches the reducer. Middleware sits in the middle of this pipeline and can intercept, inspect, transform, delay, or even stop actions before they reach the reducer.

Common middleware use cases:

  • redux-thunk (built into RTK): Allows dispatching functions (thunks) instead of plain objects — enabling async operations. This is the most commonly used middleware and is included by default in RTK's configureStore.
  • Logging: A logger middleware prints every dispatched action and the resulting state to the console — invaluable for debugging. Libraries like redux-logger do exactly this.
  • Analytics / monitoring: Track every dispatched action as a user event — send to an analytics service without touching any component code.
  • Error reporting: Wrap dispatch in try/catch to catch reducer errors and send them to Sentry or another error tracking service.
  • API middleware (redux-saga): Redux Saga is a more powerful alternative to thunks — it uses generator functions to handle complex async flows (race conditions, polling, request cancellation) as a separate middleware layer.

const loggerMiddleware = store => next => action => {
  console.group(action.type);
  console.log('Dispatching:', action);
  console.log('State before:', store.getState());
  const result = next(action); // passes action to next middleware or reducer
  console.log('State after:', store.getState());
  console.groupEnd();
  return result;
};

// Apply middleware in RTK's configureStore
const store = configureStore({
  reducer: { cart: cartReducer },
  middleware: (getDefaultMiddleware) =>
    getDefaultMiddleware().concat(loggerMiddleware),
    // getDefaultMiddleware() includes redux-thunk by default
});

Redux reducers are pure synchronous functions — they cannot perform API calls, access localStorage, or do anything asynchronous. Async operations and side effects must happen outside the reducer, before the final state update is dispatched. Redux's middleware system is what enables this.

The main approaches to async in Redux:

  • 1. RTK's createAsyncThunk (recommended): The most ergonomic and standard approach for most apps. You define an async function that performs the API call and returns data. RTK automatically dispatches three actions: pending (request started), fulfilled (success with data), and rejected (failure with error). You handle these in extraReducers inside the slice to update loading, data, and error state accordingly.
  • 2. Manual Thunks: Write your own thunk — a function that receives dispatch and getState and manually dispatches actions at each stage. More verbose but gives maximum control over the async flow (e.g., conditionally dispatching based on current state via getState()).
  • 3. Redux Saga: A separate middleware that uses ES6 generator functions to handle complex async flows — race conditions, polling, debouncing, request cancellation, parallel requests. More powerful but significantly more complex to learn. Best for enterprise apps with intricate async requirements.
  • 4. RTK Query: The highest-level solution — a complete data fetching and caching layer built into RTK. It eliminates most manual async thunk code entirely. Define your API endpoints once, and RTK Query generates hooks (useGetUsersQuery, useCreateUserMutation) that handle fetching, caching, loading states, error handling, and cache invalidation automatically.
jsx — createAsyncThunk: the standard async pattern
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';

// 1. Define the thunk — async function that returns data
export const fetchUsers = createAsyncThunk(
  'users/fetchAll',               // action type prefix
  async (_, { rejectWithValue }) => {
    try {
      const res = await fetch('/api/users');
      if (!res.ok) throw new Error('Server error');
      return res.json(); // becomes fulfilled action.payload
    } catch (err) {
      return rejectWithValue(err.message); // becomes rejected action.payload
    }
  }
);

// 2. Handle all three states in extraReducers
const usersSlice = createSlice({
  name: 'users',
  initialState: { items: [], loading: false, error: null },
  reducers: {},
  extraReducers: (builder) => {
    builder
      .addCase(fetchUsers.pending,   state => {
        state.loading = true; state.error = null;
      })
      .addCase(fetchUsers.fulfilled, (state, { payload }) => {
        state.loading = false; state.items = payload;
      })
      .addCase(fetchUsers.rejected,  (state, { payload }) => {
        state.loading = false; state.error = payload;
      });
  },
});

// 3. Dispatch from component
function UserList() {
  const dispatch  = useDispatch();
  const { items, loading, error } = useSelector(s => s.users);
  useEffect(() => { dispatch(fetchUsers()); }, [dispatch]);
  if (loading) return <Spinner />;
  if (error)   return <p className="error">{error}</p>;
  return items.map(u => <UserCard key={u.id} user={u} />);
}

Error handling in Redux follows the same unidirectional data flow as everything else — errors become state. Instead of trying to catch errors at the UI level and display them locally, you capture errors in the async logic, dispatch them as actions, store them in the Redux state, and let components read and display the error state via useSelector.

The standard pattern — error as state:

  • Include an error field in each slice's initial state (typically null when no error, or an error message string when something went wrong).
  • In async thunks (createAsyncThunk), use rejectWithValue(errorMessage) to pass the error to the rejected action's payload in a controlled way.
  • In extraReducers, handle the .rejected case by setting state.error = action.payload.
  • In the component, select the error state and conditionally render an error message.

Error reset: Always dispatch a reset action or clear the error state when the user retries or navigates away — stale error messages from a previous attempt shouldn't persist.

Global error handling with middleware: You can write middleware that intercepts any rejected action and sends it to an error monitoring service (Sentry, Datadog) globally — without adding tracking code to every slice.

jsx — complete error handling pattern
// Slice — error is part of state
const ordersSlice = createSlice({
  name: 'orders',
  initialState: { data: [], loading: false, error: null },
  reducers: {
    clearError: state => { state.error = null; }, // reset action
  },
  extraReducers: builder => {
    builder
      .addCase(fetchOrders.pending,   state => {
        state.loading = true; state.error = null; // clear on retry
      })
      .addCase(fetchOrders.fulfilled, (state, { payload }) => {
        state.loading = false; state.data = payload;
      })
      .addCase(fetchOrders.rejected,  (state, { payload }) => {
        state.loading = false;
        state.error = payload || 'Something went wrong. Please try again.';
      });
  },
});

// Component — reads error from state and displays it
function OrdersPage() {
  const dispatch          = useDispatch();
  const { data, loading, error } = useSelector(s => s.orders);

  useEffect(() => { dispatch(fetchOrders()); }, [dispatch]);

  if (loading) return <Spinner />;

  if (error) return (
    <div className="error-box">
      <p>⚠️ {error}</p>
      <button onClick={() => {
        dispatch(ordersSlice.actions.clearError());  // clear error
        dispatch(fetchOrders());                       // retry
      }}>Try Again</button>
    </div>
  );

  return data.map(o => <OrderCard key={o.id} order={o} />);
}

// Global error middleware — report all rejected actions to monitoring
const errorMiddleware = store => next => action => {
  if (action.type.endsWith('/rejected')) {
    Sentry.captureException(action.payload); // global error tracking
  }
  return next(action);
};

Flux is an application architecture pattern (not a library) originally introduced by Facebook alongside React to address the problems of bidirectional data flow in MVC applications. Redux is a concrete state management library heavily inspired by Flux, but it significantly refines and simplifies the pattern with ideas from functional programming (specifically Elm's architecture).

Key differences:

  • Single Store vs Multiple Stores: Flux allows and encourages multiple stores — one per domain (UserStore, CartStore, ProductStore). Each store manages its own state independently. Redux enforces a single store — all application state lives in one tree. Multiple domains are handled via separate reducer functions (slices) that each manage their own sub-tree.
  • Dispatcher: Flux has a central Dispatcher — a dedicated module that routes all actions to all registered stores. Every store receives every action and decides whether to handle it. Redux has no separate Dispatcher — the store itself handles action routing to reducers directly. This simplification eliminates an entire conceptual layer.
  • Reducers vs Store Methods: In Flux, stores contain both the state and the logic for updating that state (through registered handler methods). In Redux, state and update logic are separated — the store holds state, and reducers (pure functions external to the store) handle updates. This separation makes Redux reducers far more testable and composable.
  • Pure Functions vs Mutable Logic: Flux stores can use any logic — including imperative, stateful mutations. Redux requires pure reducer functions — no side effects, same input always gives the same output. This constraint is what enables time-travel debugging.
  • Library vs Pattern: Flux is a pattern — you implement it yourself or use one of many Flux implementations (Alt, Flummox, McFly). Redux is a specific, opinionated library with a standard API that you install and use as-is.
concept — flux vs redux architecture
// ── FLUX Architecture ──
// Action → Dispatcher → Store1 (handles some)
//                     → Store2 (handles others)
//                     → Store3 (handles others)
// Multiple stores, each with own state + logic
// Dispatcher routes actions to all stores

// ── REDUX Architecture ──
// Action → Single Store → Root Reducer
//                       ├── authReducer   (handles auth/* actions)
//                       ├── cartReducer   (handles cart/* actions)
//                       └── usersReducer  (handles users/* actions)
// Single store, combined reducers, no separate Dispatcher
// Pure functions, immutable updates, time-travel debugging

// Redux in code — what replaces the Dispatcher
const store = configureStore({
  reducer: {               // combineReducers happens automatically
    auth:  authReducer,   // authReducer handles 'auth/*' action types
    cart:  cartReducer,   // cartReducer handles 'cart/*' action types
    users: usersReducer,  // usersReducer handles 'users/*' action types
  }
});
// store.dispatch() is the "Dispatcher" — no separate module needed
